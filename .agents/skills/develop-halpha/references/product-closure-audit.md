# Halpha 产品闭环审计

## 适用问题

在核实业务进展、判断能否实际使用、选择下一步、检查机制复杂度或评估依赖价值时使用本方法。目标是确认产品运行路径，而不是统计文件、测试或设计完成度。

## 证据层级

对每项能力只报告已经取得证据的最高层级：

1. **设计存在：** 当前 L0–L4 定义了结果或边界。
2. **实现存在：** 产品路径中有代码、迁移或配置。
3. **直接验证：** 目标代码通过单元或集成测试。
4. **产品调用：** App、Executor、计划任务或用户路径实际调用该实现。
5. **运行可达：** 当前授权环境可以启动并到达该路径，依赖、配置和数据均可用。
6. **闭环观察：** 从用户触发到权威结果、失败或结束已经实际观察并核对。

设计、代码、测试 fixture、资格工具和手工脚本不能替代产品调用。测试通过只证明测试覆盖的输入；历史观察只证明当时的构建和环境。

## 从用户结果追踪调用链

选择任务中最重要的一条用户路径，按实际顺序核对：

- 用户从哪里开始，输入和资金边界在哪里固定；
- 状态由谁持久化，运行进程怎样发现新增或变化；
- 哪个产品循环或请求处理器调用核心服务；
- 外部动作、模拟结果或只读事实怎样产生；
- 重复、超时、断线、部分结果和重启怎样处理；
- 保护、停止、退出和接管怎样保持可用；
- 权威结果怎样返回工作台并进入可理解的终态；
- 临时责任、额度、挂单、持仓和复盘何时闭合。

直接搜索构造点和调用点。特别标出：

- 只被测试、fixture 或 `tools/**` 调用的实现；
- 已注册但没有常驻消费者的 worker、dispatcher 或服务；
- 只能通过记住 URL、手工数据库操作或一次性脚本继续的路径；
- UI 显示成功但权威业务结果仍为处理中、未知或失败的路径。

## 复杂度与删除检查

对每个对象、状态、页面、进程、依赖、摘要或治理检查回答：

1. 当前直接消费者是谁？
2. 删除后哪个当前用户结果或失败行为会改变？
3. 现有组件或更窄实现能否承担同一结果？
4. 它是否只服务测试、资格说明、未来设想或旧流程？
5. 理解、配置、迁移、测试、运行、升级、恢复和删除成本是否低于价值？

缺少当前消费者时，默认建议删除、延后或明确不支持。不要因为已有代码量、迁移或测试而保留沉没成本，也不要为删除旧机制建立兼容平台。

## 选择下一小步

优先选择能够把一条现有断链推进到更高证据层级的最小结果。通常依次考虑：

1. 接通已经存在但未被产品调用的核心责任；
2. 修复阻断主路径的真实运行错误；
3. 补齐失败、停止、重启或终态闭合；
4. 删除阻碍用户或维护者的无消费者机制；
5. 最后才新增通用基础设施或未来能力。

下一步必须说明用户价值、改动边界、直接验证和不做范围。不要按旧阶段、建设包、文档顺序或形式完整度排序。

## 审计交付

先给出当前真正可完成的用户结果，再列出：

- 已观察闭环及其证据范围；
- 最高价值断点和实际调用证据；
- test-only、tool-only、dead path 或无消费者机制；
- 建议删除、复用或暂不支持的复杂度；
- 一个最小下一结果及其验证方式；
- 当前未知和未运行检查。

不得把“代码存在”“测试通过”“工具能跑”写成“产品已完成”。
