# Halpha 设计要求

## 设计取舍

在满足用户确认的产品语义、功能正确性和必要安全边界后，选择全生命周期复杂度最低的方案。复杂度包括理解、实现、依赖、配置、测试、运行、升级、恢复和个人维护，不只计算代码量。

默认顺序是：删除、复用成熟能力、按组件能力简化、最小补充、最后才自研平台。

## 风险与流程强度

风险分为两条轴：

- 功能正确性：动作唯一、防重复、事实核对、保护、停止、接管和恢复不能因 Demo、金额小或项目早期而取消。
- 工程流程：评审、记录、观察和治理自动化只随真实账户交易动作、不可逆影响、实际并发、重复故障和维护规模增加。

独立研究、Demo、公共行情只读和不发起交易所变更请求的开发默认使用直接测试与人工确认。真实账户交易只增加直接保护其凭据、动作、核对、停止和接管的控制，不建立第二资金授权或机构发布流程。

## 所有权与最小闭环

每个业务对象、决定和可变状态只有一个所有者。其他领域只能提供检查结果、事实或交互，不得复制生命周期。

只设计当前选择且确有消费者的路径：正常行动、不行动、输入未知、重复、部分结果、停止、外部接管、恢复和结束。未来路径以不支持、人工处理或外部工具结束，不预建状态机。

失败必须说明真实主体、影响范围和可观察结果，不能只写“确保安全”或“适当处理”。未知不是失败、零值或安全状态。

## 第三方组件

计划长期维护或进入核心路径的通用能力，先检查成熟组件的公开契约、支持边界、许可证、维护和退出方式。组件满足产品结果时直接采用；只在不可妥协产品语义或功能正确性缺口处保留最小补充。

同一能力只有一个运行实现和一个事实权威。不得为未来替换建立平行框架、第二状态源或自动 fallback。组件内部缓存和投影不是 Halpha 产品事实，除非拥有领域明确采用并验证其语义。

长期组件选择及 Halpha 使用边界可以进入 L3；精确版本、配置、平台结果和当前启用状态进入 L4。

## 停止条件

当最小价值闭环、唯一所有者、正常与关键失败结果、真实账户交易边界和个人维护成本已经清楚，且剩余细节可由代码、配置和直接测试决定时，停止扩写设计。

不因文档对称、未来可能性、机构惯例或“更完整”继续增加对象、状态、进程、数据库、证据或治理步骤。
