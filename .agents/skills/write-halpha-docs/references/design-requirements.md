## 设计目标与约束顺序

设计先保证用户确认的产品语义、功能正确性和安全边界，再在满足这些约束的方案中选择总复杂度最低者。总复杂度同时包括实现、依赖、集成、配置、迁移、运行、观测、升级、回退、测试和个人维护成本，不能只比较自研代码行数。

采用以下约束顺序：

1. 当前适用 L0–L2 已拥有的产品语义、授权与不可突破边界；
2. 动作唯一、防重复、状态正确、事实核对、停止与恢复等功能正确性；
3. 已确认的安全边界和失败结果；
4. 成熟第三方组件的实际能力与约束；
5. 策略数学、算法细节、内部结构、命名和其他可替换技术选择；
6. 实现便利、既有草稿和个人偏好。

不能因为既有文档已经描述了自研方案，就把该方案视为不可妥协语义。只有上位规范或所有者明确确认的产品决定，以及功能正确性或安全所必需的约束，才可以阻止按成熟组件能力调整设计。

## 分层严格度与设计停止条件

层级越高，越必须稳定、抽象并严格遵守；层级越低，越应容纳实现中才能取得的证据。L0–L2 的产品边界、稳定语义、所有权和复杂度原则是硬约束；L3 的长期模块契约、权威边界与失败语义保持稳定，但组件接线可按公开能力作兼容调整；L4 是当前建设基线，精确参数、配置、探针和包内顺序可以依据实现证据调整。

当核心建设方向、上位语义、最小价值闭环、安全与恢复边界、唯一实现和复杂度上限已经清楚且无直接冲突时，停止继续修正文档细节，转入实现验证。只有实现差异会改变上述内容时才返回拥有该语义的上位层；普通 L4 差异只回写实际值与证据，不以无法提前百分之百预判为理由无限延长设计。

## 第三方组件优先门槛

任何拟自研能力及其中可抽离的通用部分在进入详细设计前，都必须先检索可用的成熟第三方组件。检索范围至少覆盖与能力直接相关的框架、库、官方适配器和标准基础设施；优先使用官方文档、版本发布、源码或公开 API、许可证和维护状态等一手证据。

不得仅凭以下理由跳过检索：

- 当前草稿已经写了自研对象、状态机或算法；
- 组件行为与草稿不完全一致；
- 组件内部实现不可见或不使用 Halpha 的术语；
- 自己实现看起来短小；
- 未来可能需要更换组件。

为候选组件建立最小能力映射，至少记录：

| 项目 | 必答内容 |
|---|---|
| 所需能力 | 当前真实消费者、输入、输出和失败结果 |
| 组件能力 | 组件可直接承担的行为及其公开依据 |
| 差异 | 缺失、语义不同、精度、平台、许可证、成熟度或生命周期差异 |
| 约束性质 | 不可妥协产品语义/正确性/安全，或可妥协技术/策略选择 |
| 集成成本 | 必需适配、数据转换、配置、验证、升级和运行成本 |
| 决定 | 采用、按组件能力妥协、最小补充自研或不支持 |

组件是否“成熟”由当前用途所需证据判断，而不是只看知名度。至少检查固定版本、维护与发布状态、许可证、目标平台、公开接口、关键数值或身份语义，以及失败和升级边界。证据不足时保持候选或阻断，不把未知写成组件缺失，也不立即转为自研。

## 差异、妥协与最小自研

### 决策顺序

逐项差异按以下顺序处理：

1. **直接采用：** 组件满足所需产品结果和正确性，删除对应自研设计。
2. **按组件能力妥协：** 差异只改变策略数学、技术细节、内部生命周期或其他可替换选择时，以组件实际行为为设计，更新受影响策略或技术决定并重新验证。
3. **最小补充自研：** 组件无法满足不可妥协的产品语义、功能正确性或安全边界，且没有另一成熟组件能以更低总复杂度满足时，只补充最小缺口。
4. **明确不支持：** 补充自研的成本和风险高于当前价值时，记录拒绝、人工接管或外部工具路径。

“按组件能力妥协”不是降低正确性。妥协前必须确认差异没有改变用户确认的产品结果、授权、资产边界、动作唯一性、失败语义和必要保护；妥协后重新执行对应历史、仿真、集成或验收证据，不能沿用旧设计的成功结论。

### 允许最小自研的证明

保留任何自研能力前必须同时记录：

- 已检索的合适组件和一手证据；
- 无法妥协的具体语义或组件确实缺失的能力；
- 采用组件原行为会导致的可观察错误决定或失败结果；
- 自研部分的最小责任、输入、输出和禁止扩展范围；
- 与采用组件、换组件、人工计划或不支持相比的总复杂度；
- 测试证据、退出条件、替换条件和移除路径。

不得为了未来替换方便建立完整平行抽象、通用平台或第二实现。薄适配层只允许承担真实必要的边界翻译、不可妥协产品语义或正确性补充；它不能复制组件的通用能力、内部状态或完整生命周期。

同一能力在运行时只保留一个实现和一个事实权威。不得同时保留组件实现与自研实现作为静默、自动或长期平行 fallback。人工接管、外部工具、停止和明确不支持可以作为失败结果，但不得伪装成第二套 Halpha 实现。

## 组件边界与模块实现方案

采用第三方组件后，Halpha 设计只拥有以下必要内容：

- 组件在 Halpha 中承担的能力和调用方式；
- Halpha 输入、组件输出及必要转换；
- 产品语义、事实和持久记录各自由谁拥有；
- 组件失败、超时、重试、版本不兼容和事实未知时的结果；
- 精度、身份、幂等、并发、平台、许可证和升级所需的资格化条件；
- 最小补充自研的边界，或组件完全替代后不存在 Halpha 内部实现的事实。

组件内部已经完整承担能力时，不详细设计其内部类、缓存、状态机、算法、网络生命周期或数据结构，也不为文档对称虚构 Halpha 模块。模块 L3 主要说明怎样使用组件、怎样隔离组件内部语义、怎样验证公开契约以及失败时怎样停止或接管。

组件提供的内部缓存、组合状态、估值或事件不自动成为 Halpha 的产品事实。只有上位设计明确授权且资格化证明满足所需语义时，Halpha 才可直接消费；否则它们只作为技术投影、提示或诊断信息。

长期采用的库或框架及其模块使用契约属于 L3 实现方案；当前固定版本、构建哈希、配置值、平台资格化、启用状态和阶段证据属于 L4。升级不得静默改变产品结果：版本变化必须重新检查能力差异，按影响重验，并遵循单实现原则。

## 原则在文档体系中的落点

第三方组件优先原则按最高适当层级只定义一次，不在 L0–L4 逐层复制：

- **L0：** 仅在它构成长期项目最高取舍时，记录避免重复建设、个人可维护性和不可突破产品/正确性边界；不写供应商、组件类别、比较流程或证据。
- **L1-D ARC：** 记录跨领域、跨阶段稳定复用的“先检索成熟组件、可妥协技术选择按组件能力设计、限制自研和总复杂度”的总体技术方向；不写具体库、版本或当前结论。
- **L2：** 由各领域记录本领域不可妥协的产品语义、允许下沉的技术选择、组件不得取得的事实或决定权，以及组件缺失或失败时的领域结果；不重复总体检索流程，也不写当前供应商。
- **L3：** 记录模块长期采用的具体库或框架、Halpha 使用方式、公开契约、适配边界和确有必要的最小自研。
- **L4：** 记录当前候选或固定版本、能力比较证据、许可证与平台资格化、配置、建设阻断和验收结果。

若某项含义已经由更高层拥有，下层只引用并说明消费或收缩方式。不要为了显示“已遵循组件优先”而在每篇文档重复同一原则。

## 完整性、正确性与失败路径

设计必须覆盖会改变结果的：

- 正常行动；
- 不行动或等待；
- 可接受替代；
- 输入缺失、过期、冲突或事实未知；
- 重复、重试和部分完成；
- 停止、回退、人工接管和恢复；
- 明确结束，以及结束后是否允许新版本或重启。

失败路径必须说明真实主体、作用范围和可验收结果。避免“适当处理”“确保安全”“进行通知”等没有主体和结果的表述。

功能缺陷、系统自身风险和金融风险必须分开。动作唯一、防重复、状态正确、核对和恢复属于基本功能正确性，不因资金规模小、模拟成功或组件通常可靠而降级。组件能力缺口也不能通过把失败写成“风险已知”来消除。

方案必须同时说明保留的用户价值、删除的复杂度、人工或外部路径、迁移影响和验证证据。局部替换只有在证明不改变整体责任、顺序、未知、停止、恢复和验收时才可采用。

## 消费者、概念与复杂度

新增概念、对象、状态、流程、记录、数据、字段、适配层或依赖前必须回答：

1. 当前或近期高价值消费者是谁；
2. 它支持什么决定或动作；
3. 缺少时什么失败行为会不同；
4. 更小方案、成熟组件或人工计划为何不足；
5. 理解、实现、运行、升级和维护成本是什么；
6. 何时删除、合并、降级或替换。

没有消费者，或消费价值不足以覆盖完整成本时，不创建、不执行、不保存。只出现一次或可以用普通句子说明的内容不概念化；特殊概念必须稳定、无法由普通语言准确表达、被真实场景复用，并具有唯一所有者。

整体修正优先解决责任、阶段、授权、失败、身份、依赖和事实权威问题，不以“文字改动最少”作为目标。与此同时，修复后项目复杂度不得无依据增加；新增依赖、适配层、状态、记录、进程、数据库、写入路径或运行模式必须由删除的自研能力或明确新增价值抵消，并在设计中给出可核对的前后比较。

## 个人项目边界

始终以单一所有者、自有资金、个人维护为真实约束：

- 默认顺序是删除、复用成熟组件、按组件能力简化、最小补充、自研新平台；
- 复杂度优先投入交易判断、计划、数据、执行和 UX，辅助领域不得反客为主；
- 稳定性优先依赖成熟组件、简单结构、少进程、少依赖和单一事实权威；
- 没有当前消费者时只保留边界，允许人工计划、外部工具或明确不支持；
- 不默认建设委员会、多人审批、工程资格、生产准入、统一治理矩阵、微服务平台、机构高可用或合规体系。

所有者在 Halpha 外决定并限制提供给系统的账户和资金规模；这不是 Halpha 的功能或保证。Halpha 只记录并执行所有者已配置的资金使用上限与范围、完整计划、动作前检查和启停，不自行扩大。不要把个人理财流程扩写成系统能力。

## 最小完成标准

设计可进入文档和实现前，应满足：

- 已对计划自研的通用能力完成成熟组件检索与能力映射；
- 每项差异已分类为不可妥协语义/正确性/安全或可妥协技术/策略选择；
- 可妥协差异已按组件实际能力改写并安排重新验证；
- 每项保留自研都有最小边界、证据、复杂度、退出条件和替换路径；
- 没有平行实现、平行事实权威或为了未来可能性建立的平台；
- 正常、失败、未知、停止、人工接管、恢复和结束路径按实际需要闭合；
- 实现者无需自行创造产品语义，组件内部也未被误写为 Halpha 规范事实。
