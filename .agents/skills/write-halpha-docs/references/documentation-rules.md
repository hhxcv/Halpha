## 规范效力与变更路径

### 权威范围

- 正式设计只由当前 `ACCEPTED` L0–L4 定义；proposal 不具有规范效力，不能补足正式语义。
- 中文候选必须声明候选基线、同一协调候选版本集、无当前效力和正式接受条件。
- 当前状态从 `ACCEPTED` L4 计划或事实记录读取；缺少来源和时间时保持“未知”，不从设计、代码或测试推断。
- 评审分别报告“文本自身是否一致”和“当前正式规范是否允许”，不能把二者合并。

### 直接修改与 proposal

已接受文档只有小幅修改可以直接形成新的 `ACCEPTED` 版本。小幅修改必须保持以下语义全部不变：文档身份、层级、唯一语义所有者、责任边界、规范强度、授权、行为、失败处理、验收、外部影响、稳定概念、依赖、支持范围和迁移。典型内容仅包括错字、排版、导航、元数据、链接或不改变决定的澄清。

大幅修改和新建场景必须先建立 proposal。改变上述任一语义，新增、替换或删除稳定概念/对象/状态/工作流/角色/权限/支持范围，或新建、拆分、合并、重命名、撤回规范文档及建立新的 L3/L4/ADR 场景，均属于大幅修改。按语义影响而非改动行数判断；无法可靠分类时按大幅修改处理。

直接修改不等于静默修改：仍须递增版本、记录替代关系和批准信息、同步必要直接引用与索引，并完成适用验证。

## 层级要求

### 共同规则

同一稳定语义只在最高适当层级和唯一所有者处定义一次。下层只能细化、实例化或收缩，不能重写、豁免或扩大上层含义。L0–L3 定义应当怎样，L4 记录当前做什么和实际是什么。

遇到跨层内容时拆分，不把所有细节留在同一文件：

- 长期最高取舍与不可突破边界进入 L0；
- 跨领域、跨阶段的总体产品、流程或架构方向进入对应 L1；
- 单一领域的稳定职责、产品语义和设计原则进入 L2；
- 模块、接口、Schema、状态和长期采用的组件实现方案进入 L3；
- 当前阶段、精确版本、配置、建设顺序、进度、资格化和证据进入 L4。

### L0：项目宪法

**写：**

- 长期使命、首要责任、不可突破边界、最高取舍和改变这些内容的条件；
- 规范效力、唯一权威、批准和语言共同规范等最高治理规则；
- 会长期约束所有下位设计的复杂度、所有者控制和信任边界。

**不写：**

- 产品方案、用户路径、模块、状态机、接口、字段、供应商、数值配置、测试方法、运行步骤或当前阶段；
- 可由下位简单配置解决的实例差异；
- 个人理财流程或 Halpha 外部账户和资金安排的扩写。

**准入：** 只有一项内容在阶段、实现和技术变化后仍应成立，且下位无法在不改变项目根本性质的情况下自行决定时，才进入 L0。

**完成：** 每项原则能约束下位决定；冲突顺序清楚；变更条件可识别；非规范摘要不产生独立含义。

### L1：长期总体设计

L1 只写长期、阶段无关的总体决定。四篇 L1 按 `DOC → VIS → FLOW → ARC` 的约束顺序维护：

- **L1-A DOC 文档架构：** 文档层级、职责、命名、依赖、读取、更新、创建拆分和存放规则；不写产品、流程、技术或当前实现方案。
- **L1-B VIS 目标与愿景：** 长期产品定位、目标用户、问题与价值、竞争边界、能力范围、宏观阶段及非目标；不写具体工作流、页面、模块、接口或当前阶段。
- **L1-C FLOW 核心流程与用户路径：** 长期核心场景、人与系统分工、端到端路径、外部工具交接和产品级交互立场；不写页面布局、服务、数据库、字段、完整状态机或当前实现。路径覆盖行动、不行动或等待、失败、事实未知、外部人工交接、停止和结束，只保留产品级结果。
- **L1-D ARC 总体技术要求与架构：** 系统边界、质量取舍、逻辑责任、依赖方向、主要数据与控制流、总体技术选择和演进条件；不写精确接口、数据库结构、完整领域状态、具体供应商、部署命令或当前配置。逻辑责任不等于独立服务、进程或与责任一一对应的模块。

**准入：** 内容在多个 L2/L3 和多个阶段中稳定复用，并直接改变长期产品、流程或总体技术方向。只服务一个领域的进入 L2；需要详细设计规则的进入 L3；当前选择进入 L4。

**完成：** 结论长期有效；责任不重叠；下位可确定方向；正文没有对象清单、字段、接口、当前配置或现状陈述。

### L2：领域关键内容与设计原则

先判定责任形态：

- **横向核心业务组成：** 拥有一段连续业务、业务对象完整生命周期、决定权和前后交接；
- **纵向领域：** 拥有贯穿多个业务组成和模块的一类共同约束、质量属性或技术责任。

横向所有者不复制纵向完整约束；纵向所有者不重写横向业务对象、决定和生命周期。

再判定深度：

- **边界深度：** 唯一职责、不负责范围、交付或适用边界，以及缺失或失败结果；
- **当前需求深度：** 增加当前消费者所需的稳定对象、决定、状态边界、交接和验收依据；
- **复用扩展深度：** 只有多个真实消费者出现后，才增加复用生命周期、异常、迁移和公共规则。

深度表示支持范围和通用化程度，不是篇幅。当前深度、目标深度、深化理由和当前支持范围由 L4 记录。

**写：** 领域关键问题、目标、唯一职责、稳定设计原则；会改变决定、交接、失败或全部相关 L3 的稳定语义；当前所需深度内的必要对象、状态边界和验收依据；交给 L3 与 L4 的边界。

**不写：** 类、字段、数据库列、精确 API、当前供应商、当前部署、当前参数、当前领域深度；相邻所有者的完整流程或状态副本；无消费者未来扩展；把领域代号写成运行主体。

**准入：** 说明真实消费者、已观察不足、改变的决定或失败结果、为何更小方案不足、维护成本和退出路径。无法说明时只保留边界，允许人工、外部工具或明确不支持。

**完成：** 唯一所有者和责任形态明确；所需深度有 L4 依据；输入、输出、失败与相邻边界无重复；每项稳定语义有真实 L3 或近期任务消费。

### L3：长期稳定的领域详细设计

先判定类型：

- **领域 L3（DOMAIN）：** 指定一个主要语义所有者，精确实现其 L2 语义；
- **编排 L3（ORCHESTRATION）：** 指定协调所有者，只拥有必要身份、顺序、责任交接和完成汇总。

多个领域参与不自动需要编排 L3。默认使用领域 L3 加直接依赖；只有跨领域生命周期重复出现、需要稳定身份且没有现有所有者能够承担时才建立编排。

**写：**

- 版本、适用范围、L3 类型、主要语义或协调所有者；
- 直接依赖、适用纵向约束和上位稳定语义追溯；
- 模块边界、接口、Schema、状态、事件、错误、幂等与并发、完成标准和测试要求；
- 正常、关键反例、重复或重试、停止或回退、未知、人工交接和不支持结果；
- 参与领域未达到所需深度时的人工处理、外部处理或拒绝结果；
- 经过能力比较后长期采用的第三方库或框架、组件承担的模块能力、Halpha 的使用方式、公开契约、资格化边界和最小补充自研。

组件完整替代模块内部能力时，L3 主要说明怎样调用、输入输出、事实权威、错误映射、版本兼容与验证边界；不复制组件内部类、状态机、算法、缓存、网络生命周期或数据结构，也不虚构对应 Halpha 内部实现。当前精确版本、构建哈希、配置值、启用状态和阶段证据进入 L4。

**不写：** 新产品目标、新业务原则或 L2 无依据的稳定规则；P0/P1/P2 等建设阶段、当前建设范围和顺序、当前进度或阶段证据；当前部署、凭据可用性和运行进度；组件无公开契约且 Halpha 不拥有的内部实现；为文字完整重复机器可读 Schema、类型或状态表。

**准入：** 已明确的当前或近期建设需求确实需要详细设计，并直接改变实现或验收。只需稳定原则时回 L2；只属阶段、当前精确选择、建设顺序、进度或证据时进入 L4；可由现有类型、组件公开契约和测试清楚表达时不另建长文档。

**完成：** 实现者无需创造稳定产品语义；每个输入、输出、状态推进和失败都有所有者；第三方组件与 Halpha 自研边界唯一且无平行实现；直接依赖足够而不过度；验收覆盖正常、反例、重复、停止和恢复。

### L4：分阶段落地与当前记录

L4 在当前建设范围内，把 L0–L3 长期设计转为本阶段实施目标、安排和实际记录，不重写长期设计。

L4 是可依据实际开发调整的建设基线，不是要求实现逐字复制的完整预言。实现可以在不改变 L0–L3 稳定语义、核心闭环、安全/恢复边界、唯一写路径和复杂度上限的前提下调整精确参数、组件配置、接线和包内顺序，并回写最终值与证据；触及这些上位边界时，必须先修订拥有该语义的文档。评审在核心方向基本无误且无直接冲突时停止，不为非核心 L4 细节反复建立修订。

| 类型 | 记录内容 | 适用时间字段 |
|---|---|---|
| 阶段性落地设计、计划与进度 | 本阶段目标、范围、顺序、依赖、阻断、状态和验收证据 | `as_of` 或 `decided_at` |
| 事实 | 来源、观察时间、质量、缺失、冲突与修正 | `observed_at` |
| 当前实例与配置 | 活动版本、部署、组件固定版本、实例选择和参数 | `effective_from` |
| ADR 与现状摘要 | 选项、决定理由或带日期派生视图 | `decided_at` 或 `as_of` |

只有确有时效边界时才增加失效或复核字段。

**写：** 带来源和时间的事实；上位允许范围内的当前选择、组件精确版本、配置、领域深度和支持范围；建设顺序、阻断、完成证据和已知限制；ADR 选项、决定与理由。

**不写：** 新使命、产品规则、领域原则、接口语义或长期技术边界；从代码、测试、proposal 或单次运行推断“已完成”“可用”或“生产就绪”；没有来源、时间或作用范围的事实。

**完成：** 来源、时间、适用范围和未知项清楚；当前事实不覆盖设计；缺少记录时保持未知；计划完成由实际运行、测试和用户任务证据支持。

### Proposal：候选载体

Proposal 不是独立层级，只是尚未定稿 L0–L4 内容的载体。它必须声明目标层级、候选基线、无当前效力和正式接受条件。

中文候选可与同一协调候选版本集互相对齐，但不得写成正式上位已接受。L0/L1 候选获批后再按授权建立 en-US、bundle、批准信息和正式路径；L2–L4、登记和索引只维护 zh-CN，不创建英文对应文件。未获批前不预写共同规范效力。

## 关联逻辑

### 唯一所有者与直接依赖

- 一项稳定语义只有一个最高拥有层和一个唯一所有者；其他文档只引用、规定输入条件、消费方式和缺失结果。
- 生产者只定义自身输出，消费者只定义接入要求；不在每篇文档复制完整交接双方，也不建立全连接式领域关系。
- L2 责任登记只保存身份、横向/纵向形态、规范来源、直接横向依赖、适用纵向约束和少量稳定语义，不复制正文或当前深度。
- L3 声明实际直接依赖，不因“可能有用”连接全部相邻领域；第三方组件是模块实现依赖，不取得 Halpha 产品语义所有权。
- L4 当前计划只拥有当前建设范围、顺序、依赖、领域深度、支持范围、阻断和验收证据，不拥有稳定规则。

### 读取与同步方向

修改前从目标向上读直接上位，横向读相邻所有者，向下只读真实直接消费者。修改最高拥有层后，只同步因本次语义变化而失真的直接内容：

- 目标元数据中的版本、状态、上位、负责与不负责范围；
- 直接下游引用和 L2 责任登记中的身份或少量稳定语义；
- 新增、移动或删除的概念、要求、限制和决策依据索引项；
- L4 当前计划中的建设顺序、领域深度、支持范围或验收证据；
- 正式共同规范包中的配对文本、bundle 与哈希，但仅在任务授权正式接受或同步时处理。

不要因为修改一段就机械更新所有文档。发现未定义特殊概念时，用普通语言改写、回到最高拥有层定义，或删除无消费者概念；不能在当前文档临时另建定义。

### 导航索引分工

- 概念定义只进入概念定义索引；
- 规范要求与限制只进入要求与限制索引；
- 有真实替代取舍的决定依据只进入决策与依据索引；
- 同一内容不在三个索引重复，索引不成为第二正文；
- 只有新增、移动、重命名或删除相应拥有内容时才更新索引。

## 格式、语言与登记

### 文件与最小元数据

- 规范文件名使用 `HALPHA-<类型>-<序号>-<英文短标题>.<语言>.md`，英文短标题为小写 ASCII kebab-case。
- 除 L0 无上位外，每份规范至少明确文档编号、版本、状态、层级、语言、上位文档或条款、负责范围和不负责范围；L0 使用适用范围和权威范围。
- 状态只使用 `PROPOSED`、`ACCEPTED`、`SUPERSEDED`、`WITHDRAWN`。
- 显式语义锚点在单文件内唯一；相对链接必须能到达真实目标。
- YAML 必须可解析；文档编号、层级、状态、时间和路径键与用途一致。
- 通用日期、可由状态与层级推出的当前效力，以及 L3 的当前阶段、选择进度或证据不进入正式文档头。

### 语言与共同规范包

- 只有 L0/L1 维护 zh-CN 与 en-US 共同规范文本；两者语义对齐、各自可独立阅读且无优先语言。
- L2–L4、责任登记和导航索引只维护 zh-CN，不创建 en-US 对应文件。
- L0/L1 文档头声明共同规范集标识、对应文本、bundle 和生效时间；批准、替代、对齐与摘要由 bundle 唯一拥有。
- bundle 是批准人、接受时间、可选延迟生效时间、替代关系、语言同等权威、对齐状态和摘要的唯一机器登记。
- bundle 每个正文 `body_sha256` 按 schema 3 固定规则归一化后重算；联合摘要由文档编号、版本、规范语言顺序和各正文哈希重建，bundle 只保存 `joint_set.sha256`。
- 版本、全部语言正文和 bundle 作为一个变更包处理，不单独更新其中一项。
- L2/L3 单语言 ACCEPTED 文档头记录批准人、接受时间和替代版本。

### 时间与当前记录

L4 按记录性质使用 `as_of`、`decided_at`、`observed_at` 或 `effective_from`。不得用没有来源、时间和作用范围的“当前”“已经”“可用”描述现状。组件发布事实与能力证据也要区分：官方文档说明“设计可采用”，资格化和运行记录才说明“当前可用”。

## 正文表达规则

### 结论与内容门槛

每一节先给会改变决定或动作的结论，再补使其可执行、可失败和可验收的最少要求。只保留会改变范围或非目标、责任或边界、前后顺序、输入输出与允许行为、失败停止恢复结果、验收依据的内容。

删除推理过程、常识、口号、文学性铺陈、重复规范、行业模板罗列和无消费者未来设计。理由只在存在真实替代和取舍时保留，且直接说明选择依据和放弃内容。

### 概念与命名

- 一个概念只用一个完整名称，跨文档保持一致；
- 优先常见、直白、明确的中文，避免生造隐喻、罕见词和英文对象直搬；
- 名称脱离章节上下文后仍说明“什么对象的什么”；
- “候选、决定、状态、记录、结果、方式、通知”等通用名词必须带业务对象或用途限定；
- 英文缩写只用于外部标准、文档编号、领域编号、代码标识或必须一致的技术名词，首次出现给出中文含义；
- L3 可保留实际字段名、类型名和组件公开 API 名，但必须说明中文业务语义，不能用标识符代替定义。

### 主体、责任与记录

区分抽象概念、领域责任、数据记录和运行实体。领域代号只表示语义所有权，不是运行主体；行为句写明所有者、Halpha、后台任务、执行器或具体外部系统等真实主体。

新增记录或数据必须有明确高价值消费者、决定用途、失败差异和删除条件。组件内部记录不因存在就成为 Halpha 必须保存或公开的对象。

### 路径与失败

按目标层级保留会改变结果的行动、不行动或等待、替代、输入缺失或冲突、事实未知、重复或重试、部分完成、停止、回退、人工接管、恢复和明确结束。失败写明主体、作用范围和结果，不使用“适当处理”“确保安全”等不可验收表述。

功能缺陷、系统自身风险和金融风险分开书写，不因资金规模小而降低动作唯一、状态正确、核对和恢复要求。

## 同步与编辑方法

1. 先写目标结论、层级、唯一所有者、当前消费者和不做范围。
2. 对照上位、相邻所有者和直接消费者，标出重复与缺失。
3. 先删除重复、无消费者、被组件替代和低价值内容，再补真正的边界、失败和验收。
4. 修改最高拥有层，再同步必要直接引用；不顺带格式化或整理无关文件。
5. 优先修改现有拥有者，不为一次任务新建长期文档。新建或拆分前证明独立职责、复用价值和删除条件，并按语义影响选择 proposal。
6. 完整复读修改文件及必要上下文，不依赖逐词机械替换。
7. 检查新增名词、行为主体、组件依赖、所产数据的消费者和退出条件。

评审意见按实际影响表达：指出错误层级或所有者、会导致的错误决定或失败行为，以及最小修正位置。不要用篇幅、格式偏好或抽象程度本身代替实质问题。
