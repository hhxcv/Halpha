# Halpha 文档规则

## 权威与历史

每个文档编号只在正常目标路径保存一份中文正文。文件存在即表示它是当前适用正文；Git commit 保存历史、比较和回退。不得建立 proposal、accept、bundle、archive、状态副本、责任登记或导航索引来选择另一份权威。

当前状态只从 L4 当前计划或明确事实记录读取。缺少来源、时间或适用范围时保持未知。

## 分层

| 层级 | 只负责 | 不负责 |
|---|---|---|
| L0 | 长期使命、不可突破边界、最高取舍和根本变更条件 | 产品方案、流程、对象、技术、测试和当前状态 |
| L1 | 长期总体文档规则、产品定位、核心路径和总体技术方向 | 字段、完整状态机、供应商、部署和当前建设 |
| L2 | 单一领域的稳定职责、原则、交接和失败边界 | 精确字段、当前组件、当前配置和相邻领域完整流程 |
| L3 | 有实际消费者并需要长期复用的模块、接口、状态、失败和组件使用契约 | 阶段、进度、精确版本、当前实例和实际结果 |
| L4 | 当前目标、范围、顺序、精确选择、配置、进度、事实、限制和直接结果 | 新增或改写稳定产品规则 |

`P0`、`B04` 等阶段代号，以及当前顺序、进度、证据和实例，只能进入 L4。删除阶段代号时也必须删除残留的阶段叙事，不能只换普通词。

## 唯一所有者

一项稳定语义只有一个最高拥有层和一个所有者。生产者定义输出；消费者只说明使用条件和缺失结果。相邻文档不得复制完整状态、字段、重试、恢复或授权规则。

当一个对象由横向业务领域拥有时，CAP、DAT、UX、SYS、ENG 等纵向领域只能规定自己的检查或约束，不得重新定义该对象生命周期。跨域顺序只由确有必要的协调者描述一次。

## 内容门槛

只保留会改变以下至少一项的内容：用户结果、责任归属、允许行为、失败结果、停止与接管、实现边界或当前决定。

新增概念、记录、状态、流程、组件或治理检查前，必须能说明当前消费者、缺少它时的具体差异和更小方案为何不足。删除后不改变当前行为或判断的内容应删除。

当前实现已经由代码、配置、锁文件、数据库迁移或 Git 直接拥有时，L4 引用其路径和当前结论，不复制完整 schema、版本清单或摘要。只有人工决策必须独立读取时才在 L4 保留少量值。

## 文件与语言

- 规范文件使用 `HALPHA-<类型>-<序号>-<英文短标题>.zh-CN.md`。
- 只使用中文规范；必要代码标识、外部标准和组件名称可以保留英文。
- 除 L0 外，文档头至少包含文档编号、层级、语言、上位、负责和不负责范围。
- L0 至少包含文档编号、层级、语言、适用范围和权威范围。
- 不要求显式语义锚点、固定章节模板、定义/要求/依据分类或机器配套文件。
- L4 YAML 必须包含文档编号、层级、时间和当前事实的适用范围。

## 编辑与验证

1. 写明目标含义、唯一所有者、消费者和不做范围。
2. 完整读取目标、直接上位和实际受影响的相邻/下游文档。
3. 先删重复与无消费者内容，再补最少边界。
4. 修改拥有者，再同步真正失真的直接引用和 L4。
5. 完整复读修改文件，检查普通语言、主体、未知、停止、接管和结束。
6. 运行最小机械校验；不得为一次变化新建专用治理平台。

评审分别说明文本内部是否一致、层级和所有权是否正确。结论不以篇幅、检查数量或形式完整度为依据。
