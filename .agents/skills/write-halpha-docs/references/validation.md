## 按语义影响分级

影响级别由语义变化决定，不由文件扩展名、字数或资金金额决定。

| 级别 | 文档变化示例 | 最小验证 |
|---|---|---|
| 核心影响 | L0/L1 方向；交易判断、交易计划、账户或行情事实、动作、保护、核对、资金边界、停用、恢复和常用 UX 语义 | 相关上下游全文；关键正常与反例；独立于起草步骤的复核；明确文档/发布版本回退或保留旧版路径，不建立运行时双实现 |
| 一般影响 | 单一 L2/L3 边界、普通业务规则、内部交接和非关键交互 | 目标、直接上位与相邻所有者；相关场景和引用；实际差异复读 |
| 轻量影响 | 不改变语义的格式、错字、导航索引和辅助说明 | 作者检查、链接或 YAML 等最小机械验证 |

分类不明时先按较高一级；证明影响隔离后再降级。第二视角重点从业务预期和反例重新推导，不以复述作者结论或增加 AI 数量代替独立性。

## 最佳整体修正与经验固化

评审发现问题时，不默认选择最小文字改动。先把问题归类为概念、责任、阶段、授权、失败路径、身份、依赖或当前记录问题，再比较至少一种整体修正和一种收缩或人工回退方案。最终方案必须满足：

- 由拥有该语义的最高适当层级统一定义，直接消费者不再保留同义词、平行状态或隐式例外；
- 真实价值闭环仍可达，且行动、不行动/等待、拒绝、未知、停止、人工接管、恢复和明确结束都有真实主体与结果；
- 不以微小资金、模拟成功、代码完成或单次成交降低功能正确性，也不以安全名义引入个人项目不需要的治理、平台或复杂授权；
- 每个阶段有清楚的进入条件、可用闭环、退出证据和后续边界，阶段之间不重复拥有同一能力；
- 兼容迁移、历史身份、旧记录、回退路径和测试证据已经说明；无法安全迁移时保持未知或停止，不静默转换。

只有在整体修正证明不改变上述结果时，才可采用局部修正。重复出现的判断必须回写本清单、专项 reference 或相应上位文档，避免下一次评审重新退化为局部补丁。

## 设计决策与第三方复用

对任务范围内每项拟自研通用能力逐项检查：

- 已检索适合当前用途的成熟框架、库、官方适配器或基础设施，证据来自固定版本的官方文档、发布、源码/API、许可证和维护状态；没有凭记忆或既有草稿宣称“必须自研”。
- 能力映射写清真实消费者、组件直接能力、差异、约束性质、集成成本和最终决定；证据不足保持未知或阻断。
- 差异已区分为不可妥协产品语义/功能正确性/安全边界，或可妥协策略/技术选择；没有把既有实现细节、命名或个人偏好提升为产品语义。
- 组件能够满足时已删除重复自研；差异可妥协时已按组件实际能力修改设计，并重新安排受影响的历史、仿真、集成或验收证据。
- 每项保留自研都说明组件为何无法满足不可妥协结果、最小补充边界、可观察失败、总复杂度、证据、退出条件和替换路径。
- 同一能力只有一个运行实现和一个事实权威；没有组件实现与自研实现的静默、自动或长期平行 fallback。
- 薄适配层只做必要边界翻译、不可妥协产品语义或正确性补充，没有复制组件通用能力、内部状态或完整生命周期。
- 组件完整替代模块能力时，L3 主要说明 Halpha 使用方式、输入输出、事实权威、失败、版本兼容和资格化边界，没有重画组件内部设计或虚构 Halpha 模块。
- 采用组件后的进程、数据库、写入路径、记录、运行模式和维护面没有无依据增加；新增复杂度有明确删除的自研复杂度或真实价值抵消。

## 差异范围

- 实际改动只包含任务授权的目标和必要直接引用。
- 无关工作区改动保持原样；没有顺带格式化、改名、换行或整理。
- 候选任务只修改受影响的目标 L0–L4 文档及必要直接登记、bundle 和索引；没有建立跨层 proposal 文件、`docs/proposals/`、任何 `archive/` 目录或未提交过程副本。
- 没有把候选、设计完成、代码存在或单次测试写成当前可用事实。
- 已接受文档的小幅直接修改没有改变身份、层级、所有者、责任、规范强度、授权、行为、失败处理、验收、外部影响、稳定概念、依赖、支持范围或迁移；若改变任一项且项目所有者未直接接受完整文本，任务已改为各目标文档自身的 `PROPOSED` 版本。

## 层级与责任

- 每项新增或修改含义位于最高适当层级。
- L1 没有对象、完整状态机、字段、接口、供应商、当前配置或实现状态。
- L2 没有精确字段和当前事实；当前深度与深化理由由 L4 记录。
- L3 只精确实现已有 L2 语义，声明类型、所有者、直接依赖和纵向约束；可以记录经过比较后长期采用的组件实现方案和使用契约，但不拥有其当前精确版本、配置、启用状态、P 阶段、建设范围、顺序、进度或证据。
- L4 有来源、时间和适用范围，不创造规则。
- 每项稳定语义只有一个所有者；相邻文档只写输入、消费和缺失结果。
- 横向业务责任与纵向约束责任没有互相改写。

## 正文质量

- 开头直接给结论、范围和不做内容。
- 每段会改变范围、责任、边界、顺序、失败行为或验收。
- 普通词语保持通常含义；特殊概念已在最高适当层级定义。
- 概念名称完整、直白、跨文档一致，没有依赖章节上下文的通用名词。
- 行为使用真实主体；领域代号未被写成运行实体。
- 新记录和数据有明确高价值消费者、决定用途和删除条件。
- 行动、不行动或等待、未知、失败、停止、人工接管、恢复和结束按实际需要闭合。
- 功能缺陷、系统自身风险和金融风险没有混写。
- 没有无消费者未来设计、机构模板或辅助平台过度建设。
- 没有重复设计第三方组件内部类、算法、缓存、状态机、网络生命周期或数据结构；组件内部投影也未被无依据写成 Halpha 产品事实。

## 评审停止条件

- L0–L2 的核心方向、稳定语义、所有权和复杂度原则无直接冲突，L3 的长期模块契约可指导实现。
- L4 已给出可启动当前建设的基线、阻断与验收方向；只能由实际开发确认的参数、配置和组件细节被明确标为可调整，而非继续假设性细化。
- 剩余问题不会改变核心闭环、安全/恢复责任、唯一实现、真实写路径或复杂度上限时，结论应为可接受并停止修改；把细节留给实现证据。
- 只有发现会造成错误决定、错误外部动作、责任无法闭合或上位语义冲突的问题，才继续设计修正。

## 格式与登记

- 文件名符合 `HALPHA-<类型>-<序号>-<英文短标题>.<语言>.md`；英文短标题使用小写 ASCII kebab-case。
- 元数据中的文档编号、版本、状态、层级、语言、上位、负责和不负责范围完整且与文件一致；L0/L1 文档头与 bundle 的必要绑定字段一致，L2/L3 头含批准人、接受时间和替代版本。
- 显式语义锚点在单文件内唯一；新增链接能跳到真实目标。
- YAML 能解析；文档编号、层级、状态、时间和路径键与用途一致。
- L2 责任登记只保存身份、形态、规范来源和少量稳定语义，不复制正文或当前深度。
- 当前计划只记录当前建设范围、顺序、依赖、领域深度、支持范围、阻断和验收证据，不拥有稳定语义。
- 长期采用的组件及使用契约位于相应 L3；当前固定版本、构建哈希、配置、平台资格化、启用状态和阶段证据位于 L4。
- 概念、要求与依据只进入各自索引；同一内容没有在三个索引重复。

## 语言、状态与哈希

### PROPOSED 目标文档

- 状态为 `PROPOSED`，当前效力明确为无。
- 候选基线和协调候选版本集清楚，不把 candidate 写成 ACCEPTED 上位。
- 正式接受条件明确；未提前写入批准时间、共同规范集或已对齐状态。
- 用于大幅修改或新建场景；直接位于目标 L0–L4 路径，不建立跨层 proposal 文件或专用目录。

### ACCEPTED

- 只有 L0/L1 维护 zh-CN 与 en-US 共同规范文本；两者语义对齐、各自可独立阅读且无优先语言。L2–L4、责任登记和导航索引只维护 zh-CN，不存在 en-US 对应文件。
- 文档头中的文档编号、版本、状态、语言、共同规范集标识和生效时间与 bundle 一致；配对文本和 bundle 路径有效。
- bundle 是批准人、接受时间、可选延迟生效时间、替代关系、语言同等权威、对齐状态和摘要的唯一机器登记。
- bundle 中每个正文 `body_sha256` 与按 schema 3 固定规则归一化后的正文一致。
- 联合摘要输入由文档编号、版本、规范语言顺序和各正文哈希重建；bundle 只保存 `joint_set.sha256`，该摘要必须可重算。
- 版本、正文和 bundle 作为一个变更包处理；不要只更新其中一项。
- 小幅直接修改具有新版本、替代关系、批准信息、直接引用与索引同步记录；无法证明语义未变且项目所有者未直接接受完整文本时，不得绕过目标文档 `PROPOSED` 状态。

## 机械检查脚本

从仓库根目录运行：

```powershell
# 检查本次明确修改的文件
python .agents/skills/write-halpha-docs/scripts/validate_halpha_docs.py <path...>

# 检查工作区全部已跟踪和未跟踪改动
python .agents/skills/write-halpha-docs/scripts/validate_halpha_docs.py --changed

# 重算当前已接受共同规范包正文和联合哈希
python .agents/skills/write-halpha-docs/scripts/validate_halpha_docs.py --accepted-integrity

# 修改 HALPHA-PLAN-001、建设门或真实写状态时，检查机器可读治理一致性
python governance/validate_construction_plan.py
```

文档脚本检查 UTF-8、YAML、规范文件名与核心元数据、语义锚点重复、相对链接、目标文档候选位置、禁止的 proposal/archive 目录和 bundle 哈希。治理脚本只检查当前 PLAN 的冲突阻断、建设依赖和真实写 fail-closed。两者都不判断内容是否放在正确层级、概念是否值得建立、业务路径是否完整或冲突语义是否真的关闭；这些必须人工复核。

脚本对错误返回非零状态；警告需要结合任务范围判断。传入明确路径优于在脏工作区直接使用 `--changed`，以免把所有者已有改动误算为本次范围。

## 交付结论

接受结论只使用：

- **可接受：** 语义和机械检查满足当前影响级别。
- **限范围可接受：** 明确写出适用范围、未知项和不允许扩展的部分。
- **需修复：** 指出错误所有者或层级、实际失败影响和最小修正位置。

先报告结论，再报告实际检查和证据。不要把检查数量、文档完整度或多人赞同写成质量结论。
