# Halpha 文档验证清单

## 语义检查

- 用户结果、资金边界、动作、事实、停止、接管和结束含义没有被意外改变。
- 每项稳定语义只有一个所有者；相邻文档只保留本地消费和失败结果。
- 未知没有被写成失败、零值或安全状态。
- Demo 与 Live 的身份、凭据、动作和结果仍隔离。
- 当前没有消费者的对象、状态、流程、证据和治理概念已经删除。

## 层级检查

- L0 只有使命、最高边界和取舍。
- L1 没有字段、完整状态机、供应商、当前实施或建设顺序。
- L2 没有精确字段、当前组件和相邻领域完整流程。
- L3 没有阶段、进度、精确版本、当前实例和实际结果。
- L4 有时间、范围和来源，不创造稳定规则。
- L0–L3 没有 `P0`、`B04`、`R00` 等阶段标识或同义阶段叙事。

## 复杂度检查

- 文档只使用中文，不存在配对翻译、proposal、accept、bundle、责任登记或导航索引。
- 没有为了校验器而创建第二份事实或流程。
- 代码、配置、锁文件和 Git 已拥有的详细事实没有在 L4 全量复制。
- 精确进程数、记录族数、页面数和组件接线只有确实构成长期行为时才进入 L3，否则留在 L4或代码。
- 普通 Git、直接测试和人工决定足够时，没有新增治理机制。

## 机械检查

从仓库根目录运行：

```powershell
python .agents/skills/write-halpha-docs/scripts/validate_halpha_docs.py <修改路径...>
python governance/validate_construction_plan.py  # 修改 L4 时
```

机械校验只检查 UTF-8、YAML、文件名、最小元数据、相对链接和少量层级禁词。通过不表示语义正确。

## 交付

- **可接受：** 内部一致、层级和所有权正确，验证与实际影响相称。
- **限范围可接受：** 明确剩余未知和不允许扩展范围。
- **需修复：** 指出错误所有者或层级及其实际影响。

交付先给结论，再说明修改、验证和剩余未知，不以文档数量、篇幅或检查数量证明质量。
