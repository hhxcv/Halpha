# M0 Technical Implementation Plan

## Purpose

This document defines the M0 implementation plan for Halpha.

M0 should deliver the smallest useful local report loop:

```text
market data + public information
  -> local collection or local loading
  -> locally inspectable raw artifacts
  -> AI-readable analysis artifacts
  -> Codex CLI report context
  -> Simplified Chinese Markdown research report
  -> local run archive
```

The design intentionally avoids extra architecture. M0 should prefer one working path over a broad framework.

## Scope

M0 must support one command that runs the full report loop:

```bash
python -m halpha run --config config.example.yaml
```

A successful run should create a local run directory:

```text
runs/<run_id>/
  raw/
    market.json
    public_info.json
  analysis/
    market_material.md
    public_info_material.md
    research_context.md
  codex_context/
    prompt.md
    context.md
  report/
    report.md
  run_manifest.json
```

The exact file names may change during implementation, but every M0 run must preserve these artifact roles:

| Artifact area | Responsibility |
| --- | --- |
| `raw/` | Store collected or loaded source data without analytical rewriting. |
| `analysis/` | Store deterministic, AI-readable material prepared by Halpha. |
| `codex_context/` | Store the complete prompt and context sent to Codex CLI. |
| `report/` | Store the final Simplified Chinese Markdown report generated by Codex CLI. |
| `run_manifest.json` | Store run status, timestamps, input references, artifact paths, counts, and errors. |

## Non-goals

M0 must not implement:

- trading execution;
- exchange account access;
- portfolio automation;
- dashboard UI;
- multi-user behavior;
- hosted service deployment;
- database-backed storage;
- generalized plugin architecture;
- background scheduling;
- broad data-source frameworks.

Real network collectors can be added only when they are required for the first working path. Until then, local fixture loading is acceptable as the collection boundary.

## Architecture

M0 should use a small Python-first local pipeline:

```text
config
  -> run directory creation
  -> market input collection/loading
  -> public information collection/loading
  -> raw artifact persistence
  -> AI-readable analysis material generation
  -> Codex prompt/context generation
  -> Codex CLI execution
  -> report persistence
  -> manifest update
```

Halpha owns collection, normalization, source preservation, context assembly, and local artifact management. Codex CLI owns the final report writing step.

## Technology choices

| Area | M0 choice | Reason |
| --- | --- | --- |
| Language | Python 3.11+ | Standard library coverage is sufficient for the first loop. |
| CLI | `argparse` | Avoids unnecessary CLI dependencies. |
| Config | YAML via `PyYAML` | Matches the intended configuration style. |
| Storage | Plain JSON and Markdown files | Local-first and inspectable. |
| Test runner | `pytest` | Small and conventional for Python projects. |
| Report generation | `codex exec` through `subprocess.run()` | Keeps final narrative generation outside Halpha while preserving a local context artifact. |

Runtime dependencies should be minimal:

```toml
dependencies = [
  "PyYAML>=6,<7"
]

[project.optional-dependencies]
dev = [
  "pytest>=8,<9"
]
```

No data-analysis, web, database, workflow, or UI frameworks are needed for M0.

## Configuration contract

`config.example.yaml` should stay narrow:

```yaml
run:
  output_dir: runs
  timezone: Asia/Shanghai

inputs:
  market_file: examples/inputs/market.json
  public_info_file: examples/inputs/public_info.json

report:
  title: Daily Market Intelligence Brief
  language: zh-CN

codex:
  enabled: true
  command: codex
  args:
    - exec
    - --sandbox
    - read-only
    - "-"
  timeout_seconds: 300
```

Required validation:

- `run.output_dir` is present.
- `inputs.market_file` is present.
- `inputs.public_info_file` is present.
- `report.language` is `zh-CN` for M0.
- `codex.command`, `codex.args`, and `codex.timeout_seconds` are present when `codex.enabled` is true.

Do not add environment profiles, multiple report types, or source registries in M0.

## Input data contract

M0 inputs are local JSON artifacts produced by collectors or loaded as fixtures. They are not human-curated summaries.

### Market input

`examples/inputs/market.json`:

```json
{
  "schema_version": 1,
  "artifact_type": "market_input",
  "source": "local-fixture",
  "collection_method": "programmatic_fixture",
  "collected_at": "2026-06-03T08:00:00+08:00",
  "items": [
    {
      "id": "market:BTCUSDT:2026-06-03T08:00:00+08:00",
      "symbol": "BTCUSDT",
      "as_of": "2026-06-03T08:00:00+08:00",
      "metrics": {
        "price": 68000.0,
        "change_24h_pct": 1.25,
        "volume_24h": 123456789.0
      },
      "source": {
        "name": "local-fixture",
        "url": null
      }
    }
  ]
}
```

### Public information input

`examples/inputs/public_info.json`:

```json
{
  "schema_version": 1,
  "artifact_type": "public_info_input",
  "source": "local-fixture",
  "collection_method": "programmatic_fixture",
  "collected_at": "2026-06-03T08:00:00+08:00",
  "items": [
    {
      "id": "public:example-feed:001",
      "type": "rss_item",
      "title": "Example public market event title",
      "published_at": "2026-06-03T07:30:00+08:00",
      "source": {
        "name": "example-feed",
        "url": null
      },
      "content_text": "Collector-provided public information text. In a real collector this field comes from RSS, API, HTML extraction, or another configured public source.",
      "language": "en"
    }
  ]
}
```

Important constraints:

- `public_info.json` must not describe its content as manually written or manually curated.
- The `content_text` field stores collected or fixture-loaded source text.
- If a later step summarizes `content_text`, the summary must be labeled as derived material, not raw input.
- Source name must always be retained.
- Source URL may be `null`, but missing URLs must stay explicit.

Minimum validation:

| Object | Validation |
| --- | --- |
| Top-level input | Must be a JSON object. |
| `items` | Must exist and must be a list. |
| Market item | Must include `id`, `symbol`, `as_of`, and `source.name`. |
| Public info item | Must include `id`, `title`, `published_at`, `source.name`, and `content_text`. |
| Source URL | May be `null`; must not be fabricated. |

## Analysis material contract

Analysis artifacts are optimized for AI consumption, not human reading.

They should be deterministic, sectioned, field-oriented, and low-prose. Avoid decorative tables, narrative paragraphs, and human-oriented formatting unless they improve model grounding.

### General rules

Analysis material should:

- use stable headings;
- use stable field names;
- preserve source references close to facts;
- separate raw facts, derived observations, assumptions, and uncertainties;
- use deterministic ordering;
- avoid implicit context;
- avoid invented interpretation;
- mark missing values as `null` or `unknown` explicitly;
- include artifact paths where useful for later review.

Recommended format: Markdown container plus fenced YAML records.

This keeps files easy to inspect locally while making the content more directly parseable by AI systems.

### `analysis/market_material.md`

````markdown
---
artifact_type: analysis_market_material
schema_version: 1
audience: ai
source_artifacts:
  - raw/market.json
---

# market_material

## record: market:BTCUSDT:2026-06-03T08:00:00+08:00

```yaml
record_type: market_observation
id: market:BTCUSDT:2026-06-03T08:00:00+08:00
symbol: BTCUSDT
as_of: 2026-06-03T08:00:00+08:00
metrics:
  price: 68000.0
  change_24h_pct: 1.25
  volume_24h: 123456789.0
source:
  name: local-fixture
  url: null
facts:
  - BTCUSDT price is 68000.0 as of 2026-06-03T08:00:00+08:00.
  - BTCUSDT 24h change is 1.25%.
derived_observations: []
assumptions: []
uncertainties:
  - Source URL is not provided.
```
````

Do not optimize this file for visual readability. Optimize it for source-grounded downstream generation.

### `analysis/public_info_material.md`

````markdown
---
artifact_type: analysis_public_info_material
schema_version: 1
audience: ai
source_artifacts:
  - raw/public_info.json
---

# public_info_material

## record: public:example-feed:001

```yaml
````

`derived_summary` is allowed only when generated by code or AI from `content_text`. It must not be represented as manual work.

## Codex context and prompt

`codex_context/context.md` should package the full local context:

````markdown
# codex_context

## artifact_index

```yaml
raw_market: raw/market.json
raw_public_info: raw/public_info.json
market_material: analysis/market_material.md
public_info_material: analysis/public_info_material.md
research_context: analysis/research_context.md
```

## research_context

<embed analysis/research_context.md>
````

`codex_context/prompt.md` should contain the full instruction plus the local context:

```markdown
You are the report-generation step for Halpha.

Generate a Simplified Chinese Markdown market intelligence report from the local context below.

Rules:

1. Use only the facts, source text, and derived material in the provided context.
2. Do not invent prices, events, links, sources, or certainty.
3. Preserve source awareness. When a fact is used, keep its source name nearby.
4. If a source URL is missing, say that the source URL was not provided.
5. Distinguish facts, assumptions, uncertainties, and judgment.
6. Use cautious language for market interpretation.
7. Include a risk notice.
8. The report is personal research material and is not financial advice.

Required sections:

- Title
- Core Summary
- Market Overview
- Public Information
- Synthesis
- Watch Points
- Risk Notice

<context>
...
</context>
```

Halpha should pass the generated prompt to Codex CLI and capture the final output as `report/report.md`.

## Codex CLI execution

M0 should use `subprocess.run()` and keep execution explicit:

```python
result = subprocess.run(
    [config["codex"]["command"], *config["codex"]["args"]],
    input=full_prompt,
    text=True,
    capture_output=True,
    timeout=config["codex"]["timeout_seconds"],
    check=False,
)
```

Recommended command shape:

```bash
codex exec --sandbox read-only -
```

Rationale:

- `codex exec` is the non-interactive mode for scriptable execution.
- `-` makes stdin the complete prompt.
- `read-only` is sufficient because Codex does not need to modify repository files.
- Halpha writes `report/report.md` itself from Codex stdout.

On success:

```text
result.stdout -> report/report.md
result.stderr -> optional diagnostic artifact or manifest summary
result.returncode -> run_manifest.json
```

On failure:

- write `status: failed` to `run_manifest.json`;
- include stage `codex`;
- include exit code when available;
- include a short stderr excerpt;
- return a non-zero CLI exit code.

Do not print secrets, API keys, tokens, cookies, or local auth files.

```json

  "schema_version": 1,
  "run_id": "20260603T000000Z",
  "status": "succeeded",
  "started_at": "2026-06-03T00:00:00Z",
  "finished_at": "2026-06-03T00:00:10Z",
  "config_path": "config.example.yaml",
  "inputs": {
    "market_file": "examples/inputs/market.json",
    "public_info_file": "examples/inputs/public_info.json"
  },
  "artifacts": {
    "raw_market": "raw/market.json",
    "raw_public_info": "raw/public_info.json",
    "market_material": "analysis/market_material.md",
    "public_info_material": "analysis/public_info_material.md",
    "research_context": "analysis/research_context.md",
```

Use UTC for `run_id` sorting:

```python
run_id = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%SZ")
```

If the run ID already exists, append a short suffix.

## Pipeline interface

Recommended narrow interfaces:

```python
def load_config(path: Path) -> dict: ...
def run_pipeline(config: dict, *, config_path: Path, codex_runner=run_codex_cli) -> RunContext: ...
def load_market_input(config: dict) -> dict: ...
def load_public_info_input(config: dict) -> dict: ...
def build_market_material(raw: dict) -> str: ...
def build_public_info_material(raw: dict) -> str: ...
def build_research_context(config: dict, run_id: str, market_material: str, public_info_material: str) -> str: ...
def build_codex_prompt(config: dict, research_context: str) -> str: ...
def run_codex_cli(config: dict, prompt: str, cwd: Path) -> tuple[str, dict]: ...
```

Tests should inject a fake `codex_runner` so the pipeline can be validated without Codex CLI, network access, or credentials.

## CLI behavior

Success output:

```text
Halpha run succeeded.
run_id: 20260603T000000Z
report: runs/20260603T000000Z/report/report.md
manifest: runs/20260603T000000Z/run_manifest.json
```

Failure output:

```text
Halpha run failed.
stage: codex
reason: Codex CLI not found: codex
manifest: runs/20260603T000000Z/run_manifest.json
```

Suggested exit codes:

| Exit code | Meaning |
| ---: | --- |
| 0 | Success. |
| 1 | Unclassified runtime failure. |
| 2 | Configuration error. |
| 3 | Input loading or validation error. |
| 4 | Codex execution error. |

M0 may initially use `0` and `1` only, provided the error message is actionable.

## Testing plan

`tests/test_m0_pipeline.py` should cover the smoke path:

1. Use `tmp_path` as `run.output_dir`.
2. Write temporary `market.json` and `public_info.json` fixtures.
3. Load a test config.
4. Inject a fake Codex runner that returns Simplified Chinese Markdown.
5. Run the full pipeline.
6. Assert these artifacts exist:
   - `raw/market.json`
   - `raw/public_info.json`
   - `analysis/market_material.md`
   - `analysis/public_info_material.md`
   - `analysis/research_context.md`
   - `codex_context/prompt.md`
   - `codex_context/context.md`
   - `report/report.md`
   - `run_manifest.json`
7. Assert `run_manifest.json` has `status: succeeded`.
8. Assert `report/report.md` contains Simplified Chinese content and a risk notice.

Acceptance command:

```bash
python -m pytest
python -m halpha run --config config.example.yaml
```

The test suite must not require real Codex CLI access, network access, exchange accounts, or API keys.

## Implementation sequence

### Step 1: package skeleton and CLI

Create the Python package, config loader, run directory creation, and initial manifest writing.

Done when:

- `python -m halpha run --config config.example.yaml` creates `runs/<run_id>/run_manifest.json`.
- `python -m pytest` runs.

### Step 2: local input loading

Load market and public information JSON from local files and persist raw copies.

Done when:

- `runs/<run_id>/raw/market.json` exists.
- `runs/<run_id>/raw/public_info.json` exists.
- missing required fields produce actionable errors.

### Step 3: AI-readable analysis material

Generate deterministic Markdown-plus-YAML analysis artifacts.

Done when:

- `analysis/market_material.md` exists.
- `analysis/public_info_material.md` exists.
- `analysis/research_context.md` exists.
- each material preserves source names close to the facts.

### Step 4: Codex context building

Generate the complete Codex prompt and context artifacts.

Done when:

- `codex_context/prompt.md` exists.
- `codex_context/context.md` exists.
- the prompt forbids fabricated prices, events, links, and sources.

### Step 5: Codex CLI report generation

Call Codex CLI, capture stdout, persist the final report, and update the manifest.

Done when:

- test runs pass with a fake Codex runner;
- local runs produce `report/report.md` when Codex CLI is available;
- Codex failures are recorded in `run_manifest.json`.

## M0 completion criteria

M0 is complete when:

- one command runs the full local report loop;
- raw inputs remain locally inspectable;
- analysis artifacts are AI-readable and source-aware;
- Codex prompt and context are persisted;
- the final report is Simplified Chinese Markdown;
- the run manifest records status, timestamps, inputs, outputs, counts, and Codex status;
- tests validate the pipeline without network or credentials;
- no non-goal features are introduced.
