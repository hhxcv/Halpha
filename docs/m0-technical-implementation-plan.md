# M0 Technical Implementation Plan

## Purpose

This document defines the M0 implementation plan for Halpha.

M0 should deliver the smallest useful report loop:

```text
market data + public information
  -> online data collection
  -> inspectable raw artifacts
  -> AI-readable analysis artifacts
  -> Codex CLI report context
  -> Simplified Chinese Markdown research report
  -> run archive
```

The design intentionally avoids extra architecture. M0 should prefer one working path over a broad framework.

Fixtures, static sample input files, fake collectors, and fake Codex runners are test-only tools. They must not be the product path for M0 and must not be used to claim that M0 can generate a useful report.

## Scope

M0 must support one command that runs the full report loop using real online data:

```bash
python -m halpha run --config config.example.yaml
```

A successful run should create a run directory:

```text
runs/<run_id>/
  raw/
    market.json
    text_events.json
  analysis/
    market_material.md
    text_material.md
    research_context.md
  codex_context/
    prompt.md
    context.md
  report/
    report.md
  run_manifest.json
```

The exact file names may change during implementation, but every M0 run must preserve these artifact roles:

| Artifact area | Responsibility |
| --- | --- |
| `raw/` | Store collected public source data without analytical rewriting. |
| `analysis/` | Store deterministic, AI-readable material prepared by Halpha. |
| `codex_context/` | Store the complete prompt and context sent to Codex CLI. |
| `report/` | Store the final Simplified Chinese Markdown report generated by Codex CLI. |
| `run_manifest.json` | Store run status, timestamps, source references, artifact paths, counts, and errors. |

## Product path and test path

M0 has two separate paths:

| Path | Purpose | Allowed inputs |
| --- | --- | --- |
| Product path | Produce a useful local market intelligence report. | Public market source configuration and public text source configuration. |
| Test path | Validate deterministic behavior without network, credentials, or Codex CLI. | Test fixtures, fake collectors, mocked HTTP responses, fake Codex runner. |

Rules:

- Product configuration must not default to local fixture files.
- A product run must attempt configured public data collection.
- If a product collection stage is not implemented, it must fail explicitly and write the failure to `run_manifest.json`.
- Test fixtures may exist under `tests/fixtures/` only.
- `examples/inputs/` should not be used as the M0 product input contract.
- The pipeline must never fabricate source data to make a run appear successful.

## Non-goals

M0 must not implement:

- trading execution;
- exchange account access;
- portfolio automation;
- dashboard UI;
- multi-user behavior;
- hosted service deployment;
- database-backed storage;
- generalized plugin architecture;
- background scheduling;
- broad data-source frameworks.

M0 may implement narrow real collectors for the first working path. These collectors should be explicit and source-specific, not plugin abstractions. Test fixtures may be used only in tests and examples, and must be labeled as fixtures.

## Architecture

M0 should use a small Python-first pipeline:

```text
config
  -> run directory creation
  -> public market data collection
  -> public text event collection
  -> raw artifact persistence
  -> AI-readable analysis material generation
  -> research context generation
  -> Codex prompt/context generation
  -> Codex CLI execution
  -> report persistence
  -> manifest update
```

Halpha owns collection, normalization, source preservation, context assembly, and artifact management. Codex CLI owns the final report writing step.

The pipeline scaffold should be implemented early. Stage functions may initially be present as explicit unimplemented interfaces, but an unimplemented stage must fail honestly. It must not emit placeholder source data, placeholder analysis, or a placeholder report on the product path.

## Technology choices

| Area | M0 choice | Reason |
| --- | --- | --- |
| Language | Python 3.11+ | Standard library coverage is sufficient for the first loop. |
| CLI | `argparse` | Avoids unnecessary CLI dependencies. |
| Config | YAML via `PyYAML` | Matches the intended configuration style. |
| Storage | Plain JSON and Markdown files | Simple and inspectable. |
| Market collection | Narrow public market collector | Gives M0 real report value without account access. |
| Text collection | Narrow public RSS/text collector | Gives M0 real public event context without broad crawling. |
| Test runner | `pytest` | Small and conventional for Python projects. |
| Report generation | `codex exec` through `subprocess.run()` | Keeps final narrative generation outside Halpha while preserving a reviewable context artifact. |

Runtime dependencies should stay minimal. A first implementation may use only `PyYAML` plus the Python standard library if HTTP and RSS handling are implemented narrowly. Add a package only when the current collector requires it.

```toml
dependencies = [
  "PyYAML>=6,<7"
]

[project.optional-dependencies]
dev = [
  "pytest>=8,<9"
]
```

No database, workflow framework, UI framework, or generalized data-ingestion framework is needed for M0.

## Configuration contract

`config.example.yaml` should be source-based and narrow:

```yaml
run:
  output_dir: runs
  timezone: Asia/Shanghai

market:
  enabled: true
  source: binance
  symbols:
    - BTCUSDT
    - ETHUSDT

text:
  enabled: true
  max_items: 30
  sources:
    - name: coindesk
      type: rss
      url: https://www.coindesk.com/arc/outboundfeeds/rss/
    - name: cointelegraph
      type: rss
      url: https://cointelegraph.com/rss

report:
  title: 每日市场情报简报
  language: zh-CN

codex:
  enabled: true
  command: codex
  args:
    - exec
    - --sandbox
    - read-only
    - "-"
  timeout_seconds: 300
```

Required validation:

- `run.output_dir` is present.
- `market.enabled` is present.
- `market.source` is present when `market.enabled` is true.
- `market.symbols` is a non-empty list when `market.enabled` is true.
- `text.enabled` is present.
- `text.sources` is a non-empty list when `text.enabled` is true.
- each text source has `name`, `type`, and `url`.
- `report.language` is `zh-CN` for M0.
- `codex.command`, `codex.args`, and `codex.timeout_seconds` are present when `codex.enabled` is true.

Do not add environment profiles, multiple report types, or source registries in M0.

Product config must not require:

```yaml
inputs:
  market_file: ...
  public_info_file: ...
```

Local file input keys may be used inside tests, but they must not be documented as the M0 product run contract.

## Raw artifact contract

M0 raw artifacts are produced by runtime collectors from configured public sources. They are not human-curated summaries and not product fixtures.

### Market raw artifact

`raw/market.json`:

```json
{
  "schema_version": 1,
  "artifact_type": "market_raw",
  "collector": "market",
  "collection_method": "public_http",
  "source": {
    "name": "binance",
    "url": "https://api.binance.com"
  },
  "collected_at": "2026-06-03T00:00:00Z",
  "items": [
    {
      "id": "market:binance:BTCUSDT:2026-06-03T00:00:00Z",
      "symbol": "BTCUSDT",
      "as_of": "2026-06-03T00:00:00Z",
      "metrics": {
        "price": "68000.00",
        "change_24h_pct": "1.25",
        "volume_24h": "123456789.0"
      },
      "source": {
        "name": "binance",
        "url": "https://api.binance.com"
      }
    }
  ],
  "errors": []
}
```

### Text event raw artifact

`raw/text_events.json`:

```json
{
  "schema_version": 1,
  "artifact_type": "text_events_raw",
  "collector": "text",
  "collection_method": "rss",
  "collected_at": "2026-06-03T00:00:00Z",
  "sources": [
    {
      "name": "coindesk",
      "type": "rss",
      "url": "https://www.coindesk.com/arc/outboundfeeds/rss/"
    }
  ],
  "items": [
    {
      "id": "text:coindesk:example-item-001",
      "type": "rss_item",
      "title": "Example public market event title",
      "published_at": "2026-06-03T00:00:00Z",
      "source": {
        "name": "coindesk",
        "url": "https://www.coindesk.com/arc/outboundfeeds/rss/"
      },
      "link": "https://example.com/item",
      "content_text": "Collector-provided public information text from the configured RSS item.",
      "language": "en"
    }
  ],
  "errors": []
}
```

Important constraints:

- Raw artifacts must not describe their content as manually written or manually curated.
- `content_text` stores collected source text or source-provided feed text.
- If a later step summarizes `content_text`, the summary must be labeled as derived material, not raw input.
- Source name must always be retained.
- Source URL may be `null` only when the source truly lacks a URL; missing URLs must stay explicit.
- Collector failures must be recorded. They must not be covered by fabricated fallback records.

Minimum validation:

| Object | Validation |
| --- | --- |
| Top-level raw artifact | Must be a JSON object. |
| `items` | Must exist and must be a list. |
| Market item | Must include `id`, `symbol`, `as_of`, and `source.name`. |
| Text event item | Must include `id`, `title`, `published_at`, `source.name`, and `content_text`. |
| Source URL | May be `null`; must not be fabricated. |

## Collector behavior

### Market collector

M0 should implement one narrow public market collector first.

Required behavior:

- read `market.source`;
- read `market.symbols`;
- collect public market data without account access;
- preserve source metadata close to each item;
- write `raw/market.json`;
- record collector errors in the raw artifact and manifest.

Out of scope:

- exchange account operations;
- authenticated endpoints;
- order placement;
- portfolio balances;
- multi-exchange abstraction.

### Text event collector

M0 should implement one narrow public text collector first, preferably RSS because it is simple and source-aware.

Required behavior:

- read `text.sources`;
- support `type: rss`;
- collect configured feeds;
- extract title, published time, source name, source URL, item link, and source-provided text;
- respect `text.max_items`;
- write `raw/text_events.json`;
- record per-source errors.

Out of scope:

- broad web crawling;
- search engine integration;
- paywalled content extraction;
- anti-bot bypassing;
- generalized scraping framework.

## Analysis material contract

Analysis artifacts are optimized for AI consumption, not human reading.

They should be deterministic, sectioned, field-oriented, and low-prose. Avoid decorative tables, narrative paragraphs, and human-oriented formatting unless they improve model grounding.

### General rules

Analysis material should:

- use stable headings;
- use stable field names;
- preserve source references close to facts;
- separate raw facts, derived observations, assumptions, and uncertainties;
- use deterministic ordering;
- avoid implicit context;
- avoid invented interpretation;
- mark missing values as `null` or `unknown` explicitly;
- include artifact paths where useful for later review.

Recommended format: Markdown container plus fenced YAML records.

This keeps files easy to inspect locally while making the content more directly parseable by AI systems.

### `analysis/market_material.md`

````markdown
---
artifact_type: analysis_market_material
schema_version: 1
audience: ai
source_artifacts:
  - raw/market.json
---

# market_material

## record: market:binance:BTCUSDT:2026-06-03T00:00:00Z

```yaml
record_type: market_observation
id: market:binance:BTCUSDT:2026-06-03T00:00:00Z
symbol: BTCUSDT
as_of: 2026-06-03T00:00:00Z
metrics:
  price: "68000.00"
  change_24h_pct: "1.25"
  volume_24h: "123456789.0"
source:
  name: binance
  url: https://api.binance.com
facts:
  - BTCUSDT price is 68000.00 as of 2026-06-03T00:00:00Z.
  - BTCUSDT 24h change is 1.25%.
derived_observations: []
assumptions: []
uncertainties: []
```
````

Do not optimize this file for visual readability. Optimize it for source-grounded downstream generation.

### `analysis/text_material.md`

````markdown
---
artifact_type: analysis_text_material
schema_version: 1
audience: ai
source_artifacts:
  - raw/text_events.json
---

````

`derived_summary` is allowed only when generated by code or AI from `content_text`. It must not be represented as manual work.

## Codex context and prompt

`codex_context/context.md` should package the full local context:

````markdown
# codex_context

## artifact_index

```yaml
raw_market: raw/market.json
raw_text_events: raw/text_events.json
market_material: analysis/market_material.md
text_material: analysis/text_material.md
research_context: analysis/research_context.md
```

## research_context

<embed analysis/research_context.md>
````

`codex_context/prompt.md` should contain the full instruction plus the local context:

```markdown
You are the report-generation step for Halpha.

Generate a Simplified Chinese Markdown market intelligence report from the local context below.

Rules:

1. Use only the facts, source text, and derived material in the provided context.
2. Do not invent prices, events, links, sources, or certainty.
3. Preserve source awareness. When a fact is used, keep its source name nearby.
4. If a source URL is missing, say that the source URL was not provided.
5. Distinguish facts, assumptions, uncertainties, and judgment.
6. Use cautious language for market interpretation.
7. Include a risk notice.
8. The report is personal research material and is not financial advice.

Required sections:

- Title
- Core Summary
- Market Overview
- Text Events
- Synthesis
- Watch Points
- Risk Notice

<context>
...
</context>
```

Halpha should pass the generated prompt to Codex CLI and capture the final output as `report/report.md`.

## Codex CLI execution

M0 should use `subprocess.run()` and keep execution explicit:

```python
result = subprocess.run(
    [config["codex"]["command"], *config["codex"]["args"]],
    input=full_prompt,
    text=True,
    capture_output=True,
    timeout=config["codex"]["timeout_seconds"],
    check=False,
)
```

Recommended command shape:

```bash
codex exec --sandbox read-only -
```

Rationale:

- `codex exec` is the non-interactive mode for scriptable execution.
- `-` makes stdin the complete prompt.
- `read-only` is sufficient because Codex does not need to modify repository files.
- Halpha writes `report/report.md` itself from Codex stdout.

On success:

```text
result.stdout -> report/report.md
result.stderr -> optional diagnostic artifact or manifest summary
result.returncode -> run_manifest.json
```

On failure:

- write `status: failed` to `run_manifest.json`;
- include stage `codex`;
- include exit code when available;
- include a short stderr excerpt;
- return a non-zero CLI exit code.

Do not print secrets, API keys, tokens, cookies, or local auth files.

Use UTC for `run_id` sorting:

```python
run_id = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%SZ")
```

If the run ID already exists, append a short suffix.

When a stage is unimplemented or fails, the manifest must record the failed stage and the error. The CLI must return a non-zero exit code.

## Pipeline interface

Recommended narrow interfaces:

```python
def load_config(path: Path) -> dict: ...
def run_pipeline(config: dict, *, config_path: Path, codex_runner=run_codex_cli) -> RunContext: ...
def collect_market_data(config: dict, run: RunContext) -> dict: ...
def collect_text_events(config: dict, run: RunContext) -> dict: ...
def build_market_material(raw: dict) -> str: ...
def build_text_material(raw: dict) -> str: ...
def build_research_context(config: dict, run_id: str, market_material: str, text_material: str) -> str: ...
def build_codex_prompt(config: dict, research_context: str) -> str: ...
def run_codex_cli(config: dict, prompt: str, cwd: Path) -> tuple[str, dict]: ...
```

Tests may inject fake collectors and a fake `codex_runner` so the pipeline can be validated without network access, credentials, or Codex CLI. These fakes are not product behavior.

## CLI behavior

Success output:

```text
Halpha run succeeded.
run_id: 20260603T000000Z
report: runs/20260603T000000Z/report/report.md
manifest: runs/20260603T000000Z/run_manifest.json
```

Failure output:

```text
Halpha run failed.
stage: collect_market_data
reason: stage collect_market_data is not implemented
manifest: runs/20260603T000000Z/run_manifest.json
```

Suggested exit codes:

| Exit code | Meaning |
| ---: | --- |
| 0 | Success. |
| 1 | Unclassified runtime failure. |
| 2 | Configuration error. |
| 3 | Collection or input validation error. |
| 4 | Codex execution error. |

M0 may initially use `0` and `1` only, provided the error message is actionable.

## Testing plan

`tests/test_smoke_pipeline.py` should validate behavior without becoming the product path.

Recommended tests:

1. Pipeline scaffold test:
   - use fake stage implementations;
   - run the full orchestrator;
   - verify run directory and manifest behavior.
2. Collector tests:
   - mock HTTP responses;
   - verify market and text raw artifacts;
   - do not call real network in tests.
3. Analysis tests:
   - use `tests/fixtures/raw_market.json` and `tests/fixtures/raw_text_events.json`;
   - verify AI-readable material generation.
4. Codex runner tests:
   - mock `subprocess.run()` or inject a fake Codex runner;
   - verify report persistence and failure handling.

The smoke test may use fakes and fixtures, but acceptance of M0 product behavior requires a real-source run in an environment with public network access and Codex CLI available.

Acceptance command for automated checks:

```bash
python -m pytest
```

Manual acceptance command for the product path:

```bash
python -m halpha run --config config.example.yaml
```

The automated test suite must not require real Codex CLI access, network access, exchange accounts, or API keys.

## Implementation sequence

### Step 1: pipeline scaffold, package skeleton, and CLI

Create the Python package, config loader, run directory creation, manifest lifecycle, and explicit stage interfaces.

Done when:

- `python -m halpha run --config config.example.yaml` starts the pipeline.
- unimplemented stages fail explicitly and write `run_manifest.json`.
- `python -m pytest` runs.

### Step 2: source-based config

Implement real-source configuration for public market data and public text events.

Done when:

- `config.example.yaml` uses `market` and `text` sections.
- product config does not depend on local input files.
- invalid source config produces actionable errors.

### Step 3: public market collector

Collect public market data for configured symbols and persist `raw/market.json`.

Done when:

- a real-source run writes `raw/market.json` in an online environment.
- market items include source metadata.
- failures are recorded without fabricated fallback data.

### Step 4: public text event collector

Collect public text events from configured RSS sources and persist `raw/text_events.json`.

Done when:

- a real-source run writes `raw/text_events.json` in an online environment.
- text items include source metadata and source text.
- failures are recorded without fabricated fallback data.

### Step 5: AI-readable analysis material

Generate deterministic Markdown-plus-YAML analysis artifacts from real raw artifacts.

Done when:

- `analysis/market_material.md` exists.
- `analysis/text_material.md` exists.
- each material preserves source names close to the facts.

### Step 6: research context

Generate the complete research context from analysis materials.

Done when:

- `analysis/research_context.md` exists.
- it contains market material, text material, source policy, and generation constraints.

### Step 7: Codex context building

Generate the complete Codex prompt and context artifacts.

Done when:

- `codex_context/prompt.md` exists.
- `codex_context/context.md` exists.
- the prompt forbids fabricated prices, events, links, and sources.

### Step 8: Codex CLI report generation

Call Codex CLI, capture stdout, persist the final report, and update the manifest.

Done when:

- runs produce `report/report.md` when Codex CLI is available;
- Codex failures are recorded in `run_manifest.json`.

### Step 9: real-source end-to-end run

Wire all implemented stages into the main command.

Done when:

- `python -m halpha run --config config.example.yaml` can generate a useful Simplified Chinese Markdown report from real public sources in a properly configured local environment.
- all raw, analysis, Codex context, report, and manifest artifacts are present.

### Step 10: tests and docs alignment

Add tests with fakes, mocks, and test fixtures. Update README and AGENTS only after behavior exists.

Done when:

- `python -m pytest` passes.
- README and AGENTS describe only implemented behavior.

## M0 completion criteria

M0 is complete when:

- one command runs the full report loop;
- the product path collects real public market data and real public text event data;
- raw inputs remain locally inspectable;
- analysis artifacts are AI-readable and source-aware;
- Codex prompt and context are persisted;
- the final report is Simplified Chinese Markdown generated by Codex CLI;
- the run manifest records status, timestamps, sources, outputs, counts, stages, and Codex status;
- tests validate the pipeline without network or credentials;
- fixtures are limited to tests and are not used as M0 product inputs;
- no non-goal features are introduced.
