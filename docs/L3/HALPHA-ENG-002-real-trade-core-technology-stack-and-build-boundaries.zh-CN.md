# 0. 技术结论【ENG-AUTO-SUM-001】

Halpha 使用一套 Python 后端、一套 React 前端、一个 PostgreSQL 产品数据库和一个 NautilusTrader 量化交易栈。模拟与真实写使用同一产品代码、策略类、交易计划入口、动作记录和交易所客户端路径；环境差异只由受控配置选择，不保留第二套实现。

NautilusTrader 承担行情、指标、回测、订单模型和 Binance 适配。Halpha 只保留组件不能替代的产品语义：固定计划激活、资金限制、持久动作唯一、用户停止与接管、外部事实核对和结果归属。

本文件固定长期组件边界。精确版本、当前 Windows 主机、配置、构建标识和检查结果由 L4 记录。

# 1. 第三方优先决策【ENG-AUTO-DEP-001】

成熟组件已经提供的通用能力不得在 Halpha 内再造。组件差异若只影响尚未固定的技术选择，按组件公开行为调整；若会破坏动作唯一、事实权威、资金限制、停止或用户控制，才补最小边界代码。

组件失败或不兼容时停止依赖它的能力并暴露明确错误，不自动切换到第二实现。只有实际重复故障证明单实现无法满足产品结果时，才重新评估替代。

# 2. NautilusTrader 使用契约【ENG-AUTO-NTL-001】

halpha-executor 内只有一个 TradingNode；halpha-app 不创建交易节点，也不持有 Binance 凭据。实时和回测复用同一策略类与动作提议边界。Halpha 不实现第二行情泵、指标引擎、撮合回放、订单状态机或交易连接作为后备。

NautilusTrader 的缓存、组合和风险投影属于技术输入，不替代 Halpha 的账户事实、计划激活、资金限制或动作记录。涉及资金上限、最大允许损失和归属比较的权威数值使用 Python Decimal 与 PostgreSQL NUMERIC；组件数值进入这些边界前必须显式转换并验证精度。

组件升级至少检查：

- Binance 行情与执行客户端的公开契约；
- 订单身份、状态、重连和查询行为；
- 指标、回测和实时策略输入的一致性；
- 数值转换、时间边界和异常映射；
- Windows 安装与停止行为。

# 3. App、数据与前端契约【ENG-AUTO-APP-001】

FastAPI、Pydantic、SQLAlchemy/Alembic 和 PostgreSQL 承担 HTTP、schema、事务、迁移和持久化通用能力。React、MUI、JSON Schema 表单、TanStack Query 和生成客户端承担前端通用能力。Halpha 只编写产品路由、业务校验、命令调用和必要投影，不维护第二操作界面；本地界面不可用时，用户通过 Binance 官方入口接管。

数据库是产品事实的唯一持久权威。前端缓存、量化组件缓存、日志和邮件都不是交易计划、账户、动作或结果的替代来源。迁移失败、schema 不匹配或数据库不可用时，相关写能力关闭。

# 4. 运行 profile【ENG-AUTO-ENV-001】

| profile | 能力 | 禁止 |
|---|---|---|
| DEMO | 产品闭环与模拟场所写 | 真实交易凭据和真实场所写 |
| LIVE_READ_ONLY | 公开行情、策略观察和只读输出 | 产品激活、产品数据库写、私有账户读取、交易提交 |
| LIVE_WRITE | 使用同一产品闭环执行用户已激活计划 | 未通过当前事实复核时加载秘密或提交 |

LIVE_READ_ONLY 从同一策略和数据适配路径单调移除执行客户端、产品写 repository、账户私有能力和秘密加载；它不是第二策略或第二产品。

DEMO 与 LIVE_WRITE 使用分离的配置、数据库或明确隔离的 schema、凭据引用、日志和进程互斥。任何环境身份不清或混用时，真实写关闭。

# 5. Windows、构建、迁移与备份【ENG-AUTO-BLD-001】

Windows 使用操作系统现成能力管理进程、凭据、文件权限、日志和计划启动；不为个人项目再建服务编排平台。实际 Windows 版本是否适合当前组件，由 L4 的当前安装和运行检查决定。供应商未声明常规支持时，应记录限制并用本机实际验证判断当前项目能否继续，不虚构官方支持，也不因此自动迁移平台或缩减产品目标。

每个可部署产品只有一个 product_build_id。它至少绑定：

- 产品源码版本；
- Python 与前端锁定依赖；
- 数据库迁移头；
- 前端制品；
- 非秘密运行配置。

研究、普通文档和未进入产品运行的工具不属于产品构建输入。它们的变化不改变 product_build_id；一旦研究结果进入正式策略源码或产品配置，就按普通产品变更生成新标识。

构建记录同时保留来源提交用于追溯，但来源提交不是第二个资格身份。当前标识值、输入摘要和部署位置由 L4 记录。

数据库变更通过 Alembic 串行迁移。不可逆变更前必须有可用备份；恢复检查至少证明空环境能够恢复到可读取关键状态并能安全停止。备份、恢复和回退失败时，不启用依赖它们的真实写能力。

# 6. 秘密与真实写前置检查【ENG-AUTO-SEC-001】

秘密保存在 Windows 凭据存储或同等外部秘密工具中，不能进入仓库、数据库业务表、日志、前端或构建记录。

Executor 只有在准备提交动作且以下事实均成立后，才解析真实交易秘密：

1. 当前 product_build_id 与运行输入一致；
2. 数据库和迁移可用；
3. 只有一个写执行者；
4. 当前 PlanActivation 有效，且包含账户、工具、方向、数量或资金限制、最大允许损失、有效期和允许动作；
5. 当前账户与场所事实没有阻断性未知；
6. 拟提交动作仍未存在，且提交前再次检查计划、限制和停止状态。

任一条件未知、过期或冲突时保持关闭，不解析秘密。这里不另建外置授权文件、第二资金授权对象或通用准入状态机；PlanActivation 是用户当前交易决定的唯一产品入口。

# 7. 失败与验收契约【ENG-AUTO-TST-001】

当前固定组件和构建至少验证：

- 模拟与真实写 composition 使用同一策略、计划、动作和客户端代码路径；
- 只读 profile 不具备产品写、私有账户或交易提交能力；
- 重复请求、重启和查询后重试不产生第二个 ExecutionAction 或第二次提交；
- 数据库、迁移、配置、构建标识或唯一写者异常时真实写关闭；
- 真实交易秘密在所有前置检查完成前不会被读取；
- 前端制品损坏时不产生第二业务状态，官方场所接管指引仍可取得；
- Windows 启动、停止、互斥、凭据权限、日志、备份与恢复可实际运行；
- 组件版本、许可证、安装来源和已知支持限制在 L4 可追溯。

这些检查按受影响路径执行，不要求每次修改重跑无关观察。一次成功不能证明未来收益，也不能自动扩大资金范围。
