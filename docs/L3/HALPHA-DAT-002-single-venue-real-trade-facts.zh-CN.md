# Halpha 场所交易事实契约

**层级：** L3  
**L3 类型：** DOMAIN  
**主要语义所有者：** DAT  
**所属实现模块：** venue_integration 的事实分区  
**语言版本：** zh-CN

**直接依赖：** 无  
**直接消费者：** HALPHA-ALP-002、HALPHA-TRADEPLAN-002、HALPHA-CAP-002、HALPHA-EXE-002、HALPHA-OUT-002  
**适用纵向约束：** HALPHA-SYS-001、HALPHA-ENG-001  
**本文档负责：** 场所事实记录、来源映射、归属、未知、修正和组件边界  
**本文档不负责：** 策略、计划、资金、动作决定和当前场所、账户、阈值

## 1. 设计结论

Halpha 使用一个追加式场所事实结构保存当前消费者实际引用的市场、工具、账户、订单、成交、费用、资金费、持仓和保护事实。未被产品决定使用的行情留在量化组件内，不建设全量行情仓库或第二事实源。

## 2. 最小事实结构

每条事实包含稳定事实身份、环境与场所、按用途适用的账户和工具、来源和外部对象身份、来源时间、取得时间、事实截止点、规范内容摘要、业务载荷，以及必要的激活/动作归属或修正引用。

资本数值使用确定精度并保存单位和来源精度。相同来源对象和内容幂等；同一外部身份出现不同内容时追加变化并在冲突解决前返回未知。

当前支持的事实种类由产品代码和 L4 记录，不在本契约固定数量。

## 3. 来源与归属

场所查询和带稳定身份的公开事件可以形成事实。组件缓存、组合、风险投影和合成事件只能帮助寻找或比较，不能单独证明场所受理、实际费用、持仓归零或责任闭合。

订单、成交和费用只有在稳定外部身份、环境、账户、工具和条款都匹配同一已持久动作时才归属 Halpha。用户接管和其他外部活动保持外部身份，不能补造成 Halpha 动作或精确策略收益。

## 4. 消费、修正与恢复

消费者按用途、环境、对象和截止点查询，DAT 返回满足要求的事实或未知原因。缺失、过期、冲突、时钟不可信或归属不明时，不拼接不同截止点、不填零或沿用最近值。

来源修正时追加新事实并引用原因；DAT 不创建修正任务或反向推进计划和动作。系统恢复时，消费者按未闭合责任重新查询；DAT 不拥有系统恢复顺序，也不补发错过动作。

## 5. 组件与验证

量化组件承担订阅、历史读取、工具模型、订单公开事件和连接恢复。Halpha 只保存被实际消费的事实；公开能力不足时可以在同一只读连接边界增加窄查询，不建立第二客户端、SDK、缓存或恢复循环。

至少验证来源与时间、幂等和追加修正、未知不被填零、外部身份归属、Demo/Live 隔离、重启后按责任补取，以及只读补充没有写能力。
