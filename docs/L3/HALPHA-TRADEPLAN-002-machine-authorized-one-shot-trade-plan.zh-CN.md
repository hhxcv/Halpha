# 0. 设计结论【TRADEPLAN-AUTO-SUM-001】

用户先保存一个不可变 `TradePlanVersion`，再通过一次明确激活命令创建一个 `PlanActivation`。该激活直接固定计划、代码版本、环境、账户、工具、方向、资金限制、动作范围、期限和机器执行授权；不再创建账户上限、独立机器授权、额度分配、停止版本或恢复状态机。

每次激活只运行一个交易周期。首次成交前可以按固定规则等待、撤销或重挂入场单；发生入场成交后，完全平仓并清理全部责任即结束，不自动重新入场。停止新增风险后不能恢复同一激活；用户要继续时必须建立新激活。

TRADEPLAN 只拥有三个持久记录族：`TradePlanVersion`、`PlanActivation` 和 `PlanEvent`。`ProposedAction` 内嵌在 `PlanEvent`。EXE 独占 `ExecutionAction` 及场所写入；Demo 与 Live 使用同一计划和执行链，但身份和状态完全分离。

# 1. 对象与身份【TRADEPLAN-AUTO-OBJ-001】

## 1.1 `TradePlanVersion`

计划配置在提交固定前只是页面输入，不形成权威记录。固定后形成不可变 `TradePlanVersion`，至少保存：计划身份和内容摘要；策略身份、参数及代码版本引用；环境、账户、工具、方向和持仓模式要求；入场、保护、止盈止损、退出和失败规则；目标暴露、资金限制、动作范围和期限；研究依据及已知限制。

任何会改变动作、资金后果、条件或失败行为的修改都形成新版本。固定版本不能直接产生场所写入，也不表示用户已经激活计划。

## 1.2 `PlanActivation`

`PlanActivation` 至少保存：

- 稳定 `activation_id`、幂等请求身份、固定计划和代码版本引用；
- 环境、账户、工具、方向、目标暴露和框架策略身份；
- 最大保证金、最大名义仓位、最大允许损失、单动作上限、动作范围和期限；
- 用户明确激活的时间与范围；这组字段共同表达本激活的机器执行授权；
- 生命周期、是否已有入场成交、停止新增风险时间与原因、最大损失触发结果和退出原因；
- 每条规则最后处理的来源身份、输入摘要、事实截止点和当前判断；
- 待定动作引用、当前事实截止点以及持仓、订单、保护和核对结果引用；
- 用户接管时的切换时间、当时已知持仓、订单和未决动作范围；
- 最近事实截止点、责任是否已经闭合、结果引用和状态版本。

生命周期只使用 `RUNNING | EXITING | USER_TAKEOVER | COMPLETED`。事实、条件或动作结果未知时保留在其拥有记录中，并不可逆地停止该激活新增风险，不增加第五个生命周期状态。停止新增风险不阻止保护、撤单、减仓或退出。`USER_TAKEOVER` 停止新的 Halpha 场所写但不表示已经平仓；`COMPLETED` 只在场所责任全部得到事实证明后成立。

## 1.3 `PlanEvent` 与 `ProposedAction`

条件判断只使用 `TRUE | FALSE | UNKNOWN`：成立为 `TRUE`；不成立、不适用或错过窗口为 `FALSE` 并保存理由；输入缺失、冲突、过期或规则无法计算为 `UNKNOWN`。不存在独立条件判断记录。

`PlanEvent` 是追加式计划决定记录，保存激活、规则、稳定来源身份、事实截止点、输入摘要、判断、理由、可选 `ProposedAction`、CAP 检查结果和创建时间。相同来源身份与摘要返回原事件；相同身份出现不同摘要时记录冲突和 `UNKNOWN` 判断，并停止新增风险。

`ProposedAction` 是内嵌的不可变值，至少包含环境、激活、工具、动作种类、方向、数量或全平语义、订单类型、价格或触发价、有效期、只减仓约束和来源责任。它不是场所命令；只有 CAP 接受并由 EXE 先建立唯一 `ExecutionAction` 后才可能写入场所。

# 2. 策略组件使用契约【TRADEPLAN-AUTO-CMP-001】

Executor 中的 NautilusTrader `Controller` 为每个 `RUNNING` 或 `EXITING` 激活装载一个 `HalphaStrategyAdapter`，组合固定的纯策略逻辑。组件承担行情、K 线、时钟和回调；Halpha 只注入计划参数、激活快照和事实截止点。策略输出必须先进入 TRADEPLAN，策略和回调都不能直接调用场所写能力。

组件内存、timer 或缓存不能证明条件已处理、动作已授权或激活已结束。Halpha 不复制组件事件循环、指标或订单状态机，也不维护平行策略实现。

## 2.1 重启处理

Executor 重启后，先取得单实例控制并只读核对数据库与场所，再处理任何新回调或写入。代码版本、激活内容和事实能够一致重建时继续原激活；无法证明一致时记录具体 `UNKNOWN` 事实并停止该激活新增风险，保留查询、保护、退出和用户接管入口。

重启不需要“恢复激活”命令或恢复状态机。系统不得因进程启动成功自动忽略未知，也不得建立第二激活或第二写路径来接续未决动作。

# 3. 生命周期与不变量【TRADEPLAN-AUTO-LIF-001】

## 3.1 固定与激活

创建激活的一个本地事务完成：校验计划与代码版本；读取当前环境、账户和场所事实；由 CAP 校验激活内资金限制、动作范围和期限，并拒绝同一真实账户存在第二个未闭合激活；创建一个包含全部资金与授权字段的 `PlanActivation`；记录激活事件。任一步失败都不创建激活。激活命令不调用场所，也没有后续补授权步骤。

同一环境、账户和工具在场所不能区分多个激活责任时，只允许一个未闭合激活。激活前存在未归属持仓、开放订单或写入结果未知时拒绝创建；运行中发现此类冲突时停止新增风险并等待退出或用户接管。

## 3.2 持续判断

每个回调只使用明确截止点内的事实。缺口、迟到修正或冲突不得合成事实，也不得追溯补开仓。首次成交前的撤单和重挂必须分别形成新的 `PlanEvent → CAP → ExecutionAction`；旧单结果未知时不建立新单。首次成交后不会重置交易周期，完全平仓后不会再次入场。

期限到达时停止新增风险，并对仍可能增险的开放订单按原身份撤销或查询。提交结果未知的动作只查询原身份，不重试。期限与成交竞争产生的晚到成交进入保护和退出处理，不重新授权入场。

## 3.3 最大损失

CAP 使用本激活可归属的已实现盈亏、未实现盈亏、资金费和手续费计算损失。达到上限时，同一事务在 `PlanActivation` 锁存最大损失和停止新增风险，并进入 `EXITING`；其他激活的盈亏不参与。

## 3.4 结束

`COMPLETED` 只在 EXE 与 DAT 按同一事实截止点证明以下事实后成立：场所持仓为零；本激活普通与条件订单均终结；所有 `ExecutionAction` 已确定结束且没有提交结果未知；保护、费用和晚到成交已经处理；不存在会混淆归属的外部活动。TRADEPLAN 不复制这些事实，只保存结果引用。

用户接管后，官方入口中与冻结范围匹配且只减少或关闭风险的活动可帮助证明责任结束，但不能冒充 Halpha 动作或确定策略收益。无法证明时保持具体事实或动作结果为 `UNKNOWN`、维持原生命周期并停止新增风险，同一范围不创建新激活。

# 4. 三类用户控制【TRADEPLAN-AUTO-CTL-001】

| 控制 | 立即效果 | 后续责任 |
|---|---|---|
| 停止新增风险 | 在 `PlanActivation` 内不可逆记录停止；不再形成或提交开仓、加仓和入场重挂 | 按原身份撤销或查询开放增险订单；保护、减仓和既定退出继续 |
| 退出策略 | 同时停止新增风险并进入 `EXITING` | 撤销增险订单，按当前事实形成保护、减仓或平仓动作，直到责任闭合 |
| 用户接管 | 先记录 `USER_TAKEOVER` 和切换范围，再停止该激活全部新的 Halpha 写入 | 不自动撤单或平仓；Halpha 只读核对，用户使用交易所官方入口处理 |

三项控制各自使用稳定请求身份。重复提交返回同一领域结果；相同身份但内容不同返回冲突。响应只区分 `APPLIED | REJECTED | UNKNOWN` 并返回当前激活版本，不建立通用命令回执或任务平台。网络超时后按原请求身份查询，不能生成第二控制动作。

停止新增风险后不提供恢复命令。若用户以后希望继续交易，必须先证明旧激活责任闭合，再创建新激活并重新检查当前事实、计划、资金限制和授权；用户接管本身不等于责任闭合。用户接管后的旧激活永不恢复机器写入。

# 5. 条件责任与场所委托【TRADEPLAN-AUTO-CND-001】

计划只需说明条件由 Halpha 持续判断、由场所原生订单持续判断，或不需要持续条件。同一持续条件只有一个责任方。场所条件订单提交或查询未知时，只查询原身份，不并行改由 Halpha 建立内容等价的动作。

动作使用市场单、限价单或场所条件单由计划和当前支持范围决定。提交了场所条件单不表示其来源条件由场所负责；来源条件和订单触发分别表达。

---

# 6. 并发、错误与公开能力【TRADEPLAN-AUTO-API-001】

公开能力只需：保存固定计划；以幂等请求创建激活；读取激活；消费策略提议；形成 `PlanEvent`/`ProposedAction`；提交停止新增风险、退出和用户接管；记录 EXE 关闭结果；查询时间线。TRADEPLAN 不写入或推进 `ExecutionAction`。

错误至少区分：计划版本冲突、策略或代码版本不匹配、参数无效、事实未知或冲突、归属歧义、CAP 拒绝、已停止新增风险、动作结果未知、用户接管和责任未闭合。确定拒绝与未知不能合并。

# 7. 最小验证契约【TRADEPLAN-AUTO-TST-001】

至少证明：

1. 一次幂等激活请求只建立一个包含完整资金限制与授权的 `PlanActivation`，失败不留下部分对象；
2. 策略输出只能经 `PlanEvent → CAP → ExecutionAction` 进入场所写入，且先持久化后外写；
3. 条件只产生 `TRUE | FALSE | UNKNOWN`，重复来源不重复动作，冲突来源停止新增风险；
4. 首次成交后完全平仓即结束，不能自动重新入场；
5. 停止新增风险不阻止保护、撤单、减仓和退出，且同一激活不能恢复新增风险；
6. 退出和用户接管产生不同后果，接管不自动撤单或平仓；
7. 重启先核对再继续，无法一致重建时不会写入或建立恢复状态机；
8. 未知写结果只查询原 `ExecutionAction`，不盲重试；
9. 关闭事实不足时不结束激活，也不允许同一真实账户创建新激活；
10. Demo 与 Live 共用代码和记录类型，但环境身份、账户、激活、动作及结果不可迁移。
