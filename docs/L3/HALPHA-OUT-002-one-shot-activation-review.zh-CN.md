# 0. 设计结论【OUT-RT-SUM-001】

OUT 围绕一次 `PlanActivation` 建立一条简短 `Review` 版本链，并把确实会改变后续决定的问题形成 `ImprovementHandoff`。这两个记录族是 OUT 的全部持久对象；复盘不建立独立 worker、游标、任务队列、分析仓库或第三个对象。

复盘只连接已有权威记录：ALP 的固定策略依据，TRADEPLAN 的 `PlanActivation` 与 `PlanEvent`，CAP 的授权和独占额度引用，EXE 的 `PendingAction`，DAT 的 `VenueFact`，以及适用的 UX 用户指令与回执。它不重做条件判断、资金检查、订单状态机、保护或核对，也不能通过评价结果修改这些责任。

NautilusTrader `BacktestEngine` 的 `ReportProvider` 和内置统计只属于 ALP 离线证据输入。回测报告、组件组合投影或模拟成交都不能替代真实激活的 `PlanEvent`、`PendingAction` 或 `VenueFact`，也不进入线上结果权威链。

# 1. 复盘身份与范围【OUT-RT-SCP-001】

## 1.1 唯一边界【OUT-RT-SCP-001-REQ】

一个 `activation_id` 最多对应一条 `Review` 版本链。同一不可变交易计划版本可以在用户重新授权并重新分配互斥额度后产生多个彼此独立的激活；每个激活分别形成自己的 Review 版本链。因此复盘同时保存计划版本、激活身份、固定策略依据摘要、环境、账户、场所、工具、激活起止时间和事实截止点。重复结束、重启或重复读取不得建立并行复盘。

复盘可以在以下结果后建立：未触发、本次不行动、过期、拒绝且未越过外部变化分界；动作全部或部分完成；结果仍未知；用户已接管；或激活已经明确结束。结果未知或接管时允许完成评价，但开放订单、持仓、保护、核对和额度责任必须继续可见。

OUT 不推断激活结束。开始、终止、接管及是否仍可形成动作均以 `PlanActivation` 与 `PlanEvent` 为准；场所是否实际成交、持仓、收费或平仓以 `VenueFact` 为准；写入意图与未知结果以 `PendingAction` 为准。

## 1.2 非成交与事实边界【OUT-RT-SCP-002-REQ】

非成交结果与成交结果具有同等复盘资格。未触发、拒绝、等待后结束、事实未知、权限或额度不足不得补造成交、费用、盈亏或机会成本。没有可靠 `VenueFact` 时，账户结果保持未知。

后到或修正的 `VenueFact`、`PlanEvent` 或 `PendingAction` 结果改变评价时形成新的 `Review` 版本。新版本保存变化来源和受影响结论，不把后来事实改写成激活当时已经可知。

# 2. `Review` 契约【OUT-RT-REV-001】

## 2.1 最小内容【OUT-RT-REV-001-REQ】

| 字段组 | 最小内容 |
|---|---|
| 身份与版本 | `review_id`、`review_version`、`activation_id`、内容摘要、前版引用、状态 |
| 激活边界 | 计划版本、固定策略依据摘要、环境、账户、场所、工具、开始与结束、事实截止点 |
| 输入引用 | 被采用的 `PlanActivation`、有序 `PlanEvent`、CAP 授权与独占额度引用、`PendingAction`、`VenueFact`、适用用户指令与回执 |
| 结果 | 主要结果、账户结果、部分与剩余量、开放责任和事实未知 |
| 评价 | 计划、资金与权限、执行与事实、交互、系统与维护，以及不适用或无法判断理由 |
| 改进 | 对本版本 `ImprovementHandoff` 的引用，或无需行动的理由 |

每个输入只保存稳定身份、版本或摘要和事实截止点，不复制相邻记录完整状态。`review_id` 由 `activation_id` 稳定派生；本版本的输入摘要由上述有序引用形成。相同激活和相同输入返回同一版本，相同输入身份但内容冲突时保持旧版并报告冲突。

`Review` 只使用 `DRAFT`、`COMPLETE`、`SUPERSEDED` 三个状态。输入或评价尚未明确时为 `DRAFT`；评价完成且开放责任已明确时为 `COMPLETE`；新事实形成下一版本后旧版为 `SUPERSEDED`。`COMPLETE` 不表示相邻责任已经关闭。

## 2.2 主要结果【OUT-RT-REV-002-REQ】

主要结果只取：

| 结果 | 含义 |
|---|---|
| `NO_ACTION` | 可证明该激活未越过外部变化分界 |
| `COMPLETED` | 计划内动作及适用保护、退出和核对责任均已闭合 |
| `PARTIAL` | 只完成部分动作或数量，剩余责任明确 |
| `RESULT_UNKNOWN` | 可能发生外部变化，但事实不足以确认最终结果 |
| `HANDED_OVER` | 未闭合责任已明确交给用户通过官方入口接管 |

订单显示终态不自动得到 `COMPLETED`。持仓、保护、费用、退出或核对任一未知时，必须使用 `PARTIAL`、`RESULT_UNKNOWN` 或继续显示开放责任。

## 2.3 最小评价【OUT-RT-REV-003-REQ】

每项评价只取 `AS_EXPECTED`、`ISSUE_FOUND`、`UNKNOWN` 或 `NOT_APPLICABLE`，并附最少证据：

| 评价项 | 必须回答 |
|---|---|
| 账户结果 | `VenueFact` 能证明实际发生了什么；不能可靠计算时保持未知 |
| 计划 | 策略依据、参数、条件、分支与一次性生命周期是否被正确执行 |
| 资金与权限 | 机器授权、独占额度和逐动作检查是否按 CAP 引用生效 |
| 执行与事实 | 动作是否唯一，部分、取消、保护、退出、核对和修正是否保持同一责任链 |
| 交互 | 用户是否看清环境、动作、回执、接管结果和开放责任 |
| 系统与维护 | 只有实际系统或维护问题影响结果时才评价 |

账户归因只使用能够由引用支持的市场或外部、策略或计划、用户决定、执行/数据/交互/系统、混合或证据不足类别；不估算伪精确比例。反事实只有在计划事前固定可行替代且能用当时事实和真实成本重建时才允许。

## 2.4 评价完成【OUT-RT-END-001-REQ】

`Review` 只有在身份和边界明确、适用输入已经取得或缺失影响已注明、主要结果和六项评价都有值、开放责任明确、实际问题已交接或说明无需行动时才能 `COMPLETE`。执行结果未知或用户接管可以完成评价，但必须保留 `RESULT_UNKNOWN` 或 `HANDED_OVER` 及责任方。

评价完成不得释放额度、恢复权限、结束计划、重发动作、关闭事实未知或改变用户接管。任何这些变化都只能由其语义所有者形成新记录。

# 3. `ImprovementHandoff` 契约【OUT-RT-IMP-001】

`ImprovementHandoff` 只保存稳定身份、不可变版本、来源 `Review` 版本、一个目标语义所有者、可观察问题、证据、影响范围和预期改变。身份由来源复盘版本、目标所有者和问题摘要稳定派生；相同身份内容不同必须冲突，旧版本不得原地改写。

OUT 只交付问题，不决定目标领域怎样修改。涉及多个领域时分别建立 handoff，并保留共同来源。目标消费者不存在、不会改变决定或失败行为、或者不能降低已观察成本的问题不建立记录。

Handoff 由项目所有者在现有工作台查看和处置；本契约不建立委员会、审批人、投票、自动派单、优先级评分或长期 backlog 生命周期。盈利、损失或出现次数不能自动扩大策略资格、额度或权限。

---

# 4. 接口、并发与恢复【OUT-RT-API-001】

## 4.1 命令与查询【OUT-RT-API-001-REQ】

| 操作 | 成功结果 | 保守结果 |
|---|---|---|
| `update_activation_review(activation_id, expected_version)` | 从稳定引用幂等建立或更新 `DRAFT`，输入齐备时可形成新版本 | 引用缺失或冲突时保留 `DRAFT` 和明确缺口 |
| `complete_activation_review(review_id, expected_version, evaluation)` | 形成 `COMPLETE` 版本并幂等建立适用 handoff | 完成条件不足时拒绝并返回缺口 |
| `read_activation_review(activation_id)` | 返回最新版本、历史版本、证据和开放责任 | 缺失返回未建立，不隐藏写入 |
| `list_improvement_handoffs(target_owner)` | 返回稳定 handoff 及来源 | 缺失返回空集，不自动创建 |

所有写命令必须带稳定调用身份和预期版本。重复身份与相同内容返回原结果；相同身份与不同内容返回冲突。`GET`、`HEAD`、页面加载和通知只能读取，不能隐式建立或更新复盘。

## 4.2 运行与恢复【OUT-RT-RCV-001-REQ】

既有 Halpha 协调工作在 `PlanActivation` 进入可复盘结果后调用 OUT；OUT 不拥有独立 worker、领取游标、轮询器或恢复状态机。调用失败后，协调工作以相同激活身份、输入摘要和预期版本重试；不得重放策略、资金检查或场所动作。

重启后只从 `PlanActivation`、`PlanEvent`、`PendingAction`、`VenueFact` 和必要相邻引用重建 `Review`。缓存、页面、通知、组件组合投影和离线报告不能作为恢复来源。必要引用仍不可读时保持 `DRAFT`，或在缺失影响已经明确时以 `UNKNOWN` 完成评价。

---

# 5. 失败结果【OUT-RT-FAL-001】

| 情形 | 必须结果 |
|---|---|
| 激活结束或复盘命令重复 | 返回同一复盘身份或版本，不建立并行链 |
| 激活、计划或输入摘要冲突 | 拒绝合并，保留旧版本并显示冲突 |
| 授权或独占额度引用缺失 | 资金与权限评价为 `UNKNOWN`，不推定已授权或已释放 |
| `PendingAction` 结果未决 | 主要结果为 `RESULT_UNKNOWN`，不重发动作 |
| `VenueFact` 缺失或冲突 | 账户结果保持未知，不以计划值或组件投影补齐 |
| 迟到或修正事实改变评价 | 形成新 `Review` 版本并保留旧版 |
| 部分成交或剩余责任存在 | 使用 `PARTIAL` 并逐项显示开放责任 |
| 用户接管 | 使用 `HANDED_OVER`，保留接管范围及后来事实 |
| OUT 写入失败 | 不改变策略、计划、额度、权限、动作、保护或核对 |
| 离线报告缺失或与线上事实不同 | 只影响 ALP 证据判断；不改变本次真实复盘 |

# 6. 明确不建设【OUT-RT-NON-001-REQ】

本契约不建设：

- OUT 之外的执行、数据、资金、计划或交互状态副本；
- 独立复盘 worker、扫描队列、恢复游标、聚合对象或第三个记录族；
- 跨激活统计、策略排行榜、企业 BI、自动因果归因或通用分析仓库；
- 自研历史回放、撮合、绩效报告或线上/离线结果合并；
- 自动修改策略、计划、权限、额度、事实、动作、保护、退出、核对或接管；
- 多人审批、委员会治理、自动优先级或为了填满评价而采集无消费者数据。

---

# 7. 最小验收契约【OUT-RT-TST-001】

实现至少验证：

1. 一个激活只形成一条 `Review` 版本链，重复结束、重启和并发命令保持幂等；
2. 未触发、拒绝、过期、等待后结束分别形成 `NO_ACTION`，且不补造成交、费用或盈亏；
3. 正常完成、部分成交、结果未知和用户接管分别得到正确主要结果并保留开放责任；
4. 复盘只通过稳定引用追溯 `PlanActivation`、`PlanEvent`、`PendingAction`、`VenueFact` 及适用的 ALP、CAP、UX 输入，不复制其状态；
5. 场所事实缺失、权限或额度引用缺失、输入摘要冲突都保持 `UNKNOWN` 或 `DRAFT`，不会关闭相邻责任；
6. 迟到成交、费用或事实修正形成新版本，旧版仍说明当时依据；
7. `COMPLETE` 可以保留未知或接管结果，但不能释放额度、恢复权限、重发动作或结束计划；
8. 每个实际问题只形成目标所有者明确且幂等的 `ImprovementHandoff`，没有消费者的问题不建立；
9. 离线回测报告和组件投影不能改变真实复盘结果，也不能替代 `VenueFact`；
10. OUT 写入失败与恢复只重试复盘命令，不触发任何交易动作；读取页面和通知严格只读；
11. 运行中只存在 `Review` 与 `ImprovementHandoff` 两个 OUT 记录族，没有独立 worker、队列、游标、审批流程或分析平台。

---

# 8. 复杂度与删除边界【OUT-RT-MIG-001】

本契约把 OUT 收缩为两个记录族和既有协调工作中的一次调用，不新增进程、数据库、外部写路径、后台 worker 或运行模式。复盘直接读取四个核心权威记录及必要稳定引用，不重建相邻生命周期。若不再需要结构化复盘，可在保留历史可读性后删除 OUT 写入与 handoff；交易计划、资金、执行和事实闭环不依赖 OUT 才能正确运行。
