# 0. 设计结论【CAP-AUTO-SUM-001】

用户激活计划时，一次命令同时确认环境、账户、工具、计划版本、最大保证金、最大名义仓位、最大允许损失、单动作上限、动作范围和期限。`PlanActivation` 直接保存这些内容，并以该命令本身表达只属于本次激活的机器执行授权；不另建账户上限、授权、额度、分配或停止记录，也没有外置配置门或第二授权步骤。

CAP 不拥有持久记录。它提供针对 `PlanActivation` 与当前场所事实的确定校验和计算函数；TRADEPLAN 或 EXE 把消费结果写入各自记录，不建立独立资金决定记录。为避免账户级分配，任一环境和真实账户一次只允许一个未闭合激活。

资金仍留在交易所账户中。Halpha 的资金限制不是交易所冻结、子账户隔离或损失保证。Demo 与 Live 使用同一记录形状和计算函数，但身份、账户、动作和结果不得跨环境复制或提升。

---

# 1. `PlanActivation` 内嵌资金与授权【CAP-AUTO-OBJ-001】

一次激活直接固定：

- 环境、账户、工具、方向、计划版本和适用计价资产；
- 最大保证金、最大名义仓位、最大允许损失和单动作最大名义金额；
- 允许的入场、撤单、保护、止盈、减仓和平仓动作及订单限制；
- 授权开始、截止时间、用户激活命令引用和代码版本引用；
- 最近资金检查与损失输入截止点、最大损失是否已触发、是否已停止新增风险及原因。

这些字段属于同一个激活，不是账户上限、授权、额度或分配对象。激活内容、环境或期限不得原地扩大；需要继续或改变范围时，新建激活。最大损失触发和停止新增风险在该激活内不可逆。场所余额只用于判断动作当前是否可行，不能自动扩大激活内限制。

---

# 2. 激活时检查【CAP-AUTO-ACT-001】

创建激活时，在同一环境和账户下锁定仍未结束的 `PlanActivation`。存在任何未闭合激活、未归属持仓或订单、写入结果未知或归属冲突时，不创建新激活。不存在这些责任时，校验所有限制为正、单位与工具一致、动作范围受计划约束、期限有效，并确认当前场所余额、保证金与规则能够承载计划预览。

运行中不得提高激活限制或扩大动作范围。需要调整时，必须先证明旧激活责任闭合，再由用户新建激活；用户接管本身不等于闭合。由于同一真实账户没有并发激活，本契约不需要额度求和、分配、预留或跨激活损益抵消。

---

# 3. 动作分类【CAP-AUTO-CLS-001】

CAP 只区分三种检查结果：

| 结果 | 含义 | 处理 |
|---|---|---|
| `INCREASE` | 动作可能增加最坏风险敞口 | 必须通过授权、期限、停止状态、激活内资金限制和损失检查 |
| `NON_INCREASE` | 可证明只是查询、撤销既有增险订单、保护、减仓或平仓且不会反向开仓 | 不增加受限敞口；仍须匹配激活、环境、场所事实和允许动作 |
| `UNKNOWN` | 无法证明风险不会增加 | 拒绝写入并停止该激活新增风险，等待原责任核对或用户接管 |

订单名称不决定分类。限价单可能增险，条件单可能减险；分类以当前持仓、开放订单、在途动作和最坏可执行数量为准。用户停止新增风险后，`NON_INCREASE` 的保护、撤单、减仓和平仓不得被误阻；用户接管后则停止该激活全部新的 Halpha 场所写，只保留查询和核对。

---

# 4. 同一轻量检查【CAP-AUTO-CHK-001】

同一个无副作用检查函数在两个时点运行：TRADEPLAN 准备 `ProposedAction` 时，以及 EXE 即将执行已经持久化的 `ExecutionAction` 前。后一次必须重读最新激活、场所事实和开放责任；这只是防止事实变化，不形成审批流程或新记录。

增加风险动作依次检查：身份和环境一致；激活仍有效且未停止新增风险；动作在激活范围和期限内；事实足够新且归属明确；单动作金额未越界；计入持仓、开放订单和在途动作后的保证金与名义仓位未越过激活限制；本激活损失未达到上限；场所余额和规则仍允许。任何输入缺失、冲突或过期都返回 `UNKNOWN`，不能按零或最近值放行。

后一次检查失败时，原 `ExecutionAction` 进入确定拒绝或未知核对，不新建动作、不改变订单类型，也不通过组件自动重试。金额、数量和阈值使用 `Decimal`；组件风控可以更保守地拒绝，但不能扩大 CAP 允许范围。

---

# 5. 损失归属与停止【CAP-AUTO-LOSS-001】

本激活净结果只使用可追溯到该激活的已实现盈亏、未实现盈亏、实际资金费和实际手续费：

```text
activation_net = realized_pnl + unrealized_pnl + funding - commission
activation_loss = max(-activation_net, 0)
```

达到 `max_allowed_loss` 时，CAP 返回触发结果；TRADEPLAN 在同一事务把 `PlanActivation.max_loss_reached` 和停止新增风险原因写为不可逆事实，并进入退出路径。迟到费用可以增加最终损失，但盈利不能解除已经触发的停止。其他激活或外部动作的盈亏不得计入；归属未知时停止新增风险并核对。

最大保证金和最大名义仓位只限制 Halpha 主动增加风险。行情、费用或场所规则使现有持仓自然越界时，只阻止后续增险，不自动平仓，也不阻止保护、减仓和退出。最大允许损失是触发退出的控制值，不保证最终损失不会因滑点、跳空、强平或故障超过它。

---

# 6. 故障与环境边界【CAP-AUTO-CMP-001】

数据库不可用、激活字段不一致、事实过期或无法归属时，不以内存值继续写入。重启后从激活和场所事实重算；一致时可继续原激活，不一致时停止新增风险并显示 `UNKNOWN`，不建设恢复状态机。

Demo 记录只能写入其模拟端点，Live 记录只能写入其真实端点。两者调用同一检查代码，但环境、账户、凭据引用、激活和 `ExecutionAction` 身份完全分离。Demo 成功不能成为 Live 授权或资本事实。

---

# 7. 公开能力、错误与验证【CAP-AUTO-API-001】

公开能力只需：校验新激活的资金与授权字段并拒绝同账户第二个未闭合激活；分类并检查 `ProposedAction`；场所写入前复核；更新本激活损失；判断旧激活责任是否闭合。所有能力使用稳定身份、预期版本和事实截止点。

稳定结果至少区分：同账户激活冲突、授权范围不符、授权过期、已停止新增风险、单动作、保证金或名义仓位超出、最大损失触发、归属未知、估值未知和版本冲突。

至少验证：

1. 一次激活命令只创建一个包含全部资金限制和授权的 `PlanActivation`，重复提交返回同一结果；
2. 同一真实账户的第二个未闭合激活被拒绝，失败不会留下部分对象；
3. 停止新增风险不会阻止可证明的保护、撤单、减仓和平仓；
4. 本激活损失口径正确，触发后重启仍不可恢复新增风险；
5. 提议时与写入前使用同一检查函数，未知不重试或放行；
6. Demo 与 Live 共用代码但不能交叉引用、复制或提升身份；
7. 用户接管后无新 Halpha 写入，未闭合责任仍阻止同一真实账户创建新激活。
