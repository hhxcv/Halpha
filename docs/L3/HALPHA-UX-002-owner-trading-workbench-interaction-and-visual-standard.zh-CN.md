# 0. 交互结论【UX-AUTO-SUM-001】

工作台围绕一个闭环组织：用户选择代码策略、配置参数和互斥额度，复核后激活一次交易循环；Halpha 在用户离开后持续判断、下单和管理退出；用户随时查看真实状态，并可停止新增风险、退出策略或接管。激活是一次高后果机器授权，激活后的计划内动作不再逐单弹出确认。

界面必须持续区分计划、用户指令、Halpha 处理回执、场所事实、动作未决和核对结果。已接收、已提交、场所已接受、部分成交、保护已确认、已经平仓和责任已结束分别显示；任何前态不得冒充后态。

前端直接采用 React、Material UI（MUI）与 Emotion、React JSON Schema Form 的 MUI 主题与 AJV8、React Router、TanStack Query、openapi-typescript 和 openapi-fetch。组件承担布局、表单、路由、非持久查询和 API 客户端，Halpha 不重复实现这些通用能力。服务器业务校验和权威记录始终优先于浏览器校验与缓存。

同一 `halpha-app` FastAPI 进程还提供服务端渲染的 `/operations` 有限操作入口，供 React 静态资源或 SPA 路由不可用时在目标主机本地完成恢复判断与高价值控制。它不是第二工作台：必须复用同一所有者认证、会话、CSRF、Host/Origin、限频、`Command`/`Receipt`、领域处理器和 PostgreSQL 投影；不得增加第二 API、第二业务状态、第二认证口令、第三进程或远程应急通道。

本文件只定义单所有者 Web 工作台及其同进程有限操作入口。上位设计保留其他交互形态接入同一业务流的兼容边界，但不构成其他形态已获支持、阶段承诺或本文件中的预先设计；未来若形成真实建设范围，必须由适用 L3 另行定义该形态的安全与技术契约，并继续复用同一业务对象、命令、回执、领域处理器和权威状态。

UX 持久记录族只有 `Task`、`Command`、`Receipt` 和 `Notification`。页面投影、表单草稿、路由状态、会话、查询缓存和时间线不建立第五个权威记录族。

# 1. 组件实现方案【UX-AUTO-CMP-001】

## 1.1 采用组件及使用边界【UX-AUTO-CMP-001-REQ】

| 交互能力 | 采用组件 | 使用方式 | 权威与失败边界 |
|---|---|---|---|
| 页面、普通表格、表单控件、反馈与主题 | React、MUI、Emotion | 使用 MUI 现成组件和主题能力组合任务页面；只补充交易领域文案与少量领域控件 | 组件状态只用于呈现；刷新后从服务器重建 |
| 有限操作入口 | FastAPI/Starlette 服务端 HTML | 同一 App 内渲染 `/operations`；复用正常领域查询、命令和回执，不依赖 React/Node/Vite 静态制品 | 只允许本地主机、同源、同认证与 CSRF；不得形成第二控制面或宽化权限 |
| 策略参数表单 | RJSF MUI、AJV8 | 按 ALP JSON Schema 生成控件，AJV8 提供即时类型和局部约束反馈 | 服务器 Pydantic 与拥有领域校验是唯一接受结论；客户端成功不能启用计划 |
| 路由与中断返回 | React Router | 稳定对象路由、深链和返回位置 | 路由参数不授予权限；打开页面必须重新查询当前版本 |
| 查询、轮询和过期 | TanStack Query | 只管理内存查询、轮询、失效、加载和错误 | 不持久化、不离线提交、不乐观更新业务状态；缓存永不成为事实 |
| API 类型与调用 | openapi-typescript、openapi-fetch | 从 FastAPI OpenAPI 生成类型和客户端；命令提交原样返回 `Command`/`Receipt` | 不手写平行业务 DTO 或状态枚举；schema 不兼容时页面阻断命令 |
| 会话、口令与 CSRF | Starlette 会话与可信 Host 中间件、pwdlib Argon2、asgi-csrf、limits | 签名短时会话、成熟口令哈希、签名双提交 CSRF 和非权威登录限频 | 接受签名双提交而非自研 session-bound CSRF；失败拒绝命令但不改变业务状态 |
| 邮件通知 | ENG 的标准库 SMTP 适配 | `NotificationDispatcher` 发送最小任务提示和只读深链 | 邮件不含可执行交易动作；投递结果不改变任务来源或交易结果 |
| 浏览器与可访问性验证 | Playwright、axe、Vitest | 验证关键任务、真实渲染、键盘、焦点、对比度与前端逻辑 | 组件测试不替代服务器领域和真实外部验证 |

Halpha 只保留以下最小前端自研：页面组合、交易领域标签、RJSF 的单位/Decimal/动作范围控件、服务器错误定位、四个交互记录族的投影，以及同源检查所需提示。不得建立自研组件库、表单引擎、请求缓存、路由器、认证框架或 API SDK。

工作台不采用专业 K 线/图表库、高级数据表、行情终端框架、可视化策略编辑器或脚本编辑器。价格、订单、事实和动作关系用服务端投影的普通表格、摘要和时间线表达；需要专业分析时由外部工具承担，不在 Halpha 建第二终端。

## 1.2 浏览器状态限制【UX-AUTO-CMP-002-REQ】

TanStack Query 只保存当前浏览器进程内的查询副本，每项事实查询必须显示服务器截止点和视图取得时间。轮询停止、窗口离线、服务器错误或截止点不再推进时，页面立即显示“视图可能过期”，并禁止激活、恢复增险和普通资本配置等可能增加风险的命令，直至重新取得当前对象和命令预览。停止新增风险、退出策略和用户接管仍必须保持可提交；服务器按原幂等身份与当前权威状态保守处理，页面不得因陈旧缓存宣称它们已经生效。

命令不经过离线队列或乐观更新。openapi-fetch 超时后页面按原 `command_id` 或幂等键查询 `Receipt`；不得创建第二命令来猜测第一次是否成功。浏览器关闭后只允许从服务器记录恢复，不能从 localStorage、Service Worker 或 TanStack Query 持久缓存恢复授权、额度、控制或事实。

# 2. 四个交互记录族【UX-AUTO-REC-001】

## 2.1 最小契约【UX-AUTO-REC-001-REQ】

四类记录都包含 `schema_version`、稳定身份、适用环境/用户范围和内容摘要。`Task`、`Receipt` 与 `Notification` 的可变状态使用 `state_version` 和 expected-version 比较交换推进；版本不符返回冲突，不以后写覆盖。schema 未知或摘要不一致时拒绝消费，不从浏览器投影猜测迁移。

| 记录族 | 稳定身份与内容 | 状态 | 高价值消费者 |
|---|---|---|---|
| `Task` | `task_id`、由任务种类/来源种类/稳定来源引用/责任代次/范围组成的稳定责任键、优先级、到期/无响应结果、来源版本与摘要、`state_version` | `OPEN | ACKNOWLEDGED | RESOLVED` | 用户待办、通知和故障入口 |
| `Command` | `command_id`、幂等键、目标身份与预期版本、不可变意图、范围、参数、提交时间与内容摘要 | 不可变 | 拥有领域命令处理器与审计查询 |
| `Receipt` | 从一个 Command 稳定派生的 `receipt_id`、处理所有者、理由码、结果或未决责任引用、`state_version` | `RECEIVED | PROCESSING | EFFECTIVE | REJECTED | UNKNOWN` | 命令提交页面、恢复查询与 Task |
| `Notification` | `notification_id`、稳定来源语义身份、Task 引用、来源业务时间、收件路由引用、内容摘要、尝试与领取版本 | `PENDING | DELIVERED | ABANDONED` | App notification dispatcher 与工作台 |

相同稳定身份或幂等键但内容摘要不同必须冲突。Command 第一次成功插入时冻结提交时间；重复 HTTP、刷新和恢复返回原记录。Receipt 对 Command 唯一，正常推进为 `RECEIVED → PROCESSING → EFFECTIVE | REJECTED | UNKNOWN`；`UNKNOWN` 只能按原命令和业务身份查询后推进为确定终态，确定终态不能重开或另建回执。

Task 的 `ACKNOWLEDGED` 只表示用户已看见，不关闭来源责任；只有来源拥有领域给出闭合引用才可 `RESOLVED`。同一责任在未闭合期间只保留一个 Task；真正出现新责任时使用新的责任代次，不以普通内容刷新复活旧 Task。

Notification 同时承担业务通知和 outbox，不另建通知事件或邮件任务记录。来源事务在需要提示时幂等插入同一 Notification；来源已有业务提交时间时直接引用，没有时在该 Notification 首次插入时冻结生成时间，重试不得改写。重复来源保持同一 Notification；只允许 `PENDING → DELIVERED | ABANDONED`，投递失败在原身份上有界重试，放弃后幂等确保一个可见 Task。领取、退避和尝试计数不改变来源语义身份。邮件打开、投递或回复均不改变业务状态。

## 2.2 共同命令闭环【UX-AUTO-REC-002-REQ】

所有会改变状态的页面都遵循同一闭环：

```text
重新读取对象、事实截止点、授权与预期版本
→ 形成当前后果预览
→ 在一个事务中创建不可变 Command 与 RECEIVED Receipt
→ 拥有领域以同一业务事务处理并推进 Receipt
→ EFFECTIVE / REJECTED，或 UNKNOWN 后按原身份查询
→ 重新读取业务对象并更新/闭合 Task
```

HTTP 2xx、按钮变为禁用、邮件送达或 Receipt 处于 `RECEIVED/PROCESSING` 都不表示业务生效。涉及动作时，Receipt 只说明用户指令处理；同环境 `ExecutionAction` 与相应 `VenueFact` 才说明执行和环境事实。Demo/LIVE 使用同一动作类型和状态标签，但页面必须持续展示环境、账户和授权效力，不能合并或跨环境引用。

只改变 App 本地域对象的计划创建、修改、固定和激活命令，由 App 在创建 Command/Receipt 的同一事务内完成领域处理并把 Receipt 直接提交为 `EFFECTIVE | REJECTED`；不留下无人处理的 `RECEIVED`。停止新增风险、退出、用户接管、恢复新增风险和恢复激活先由 App 同步持久化 Command 与对应领域控制状态，Receipt 为 `PROCESSING`；现有 HalphaCoordinator 观察该领域状态并推进既有取消、退出或 Strategy 生命周期，在取得拥有领域效果证据后通过 UX 公共 `record_command_result(command_id, evidence_ref, expected_version)` 边界推进原 Receipt。该调用不让 EXE 直接拥有 UX 表，也不增加 Command worker；重启后 HalphaCoordinator 从 PlanActivation 的稳定 command 引用恢复同一责任。

停止新增风险只有在全部既有增险责任终态后才为 `EFFECTIVE`；无法确认则为 `UNKNOWN`。退出命令只有在 `EXITING` 已持久化、全部既有增险责任终态且唯一退出责任已建立后才为 `EFFECTIVE`，但不表示已经平仓；任一条件无法确认时为 `UNKNOWN`。用户接管只有在状态已持久化、Strategy 适配器已停止且不会启动新调用后为 `EFFECTIVE`；在途调用仍单独显示。恢复新增风险只有当前事实和 CAP 重新检查通过且新 StopStateVersion 已提交后为 `EFFECTIVE`。

# 3. 信息架构与路由【UX-AUTO-IA-001】

## 3.1 全局框架【UX-AUTO-IA-001-REQ】

桌面工作台使用以下稳定结构：

```text
状态栏：环境｜账户｜事实截止点｜新增风险状态｜开放责任
导航：总览｜策略计划｜资金与授权｜复盘｜设置
主任务区：当前页面唯一主要任务
检查器：来源、版本、原始事实、错误和责任详情
```

环境、账户和事实截止点在所有资本页面持续可见。真实与模拟环境同时使用文字、图标和色彩，不得只靠颜色。切换环境、账户或工具必须重新取得对象和命令预览，不能静默复用待提交的环境相关输入。

## 3.2 稳定路由【UX-AUTO-IA-002-REQ】

| 路由 | 用户任务 |
|---|---|
| `/overview` | 查看活动激活、开放 Task、未知和最近通知 |
| `/plans` | 查看代码策略、计划草案、固定版本和激活历史 |
| `/plans/new` | 选择策略并配置计划 |
| `/plans/:plan_version_id/activate` | 复核一次激活与机器授权 |
| `/activations/:activation_id` | 观察一个激活并使用三类控制 |
| `/capital` | 查看账户边界、互斥额度和机器授权 |
| `/reviews/:review_id` | 查看一次激活复盘 |
| `/settings` | 查看环境、连接、邮件通知和只读运行配置状态 |
| `/operations` | React 静态资源或 SPA 不可用时，在同一 App 内查看关键运行/恢复状态并提交有限操作 |

`/operations` 必须显示环境、账户、`run_state`/暂停原因、服务器事实截止点、开放责任、持仓、订单、保护、核对状态、授权期限、恢复允许或拒绝原因，以及 CAP 的 `NEW_FUNDING`、`PROTECTION`、`RISK_REDUCTION_OR_ORDER_MANAGEMENT`、`ALL_WRITES` 四类停止。它只提供 `RESUME_ACTIVATION`、`EXIT_STRATEGY`、`USER_TAKEOVER` 的当前后果预览、提交和原 Receipt 查询；登录会话缺失时可渲染使用同一所有者口令与 CSRF 的最小登录表单，禁止新增“应急口令”。该入口不得成为邮件、移动或远程命令通道。

页面可以有折叠区和检查器，但不为每个自动动作建立人工确认路由，也不建立独立通知中心或专业交易终端。

# 4. 计划配置与激活【UX-AUTO-PLAN-001】

## 4.1 策略与参数【UX-AUTO-PLAN-001-REQ】

策略选择页只列出当前构建登记的代码策略身份、版本、逻辑摘要、事实与动作需要、适用范围和 ALP 提供的证据可用性。UI 不自行选择“最新”证据，不扩大或拼接部分重叠范围；服务器返回不具备真实使用资格时，激活按钮保持禁用并列出阻断理由。

RJSF 按 ALP schema 生成参数表单。每项显示名称、含义、单位、允许范围和当前值；Decimal 使用文本输入与显式单位，不经 JavaScript number 隐式转换。AJV8 错误就地展示，服务器交叉约束和领域错误仍是最终结论，并映射回字段或页面阻断区。

策略参数与以下计划内容分组显示，不能混成任意 JSON：

1. 环境、账户、工具和方向；
2. 入场、保护、止盈止损、时间退出和失败行为；
3. 最大保证金、最大名义仓位和最大允许损失；
4. 允许动作、有效期和邮件通知。

保存草案不激活、不占用额度、不授予机器权限。固定版本前显示全部缺失和差异；固定后只读，修改只能复制成新草案。

## 4.2 激活复核【UX-AUTO-ACT-001-REQ】

激活页在同一页面完整显示策略与实现摘要、计划版本与参数、环境/账户/工具/方向、事实截止点、入场/保护/退出、动作范围、有效期、其他活动额度和所有未知。页面明确说明：

- 最大保证金和最大名义仓位只限制 Halpha 主动增加风险；行情导致已持仓自然越界只阻止后续增险，不自动减仓或平仓；
- 最大允许损失只计算本激活归属的已实现盈亏、未实现盈亏、手续费和资金费净合并；其他策略不参与；触线后停止新增风险并承担本激活全部退出责任；
- 激活后用户离线时，Halpha 可以在列明范围内向所属环境提交动作；LIVE 还必须明确显示真实资本效力与真实写门，交易场所支持的条件由场所监控，其余条件由 Halpha 持续监控；
- 激活成功只表示计划、授权和互斥额度已建立，不表示已经下单、成交或获利。

主按钮固定为“激活并授予机器权限”。提交前需要一次完整后果预览和显式动作，但不增加机构审批或逐动作确认。双击、超时和响应丢失复用同一 Command；结果未知时只提供原身份查询，不提供“再次激活”。

# 5. 运行观察与时间线【UX-AUTO-RUN-001】

## 5.1 运行页【UX-AUTO-RUN-001-REQ】

运行页按以下顺序展示：激活状态与事实截止点；环境/账户/工具/方向；三轴额度及实际使用；当前条件与下一判断时间；持仓、开放订单与保护覆盖；动作未决和用户责任；三类控制；时间线。存在敞口且保护未确认时，页面顶部显示不可关闭的高优先级提示，普通盈亏或策略信息不得遮盖。

条件行显示规则名称、`Halpha 监控 | 场所监控`、所需窗口、最近结果、输入截止点和下一可判断时间。`UNKNOWN` 是独立状态，不能显示成“未触发”。订单状态不能推定持仓，部分成交必须同时显示已成交量、剩余量、当前持仓和保护状态。

## 5.2 唯一时间线投影【UX-AUTO-RUN-002-REQ】

时间线只引用并组合以下权威记录：

| 来源 | 页面含义 |
|---|---|
| `PlanActivation` | 当前激活、运行游标、控制状态、保护摘要和结束责任 |
| `PlanEvent` | 策略判断、用户控制和领域决定的持久事件 |
| `ExecutionAction` | 使用同一状态词展示待执行、已提交、结果未决或已闭合的动作责任；每行固定显示 Demo/LIVE、账户、授权效力和场所端点类别，不按环境换对象名称 |
| `VenueFact` | 订单、成交、持仓、资金和场所查询事实 |

投影顺序为“计划事件 → ExecutionAction → 场所事实 → 核对结论”。页面不建立任何额外时间线对象；全部细节从上述逻辑记录的版本、理由码和引用读取。来源冲突、Demo/LIVE 环境不符或顺序无法确认时显示未知，不由前端猜测排序或闭合。

Demo 激活与复盘页面必须把“系统流程与机制验证”作为首要证据区，显示计划→CAP→EXE→DAT→OUT 链、动作状态、防重复、保护、核对、停止、恢复、接管和环境隔离结果；“策略行为验证”作为次要证据区。策略区必须常驻说明模拟结果不证明 LIVE 的流动性、排队、冲击、滑点、费用、资金费率、延迟、权限、可用性或真实 Alpha，且不能用绿色成功状态弱化该说明。

# 6. 三类控制与恢复【UX-AUTO-CTL-001】

三类控制始终分开展示并提交不同 Command：

| 控制 | 激活后的业务后果 | 页面生效条件 |
|---|---|---|
| 停止新增风险 | 禁止新的开仓、加仓和入场重挂；关闭仍可取消的增险入场责任；Strategy 继续运行以处理保护、减仓和退出 | Receipt 生效且服务器重读显示新增风险已禁止；不宣称持仓已平 |
| 退出策略 | 激活进入退出过程，停止新增风险，并按计划通过持久动作门取消、减仓或平仓；Strategy 在退出责任闭合前继续产生允许的退出提议 | 页面展示每一步、未决和最终场所核对；不承诺价格或即时成交 |
| 用户接管 | 先持久化 `USER_TAKEOVER`；`HalphaCoordinator` 拒绝新的 READY 动作，并以不管理场所订单的行为停止并移除对应 HalphaStrategyAdapter；Halpha 不自动撤单或平仓，用户在 Binance 官方入口处理，Halpha 只读观察与核对 | 只有接管已持久化、适配器已停止且不会启动新调用后显示“用户接管已生效”；在途结果仍可迟到并保持显著可见 |

每个控制都有独立后果摘要、明确按钮和原 Command/Receipt 查询。保护缺口、动作未决或 Executor 不可达时，不得显示“已经安全停止”或“已经退出”。用户接管后同一激活不再提供恢复机器写入，只显示只读核对、开放责任和新建计划入口。

“恢复机器增险”与“恢复激活”是两个独立高后果命令。前者只处理可恢复的新增风险停止；后者只清除 `MANUAL_PLAN_RESUME` 的 `WRITER_CONTINUITY_LOST` 暂停。两者的预览都必须包含暂停/停止原因、当前场所事实、保护状态、授权版本、额度、四类 CAP 停止和重新检查结果；`RESUME_ACTIVATION` 遇到用户停止、授权撤销或失效、`ALL_WRITES`、用户接管、最大损失、过期、条款变化或未知事实必须拒绝。页面加载、App/Executor 重启、轮询成功、SPA 重新可用或邮件链接不能自动恢复。

该暂停期间页面明确说明既有场所原生订单和保护仍可继续，Halpha 只查询、核对和保存迟到事实，不会形成新的场所写。用户可以直接选择“退出策略”，经已允许的撤单、保护或风险减少链处理，而不先恢复普通策略；也可以选择“用户接管”。在用户明确选择前，不得以“正在恢复”掩盖实际 `PAUSED`。

# 7. 资金、授权与结束【UX-AUTO-CAP-001】

资金页分开显示 Binance 账户事实、Halpha 账户边界和每个激活的互斥额度。额度分别展示最大保证金、最大名义仓位和最大允许损失，不能合并成“可用资金”或暗示交易所实际隔离了资金。

资金页还按 `NEW_FUNDING`、`PROTECTION`、`RISK_REDUCTION_OR_ORDER_MANAGEMENT`、`ALL_WRITES` 四类分别显示停止来源及并集结果；`ALL_WRITES` 必须覆盖前三类，其他类别不得被前端折叠成含义不明的单个开关。每个动作同时显示独立 `max_action_notional` 与激活/账户聚合上限检查，拆单、并发、重试和部分成交不得被呈现为规避单动作上限的方式。

保证金或名义仓位自然越界显示“阻止后续增险，不自动退出”；最大允许损失触线显示“本激活已停止增险并进入全部退出”，列出已实现、未实现、手续费、资金费、净损失和事实截止点。其他激活的损益与额度保持分离。

额度释放只有在 CAP 已引用 EXE 按唯一完整闭合契约形成的最终 `closure_digest` 后才显示完成。UX 不复制或缩短闭合清单；前端不得根据“策略停止”、用户接管、订单取消请求、局部订单状态或余额变化自行释放或显示结束。

# 8. Task 与邮件通知【UX-AUTO-NOT-001】

Task 优先顺序为：可能重复动作、动作未决、保护缺口和用户接管责任；有明确时限的决定；正在运行的控制或责任变化；计划准备；普通信息。同一来源只进入一个未闭合 Task，同级按到期时间、可证明后果和稳定身份排序。没有 Task 时显示“当前无待办”、事实截止点和写入状态，不生成内容流。

Notification 至少覆盖激活、条件触发、动作提交、成交、保护缺口、停止、退出、动作未知、用户接管和额度释放。需要用户处理的来源先幂等确保 Task；邮件只包含事件摘要、环境/工具、时间、Task 身份和返回工作台的只读路径，不包含秘密、完整账户信息、会话令牌或可执行资本命令。

深链打开后必须重新认证并重读当前 Task、对象版本、事实截止点和有效期。任务已解决、内容已变化、会话已过期或当前事实不可确认时，页面拒绝旧预览并显示当前结果。邮件发送失败不重放交易、不扩大权限；Notification 放弃后形成可见 Task，不增加通知 worker 或第二 outbox。

# 9. Web 安全【UX-AUTO-SEC-001】

本节只适用于本契约定义的 Web 工作台和同进程 `/operations`，不得外推为其他客户端形态的认证或网络方案；未来形态不在本文件中预先设计。

工作台只认证单一所有者。pwdlib Argon2 校验口令；Starlette 签名 cookie 建立绝对到期的短时会话；asgi-csrf 对包括登录在内的所有不安全 HTTP 方法执行签名双提交校验。Halpha 接受该组件的双提交模型，不自研 session-bound CSRF 状态或会话数据库记录。Starlette 可信 Host 与 App 同源检查共同拒绝异常 Host、Origin/Referer；`GET/HEAD`、资源预取、邮件打开和跨站表单不得改变状态。

limits 只限制失败登录、激活/恢复增险的重复尝试和普通入口滥用，不作为授权或动作防重。停止新增风险、退出策略和用户接管的第一次合法请求必须可达；它们的重复与并发由 Command 幂等和预期版本控制。精确口令参数、cookie 属性与期限、CSRF 名称、允许来源和限频值由 L4 固定。

认证成功只允许进入交互层，不授予计划激活、额度或 Binance 动作。高后果的激活、恢复增险、退出和用户接管仍需各自当前预览与明确 Command。会话失效或秘密轮换后旧会话拒绝；离线重置走主机上的人工恢复路径，不通过邮件或浏览器建立第二认证通道。

浏览器响应、DOM、日志、剪贴板和邮件不得出现 Binance、数据库、SMTP 或会话签名秘密。安全组件失效时拒绝受影响 Web 命令，页面明确指向 Binance 官方入口的人工接管；不得静默降级为无认证接口。

# 10. 视觉、响应式与可访问性【UX-AUTO-VIS-001】

MUI 主题使用中性基底、单一交互强调色和少量语义色；真实环境、未知、保护缺口和用户接管同时使用文字、图标与色彩。数字、单位、正负号和时间采用稳定对齐；颜色不表示建议或安全，盈亏动画、渐变、玻璃拟态、行情闪烁和游戏化效果不使用。

主要支持桌面宽度。窄屏时导航折叠、检查器进入抽屉、普通表格可水平滚动或转为逐项列表；激活后果、额度、保护缺口和三类控制不得隐藏或缩成无说明的一键动作。关键控件支持键盘、明确焦点、关联标签和错误、足够对比度及屏幕阅读器状态播报；动效尊重减少动态效果设置。

MUI 组件优先于自建组件；仅在真实任务无法由现成控件表达且没有更小文本/布局方案时新增领域控件。无消费者控件、专用设计系统、专业图表和高级数据表必须删除或不建设。

# 11. 浏览器验收【UX-AUTO-TST-001】

Vitest 覆盖纯前端映射，Playwright 与 axe 在真实浏览器至少验证：

1. RJSF 参数控件、Decimal 文本、AJV8 即时错误和服务器交叉错误不会被混淆；固定版本后不能原地编辑；
2. 激活页完整显示策略、事实、动作、三轴额度、自然越界、激活独立损失、保护、有效期、通知和用户接管后果；
3. 双击、超时、刷新、返回和 schema 不兼容不会建立第二激活或第二 Command；UNKNOWN 按原身份查询；
4. 激活后没有逐动作确认；运行页正确区分未触发、UNKNOWN、待执行、场所已接受、部分成交、持仓、保护和完全结束；
5. 时间线只由 `PlanActivation`、`PlanEvent`、本环境的 `ExecutionAction` 和 `VenueFact` 重建，Demo/LIVE 环境或来源冲突不被前端猜测闭合；Demo 复盘明确以系统机制证据为主、策略行为证据为次并展示不可外推项；
6. 停止新增风险、退出策略、用户接管、恢复增险和恢复激活具有不同预览、Command、Receipt 和结果；用户接管不自动撤单/平仓且不显示恢复入口；`RESUME_ACTIVATION` 只清除写者连续性暂停，不清除其他停止；
7. Task 确认不误关来源责任，Receipt UNKNOWN 不重复业务效果，Notification 去重、重试、放弃和深链不形成第二状态或执行动作；
8. TanStack Query 轮询中断、缓存过期、服务器断开和离线页面不把旧事实冒充当前事实，也不允许离线资本命令；
9. 未认证、会话到期、CSRF 缺失/错误、Host/Origin 不符、跨站请求、GET/预取和旧邮件深链均不能改变状态；停止、退出和用户接管的第一次合法请求仍可达；
10. 桌面与窄屏、键盘、焦点、错误关联、对比度、屏幕阅读器播报和减少动态效果符合选定基线；DOM、网络、日志和错误页没有长期秘密；
11. React/Vite 静态制品不可用或 SPA 路由失败时，`/operations` 仍能通过同一认证、CSRF、Command/Receipt 和领域处理器展示关键运行/恢复状态，提交三个有限操作并查询原回执；正常与有限入口的并发提交保持同一幂等结果，且不存在第二 API、状态、口令或进程。

浏览器组件测试证明交互契约，不证明 Binance、SMTP、数据库恢复或交易策略当前可用；相应当前资格化只由 L4 记录。

# 12. 迁移、复杂度与 L4 边界【UX-AUTO-MIG-001】

迁移到本设计时删除逐动作确认页、浏览器自建表单/请求缓存/API 类型、专业图表或高级表格候选，以及四类权威记录之外的额外时间线对象。历史内容只读投影到现有记录，不伪造新交互状态。

前端仍只有一个 Web 应用、一个 API、四个交互记录族和一个 App `NotificationDispatcher`。同进程 `/operations` 只是同一应用边界的服务端渲染入口，不是第二 Web 应用或运行模式。TanStack Query、RJSF、生成客户端和成熟安全组件分别替代自研查询、表单、DTO 与认证/CSRF 计划，不增加产品进程、写入路径或事实权威。

L4 只记录当前精确组件版本、锁与构建摘要、路由启用范围、cookie/CSRF/限频参数、主题变量、目标浏览器与屏幕、SMTP 配置引用、页面建设状态、测试证据和已知限制。本 L3 不声明任何页面、组件或安全边界已经部署或可用于真实资金。
