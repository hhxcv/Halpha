**层级：** L3  
**L3 类型：** DOMAIN  
**主要语义所有者：** EXE  
**所属实现模块：** `venue_integration` 的 EXE 语义分区；与 DAT 同模块但状态所有权分离  
**语言版本：** zh-CN  
**上位文档或条款：** HALPHA-EXE-001  
**直接依赖：** HALPHA-ALP-002、HALPHA-TRADEPLAN-002、HALPHA-CAP-002、HALPHA-DAT-002  
**直接消费者：** HALPHA-OUT-002、HALPHA-UX-002  
**适用纵向约束：** HALPHA-SYS-001、HALPHA-ENG-001  
**本文档负责：** Demo 与 Live 共用的唯一 `ExecutionAction` 写路径；动作持久身份、单写者、先持久后外写、防重复、查询后再判断、部分结果、保护、退出、核对、责任闭合、重启和用户接管  
**本文档不负责：** 定义策略或计划条件；授予机器执行范围或计算资金上限；拥有场所事实；规定当前场所、账户、工具、订单参数、组件精确版本、配置、凭据、限频、超时或检查结果

# 0. 设计结论【EXE-AUTO-SUM-001】

所有 Halpha 场所写入只允许经过同一条链：

```text
策略提议、计划条件或用户控制
→ TRADEPLAN 形成 PlanEvent 与 ProposedAction
→ CAP 检查当前 PlanActivation 内资金限制与事实
→ EXE 在同一事务中持久化唯一 ExecutionAction
→ 提交前重新检查
→ 场所写入
→ DAT 形成 VenueFact，EXE 查询并闭合原动作
```

EXE 只拥有一个持久记录族 `ExecutionAction`、一个私有可写仓储和一条场所客户端构造路径。提交尝试、外部身份、未知结果、保护和核对证据都保存在原动作中，不另建写入控制、提交尝试、保护任务、核对事项、资格状态或恢复对象。

Demo 与 Live 调用相同代码和记录类型，但使用完全分离的环境、账户、端点、凭据引用、数据库和动作身份。没有场所写客户端的只读运行不进入 EXE 动作边界，也不能把观察结果转换成 `ExecutionAction`。

---

# 1. `ExecutionAction`【EXE-AUTO-ACT-001】

## 1.1 最小内容【EXE-AUTO-ACT-001-REQ】

每个动作至少保存：

- 稳定 `execution_action_id`、环境、账户、激活和来源 `PlanEvent`；
- 动作种类、风险方向、工具、方向、数量或全平语义、订单类型、价格或触发价、有效期和只减仓约束；
- 规范条款摘要、建立动作时的 CAP 检查摘要和事实截止点；
- 场所客户端身份；下单动作使用一次生成并持久化的 UUID，撤单动作引用原订单身份；
- 当前责任状态、版本、调用开始时间以及场所订单、成交、费用、持仓和保护事实引用；
- 未知原因、下一次查询时间和责任闭合引用。

同一环境、激活、来源身份、动作种类和规范条款只能对应一个动作。事务结果未知时按来源身份读取原动作，不创建第二个。客户端身份只用于稳定关联，不编码授权、计划或资金语义。

## 1.2 单一责任状态【EXE-AUTO-STA-001-REQ】

`ExecutionAction.state` 只使用：

`READY | SUBMITTING | UNKNOWN | OPEN | NOT_SUBMITTED | CLOSED | HANDED_OVER`。

- `READY`：动作已持久化，尚未越过场所写入边界；
- `SUBMITTING`：调用前状态已经提交，此后崩溃按可能已调用处理；
- `UNKNOWN`：可能已调用或场所结果冲突，只能查询原身份；
- `OPEN`：场所已经接受或动作出现部分结果，仍有订单、成交、保护或核对责任；
- `NOT_SUBMITTED`：有证据证明没有发生场所调用或场所明确拒绝且无外部变化；
- `CLOSED`：订单、成交、费用、持仓和保护对本动作的影响均已得到确定事实；
- `HANDED_OVER`：用户接管前尚未调用的动作不再由 Halpha 提交。

场所订单自身的接受、工作、部分成交、成交、取消、拒绝和过期状态由 `VenueFact` 保存，不在 `ExecutionAction` 再建一套平行订单状态机。一个动作最多越过场所写入边界一次。

# 2. 建立与提交【EXE-AUTO-SUB-001】

## 2.1 本地原子提交【EXE-AUTO-TXN-001-REQ】

TRADEPLAN 形成不可变 `PlanEvent`/`ProposedAction`，CAP 使用同一事实截止点检查当前 `PlanActivation` 内资金限制。CAP 接受时，EXE 在同一环境事务中建立单向引用该事件的 `ExecutionAction`；任一步失败全部回滚。EXE 不直接修改计划、激活或 DAT 私有记录。

事务提交前禁止调用订单工厂、下单、撤单或其他场所写方法。相同来源和条款返回原动作；相同来源出现不同条款时拒绝并停止该激活新增风险。

## 2.2 唯一场所调用【EXE-AUTO-SUB-001-REQ】

处理 `READY` 动作时依次执行：

1. 重读当前 `PlanActivation`、前置动作和所需 `VenueFact`；
2. 调用同一 CAP 函数重新检查动作；后一次可以否决前一次；
3. 在事务中写入请求摘要、调用开始时间和 `SUBMITTING`，提交后才调用场所；
4. 使用原客户端身份调用唯一执行客户端；策略逻辑和普通回调都不能取得该写能力；
5. 保存直接结果，并按原客户端身份查询订单与场所结果；
6. 通过 DAT 写入有来源的 `VenueFact`，再推进原 `ExecutionAction`。

执行客户端、HTTP 库和 SDK 的自动写重试必须关闭。读取与查询可以有界重试，但不能更换动作、客户端身份、目标订单或条款。会在组件内部生成未持久子动作的便捷方法不得使用；每个可能改变场所的子动作都必须先有自己的 `ExecutionAction`。

## 2.3 查询后再判断【EXE-AUTO-UNK-001-REQ】

| 情形 | 唯一处理 |
|---|---|
| 动作事务前失败 | 无场所效果；相同来源可重新计算 |
| 动作已提交但仍为 `READY` | 重读当前事实和 CAP 后处理原动作 |
| `SUBMITTING` 后崩溃、超时或断线 | 进入 `UNKNOWN`，只查询原客户端身份，不再次写入 |
| 场所证明原身份不存在且请求未进入 | `NOT_SUBMITTED`，原动作不复活 |
| 场所已经接受或出现部分结果 | `OPEN`，按原身份继续核对 |
| 风险减少责任明确未提交但责任仍存在 | TRADEPLAN 可按新事实形成一个新的风险减少动作；原动作保留终态 |

任何 `UNKNOWN` 都不可逆地停止所属激活新增风险。超时和金额较小都不能证明未发生外部变化。

# 3. 订单、部分结果与保护【EXE-AUTO-ORD-001】

条件、撤单和重挂都由 TRADEPLAN 分别形成动作。旧订单只有在终态、取消与成交竞争已经核对、当前持仓和剩余目标明确后，才可建立替代订单；旧结果未知时不建立新单。减仓和平仓数量不得超过同一事实截止点已核对的可减持仓，无法证明不会反向开仓时不提交。

首次风险敞口或新增部分成交被事实确认后，计划要求的保护必须形成独立动作。保护尚未由场所事实确认前不得继续扩大敞口。场所原生保护只有在身份、查询、撤销、部分执行、只减仓或全平含义以及 Halpha 离线后的持续行为均已验证时才可使用。

保护提交或查询未知时保留原保护责任、停止新增风险并只查询原身份，不提交等价的第二保护。替换保护时，旧保护保持有效直到新保护得到场所确认；若新旧并存可能反向开仓或过度减仓且没有已验证替换方式，则不支持替换。

期限、退出或停止新增风险要求处理开放增险订单时，已经 `SUBMITTING` 或 `UNKNOWN` 的动作仍只查询原身份。取消后的晚到成交形成新的已确认敞口，进入保护和退出，不重新授予入场机会。持仓归零后，全部相关入场、保护和退出订单仍须按原身份核对到确定结果。

# 4. 最大损失与三项控制【EXE-AUTO-CTL-001】

EXE 把本激活可归属的已实现盈亏、未实现盈亏、实际手续费和实际资金费交给 CAP。达到最大允许损失时，TRADEPLAN 在 `PlanActivation` 内锁存停止新增风险并进入 `EXITING`；EXE 只执行随后形成的撤单、保护、减仓和平仓动作，不建立第二损失或停止记录。

三项用户控制的执行边界是：

- `STOP_NEW_RISK`：计划先不可逆记录停止；EXE 不再提交增加风险动作，继续查询原动作并允许经 CAP 证明不会增险的保护、撤单、减仓和退出；
- `EXIT_STRATEGY`：计划进入 `EXITING`；每个撤单、保护、减仓或平仓仍分别经过 CAP 和唯一 `ExecutionAction`；
- `USER_TAKEOVER`：计划先持久化接管范围和切换时间；EXE 停止该激活全部新的 Halpha 写入，不自动撤单或平仓，只查询在途动作和记录后续场所事实。

三项控制使用各自稳定请求身份和计划领域结果，不依赖通用命令、回执或任务平台。用户停止新增风险或接管后不能恢复同一激活；旧责任闭合后需要继续交易时建立新激活。

# 5. 核对与责任闭合【EXE-AUTO-REC-001】

NautilusTrader 负责其公开支持的启动核对、订单事件、持仓核对和单订单查询。Halpha 不复制组件订单状态机；组件缓存、组合投影或合成事件不能证明成交、费用、持仓归零或责任闭合。

Halpha 只依据 `ExecutionAction` 的环境、激活、客户端身份和条款摘要归属自身动作。未知客户端身份、摘要冲突或无法唯一匹配的订单、成交和持仓属于外部或未知活动；Halpha 不自动认领、撤销或计入本激活结果，并停止相关新增风险。

`ExecutionAction` 只有在其订单终态、成交与费用完整、对持仓和保护的影响明确，且没有可能已调用但结果未知的责任时才进入 `CLOSED`。

EXE 对一个激活只返回非持久的闭合判断与事实引用。只有同一截止点同时证明以下内容时，结果才为 `CLOSED`：

1. 场所持仓为零；
2. 本激活普通和条件订单全部终结，且查询没有开放订单；
3. 所有相关 `ExecutionAction` 均为 `CLOSED | NOT_SUBMITTED | HANDED_OVER`，没有 `SUBMITTING | UNKNOWN`；
4. 保护、晚到成交和费用已经处理；
5. 不存在会混淆归属的新增、反向或未知外部活动。

条件不足时返回 `UNKNOWN` 和缺口，TRADEPLAN 保存引用但不能结束激活。用户接管范围内、可证明只减少或关闭风险的官方入口活动可以帮助证明持仓和订单责任闭合，但不能冒充 Halpha 动作或补造精确收益。

# 6. 重启与单写者【EXE-AUTO-RCV-001】

每个写入范围同时只能有一个 Halpha 执行器。单宿主可以使用操作系统互斥；无法证明唯一性时停止所有新的 Halpha 场所写入，只保留读取、查询、核对和用户接管。

重启后依次执行：取得单写者控制；读取未闭合激活和动作；按原客户端身份查询场所；保存迟到订单、成交、持仓、费用和保护事实；处理 `SUBMITTING`/`UNKNOWN`；重新检查计划、代码版本、激活内资金限制和当前事实。全部一致时可继续原激活及其既有责任；无法证明一致时不可逆地停止新增风险，只保留查询、保护、退出和用户接管。

重启不建立恢复状态、恢复命令或第二激活。它不重放旧动作，也不使已过期、已停止或已接管的激活恢复写入。已有风险需要保护或退出时，TRADEPLAN 按当前事实形成新动作，每个动作仍经过 CAP 和先持久后外写。

# 7. 公开能力、错误与验证【EXE-AUTO-API-001】

公开能力只需：建立动作；处理原动作；应用场所事实；收集本激活损失输入；判断动作和激活责任是否闭合；应用用户接管。所有写能力使用稳定身份、预期版本和数据库事务。

错误至少区分：CAP 拒绝、重复身份冲突、前置动作未闭合、确定未提交、提交结果未知、场所拒绝、持仓未知、保护未知或缺口、外部活动和用户接管。确定拒绝与未知不能合并。

至少验证：

1. Demo 与 Live 的所有场所写都经过同一 EXE 代码、私有仓储和执行客户端路径，环境、账户、凭据和动作不可交叉引用；
2. 每个动作先持久化，再以原客户端身份最多调用一次；写自动重试关闭；
3. 事务、提交、超时、断线和重启各窗口都不会产生第二动作或第二场所调用；
4. `UNKNOWN` 只查询原身份并停止新增风险，不以超时推定未提交；
5. 部分成交、取消竞争、晚到成交和重挂不会超过剩余目标或反向开仓；
6. 保护缺口阻止新增风险，未知保护不产生内容等价的第二保护；停止新增风险不误阻保护、撤单、减仓和退出；
7. 最大损失只由激活内事实触发，EXE 不建立第二锁存或停止对象；
8. `STOP_NEW_RISK`、`EXIT_STRATEGY` 和 `USER_TAKEOVER` 后果不同，且不存在恢复激活命令或通用回执依赖；
9. 单写者丢失、数据库不可用、事实冲突或归属未知时没有新场所写入；
10. 激活闭合不足时返回 `UNKNOWN`，不结束计划或允许同一真实账户创建新激活；
11. 数据库中只有 `ExecutionAction` 一个 EXE 记录族，没有写控制、提交尝试、保护任务、核对事项或恢复状态机。

# 8. 非目标与 L4 边界【EXE-AUTO-NON-001】

EXE 保持一条场所写链、一个 `ExecutionAction` 记录族和基于当前事实的即时闭合判断；不建设第二写管线、自研场所协议栈或订单状态机、通用保护/核对平台、独立保护服务、闭合摘要对象或恢复激活流程。动作唯一、查询后再判断、未知时停止新增风险、保护、退出和用户接管边界保持不变。

L4 只记录当前场所、账户、工具、订单与保护范围、组件精确版本和配置、限频、超时、宿主实例、凭据边界及实际检查结果；本 L3 不声明这些当前已经可用。
