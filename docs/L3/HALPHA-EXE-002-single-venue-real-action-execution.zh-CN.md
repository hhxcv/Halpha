# Halpha 场所动作执行契约

**文档编号：** HALPHA-EXE-002  
**层级：** L3  
**L3 类型：** DOMAIN  
**主要语义所有者：** EXE  
**所属实现模块：** venue_integration 的执行分区  
**语言版本：** zh-CN  
**上位文档或条款：** HALPHA-EXE-001  
**直接依赖：** HALPHA-TRADEPLAN-002、HALPHA-CAP-002、HALPHA-DAT-002  
**直接消费者：** HALPHA-OUT-002、HALPHA-UX-002  
**适用纵向约束：** HALPHA-SYS-001、HALPHA-ENG-001  
**本文档负责：** 外部动作记录、状态、提交、防重复、保护、核对和本地恢复行为  
**本文档不负责：** 计划与激活生命周期、资金计算、事实生产和当前场所参数

## 1. 动作记录

每个外部动作使用一个不可变动作记录，至少保存稳定身份、环境、账户、激活、来源计划事件、动作条款摘要、建立时检查结果、场所客户端身份、当前责任状态、外部事实引用、未知原因和闭合引用。

同一环境、激活、来源和规范条款只能对应一个动作。修改、撤单、补保护、减仓和平仓分别形成新动作并引用原因动作，不覆盖原意图和外部身份。

## 2. 状态

动作责任只使用以下状态：

- `READY`：已持久化，尚未越过场所写边界；
- `SUBMITTING`：调用前标记已经提交，此后崩溃按可能调用处理；
- `UNKNOWN`：可能调用或外部事实冲突，只查询原身份；
- `OPEN`：场所已接受或出现部分结果，责任未闭合；
- `NOT_SUBMITTED`：能够证明没有发生外部变化；
- `CLOSED`：该动作对订单、成交、费用、持仓和保护的影响已核对；
- `HANDED_OVER`：用户接管后未调用动作不再由 Halpha 提交。

场所订单自身状态由 DAT 事实表达，不在动作记录复制第二订单状态机。

## 3. 建立与提交

TRADEPLAN 形成不可变动作意图，CAP 返回检查结果，EXE 在本地事务中建立动作记录。事务提交前禁止调用场所。处理 `READY` 时，执行角色重新读取当前激活和事实、再次调用 CAP、写入 `SUBMITTING`，提交后使用原客户端身份调用唯一执行客户端，再由 DAT 保存结果事实。

执行客户端的自动写重试关闭。读取可以有界重试，但不能更换动作、身份、目标或条款。`SUBMITTING` 后超时、断线或崩溃进入 `UNKNOWN`，只查询原身份。

## 4. 部分结果与保护

旧订单只有在终态和取消/成交竞争已经核对后，计划才可形成替代动作。减仓和平仓不得超过当前可证明持仓；无法证明不会反向开仓时不提交。

风险敞口出现后，计划所需保护形成独立动作。保护由场所事实确认前不继续扩大敞口。保护未知时只查询原身份，不提交内容等价的第二保护。

## 5. 用户控制与闭合

TRADEPLAN 持久化停止、退出或接管决定后，EXE 按当前动作状态处理：停止不再提交增险动作；退出继续处理保护和减仓动作；接管停止该激活新的 Halpha 场所写，只读核对在途结果。

动作只有在订单、成交、费用以及对持仓和保护的影响明确，且没有可能已调用但结果未知的责任时闭合。激活整体是否闭合由 TRADEPLAN 消费 EXE 与 DAT 的结果决定。

## 6. 恢复与验证

系统恢复顺序由 SYS 执行。EXE 在获得单写者和恢复调用后读取未闭合动作、查询原客户端身份、保存迟到事实并推进原记录；它不创建恢复状态、第二激活或第二写路径。

至少验证事务与场所调用各故障窗口、重复输入、部分成交、取消竞争、保护未知、用户控制、数据库不可用、Demo/Live 隔离和单写者丢失均不会产生第二动作或第二场所调用。
