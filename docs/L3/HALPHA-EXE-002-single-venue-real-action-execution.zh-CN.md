**层级：** L3  
**L3 类型：** DOMAIN  
**主要语义所有者：** EXE  
**所属实现模块：** venue_integration 的执行分区  
**语言版本：** zh-CN  
**上位文档或条款：** HALPHA-EXE-001  
**直接依赖：** HALPHA-TRADEPLAN-002、HALPHA-CAP-002、HALPHA-DAT-002  
**直接消费者：** HALPHA-OUT-002、HALPHA-UX-002  
**适用纵向约束：** HALPHA-SYS-001、HALPHA-ENG-001  
**本文档负责：** 外部动作记录、状态、提交、防重复、保护、核对和本地恢复行为  
**本文档不负责：** 计划与激活生命周期、资金计算、事实生产和当前场所参数

## 1. 动作记录

每个外部动作使用一个不可变动作记录，至少保存稳定本地动作身份、环境、账户、激活、来源计划事件、动作类型、规范条款摘要、建立时检查结果、目标订单身份、场所接口支持时的独立请求身份、当前责任状态、外部事实引用、未知原因和闭合引用。新建与重挂订单必须具有稳定场所客户端订单身份；改单和撤单不能虚构交易所并不接受的第二客户端订单身份，而以本地动作身份、原目标身份、期望条款和事实截止点保持幂等责任。

同一环境、激活、来源、动作类型、目标订单身份和规范条款只能对应一个动作。修改、撤单、补保护、减仓和平仓分别形成新动作并引用原因动作，不覆盖原意图和外部身份。多个订单档位共享计划摘要只用于归属和读取，不把各档的提交与未知结果合并成一个动作状态。

## 2. 状态

动作责任只使用以下状态：

- `READY`：动作责任已在本地持久化，尚未向交易所提交变更请求；
- `SUBMITTING`：调用前标记已经提交，此后崩溃按可能调用处理；
- `UNKNOWN`：可能调用或外部事实冲突，只查询原身份；
- `OPEN`：场所已接受或出现部分结果，责任未闭合；
- `NOT_SUBMITTED`：能够证明没有发生外部变化；
- `CLOSED`：该动作对订单、成交、费用、持仓和保护的影响已核对；
- `HANDED_OVER`：用户接管后，能够证明从未调用的动作不再由 Halpha 提交。

场所订单自身状态由 DAT 事实表达，不在动作记录复制第二订单状态机。已经进入 `SUBMITTING | UNKNOWN | OPEN` 的动作不会因接管改写为 `HANDED_OVER`；它保持原责任并只读核对，直到外部事实足以推进为原有终态。

## 3. 建立与提交

TRADEPLAN 形成不可变动作意图，CAP 返回检查结果，EXE 在本地事务中建立动作记录。需要一次建立同一订单计划的全部 `PlanEvent` 和动作时，无状态事务协调器组合 TRADEPLAN 与 EXE 仓储；记录所有权仍分别属于两个领域。事务提交前禁止向交易所请求变更。处理 `READY` 时，执行角色重新读取当前激活和事实、再次调用 CAP、把责任状态持久化为 `SUBMITTING`；新建订单使用动作已保存的场所客户端订单身份，改单或撤单使用已保存的目标身份和期望条款调用唯一执行客户端，再由 DAT 保存结果事实。

执行客户端不得自动重试交易所变更请求。查询可以有界重试，但不能更换动作、身份、目标或条款。`SUBMITTING` 后超时、断线或崩溃进入 `UNKNOWN`。新建订单只查询原客户端订单身份。改单或撤单只有在直接场所响应，或出现晚于调用前目标事实版本或截止点且能证明目标订单取得期望条款或终态的权威事实时，才可确认结果；场所没有独立请求身份，或查询不能区分“请求未到达”和“请求发生后又成交或变化”时，继续保持 `UNKNOWN`，只查询原目标订单及迟到成交。

## 4. 部分结果与保护

原生改单只修改当前场所明确支持且仍开放的原订单，并使用独立动作责任记录本次请求；原订单身份和历史条款保持可读。撤销后重挂只有在旧订单终态和取消/成交竞争已经核对后才可形成替代动作。改单或撤单结果未知时按上一节核对目标身份和期望条款，不得同时重挂。减仓和平仓不得超过当前可证明持仓；无法证明不会反向开仓时不提交。

风险敞口出现后，计划所需保护形成独立动作。串行多档在当前实际持仓的保护由场所事实确认前不开放下一档；并行多档只有在预先确认的场所保护能够覆盖所有仍可能成交档位的最大持仓时才可提交。任一入场、撤单或保护结果未知时，尚未提交档位停止，仍开放档位保持累计资金与保护责任直到其终态。保护未知时只查询原身份，不提交内容等价的第二保护。

## 5. 用户控制与闭合

TRADEPLAN 持久化停止、退出或接管决定后，EXE 按当前动作状态处理：停止不再提交增险动作；退出继续处理撤销、保护和减仓动作；接管后不再为该激活发起新的交易所变更请求，只有从未调用的动作可转 `HANDED_OVER`，已调用或可能已调用的动作保持原状态并只读核对。

动作只有在订单、成交、费用以及对持仓和保护的影响明确，且没有可能已调用但结果未知的责任时闭合。激活整体是否闭合由 TRADEPLAN 消费 EXE 与 DAT 的结果决定。

## 6. 恢复与验证

系统恢复顺序由 SYS 执行。EXE 在确认唯一执行者并获得恢复调用后读取未闭合动作、查询原客户端身份、保存迟到事实并推进原记录；它不创建恢复状态、第二激活或第二交易所变更请求路径。

至少验证事务与交易所调用各故障窗口、重复输入、多档部分提交、部分成交、改单与取消竞争、保护未知、用户控制、数据库不可用、Demo/Live 隔离和唯一交易所请求执行者丢失均不会产生第二动作或第二次交易所调用。
