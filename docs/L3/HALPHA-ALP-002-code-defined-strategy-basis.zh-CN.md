# 0. 设计结论【ALP-CODE-SUM-001】

Halpha 使用 NautilusTrader 的策略生命周期、行情回调、原生指标和回测能力，不另建策略框架、指标库、撮合器或绩效平台。每个运行中的 `PlanActivation` 对应一个 `HalphaStrategyAdapter`；它组合一份纯策略逻辑，接收规范输入并产生零个或一个瞬态 `StrategyProposal`。

纯策略逻辑没有订单或场所写能力。`StrategyProposal` 只有经过 TRADEPLAN 形成 `PlanEvent`/`ProposedAction`、CAP 检查并由 EXE 先持久化唯一 `ExecutionAction` 后，才可能进入场所写路径。相同策略逻辑和参数用于实时运行与离线回测，差异只在输入来源和回测模型；差异与限制必须如实显示，不能用另一套策略实现补齐。

---

# 1. 组件采用与最小自研边界【ALP-CODE-CMP-001】

## 1.1 NautilusTrader 使用范围【ALP-CODE-CMP-001-REQ】

| 能力 | 采用方式 | Halpha 保留的边界 |
|---|---|---|
| 策略生命周期与回调 | `Strategy`、`Controller` 与运行时数据事件 | 每个激活一个适配器；组件内存不是计划或事实权威 |
| K 线、时钟与原生指标 | 组件公开的数据和指标能力 | 只校验输入来源、连续性和边界，不复制公式或事件循环 |
| 历史读取与回测 | `BarDataWrangler`、`ParquetDataCatalog`、`BacktestEngine` | 只用于离线验证，不成为产品数据库或实时事实 |
| 报告与统计 | 组件公开报告能力 | 作为普通研究材料，不自行授权计划或场所动作 |

组件的缓存、组合、风控投影、订单状态或绩效结果都不是 Halpha 的计划、资金、事实或执行权威。当前组件版本、配置和实际检查结果由 L4 记录；缺少所需公开能力时明确不支持，不增加自动备用实现。

## 1.2 保留的最小 Halpha 能力【ALP-CODE-CMP-002-REQ】

Halpha 只保留：随产品代码发布的策略清单、不可变策略身份和代码版本引用；参数 schema 与规范化；一个把框架回调转换为纯逻辑输入和 `StrategyProposal` 的适配器；以及研究材料的适用范围与限制。它们不形成动态插件系统、独立服务、数据库记录族或通用研究平台。

# 2. 策略定义、参数与输出【ALP-CODE-OBJ-001】

## 2.1 代码策略定义【ALP-CODE-OBJ-001-REQ】

每个代码策略至少固定：

- 策略身份、版本和代码版本引用；
- 参数 schema、单位、范围与交叉约束；
- 所需输入、允许的最大过期、连续性要求和未知时的不输出规则；
- 使用的组件指标、可提出的动作种类、适用范围和已知限制。

策略逻辑、参数含义、输入语义或动作语义改变时形成新策略版本。参数值由不可变 `TradePlanVersion` 保存；运行中的参数不得热更新。策略只从产品代码中的显式清单装载，不扫描目录、不下载或执行任意代码。

## 2.2 不可变 `StrategyProposal`【ALP-CODE-ACT-001-REQ】

`StrategyProposal` 是一次稳定输入产生的瞬态值，不是记录、授权或场所命令。它携带激活与策略身份、规则和触发身份、输入摘要、动作内容、风险方向、理由和有效期。相同触发身份和输入必须产生相同结果；相同身份出现不同摘要时为冲突。

输入缺失、过期、不连续、冲突、组件异常或数值无效时，不得产生增加风险的提议。停止新增风险后只允许保护、撤单、减仓或退出提议；退出责任闭合或用户接管后不再产生提议。

策略代码不得取得 `OrderFactory`、执行客户端或等价写入口。适配器只有在 EXE 提供一个已经持久化且身份、环境、激活和条款完全匹配的 `ExecutionAction` 时，才可调用私有写门；策略回调本身不能调用该门。

## 2.3 事实与恢复边界【ALP-CODE-DAT-001-REQ】

实时计算可使用组件内存状态，但被采纳的触发身份、输入摘要和决定保存在 `PlanEvent`，外部结果保存在 `VenueFact`。重启后从 `PlanActivation`、`PlanEvent` 和满足用途要求的事实重建；无法证明等价时停止新增风险并报告 `UNKNOWN`，不得使用最近值、零值、自研指标或第二运行器继续。

## 2.4 本地接口【ALP-CODE-API-001-REQ】

ALP 只需提供策略描述、参数校验和适配器构造三个本地能力。前两项确定且无场所副作用；构造只接受固定计划和激活上下文。实时与离线使用同一 `StrategyProposal` 输出契约。

# 3. 离线验证使用契约【ALP-CODE-EVD-001】

## 3.1 同一策略类【ALP-CODE-EVD-001-REQ】

离线回测必须使用与实时运行相同的适配器、纯逻辑、参数规范化和原生指标。历史数据缺少实时事实时，只能使用明确标注来源的代理输入，并在结果中说明差异；不得把代理写成实时事实，也不得为此增加第二行情源或策略实现。

组件承担回放、撮合、费用、滑点、延迟和报告中其实际支持的部分。未建模的资金费、成交机制或其他差异直接标为未建模。回测成功、模拟盈利或单次真实盈利均不证明未来收益，也不自动扩大资金限制或权限。

## 3.2 相称检查【ALP-CODE-EVD-002-REQ】

检查强度由结果用途决定：普通研究只需可重演输入、参数、代码版本、结果与限制；计划准备还需证明拟用策略版本和参数在声明范围内通过相关逻辑、边界和故障检查。检查结果是普通事实，不建立证据身份、资格状态或第二构建摘要。

至少检查：一次激活不会在完成后自动重新入场；提议不能绕过 TRADEPLAN、CAP 和 EXE；未知输入不增加风险；策略代码没有场所写能力；同一规范输入在线与离线产生相同提议。

# 4. 失败、兼容与停止【ALP-CODE-OPS-001】

| 情形 | 结果 |
|---|---|
| 策略身份、参数或组件公开契约不匹配 | 拒绝启动，不回退到相近版本 |
| 数据缺失、过期、乱序或冲突 | 不增加风险，显示原因与 `UNKNOWN` |
| 策略处理异常 | 停止该激活的新风险提议；已有责任继续保护、退出或接管 |
| 策略尝试直接写场所 | 拒绝并停止该实例，不转送第二写路径 |
| 重复回调 | 按同一触发身份返回同一结果，不产生第二动作 |
| 用户接管 | 停止并移除适配器，不自动撤单或平仓 |
| 离线运行失败 | 本次研究结果不可用，不改变已有运行责任 |

组件升级若可能改变输入、指标、回调顺序或提议结果，必须更新代码版本引用并重新执行受影响检查。运行时只保留一个组件版本和一套策略逻辑。

---

# 5. 最小验收契约【ALP-CODE-TST-001】

实现至少证明：

1. 每个运行激活只有一个适配器，同一激活不会自动开始第二个交易周期；
2. 实时与离线对同一规范输入产生相同 `StrategyProposal`；
3. 静态检查与运行测试都证明纯策略逻辑无法场所写入；
4. 参数边界、输入缺口、重复、乱序和异常不会产生重复或增加风险的动作；
5. 重启可从 Halpha 权威记录恢复，不能恢复时结果为 `UNKNOWN`；
6. 没有 DSL、动态插件、通用优化平台、自研回放撮合或第二量化栈。

---

# 6. 复杂度与删除边界【ALP-CODE-CPLX-001】

本契约不新增进程、数据库、持久记录族、场所写路径或运行模式。NautilusTrader 承担通用运行和回测能力，Halpha 只保留策略身份、参数校验、薄适配和必要的产品边界。没有代码策略消费者时，在开放激活责任闭合后可删除该清单与适配，不影响 TRADEPLAN、CAP 或 EXE 的权威记录。
