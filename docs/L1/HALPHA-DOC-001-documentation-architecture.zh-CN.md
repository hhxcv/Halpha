# 0. 总体结构【DOC-STR-001】

## 文档层级结构定义【DOC-STR-001-DEF】

~~~text
                         L1-A 文档架构
                         （只管理文档）
                                │
L0 宪法 → L1-B 目标与愿景 → L1-C 业务流程与用户路径
                                ↓
                       L1-D 总体技术要求与架构
                                ↓
                L2 领域关键内容与设计原则
                                ↓
                       L3 领域长期稳定的详细设计
                                ↓
                  L4 分阶段落地、事实与现状记录
~~~

## 层级使用要求【DOC-STR-001-REQ】

使用规则只有三条：

1. 下层服从上层，同一主题只在最高拥有层定义一次。
2. L0–L3 定义应当怎样；L4 记录当前做什么和实际是什么。
3. 设计变化先修改拥有该语义的文档，再复核受影响的下游。

# 1. 层级定位与职责【DOC-LVL-001】

层级首先按“哪一层拥有哪类设计语义”划分，不按时间划分。长期或当前只是用来判断设计是否稳定、是否已进入落地，不单独构成一层职责。

## 1.1 各层定位、唯一职责与落地区分【DOC-OWN-001-DEF】

| 层级 | 核心定位 | 唯一职责 |
|---|---|---|
| L0 | 项目级原则 | 定义长期使命、最高边界、不可突破的原则和最高取舍。 |
| L1 | 总体原则 | 分别定义文档、产品、核心路径和总体技术的长期方向、原则与演进条件。 |
| L2 | 领域关键内容 | 定义每个领域要解决的关键问题、领域目标、关键能力、责任边界和设计原则；横向核心业务组成与纵向领域只说明责任形态，不改变这项职责。 |
| L3 | 长期稳定的领域详细设计 | 以 L2 为依据，定义模块边界、接口、状态、完成标准、测试要求，以及经工程门槛选择并预期跨阶段复用的具体库或框架及使用契约；可以限定设计范围，但不拥有建设阶段、当前精确版本或启用状态。 |
| L4 | 分阶段落地与实施 | 在当前建设范围内，把 L0–L3 的长期设计收束为本阶段的实施目标、建设范围、顺序、依赖、精确版本与实例选择、当前配置、资格证据、进度、验收证据和实际事实。 |

L0–L3 形成长期且可复用的原则、领域内容和详细设计。L4 不是重写这些设计，而是为当前阶段选择适用范围、组合已有设计并明确实施目标；它不得新增或改变 L0–L3 的稳定规则。建设阶段标识、当前建设范围、顺序、进度和证据只在 L4 记录。

## 1.3 L1 的依赖顺序【DOC-L1-001】

四篇 L1 文档始终按以下语义约束顺序建立和维护：

1. HALPHA-DOC-001-documentation-architecture：确定文档形式。
2. HALPHA-VIS-001-goals-and-vision：确定长期产品目标和方向。
3. HALPHA-FLOW-001-core-workflows-and-user-journeys：选择达成目标的长期核心路径。
4. HALPHA-ARC-001-technical-requirements-and-architecture：选择支撑这些路径的长期总体技术方案。

后者服从前者，但分别拥有产品、流程和技术语义。四篇文档可以并行起草，正式生效必须按上述顺序进行；下位文档只能引用已经 ACCEPTED 的直接上位版本。

## 1.4 L2–L4 的使用【DOC-L24-001】

### L2–L4 对象与分类定义【DOC-L24-001-DEF】

- **L2 稳定语义责任地图：** L1 遗留给下位细化的每项稳定语义都有一个明确的 L2 语义所有者；它描述责任覆盖，不代表等量设计或实现。
- 只有当前或近期消费者依赖的 L2 才深化到足以约束 L3；较晚职责先保留清晰边界。

L2 领域或其独立子适用范围使用三种深度级别：

| 深度 | 稳定含义 |
|---|---|
| L2 边界深度 | 唯一职责、明确不负责、交付或适用边界，以及缺失或失败时的结果 |
| L2 当前需求深度 | 当前消费者所需的稳定对象、决定、状态边界、交接和验收依据 |
| L2 复用扩展深度 | 多个真实消费者出现后所需的复用生命周期、异常、迁移和稳定公共规则 |

深度表示支持范围、通用化和自动化程度，不表示文档篇幅或流程数量。

L2 采用两个相互独立的职责维度：

| 形态 | 负责内容 | 不负责内容 |
|---|---|---|
| 横向核心业务组成 | 一段连续业务、业务对象的完整生命周期、决定权及其前后交接；该业务可以由一个或多个模块实现 | 跨全部业务重复定义同一种数据、交互、架构、工程或运行要求 |
| 纵向领域 | 贯穿多个业务组成和模块的一类共同约束、质量属性或技术职责 | 重新定义横向业务对象、业务决定及其生命周期 |

L2 按业务或共同约束划分稳定责任，不按软件模块划分责任。模块边界由 L3 在 SYS 的系统组成与运行边界约束下定义；L2 只提供模块必须实现的稳定业务语义和责任边界。

L3 分为两类：领域 L3 指定一个主要语义所有者并精确实现其 L2 语义；编排 L3 指定协调所有者，只拥有必要身份、顺序、责任交接和完成汇总。长期采用的具体第三方库或框架属于对应模块的 L3 实现方案，但第三方不取得 Halpha 的产品语义、事实或决定所有权。

L4 使用以下记录类型：

| 类型 | 记录内容 |
|---|---|
| 阶段性落地设计、计划与进度 | 本阶段实施目标、当前建设范围、顺序、依赖、阻断、完成状态和验收证据 |
| 事实、实例与配置 | 来源、观察或生效时间、活动版本、第三方依赖的精确版本与构建标识、部署、当前参数和平台资格证据 |
| L2 深度与支持范围 | 当前深度、目标深度、真实消费者、深化理由和明确不支持范围 |
| ADR 与现状摘要 | 决定理由或带日期的派生视图，不拥有新的产品或系统规则 |

### L2–L4 使用要求【DOC-L24-001-REQ】

L2 必须先建立领域关键内容、目标、责任边界和稳定设计原则，再按建设需要完善具体内容。责任地图登记的各领域不要求达到相同设计深度；已启用路径必须对其真实职责正确，尚未深化的能力可以人工处理、使用外部工具或明确不支持。

全部领域默认从“L2 边界深度”开始；出现当前真实消费者后才进入“L2 当前需求深度”，“L2 复用扩展深度”不默认采用。领域不得为了体系对称、文档整齐或未来可能性自动升级。深化前必须说明真实消费者、已观察不足、当前需求、业务收益、项目所有者可承担的维护能力、全生命周期成本、为何更小方案不足以及退出路径。当前深度、目标深度、支持范围和深化理由只在 L4 记录。

同一稳定语义只能有一个语义所有者。其他 L2 可以声明输入条件、消费方式和缺失时行为，但不得复制拥有者的规则、状态体系或完整流程。生产者只定义自身输出，消费者只定义接入要求；文档之间不建立全连接式交接说明。

一个设计涉及多个 L2 不自动构成编排 L3。默认以领域 L3 加直接依赖表达；只有重复出现、需要稳定身份且无法由现有领域承担的跨领域生命周期，才使用 `ORCHESTRATION`。

每篇 L3 必须声明类型和语义所有者，列出直接依赖及适用纵向约束，追溯主要业务语义和跨领域要求，并将已有 L2 的领域关键内容、目标和设计原则落实为长期稳定的详细设计。当前或近期建设需求只决定是否需要起草或深化详细设计，不得在 L3 写成 P 阶段、当前建设范围、建设顺序、进度或证据。若实现所需稳定规则在 L2 中没有依据，必须先修改拥有该语义的 L2。

经过 ARC 与 ENG 的能力检索和差异判定后，L3 可以把具体第三方库或框架记录为模块的长期实现方案，并规定 Halpha 怎样调用、输入输出、事实权威、必要转换、失败结果、版本兼容边界、资格化条件和确有必要的最小自研。精确版本、构建哈希、当前配置、目标操作系统验证、许可证核对、启用状态和本阶段证据只由 L4 记录。

第三方库已经完整承担模块内部能力时，L3 主要说明 Halpha 的使用方式、边界、失败与退出，不复制供应商内部类、算法、缓存、状态机、网络生命周期或数据结构，也不为了文档对称虚构一套 Halpha 内部实现。L3 不得同时保留第三方实现与自研实现作为静默、自动或长期平行替代；组件不可用时采用停止、外部人工接管或明确不支持等上位允许结果。

当前建设计划的唯一入口固定为：

~~~text
docs/L4/HALPHA-PLAN-001-current-construction-plan.yaml
~~~

该入口记录当前建设范围并引用必要的进度、配置和证据。L4 不能改变 L0–L3；缺少记录时，当前状态视为未知。

# 2. 文档格式与存放位置【DOC-FMT-001】

## 2.1 命名【DOC-NAM-001】

当前规范文件使用：

~~~text
HALPHA-<类型>-<序号>-<英文短标题>.<语言>.md
~~~

其中：

- `HALPHA-<类型>-<序号>` 是稳定文档编号，也是文档身份；
- `英文短标题` 只使用小写英文字母、数字和连字符（kebab-case），用于快速说明文档领域，不参与确定文档身份；
- 同一文档的所有语言版本使用相同英文短标题；
- 标题措辞小幅变化不要求改名；文档职责实质变化时，应评估新建文档编号；
- `语言` 使用 BCP 47 标准语言标记，例如 `zh-CN`、`en-US`；
- 版本写入元数据，不写入根目录当前文件名；历史版本由归档路径或归档文件名标明。

语言中立的机器可读登记沿用相同文档编号和英文短标题，可在扩展名前增加用途标记，例如 `.bundle.yaml`；它不得承载只存在于一种语言中的自然语言规范。

### 当前文档类型代码

文档编号中的类型代码必须使用下表的唯一写法。代码是文档身份的一部分，大小写敏感；不得为同一英文含义另造缩写或在正文、锚点、登记和文件名之间混用。`TRADEPLAN` 是“交易计划”复合代码，`PLAN` 只保留给 L4“当前建设计划”；`PLN` 不再使用。

| 类型代码 | 英文含义 | 中文职责 |
|---|---|---|
| `CON` | Constitution | 项目宪法 |
| `DOC` | Documentation architecture | 文档架构 |
| `VIS` | Vision | 目标与愿景 |
| `FLOW` | Core workflows | 核心业务流程与用户路径 |
| `ARC` | Architecture | 总体技术要求与架构 |
| `ALP` | Alpha research | Alpha 研究、经济证据与策略 |
| `CTX` | Context | 候选与决定上下文 |
| `DAT` | Data | 权威事实、市场数据与时间 |
| `CAP` | Capital | 资本、风险与权限 |
| `TRADEPLAN` | Trading plan | 交易计划与条件生命周期 |
| `POR` | Portfolio | 多项资金用途比较与组合边界 |
| `EXE` | Execution | 执行、保护、核对与恢复 |
| `OUT` | Outcomes | 结果、归因与学习 |
| `UX` | User experience | 用户交互与控制界面 |
| `SYS` | System | 系统组成与运行实体边界 |
| `ENG` | Engineering | 工程质量与构建边界 |
| `PLAN` | Current construction plan | L4 当前建设计划；固定入口仍为 `HALPHA-PLAN-001` |

文档类型代码不等同于英文短标题：短标题只用于阅读导航，类型代码决定唯一所有者和文档身份。新增类型必须先修改本表和责任登记，再创建文件；未列入本表的代码不得进入新的当前规范文件名或稳定语义锚点。历史归档或已接受版本中的 `PLN` 与 `OPS` 身份及其历史锚点可以保留，但候选和新版本不得继续建立这些已退役身份；迁移时必须显式记录身份变化，不得同时写入两套新旧编号。

### 中文术语表述

中文规范正文优先使用能够直接说明业务含义的常用中文，不为已有直白说法另造抽象名词或隐喻。例如，“资本包络”应写成“资金使用上限与范围”，“最终资本门”应写成“提交前再次检查资金与权限”，“动作物化”应写成“形成并保存动作记录”。

英文缩写或英文名称只在外部标准、文档编号、领域编号、代码标识或确有必要保持一致的技术名词中使用；首次出现时应给出中文含义。L3 可以保留实际字段名或类型名，但必须明确对应的中文业务语义，不能用字段名代替业务定义。同一概念在各文档中使用同一中文名称；程度差异另行说明，不写入领域或对象名称。

### 语义所有者与项目所有者的区分【DOC-SEM-001-DEF】

“项目所有者”表示承担项目建设、设计和建设计划责任的真人主体；“用户”表示投入交易资本并使用产品、作出产品使用和资本控制决定的真人主体；同一真人可以兼任二者，但语义不能合并。“开发者”表示执行开发、验证或发布工作的主体，可以是 AI 工具或项目所有者，但不是用户。“语义所有者”“主要语义所有者”“协调所有者”表示文档或 L2/L3 的责任归属，不表示真人、账号、运行进程或授权。责任归属只说明谁定义和维护含义，不能作为实际行为主体；实际行为必须由用户、项目所有者、开发者、Halpha、后台任务、执行器或具体外部系统执行。元数据或表格出现这些名称时，必须根据上下文写出完整角色名称。

### 概念、行为主体与记录要求【DOC-SEM-001-REQ】

1. 具有稳定特殊含义的概念必须先由最高适当层级和唯一语义所有者定义，再被其他章节引用；未定义的特殊概念不得作为要求、状态、对象或理由使用。
2. 常见词语保持通常含义，不为局部设计赋予隐藏的项目含义。确需区分时使用带限定条件的直白名称；只出现一次且没有独立消费者的内容直接用普通语句说明，不建立概念。
3. 概念名脱离原文档、章节标题和领域编号后，仍必须明确它描述的业务对象和作用。“候选、决定、状态、记录、结果、方式、通知”等通用名词必须带必要的对象或用途限定；跨文档引用使用同一完整名称，不依赖引用处重新声明或临时改名。
4. 领域编号只表示谁拥有、定义或负责某项语义，不作为实际行为主体。形成、读取、提交、停止、核对或通知等行为必须明确由用户、项目所有者、开发者、Halpha、后台任务、执行器或具体外部系统完成。
5. 新概念、对象、记录行为及其产生的数据必须说明真实消费者、改变的决定或失败行为和删除条件；没有消费者，或消费价值不足以覆盖理解、实现、运行和维护成本时，不创建或不记录。
6. L2 只规定会改变决定、责任或失败处理的记录语义；具体字段、存储和传递形式进入 L3，当前实际来源、配置、限制和例外进入 L4。

只有 L0 和 L1 维护 zh-CN 与 en-US 共同规范文本及 bundle。L2、L3、L4、责任登记和导航索引只维护 zh-CN，不创建 en-US 对应文件；外部标准、代码标识和必要技术名词仍可按正文术语规则使用英文。

## 2.2 最小元数据【DOC-MET-001】

除 L0 没有上位文档外，每份规范文档至少声明：

- 文档编号；
- 版本；
- 状态；
- 层级；
- 语言版本；
- 上位文档或条款；
- 本文档负责；
- 本文档不负责。

L0 以适用范围和权威范围代替普通的上位、负责与不负责字段。其他文档的上位、负责和不负责范围由 Markdown 文档头拥有，不在机器登记中重复维护。

L0/L1 共同规范文本的文档头还必须声明共同规范集标识、配对文本、共同规范集登记和生效时间。共同规范 bundle 是批准、替代、语言权威、对齐状态、正文摘要和联合摘要的唯一机器登记；它必须声明 schema 版本、文档编号与版本、共同规范集标识、状态、对齐状态、共同规范语言顺序与同等权威、正文路径与摘要、联合摘要、批准人、接受时间和替代版本。只有延迟生效时才在 bundle 另列生效时间，否则生效时间等于接受时间。

文档编号、版本、状态、语言、共同规范集标识和生效时间是文本头与 bundle 的必要绑定字段，必须相互一致。标题、上位关系、逐文件共同规范角色、独立 AI 使用条件和语言最终解释权不在 bundle 重复声明；共同规范语言同等权威由一个 bundle 级字段表达。

schema 3 bundle 的摘要必须按以下固定规则重算：

1. 读取每份规范正文文件的原始字节，按带可选 UTF-8 BOM 的 UTF-8 解码；把 `CRLF` 和单独的 `CR` 统一为 `LF`，不进行其他空白折叠、删减或 Unicode 归一化。
2. 规范正文从第一行符合“一级 Markdown 标题、空白、十进制章节号，且章节号后紧接句点或空白”的编号章节标题开始，包含该标题行直到文件末尾；文本头和它之前的标题不进入摘要。正文末尾是否存在换行属于摘要输入，不得补写或删除。
3. 将规范正文按 UTF-8 编码后计算 SHA-256；`files.<language>.body_sha256` 保存小写十六进制摘要。`files` 的语言键顺序必须与 `normative_languages` 完全一致。
4. 联合摘要输入依次为文档编号、`LF`、版本、`LF`，再按 `normative_languages` 顺序为每种语言追加 `<language>:<body_sha256><LF>`。对该完整 UTF-8 输入计算 SHA-256，并把小写十六进制摘要保存为 `joint_set.sha256`。

schema 3 bundle 不重复保存上述摘要规则或可重建的联合摘要输入；任何实现只要遵循本节规则，都必须能够仅用正文、文档编号、版本和规范语言顺序重算正文摘要与联合摘要。

L2/L3 当前单语言 ACCEPTED 规范在文档头另行声明批准人、接受时间和替代版本。只有延迟生效时才增加生效时间。通用“日期”、由状态和层级即可推出的“当前效力”、重复的设计接受说明，以及 P 阶段、当前建设选择、进度或证据不得写入 L1–L3 元数据；实现状态只从 L4 读取。

状态沿用 PROPOSED、ACCEPTED、SUPERSEDED 和 WITHDRAWN。该生命周期用于 L0–L3 规范文档，以及确需批准的 L4 当前约定或 ADR。

L4 时间记录必须使用与内容类型相匹配的时间语义：事实说明观察时点，当前计划和摘要说明截止或决定时点，当前实例或配置说明生效时点；只有确有时效边界时才记录失效或复核时点，不为形式完整填写无意义信息。当前计划使用 `as_of`、批准人和接受时间；接受后立即生效时不重复记录生效时间。机器登记和当前计划不得保存指向自身文件的路径字段。具体字段名和类型由相应机器规范定义。

## 2.3 文件位置【DOC-LOC-001】

~~~text
docs/
  L0/
  L1/
  L2/
  L3/
  L4/
  proposals/
~~~

目录规则只有三条：

- `L0`–`L4` 保存各层当前定稿文档及其机器可读配套文件；
- 需要保留的历史版本放入对应层级的 `archive/` 子目录；
- `proposals/` 以单层目录保存尚未定稿的候选和参考输入，其中内容没有规范效力。候选获批后移入对应层级，不再使用的内容可以删除。

# 3. AI 最小读取规则【DOC-AIR-001】

AI 处理任务时：

1. 状态任务从 L4 当前建设计划进入；缺少记录时回答“未知”，不得从代码或长期设计推断现状。
2. 实现任务从当前建设范围进入，先读取主要语义所有者对应的领域 L3，包括其中长期采用的第三方能力及使用边界，再读取该 L3 的直接依赖、实际适用的纵向 L2、必要 L1，以及 L4 中精确版本、当前配置和资格证据；只有确有编排 L3 时才读取协调所有者和参与者对应的文档。未被真实依赖的文档不读取。
3. 正式任务只使用当前 ACCEPTED 文档；`proposals/` 仅用于明确的起草、评审、修订或隔离原型。
4. L1–L3 不能说明当前状态，L4 不能补写产品或系统规则；语义缺失或当前定稿相互冲突时停止实现并报告。
5. 普通任务读取直接引用的 L0 条款；规范批准、总体复杂度方向、真实资金使用上限或适用范围变化时读取当前工作语言的 L0 全文。

# 4. 进入实现【DOC-IMP-001】

项目进入正式实现或持久集成前必须满足：

- 四篇 L1 均有当前 ACCEPTED 版本；
- 当前建设范围已登记在 L4，并引用适用设计、依赖和阻断；
- 完整 L2 责任地图已登记在非规范机器索引 `docs/L2/l2-responsibility-map.registry.yaml`；它只需列出当前责任的身份、形态、规范文档和少量稳定语义，不复制正文、不维护复杂跨领域关系，也不记录当前深度；
- 当前建设范围所需的主要 L2、直接依赖、适用纵向 L2 和 L3 均有当前 ACCEPTED 版本，并达到 L4 声明的必要深度；若该范围确有编排 L3，再要求其协调与参与领域满足同样条件；
- L3 未引入上述 L2 中没有依据的稳定规则；稳定验收语义已经由 L2/L3 明确，当前事实、配置、支持与不支持范围已经由 L4 记录。
- 对拟自研能力已按 ENG 完成成熟第三方能力检索、差异分类与复杂度比较；L3 已删除可直接复用的内部重复设计并界定最小补充，L4 已记录当前精确依赖和进入实现所需的资格证据。证据不足时保持阻断或明确不支持，不以平行自研替代资格缺口。

不要求预先把全项目所有 L2 写到同等深度，也不要求预先完成全项目 L3；完整的是职责覆盖，不是未来细节。未被当前建设范围消费的领域可以保持“L2 边界深度”，甚至长期不形成独立运行模块。隔离原型可以使用指定的提议版本，但进入正式集成前补齐当前建设范围的实际依赖。开始实现、测试完成和用户提高 Halpha 真实资金操作权限是不同事实，但不为此建立统一工程认可与准入体系。

# 5. 更新位置【DOC-UPD-001】

修改拥有该语义的最高层：使命和最高边界改 L0；文档规则改 L1-A；产品方向改 L1-B；业务路径改 L1-C；总体技术改 L1-D；领域原则改 L2；详细设计规则及长期采用的具体库或框架与使用契约改 L3；精确版本、当前配置、平台与许可证资格证据、当前计划和事实改 L4。修改后只复核受影响的下游。

# 6. 创建、拆分与复核【DOC-SPL-001】

L2 文档应当拥有一个可独立命名的横向核心业务组成或纵向领域；L3 文档应当拥有一个明确实现适用范围，并按类型声明主要语义所有者或协调所有者。当前或近期建设需求只决定是否起草或深化某份 L2/L3 文档及其设计范围；它不属于 L3 的阶段规划，不决定某项稳定职责是否存在，也不要求责任地图各领域保持对称篇幅、对象数量或实现投入。具体 P 阶段、当前建设范围和顺序由 L4 拥有。

不为每个任务、普通类、第三方库内部对象或可直接由公开契约、数据结构、类型和测试表达的内容建立长期文档。出现以下情况时调整边界：同一稳定语义被多篇拥有时合并或指定唯一语义所有者；一篇同时拥有多个可独立演进的职责时拆分；文档失去独立职责时合并或删除；领域深化成本持续高于真实消费价值时收缩到较浅深度。已接受文档失去独立职责时，必须通过协调 proposal 把该版本标记为 `SUPERSEDED` 并归档，同时迁移全部直接消费者；不得用 `WITHDRAWN` 抹去曾经生效的版本。边界或深度调整后必须复核全部受影响 L3 的语义所有者、直接横向依赖和纵向约束。

## 6.1 已接受文档的小幅修改【DOC-CHG-001】

已接受文档的小幅修改可以直接形成新的 `ACCEPTED` 版本，不要求先建立 proposal。小幅修改按语义影响判断，不按改动行数判断；它必须同时满足以下条件：

- 保持文档身份、层级、唯一语义所有者、负责范围和不负责范围不变；
- 不改变规范强度、授权、允许或禁止的动作、正常/失败/未知/停止/恢复/结束行为、验收标准或外部影响；
- 不新增、替换或删除稳定概念、对象、状态、工作流、角色、权限、支持范围、依赖方向或迁移要求；
- 只进行错别字、排版、导航、元数据、链接修正，标题措辞微调，或不改变决定与失败处理的澄清。

直接修改不等于静默修改。新版本仍必须记录版本和替代关系，取得项目所有者批准，完成直接下游和索引同步，并通过适用验证；共同规范文本还必须同步全部语言正文、bundle 和哈希后一起生效。

## 6.2 必须建立 proposal 的修改【DOC-CHG-002】

大幅修改和新建场景必须先建立 proposal，经评审与项目所有者批准后才能进入正式路径。以下任一情况都属于该范围：

- 改变文档身份、层级、唯一语义所有者、责任边界、依赖方向或规范强度；
- 改变授权、允许或禁止的动作、正常/失败/未知/停止/恢复/结束行为、验收标准、外部影响或迁移要求；
- 新增、替换或删除稳定概念、对象、状态、工作流、角色、权限、支持范围或规范场景；
- 新建、拆分、合并、重命名或撤回规范文档，或者建立新的 L3 适用范围、L4 阶段性场景或 ADR 决策场景。

proposal 的目标层级、候选基线、协调候选集、无当前效力和正式接受条件必须明确。无法可靠判断是小幅修改还是大幅修改时，按大幅修改处理并建立 proposal。
