# 0. 总体结构【DOC-STR-001】

## 文档层级结构定义【DOC-STR-001-DEF】

~~~text
                         L1-A 文档架构
                         （只管理文档）
                                │
L0 宪法 → L1-B 目标与愿景 → L1-C 业务流程与用户路径
                                ↓
                       L1-D 总体技术要求与架构
                                ↓
                L2 领域关键内容与设计原则
                                ↓
                       L3 领域长期稳定的详细设计
                                ↓
                     L4 当前实施与事实记录
~~~

## 层级使用要求【DOC-STR-001-REQ】

使用规则只有三条：

1. 下层服从上层，同一主题只在最高拥有层定义一次。
2. L0–L3 定义应当怎样；L4 记录当前做什么和实际是什么。
3. 设计变化先修改拥有该语义的文档，再复核受影响的下游。

# 1. 层级定位与职责【DOC-LVL-001】

层级首先按“哪一层拥有哪类设计语义”划分，不按时间划分。长期或当前只是用来判断设计是否稳定、是否已进入落地，不单独构成一层职责。

## 1.1 各层定位、唯一职责与落地区分【DOC-OWN-001-DEF】

| 层级 | 核心定位 | 唯一职责 |
|---|---|---|
| L0 | 项目级原则 | 定义长期使命、最高边界、不可突破的原则和最高取舍。 |
| L1 | 总体原则 | 分别定义文档、产品、核心路径和总体技术的长期方向、原则与演进条件。 |
| L2 | 领域关键内容 | 定义每个领域要解决的关键问题、领域目标、关键能力、责任边界和设计原则；横向核心业务组成与纵向领域只说明责任形态，不改变这项职责。 |
| L3 | 长期稳定的领域详细设计 | 以 L2 为依据，定义当前消费者需要长期复用的模块边界、接口、状态、失败结果、测试要求和具体组件使用契约；可以限定设计范围，但不拥有当前精确版本、配置、进度或启用状态。 |
| L4 | 当前实施与事实记录 | 记录当前实施目标、范围、必要顺序与依赖、精确版本和实例选择、配置、进度、直接验证结果、已知限制和实际事实。 |

L0–L3 形成长期且可复用的原则、领域内容和详细设计。L4 不是重写这些设计，而是记录当前适用范围、实施目标和实际结果；它不得新增或改变 L0–L3 的稳定规则。当前实施标识、范围、顺序、进度和验证结果只在 L4 记录。

## 1.3 L1 的依赖顺序【DOC-L1-001】

四篇 L1 文档始终按以下语义约束顺序建立和维护：

1. HALPHA-DOC-001-documentation-architecture：确定文档形式。
2. HALPHA-VIS-001-goals-and-vision：确定长期产品目标和方向。
3. HALPHA-FLOW-001-core-workflows-and-user-journeys：选择达成目标的长期核心路径。
4. HALPHA-ARC-001-technical-requirements-and-architecture：选择支撑这些路径的长期总体技术方案。

后者服从前者，但分别拥有产品、流程和技术语义。四篇文档可以并行起草；修改和引用必须按上述约束顺序读取直接上位文档。

## 1.4 L2–L4 的使用【DOC-L24-001】

### L2–L4 对象与分类定义【DOC-L24-001-DEF】

- **L2 稳定语义责任地图：** L1 遗留给下位细化的每项稳定语义都有一个明确的 L2 语义所有者；它描述责任覆盖，不代表等量设计或实现。
- L2 始终说明唯一职责、不负责范围和失败边界；只增加当前消费者会使用的对象与规则。多个真实消费者重复需要时才抽取公共生命周期或复用规则，不设置额外状态或升级流程。

L2 采用两个相互独立的职责维度：

| 形态 | 负责内容 | 不负责内容 |
|---|---|---|
| 横向核心业务组成 | 一段连续业务、业务对象的完整生命周期、决定权及其前后交接；该业务可以由一个或多个模块实现 | 跨全部业务重复定义同一种数据、交互、架构、工程或运行要求 |
| 纵向领域 | 贯穿多个业务组成和模块的一类共同约束、质量属性或技术职责 | 重新定义横向业务对象、业务决定及其生命周期 |

L2 按业务或共同约束划分稳定责任，不按软件模块划分责任。模块边界由 L3 在 SYS 的系统组成与运行边界约束下定义；L2 只提供模块必须实现的稳定业务语义和责任边界。

L3 分为两类：领域 L3 指定一个主要语义所有者并精确实现其 L2 语义；编排 L3 指定协调所有者，只拥有必要身份、顺序、责任交接和完成汇总。长期采用的具体第三方库或框架属于对应模块的 L3 实现方案，但第三方不取得 Halpha 的产品语义、事实或决定所有权。

L4 使用以下记录类型：

| 类型 | 记录内容 |
|---|---|
| 当前实施 | 目标、范围、必要顺序与依赖、阻断、进度和直接验证结果 |
| 事实、实例与配置 | 来源和时间、活动版本、第三方依赖的精确版本与构建标识、部署、当前参数和平台验证结果 |
| 支持决定 | 真实消费者、当前支持与不支持范围、必要理由和带日期的当前摘要 |

### L2–L4 使用要求【DOC-L24-001-REQ】

L2 必须先说明领域关键问题、唯一职责、不负责范围和失败结果，再按当前消费者需要补足对象、决定、交接和验收。责任地图中的领域不要求对称篇幅、对象数量或实现投入；未选择的能力可以人工处理、使用外部工具或明确不支持。

新增规则、对象或公共生命周期前，必须说明当前消费者、它改变的决定或失败结果、为何更小方案不足、个人维护成本和删除条件。体系对称、文档整齐和未来可能性都不是理由。L4 只记录当前支持范围、实际选择和结果。

同一稳定语义只能有一个语义所有者。其他 L2 可以声明输入条件、消费方式和缺失时行为，但不得复制拥有者的规则、状态体系或完整流程。生产者只定义自身输出，消费者只定义接入要求；文档之间不建立全连接式交接说明。

一个设计涉及多个 L2 不自动构成编排 L3。默认以领域 L3 加直接依赖表达；只有重复出现、需要稳定身份且无法由现有领域承担的跨领域生命周期，才使用 `ORCHESTRATION`。

每篇 L3 必须声明类型和语义所有者，列出直接依赖及适用纵向约束，追溯主要业务语义和跨领域要求，并将已有 L2 原则落实为长期稳定的详细设计。实际消费者只决定是否需要起草或扩展详细设计；当前实施标识、范围、建设顺序、进度和验证结果不得进入 L3。若实现所需稳定规则在 L2 中没有依据，必须先修改拥有该语义的 L2。

经过 ARC 与 ENG 的能力检索和差异判定后，L3 可以把具体第三方库或框架记录为模块的长期实现方案，并规定 Halpha 怎样调用、输入输出、事实权威、必要转换、失败结果、兼容边界和确有必要的最小补充。精确版本、构建标识、当前配置、目标操作系统与许可证检查、启用状态和实际验证结果只由 L4 记录。

第三方库已经完整承担模块内部能力时，L3 主要说明 Halpha 的使用方式、边界、失败与退出，不复制供应商内部类、算法、缓存、状态机、网络生命周期或数据结构，也不为了文档对称虚构一套 Halpha 内部实现。L3 不得同时保留第三方实现与自研实现作为静默、自动或长期平行替代；组件不可用时采用停止、外部人工接管或明确不支持等上位允许结果。

当前实施计划的唯一入口固定为：

~~~text
docs/L4/HALPHA-PLAN-001-current-construction-plan.yaml
~~~

该入口记录当前范围、进度、配置和验证结果。L4 不能改变 L0–L3；缺少记录时，当前状态视为未知。

# 2. 文档格式与存放位置【DOC-FMT-001】

## 2.1 命名【DOC-NAM-001】

规范文件使用：

~~~text
HALPHA-<类型>-<序号>-<英文短标题>.<语言>.md
~~~

其中：

- `HALPHA-<类型>-<序号>` 是稳定文档编号，也是文档身份；
- `英文短标题` 只使用小写英文字母、数字和连字符（kebab-case），用于快速说明文档领域，不参与确定文档身份；
- 同一文档的所有共同规范语言文本使用相同英文短标题；
- 标题措辞小幅变化不要求改名；文档职责实质变化时，应评估新建文档编号；
- `语言` 使用 BCP 47 标准语言标记，例如 `zh-CN`、`en-US`；

### 文档类型代码

文档编号中的类型代码必须使用下表的唯一写法。代码是文档身份的一部分，大小写敏感；不得为同一英文含义另造缩写或在正文、锚点、登记和文件名之间混用。`TRADEPLAN` 是“交易计划”复合代码，`PLAN` 只保留给 L4“当前建设计划”。

| 类型代码 | 英文含义 | 中文职责 |
|---|---|---|
| `CON` | Constitution | 项目宪法 |
| `DOC` | Documentation architecture | 文档架构 |
| `VIS` | Vision | 目标与愿景 |
| `FLOW` | Core workflows | 核心业务流程与用户路径 |
| `ARC` | Architecture | 总体技术要求与架构 |
| `ALP` | Alpha research | Alpha 研究、经济证据与策略 |
| `DAT` | Data | 权威事实、市场数据与时间 |
| `CAP` | Capital | 资本、风险与权限 |
| `TRADEPLAN` | Trading plan | 交易计划与条件生命周期 |
| `EXE` | Execution | 执行、保护、核对与恢复 |
| `OUT` | Outcomes | 结果、归因与学习 |
| `UX` | User experience | 用户交互与控制界面 |
| `SYS` | System | 系统组成与运行实体边界 |
| `ENG` | Engineering | 工程质量与构建边界 |
| `PLAN` | Current construction plan | L4 当前建设计划；固定入口仍为 `HALPHA-PLAN-001` |

文档类型代码不等同于英文短标题：短标题只用于阅读导航，类型代码决定唯一所有者和文档身份。新增类型必须先修改本表和责任登记，再创建文件；未列入本表的代码不得进入规范文件名或稳定语义锚点。

### 中文术语表述

中文规范正文优先使用能够直接说明业务含义的常用中文，不为已有直白说法另造抽象名词或隐喻。例如，“资本包络”应写成“资金使用上限与范围”，“最终资本门”应写成“提交前再次检查资金与权限”，“动作物化”应写成“形成并保存动作记录”。

英文缩写或英文名称只在外部标准、文档编号、领域编号、代码标识或确有必要保持一致的技术名词中使用；首次出现时应给出中文含义。L3 可以保留实际字段名或类型名，但必须明确对应的中文业务语义，不能用字段名代替业务定义。同一概念在各文档中使用同一中文名称；程度差异另行说明，不写入领域或对象名称。

### 语义所有者与项目所有者的区分【DOC-SEM-001-DEF】

“项目所有者”表示承担项目建设、设计和建设计划责任的真人主体；“用户”表示投入交易资本并使用产品、作出产品使用和资本控制决定的真人主体；同一真人可以兼任二者，但语义不能合并。“开发者”表示执行开发、验证或发布工作的主体，可以是 AI 工具或项目所有者，但不是用户。“语义所有者”“主要语义所有者”“协调所有者”表示文档或 L2/L3 的责任归属，不表示真人、账号、运行进程或授权。责任归属只说明谁定义和维护含义，不能作为实际行为主体；实际行为必须由用户、项目所有者、开发者、Halpha、后台任务、执行器或具体外部系统执行。元数据或表格出现这些名称时，必须根据上下文写出完整角色名称。

### 概念、行为主体与记录要求【DOC-SEM-001-REQ】

1. 具有稳定特殊含义的概念必须先由最高适当层级和唯一语义所有者定义，再被其他章节引用；未定义的特殊概念不得作为要求、状态、对象或理由使用。
2. 常见词语保持通常含义，不为局部设计赋予隐藏的项目含义。确需区分时使用带限定条件的直白名称；只出现一次且没有独立消费者的内容直接用普通语句说明，不建立概念。
3. 概念名脱离原文档、章节标题和领域编号后，仍必须明确它描述的业务对象和作用。“候选、决定、状态、记录、结果、方式、通知”等通用名词必须带必要的对象或用途限定；跨文档引用使用同一完整名称，不依赖引用处重新声明或临时改名。
4. 领域编号只表示谁拥有、定义或负责某项语义，不作为实际行为主体。形成、读取、提交、停止、核对或通知等行为必须明确由用户、项目所有者、开发者、Halpha、后台任务、执行器或具体外部系统完成。
5. 新概念、对象、记录行为及其产生的数据必须说明真实消费者、改变的决定或失败行为和删除条件；没有消费者，或消费价值不足以覆盖理解、实现、运行和维护成本时，不创建或不记录。
6. L2 只规定会改变决定、责任或失败处理的记录语义；具体字段、存储和传递形式进入 L3，当前实际来源、配置、限制和例外进入 L4。

只有 L0 和 L1 维护 zh-CN 与 en-US 共同规范文本。两份文本同等权威、各自完整且无优先语言。L2、L3、L4、责任登记和导航索引只维护 zh-CN，不创建 en-US 对应文件；外部标准、代码标识和必要技术名词仍可按正文术语规则使用英文。

固定 L1 文件名如下：

|---|---|---|
| 设计文档架构 | `HALPHA-DOC-001-documentation-architecture.zh-CN.md` | `HALPHA-DOC-001-documentation-architecture.en-US.md` |
| 目标与愿景 | `HALPHA-VIS-001-goals-and-vision.zh-CN.md` | `HALPHA-VIS-001-goals-and-vision.en-US.md` |
| 核心业务流程与用户路径 | `HALPHA-FLOW-001-core-workflows-and-user-journeys.zh-CN.md` | `HALPHA-FLOW-001-core-workflows-and-user-journeys.en-US.md` |
| 总体技术要求与架构 | `HALPHA-ARC-001-technical-requirements-and-architecture.zh-CN.md` | `HALPHA-ARC-001-technical-requirements-and-architecture.en-US.md` |

## 2.2 最小元数据【DOC-MET-001】

除 L0 没有上位文档外，每份规范文档至少声明：

- 文档编号；
- 层级；
- 语言版本；
- 上位文档或条款；
- 本文档负责；
- 本文档不负责。

L0 以适用范围和权威范围代替普通的上位、负责与不负责字段。L0/L1 共同规范文本的文档头还必须声明配对文本；两份文本共享稳定文档编号、章节结构和语义锚点。其他文档的上位、负责和不负责范围由 Markdown 文档头拥有，不在机器登记中重复维护。

文档头按本节声明结构。当前实施标识、选择、进度或验证结果不得写入 L1–L3 元数据；实现状态只从 L4 读取。

L4 时间记录必须使用与内容类型相匹配的时间语义：事实说明观察时点，当前计划和摘要说明截止或决定时点，当前实例或配置说明适用起点；只有确有时效边界时才记录失效或复核时点，不为形式完整填写无意义信息。机器登记和当前计划不得保存指向自身文件的路径字段。具体字段名和类型由相应机器规范定义。

## 2.3 文件位置【DOC-LOC-001】

~~~text
docs/
  L0/
  L1/
  L2/
  L3/
  L4/
~~~

每个稳定文档编号在对应层级的目标路径维护正文；该正文表达该文档的设计，实际 Git commit 用于历史追溯、比较和恢复。只有产品或当前事实的真实消费者需要时，才维护机器可读配套文件。

# 3. AI 最小读取规则【DOC-AIR-001】

AI 处理任务时：

1. 状态任务从 L4 当前建设计划进入；缺少记录时回答“未知”，不得从代码或长期设计推断现状。
2. 实现任务从当前目标和范围进入，先读取主要语义所有者对应的领域 L3，再读取该 L3 的直接依赖、实际适用的纵向 L2、必要 L1，以及 L4 中精确版本、当前配置和验证结果；只有确有编排 L3 时才读取协调所有者和参与者对应的文档。未被真实依赖的文档不读取。
3. 设计与实现任务读取对应的目标文档；所需文档缺失时报告缺口，不以推断补足。
4. L1–L3 不能说明当前状态，L4 不能补写产品或系统规则；语义缺失或当前定稿相互冲突时停止实现并报告。
5. 普通任务读取直接引用的 L0 条款；修改规范治理、总体复杂度方向、真实资金使用上限或适用范围时读取当前工作语言的 L0 全文。

# 4. 进入实现【DOC-IMP-001】

开始实现只需满足与该结果直接相关的条件：

- 当前设计能够确定产品结果和唯一语义所有者；
- L4 已记录当前目标、范围、精确选择和已知阻断；
- 当前消费者所需的 L2/L3 规则足以判断正常与失败结果，且 L3 没有创造 L2 缺失的稳定语义；
- 新增或替换通用能力时，已按 ENG 比较成熟能力并选择当前可维护的最小方案。

不要求预先设计无当前消费者的能力，也不要求为了实现一个结果补齐全项目文档、统一工程认可或准入体系。设计缺口只阻断依赖它的行为；路径、数据库、凭据和外部写影响相互独立的工作可以继续。开始实现、测试完成和用户提高 Halpha 真实资金操作权限仍是不同事实。

# 5. 更新位置【DOC-UPD-001】

修改拥有该语义的最高层：使命和最高边界改 L0；文档规则改 L1-A；产品方向改 L1-B；业务路径改 L1-C；总体技术改 L1-D；领域原则改 L2；详细设计规则及长期采用的具体库或框架与使用契约改 L3；精确版本、当前配置、平台与许可证验证结果、当前计划和事实改 L4。修改后只复核受影响的下游。

# 6. 创建、拆分与复核【DOC-SPL-001】

L2 文档应当拥有一个可独立命名的横向核心业务组成或纵向领域；L3 文档应当拥有一个明确实现适用范围，并按类型声明主要语义所有者或协调所有者。实际消费者只决定是否起草或扩展某份 L2/L3 文档及其设计范围，不决定某项稳定职责是否存在，也不要求责任地图各领域保持对称篇幅、对象数量或实现投入。当前范围、顺序和进度由 L4 拥有。

不为每个任务、普通类、第三方库内部对象或可直接由公开契约、数据结构、类型和测试表达的内容建立长期文档。出现以下情况时调整边界：同一稳定语义被多篇拥有时合并或指定唯一语义所有者；一篇同时拥有多个可独立演进的职责时拆分；文档失去独立职责时合并或删除；维护成本持续高于真实消费价值时删减到仍有用的最小范围。文档失去独立职责时，必须在同一工作变更中修改受影响的目标文档并迁移全部直接消费者。边界或范围调整后必须复核全部受影响 L3 的语义所有者、直接横向依赖和纵向约束。

## 6.1 直接修改与影响验证【DOC-CHG-001】

所有文档变化都直接修改稳定文档编号对应的目标文件。验证强度按语义影响判断，不按改动行数判断：

- 不改变身份、层级、所有者、责任、规范强度、行为、失败处理、验收、外部影响、稳定概念、依赖、支持范围或迁移的变化，使用轻量检查；
- 改变其中任一语义的变化，必须读取直接上位、相邻所有者和直接消费者，覆盖关键正常与反例，并按实际影响完成独立复核；
- 新建、拆分、合并、重命名或删除文档时，必须证明独立职责、迁移直接消费者，并删除失真的旧引用。

直接修改不等于静默修改。必须同步受影响的直接下游、责任登记和索引，并完成适用验证。L0/L1 的任何含义或结构变化必须在同一工作变更中同步两份语言正文。

## 6.2 跨文档同步【DOC-CHG-002】

跨层或跨文档变化先修改拥有该语义的最高文档，冻结含义后再同步必要的直接消费者、责任登记、索引和 L4 当前计划。无法在授权范围内完成必要同步时，停止并报告缺口。
