# Halpha 设计文档架构

**文档编号：** HALPHA-DOC-001  
**层级：** L1-A  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001  
**本文档负责：** 文档层级、唯一所有者、当前正文、历史和最小维护规则  
**本文档不负责：** 产品、业务流程、技术方案和当前实施内容

## 1. 一份当前正文

每个文档编号只保留一份中文正文。文件存在即表示当前适用；直接修改该文件，Git commit 保存历史、比较与回退。

不得建立翻译配对、proposal、accept、bundle、archive、状态副本、责任登记或导航索引。代码、测试、配置和当前记录不能成为平行设计权威。

## 2. 文档层级

| 层级 | 唯一职责 |
|---|---|
| L0 | 长期使命、不可突破边界、最高取舍和根本变更条件 |
| L1 | 长期总体文档规则、产品定位、核心路径和总体技术方向 |
| L2 | 单一领域的稳定职责、原则、交接和失败边界 |
| L3 | 有实际消费者并需要长期复用的详细设计和组件使用契约 |
| L4 | 当前工作焦点、精确选择、配置、事实、限制和直接结果 |

L0–L3 定义长期应当怎样；L4 记录当前正在处理什么和实际是什么。L4 始终只保留一个滚动的当前焦点，焦点变化时直接更新，历史由 Git 保存。

## 3. 唯一所有者

同一稳定语义只由最高适当层级的一个文档定义。其他文档只说明输入、消费方式和缺失结果，不复制对象生命周期、完整状态、授权、重试或恢复规则。

L2 按实际领域划分责任；L3 只细化已有 L2 语义。是否存在某个 L2/L3 由真实消费者决定，不要求领域数量、篇幅和实现投入对称。

## 4. 当前计划

当前工作的唯一文档入口是 `docs/L4/HALPHA-PLAN-001-current-plan.yaml`。它记录当前焦点和事实，不新增稳定规则。缺少来源、时间或记录时，当前状态为未知。

代码、配置、锁文件、迁移和 Git 已直接拥有的详细事实，L4 原则上只引用路径和当前结论，不复制完整内容。

## 5. 文件和最小元数据

规范文件使用 `HALPHA-<类型>-<序号>-<英文短标题>.zh-CN.md`。类型和序号只用于稳定识别，不构成额外审批或生命周期。

除 L0 外，文档头至少说明文档编号、层级、语言、上位、负责和不负责范围。L3 另说明主要所有者、直接依赖和实际适用约束。章节使用普通中文标题，不要求语义锚点或定义/要求/依据分类。

## 6. 修改与验证

修改先确定最高层级和唯一所有者，完整读取目标与真正受影响的直接上下游。先删除重复和无消费者内容，再修改拥有者并同步失真的直接引用。

验证强度按实际影响决定：普通文案作者检查；业务和实现边界运行相关测试；真实动作、资金、停止、接管和跨域冲突从正常路径与关键反例复核。不得为一次修改新建状态、登记或治理平台。

机械校验只证明编码、格式、链接和 YAML 可读，不证明设计正确。
