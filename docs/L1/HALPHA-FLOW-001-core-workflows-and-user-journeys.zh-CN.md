# 0. 流程定位与核心承诺【FLOW-SUM-001】

## 0.1 流程选择依据【FLOW-SUM-001-RAT】

Halpha 的日常主线是帮助用户形成判断、比较方案、制定计划、正确行动、看清结果并持续改进，而不是让用户先完成一套独立的控制或治理流程。

用户可以谨慎观察、频繁干预和随时停用。即使长期运行表现良好，流程也不得以无人值守为成立前提，或让用户失去停止和外部接管能力。

## 0.2 核心流程要求【FLOW-SUM-001-REQ】

可供 Halpha 使用的账户和资金规模由用户在系统外决定。FLOW 只规定用户在 Halpha 内怎样配置并使用资金使用上限与范围、授权、计划、停止和恢复，不把个人总投入决策设计成产品流程。真实动作路径分别落实 CON-CMP-003 的三类控制要求；辅助机制不得比核心交易任务更复杂。

流程设计必须满足：

- 用户进入后能够先看清“发生了什么、是否值得行动、Halpha 是否可用”；
- 常用任务少步骤完成，专业细节按需展开；
- 真实动作通常来自已固定并启用的完整交易计划；用户明确作出并已固定的已有风险敞口保护或降低风险决定，以及用户明确提交且可证明不增加或转换风险的撤销、保护、划转或降低风险指令可以不依赖交易计划，但必须固定对象、范围和原因，并进入同一资金使用上限与范围、Halpha 真实资金操作权限、执行与核对路径；
- 需要触发后人工判断的动作只在用户在线且完整人工授权路径在适用时限内可行时继续；机器授权路径必须在触发前取得明确、有界的授权；
- Halpha 发起的交易场所或账户变化只经过单一且隔离的外部写入边界；用户经官方入口独立操作按外部活动接入；
- 交易所模拟盘与真实资金环境使用同一交易计划、CAP 检查、EXE 执行、DAT 核对和 OUT 复盘流程；差异只通过明确的环境配置、账户与授权效力表达，不另建一套简化模拟流程；
- 事实未知、路径不可行或权限不足时保持等待、不交易或停用，不猜测成功；只有用户明确指示且可证明不增加或转换风险的收缩动作可以继续；
- 停止和外部人工接管始终可达。

# 1. 有限注意力下的路径连续性【FLOW-TIM-001】

FLOW 只规定产品级路径：用户既能快速处理当前状态和有时限决定，也能连续完成研究、计划和复盘；故障时优先停止影响、核对外部状态并取得人工接管。具体待办排序和交互时限由下位设计定义，实际表现由 L4 记录。

无异常时，用户可以直接进入最有价值的研究、计划或复盘任务。异常、有时限决定和可能改变交易场所或账户的动作必须可直达，但不把正常产品入口变成技术状态页。

## 1.1 中断与恢复

候选、研究和计划草案可以保存后继续。恢复时必须突出期间变化、过期输入和需要重做的决定。已经发生或可能发生的真实动作不得因恢复草案而重放；Halpha 必须先读取并核对外部状态。

# 2. 重复使用场景【FLOW-MAP-001】

| 场景 | 主要路径 | 允许的结束 |
|---|---|---|
| 快速状态检查 | 账户与持仓 → 活跃计划与未决动作 → Halpha 可用性 → 高价值待办 | 无事退出，或进入一个明确任务 |
| 候选捕获 | 保存来源与时点 → 补足必要信息 → 去重与筛选 | 等待、结束、进入研究或计划 |
| Alpha 研究 | 固定研究任务与评价边界 → AI 生成、运行、反证、筛选与复核 → 最终候选材料 → 人工最终审查 | 无合格候选、拒绝、返回新研究、继续观察或选择进入正式策略实现与资格验证 |
| 制定计划 | 判断依据 → 进入、退出、调整与保护 → 数量、成本、时效与失败处理 | 草案、固定并启用，或放弃 |
| 人工确认动作 | 当前预览 → 用户当前确认 → 资金使用上限与范围及 Halpha 真实资金操作权限检查 → 真实动作 → 核对 | 等待、拒绝、不交易、部分或完整结果、结果未决 |
| 固定策略运行 | 策略输入 → 完整计划 → 条件评价 → 当前事实与授权复核 → 动作与核对 | 不行动、暂停、缩小范围、继续或停用 |
| 保护或降低风险 | 已固定的保护/降低风险决定或用户明确指令 → 当前事实 → 资金使用上限与范围及 Halpha 真实资金操作权限检查 → 动作与核对 | 完成、拒绝、未知、外部接管或保持停用 |
| 外部人工操作回归 | 读取场所事实 → 标明外部来源 → 关联现有计划或后续决定 | 事实可被后续判断引用 |
| 短复盘 | 原决定 → 实际结果与成本 → 判断、计划、数据、执行和交互差异 | 无需改变，或形成可行动改进 |
| 故障与恢复 | 停用受影响范围的新真实动作 → 外部核对 → 修复或回退 → 核对恢复条件 | 保持停用，或由用户显式恢复适用的 Halpha 真实资金操作权限 |
| 停用与退出 | 停用新真实动作 → 看清遗留责任 → 撤销写权限与凭据 → 导出必要资料 | 外部账户保持独立可控 |

# 3. 外部工具与 Halpha 边界【FLOW-TOOL-001】

## 3.1 外部工具分工依据【FLOW-TOOL-001-RAT】

成熟外部工具优先承担其擅长的任务：交易场所负责官方账户操作，专业图表负责深度视觉分析，新闻和数据源提供原始材料，研究环境支持探索。通用 AI 可以在 ALP 已确定的研究与评价边界内，承担非权威的任务形成、假设生成、研究实现、运行编排、比较、筛选、反证、复核和材料整理；它不因此取得事实、经济证据判断、正式策略晋升、交易计划或真实动作权威。

## 3.2 Halpha 自持与交接要求【FLOW-TOOL-001-REQ】

Halpha 必须保持研究和计划上下文、当前采用的策略或计划、用户配置的资金使用上限与范围和 Halpha 真实资金操作权限、Halpha 发起的真实动作及未决责任、核对后的必要事实，以及从计划、动作和结果形成的学习。

外部能力足够时不重复建设。交接必须使用户看清对象、时点、适用计划、动作边界和返回方式；返回结果按外部事实核对，不能因用户点击“完成”就成为账户事实。

# 4. 用户交互与控制【FLOW-UX-001】

## 4.1 六类主要用户任务域

### 六类用户任务域定义【FLOW-UX-001-DEF】

以下是长期任务域，不是页面、菜单或软件模块名称。

| 用户任务域 | 用户要回答的问题 | 典型任务 |
|---|---|---|
| 市场观察与实时情报 | 市场、事件和关注对象发生了什么 | 观察行情与事件、发现线索、查看计划相关信号 |
| 策略研究与验证 | 某个观点或策略为什么可能成立 | 假设、数据探索、反证、回测和版本比较 |
| 交易计划与演练 | 如果行动，怎样进入、退出、保护、调整和停止 | 制定、比较、预览、模拟、固定和启用计划 |
| 交易执行与持仓管理 | 当前计划、订单或持仓需要什么行动 | 确认、执行、保护、调整、取消、退出和异常处理 |
| 账户与交易记录 | 客观上发生了什么，事实是否可核对 | 查询余额、订单、成交、持仓、费用和外部活动 |
| 回顾分析与学习 | 原判断和计划质量怎样，下一次改变什么 | 比较计划与实际、分析偏差、形成改进事项 |

### 用户任务域使用要求【FLOW-UX-001-REQ】

L3 可以把多个任务域组合在同一界面，也可以为高频任务提供独立入口，但必须保持任务上下文连续。“账户与交易记录”提供事实，“回顾分析与学习”提供有证据边界的解释；二者不能合并成一份只看盈亏的报告。

“交易计划与演练”把研究结果、用户判断或持仓处置需要变成可预览、可模拟、可固定和可启用的计划；“交易执行与持仓管理”承接已经触发或正在发生的动作与责任。一个下单按钮不能替代计划制定。

## 4.2 当前状态与待处理事项是跨域入口

用户进入 Halpha 时必须能看清当前订单和持仓、活跃计划、有时限决定、异常和重要变化，并从跨域入口直达相应计划、执行、事实、研究或复盘任务。L1 不固定该入口的页面名称或导航形态；没有待办时，用户可以退出或直接进入任何高价值任务。

## 4.3 真实使用路径允许直接进入和回流

一个完整的发现—行动—学习环可以表达为：

~~~text
市场观察与实时情报
        ↓（需要时研究，也可由充分的用户判断直接开始）
策略研究与验证 ───────────────┐
        ↓                      │
交易计划与演练                 │
        ↓                      │
交易执行与持仓管理             │
        ↓                      │
账户与交易记录                 │
        ↓                      │
回顾分析与学习 ────────────────┘ 返回情报、研究或新计划
~~~

这不是强制漏斗。已有持仓、保护需要和订单异常可以直接进入执行与持仓管理；用户可以直接查询事实；没有合格机会时以等待、保持现金或不交易结束；只有产生可行动问题时才需要回流。

## 4.4 典型任务路径

| 用户意图 | 合理路径 | 正常结束 |
|---|---|---|
| 快速确认是否有事 | 当前状态与待办 → 相关计划、持仓、订单、情报或异常 | 处理一个事项，或无事退出 |
| 把观点变成交易 | 用户判断或情报 → 可选研究 → 计划 → 演练 → 在交易所模拟盘或真实资金环境按同一交易链执行 | 不交易、继续观察、模拟执行或真实执行 |
| 响应活跃计划 | 通知或条件触发 → 当前事实与计划分支 → 确认、等待、拒绝或按已授权规则行动 | 决定被记录；动作进入跟踪与核对 |
| 管理已有订单或持仓 | 订单或持仓 → 依据、当前事实和未决责任 → 保护、调整、取消或退出 | 责任闭合，或转交外部人工处理 |
| 研究和验证策略 | 固定任务与判定依据 → AI 探索、反证、比较和筛选 → 样本外、复现与最终候选材料 → 人工最终审查 | 无候选、淘汰、返回新研究、观察或选择进入正式策略实现与资格验证 |
| 查询与复盘 | 事实时间线 → 核对状态 → 与原决定比较 | 退出，或形成新的研究、计划或改进事项 |

这些路径可以由搜索、保存视图、通知、深链接或对象关系启动。L1 要求上下文连续和结果可回查，不冻结页面数量。

## 4.5 命令必须清楚

确认当前拟执行动作、启用或修改计划、启用或停用固定策略、扩大或缩小资金使用上限与范围、停用新真实动作、进入外部人工接管、恢复适用的 Halpha 真实资金操作权限，必须保持为不同命令。降低资金使用范围或 Halpha 真实资金操作权限应快速完成；提高资金使用上限、扩大范围或提高 Halpha 真实资金操作权限时必须显示当前配置与后果，但不建立多层审批。单笔确认不得顺带改变长期资金使用上限与范围或 Halpha 真实资金操作权限。

## 4.6 信息层次

默认显示当前结论、关键依据、主要未知、下一动作和 Halpha 是否可用；来源、历史、模型、执行尝试和技术信息按需展开。事实、估计、AI 建议、用户决定和场所结果必须可辨。

## 4.7 业务流程与交互形态无关【FLOW-UX-002】

### 平台无关业务流程要求【FLOW-UX-002-REQ】

Halpha 的业务流程不依赖具体交互平台或客户端形态。任何受支持的交互形态都必须读取同一权威状态、保持同一业务对象与路径上下文、调用同一业务命令以及适用的授权、资金检查、执行和核对链，并把结果写回同一业务状态。形态切换不得复制、迁移、分叉或猜测业务状态，也不得改变决定、授权、失败、未知、停止、恢复和结束语义。具体支持哪些交互形态及其当前建设范围由下位设计和 L4 记录。

# 5. 研究或交易计划候选、研究与计划【FLOW-TRADEPLAN-001】

## 5.1 研究或交易计划候选去向

候选保留足以理解来源、时点、对象、初始判断和当前相关性的上下文。用户或 Halpha 按 CTX 规则决定结束、等待、进入经济研究，或在依据充分时直接形成交易计划；不是每个候选都必须经过正式研究。

## 5.2 AI 主导研究与最终人工评选【FLOW-AIR-001-REQ】

在已由用户明确或经 CTX 路由进入 ALP 正式研究的问题，以及 ALP 已确定的研究、评价、资源与停止边界内，AI 可以作为主要研究工作者，自主完成具体研究任务与用例的形成以及中间研究、筛选、反证、复核、重跑和最终候选材料编排。处于该边界内的任务形成、中间运行、淘汰和失败不要求用户或项目所有者逐项批准；没有合格候选是正常结束，不得为了产生交付而降低口径。具体预登记、数据角色、检验、隔离、工具和证据包方法只由 ALP 及下位设计拥有。

AI 的中间评价、排序和审查始终是需要核对的派生判断，必须服从 ALP 对证据、完整搜索、失败保留和结论生效的规则；同源 AI 互相赞同不增加经济证据。AI 不得改变已固定边界、删除失败、豁免硬门槛、把自身输出评为可信或使自己的正向判断生效。

只有 ALP 规定的最终候选材料才提交项目所有者作必要结果审查。项目所有者可以拒绝、要求在新边界下重新研究，或形成/确认与材料相称的 ALP 经济证据判断并选择进入正式策略实现与资格验证。项目所有者可以随时停止自动研究或抽样复核过程，但抽样复核不是每个中间步骤的必经批准。

最终候选审查只决定是否进入正式策略实现与资格验证。它不得同时替代正式源码资格、固定交易计划版本、交易计划启用、真实资金环境选择、资金使用上限与范围、机器授权或当前真实动作确认；即使项目所有者与用户是同一真人，这些决定也必须继续按各自拥有领域分别形成。AI 可以实施或检查功能与实现验证、分析经济材料并编排重演，但不能以“AI 已验证”跨过上述任一边界。

## 5.3 研究进入计划

Halpha 按 ALP 规则形成策略或研究结论时，必须表达机制、适用范围、反证、成本、流动性边界、证据局限和失效条件。研究可以改善判断或形成交易计划依据，但不能直接产生订单。

## 5.4 真实动作前的必要决定

真实动作通常必须引用完整交易计划，计划完整性和版本边界由 TRADEPLAN 定义。例外仅限用户明确作出并已固定的已有风险敞口保护或降低风险决定，以及用户明确提交且可证明不增加或转换风险的撤销、保护、划转或降低风险指令；例外必须在动作前固定对象、范围、原因和允许结果，并经过与计划动作相同的资金使用上限与范围、Halpha 真实资金操作权限、执行和核对路径。

计划可以承接用户决定、研究依据或必要的多项资金用途选择。启用前必须能够预览、演练、固定或以不行动结束；不完整内容只能保留为草案；实质变化形成新版本，不能在真实动作后补写原决定。

## 5.5 多项资金用途比较只在必要时进入

普通单项计划和单一目标验证不要求多项资金用途比较。只有多个真实资金用途竞争同一资金、现有持仓与拟行动需要比较共同风险敞口或再平衡时，才按 POR 边界进入比较；用户作出选择后，仍按 TRADEPLAN 形成适用计划。

# 6. 运行方式、交易计划启用、事件与交易动作【FLOW-RUN-001】

## 6.1 交易记录环境与 Halpha 真实资金操作权限分工

历史研究与历史行情回放用于研究和验证；交易所模拟盘与真实资金环境用于交易路径。交易所模拟盘的首要目的，是验证将用于真实资金环境的系统流程、授权检查、执行状态、核对、停止、恢复和证据机制；策略行为验证是次要目的。模拟结果不能直接证明真实市场中的流动性、排队、冲击、滑点、费用、资金费率、延迟、权限、可用性或真实 Alpha 表现。

交易所模拟盘与真实资金环境必须使用同一从 TRADEPLAN 经 CAP、EXE、DAT 到 OUT 的业务流程和稳定动作语义，同时让环境、账户、授权、动作和结果持续可辨。由模拟进入真实资金环境不是同一运行身份的升级或迁移；用户必须新建真实环境身份并重新选择适用计划、资金使用上限与范围和 Halpha 真实资金操作权限。模拟决定、授权、动作或结果身份不得迁移为真实决定、授权、动作或事实。具体存储、凭据、实例和适配隔离由 ARC 及下位设计定义。

## 6.2 交易计划启用与观察

计划启用后，Halpha 可以评价条件、必要事实的完整性与时效以及外部事件。条件满足只产生当前计划分支，不证明应当交易；动作仍须结合最新事实和适用授权路径。

## 6.3 两种授权路径与业务场景

真实动作只有人工授权和机器授权两种路径；真实验证、保护和降低风险是用途或风险场景，不是第三种授权。

### 人工授权路径

人工授权由用户对当前具体动作或明确受控范围作出决定。需要触发后人工判断时，只有用户在线、能够理解和响应，并且从通知到提交的完整路径在适用时限内可行时，动作才可继续。否则只能等待、过期、阻断或不交易，不能自动转为机器授权。

### 机器授权路径

机器授权必须在触发前由用户授予，明确作用域、期限、动作范围和失败结果。Halpha 只能在授权交集内按确定规则行动；关键事实未知、时效不可行、出现必须由人承担的判断或授权边界不清时，必须停止相应动作并进入核对、人工接管或不交易。

### 外部人工操作与工程验证

用户通过交易场所官方入口的操作属于外部活动。Halpha 提供必要交接上下文，返回后读取并核对场所事实，不能把外部操作事后改写成 Halpha 动作。交易所模拟盘可以用与人工授权或机器授权相同的路径形态验证系统机制，但其授权仅具模拟验证效力；工程或真实验证可以使用任一授权路径，验证目的不能提高普通资金使用上限、扩大范围或提高 Halpha 真实资金操作权限。

## 6.4 统一交易动作链

~~~text
已启用交易计划的当前分支，或用户明确作出并已固定的允许保护/降低风险决定，或用户明确提交且可证明不增加或转换风险的撤销、保护、划转/降低风险指令
→ 读取当前关键事实并确认当前环境的人工授权或机器授权路径可用
→ Halpha 按 CAP 规则检查当前环境限定的授权效力、资金使用上限与范围、停用状态和适用限额；真实资金环境还必须检查 Halpha 真实资金操作权限
→ Halpha 按 EXE 规则在外部写入前形成环境限定的 ExecutionAction 并建立唯一处理责任
→ 当前环境单一且隔离的外部写入边界刷新关键事实、重复 CAP 检查并向该环境绑定的场所端点提交动作
→ 部分结果、超时和结果未决保持可见并由 EXE 推进
→ Halpha 按 DAT 规则核对外部事实，并把结果交回适用计划、保护责任或结果学习路径
~~~

交易所模拟盘与真实资金环境的所有允许来源进入同一计划、CAP 检查、执行、核对和停止路径；环境只改变配置、外部端点和授权的资本效力，不改变流程所有权、动作状态推进或失败处理。该链不为未来动作预先占用资金，也不建立独立或最终资金审批。人工授权路径必须确认当前动作受到用户当前决定或仍有效的明确受控范围决定支持；机器授权路径必须在触发前已有授权并在提交前保持有效。真实资金环境必须额外满足全部真实资金边界；模拟验证授权不能满足真实资金检查。具体动作身份、失败和重试语义由 EXE 定义。

## 6.5 停止与人工干预

用户可以立即停用全局、账户或策略范围内的 Halpha 新真实动作。停用不自动撤单、划转或平仓；需要真实处置时，必须形成新的允许来源动作，或由用户经官方入口处理。

人工干预后，Halpha 先读取外部订单和持仓，再识别适用计划或保护责任；任何后续动作决定仍须按正常决定与授权路径取得。重启或恢复不能让旧拟执行内容、旧确认或旧处理记录自动产生外部动作；外部事实不清时，受影响范围保持停用。

# 7. 结果、学习与资金使用上限、范围变更【FLOW-LRN-001】

计划结束或重要真实动作责任闭合后，短复盘回答原判断、实际结果、成本，以及判断、计划、数据、执行、UX 或稳定性中的可行动偏差。OUT 拥有复盘和跨记录归因语义；复盘结论不能直接改写策略、计划、授权或资金使用上限与范围。

用户要求提高资金使用上限或扩大范围时，Halpha 应汇总与该决定相关的判断质量、计划表达、事实可靠性、执行正确性、UX 可控性和恢复表现。Halpha 不得根据盈利、流程完成度或模型结论自动扩大范围，也不判断个人应增加多少总投入。

# 8. 接入、真实验证、故障与退出【FLOW-REC-001】

## 8.1 最小接入

最小接入先建立只读连接和外部账户控制路径，使用户看清账户事实、配置资金使用上限与范围和 Halpha 真实资金操作权限、验证凭据撤销和人工接管，再在隔离环境走通计划与动作路径。只有用户明确选择真实环境、验证目标和资金使用上限与范围后，才进入真实验证。接入不等待全部领域深化，也不建立统一生产准入包。

## 8.2 真实验证流程

每次真实验证必须有明确目标、真实环境、授权路径、资金使用上限与范围、预期动作、停止条件和证据要求，并保持停止和外部接管可用。结果可以是支持、否定、无法判断、业务或技术问题、结果未决、事实未知或主动停止；一次成功或盈利不能证明成熟。

### 验证计划定义【FLOW-VAL-001-DEF】

验证计划组织验证目标、环境、资金使用上限与范围、授权路径、停止条件和证据要求，但不替代真实动作通常所需的交易计划，也不替代例外动作所需的固定决定或指令。是否需要独立验证计划对象或自动编排，由下位设计和 L4 根据真实消费者决定。

## 8.3 故障与恢复

故障采用短闭环：停用受影响范围的新真实动作 → 核对外部订单和持仓 → 保存必要材料 → 修复或回退 → 核对未决责任、原授权效力与运行资格 → 按当前阶段已接受的恢复方式等待用户命令或在严格证据门通过后继续。恢复已经失效、撤销或收缩的权限必须由用户决定；仍有效机器授权的运行资格是否可以在原范围继续由下位已接受设计分阶段明确。未知或未决未消除时保持停用；只有用户明确指示且可证明不增加或转换风险的收缩动作可以继续。

## 8.4 Halpha 不可访问

用户通过交易场所官方入口查看账户、撤销 Halpha 写入能力，并在必要时撤单、保护或处置持仓。Halpha 恢复后默认不产生新真实动作，先读取权威状态和外部事实，证明唯一写入者并避免重放旧记录。人工路径的每个动作仍必须取得新的当前人工授权，旧确认不得复用。机器路径只有在原机器授权仍有效、构建/配置/凭据/账户身份与持久写入连续性未改变、订单/成交/仓位/保护/未知均已核对、不会补发错过信号，并且当前阶段已接受设计明确允许自动继续时，才可在原范围继续；否则必须等待用户显式继续同一激活、退出、接管或建立新授权。用户停止、撤销、接管、授权到期或上述证据不能证明时不得自动继续。

## 8.5 停用与退出

退出时停用新真实动作、看清遗留订单与持仓、撤销凭据、导出必要计划和交易记录，并确认账户可在外部独立控制。复杂退场、继任和机构式归档不是个人项目要求。

# 9. 横向业务责任与纵向约束责任交接【FLOW-HOF-001】

## 9.1 两个责任维度【FLOW-HOF-001-DEF】

### 六项横向业务责任

| 领域 | 唯一主责 | 长期进入条件或产品作用 |
|---|---|---|
| CTX | 研究或交易计划候选及其去向决定 | 捕获、筛选并决定进入研究、计划、等待或结束 |
| ALP | Alpha 研究、经济证据与策略 | 提高判断和盈利能力 |
| POR | 多项资金用途比较 | 只在多个真实用途需要比较时进入 |
| TRADEPLAN | 交易计划与条件生命周期 | 把依据变成完整、可运行和可结束的计划 |
| EXE | 环境限定的执行动作、场所结果、保护与核对推进 | 对 Halpha 在交易所模拟盘或真实资金环境发起的场所或账户变化保持唯一责任和基本正确性 |
| OUT | 综合结果、归因与改进交接 | 只在结果能改变后续决定时形成复盘与问题 |

### 五项纵向约束

五项纵向约束由 ARC 定义，并分别由 CAP、DAT、UX、SYS 和 ENG 拥有稳定语义。FLOW 只确认横向路径受适用约束，不复制纵向领域规则。

## 9.2 责任地图使用与下位要求【FLOW-HOF-001-REQ】

六项横向业务责任与五项纵向约束责任共同组成十一项责任地图，但不要求十一模块、十一服务或同等建设深度。真实消费者、业务价值和维护成本决定是否深化；当前选择与投入由 L4 记录。

### 下位交接

L2、L3 和 L4 的通用职责、准入与现状记录边界以 HALPHA-DOC-001 为准。FLOW 只把本文定义的六项横向业务责任、主要交接、任务上下文连续，以及行动、不行动或等待、事实未知、停止、外部人工接管、恢复和结束的产品级结果交给适用下位设计；下位文档的建立、深化与当前记录按 DOC 执行。

FLOW 的直接交接还要求所有下位路径保持 CON-CMP-003 的三类控制要求及其领域分工，不得重新引入为未来动作预先占用资金或两阶段资金与真实动作审批；其他层级与治理约束直接引用 CON 与 DOC，不在 FLOW 重写。
