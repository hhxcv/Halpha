# 0. 流程定位与核心承诺【FLOW-SUM-001】

## 0.1 流程选择依据【FLOW-SUM-001-RAT】

Halpha 服务于一条直接的个人交易主线：看清事实，形成判断，制定计划，正确行动，核对结果并改进。研究、控制和工程措施都应帮助这条主线，而不成为用户必须先完成的独立治理流程。

用户可以不交易、停止 Halpha 或转到交易场所官方入口接管。系统运行时间更长，不改变用户的最终决定权。

## 0.2 核心流程要求【FLOW-SUM-001-REQ】

- 用户进入后能够看清账户、持仓、订单、活跃计划、重要异常和是否需要行动；无事时可以直接退出或进入研究、计划和复盘。
- 真实动作通常来自已经固定的完整交易计划；已有风险敞口的保护、退出或降低风险，也可以来自用户针对明确对象和范围作出的直接指令。
- 一次明确的计划激活可以同时授予仅属于该次激活的有界机器执行范围。该范围必须固定环境、账户、计划、工具、资金限制、期限和允许动作；它不改变计划，也不适用于后续激活。
- 每个外部动作都必须使用当前事实，经过适用的资金与权限检查，由唯一执行路径提交，并随后核对交易场所事实。重复请求不得产生重复动作。
- 事实未知、权限不足或路径不可行时，不新增风险；只有用户明确要求且能够证明不会增加或转换风险的收缩动作可以继续。
- 交易所模拟盘与真实资金环境使用同一业务链，但环境、账户、授权、动作和结果始终分离，模拟决定和结果不得迁移为真实资金事实。
- 停止、退出和交易场所官方入口的人工接管始终可达。

# 1. 有限注意力下的路径连续性【FLOW-TIM-001】

用户应能从当前状态直接进入最有价值的任务。异常和有时限事项优先出现，但正常入口不应变成技术状态页。

研究和计划草案可以中断后继续；恢复时突出期间变化、过期输入和需要重做的决定。已经发生或可能发生的外部动作不得因恢复、重启或重复点击而重放，Halpha 必须先核对外部事实。

---

# 2. 重复使用场景【FLOW-MAP-001】

| 用户目的 | 主要路径 | 正常结束 |
|---|---|---|
| 快速查看 | 当前事实与待办 → 相关计划、订单、持仓或异常 | 处理一项，或无事退出 |
| 研究策略 | 明确问题和边界 → 探索、反证和比较 → 人工选择 | 放弃、继续研究或采用结果 |
| 制定计划 | 判断依据 → 进入、退出、保护、规模、期限和失败处理 → 预览 | 保存草案、固定或放弃 |
| 运行计划 | 激活计划 → 观察条件 → 当前事实与动作检查 → 执行和核对 | 不行动、继续、停止新增风险、退出或接管 |
| 处理已有风险 | 当前订单或持仓 → 保护、降低风险或退出决定 → 执行和核对 | 完成、保持停用或外部接管 |
| 复盘改进 | 原判断和计划 → 实际结果与成本 → 可行动差异 | 无需改变，或形成新研究、计划或改进 |

这些路径不是强制漏斗。用户可以直接研究、查询事实、处理持仓或复盘；没有合适机会时，以等待、保持现金或不交易结束。

# 3. 外部工具与 Halpha 边界【FLOW-TOOL-001】

## 3.1 外部工具分工依据【FLOW-TOOL-001-RAT】

交易场所负责官方账户操作，专业图表和研究环境负责其擅长的分析，新闻和数据源提供原始材料。Halpha 不重复建设已经足够的外部能力。

AI 可以在用户提出的研究问题和当前可用数据、工具边界内直接承担探索、实现、运行、比较、反证和材料整理；只有资源消耗或结论用途需要时才补充相称的资源和停止边界。AI 不取得事实、策略采用、交易计划或真实动作的决定权。

## 3.2 Halpha 自持与交接要求【FLOW-TOOL-001-REQ】

Halpha 保持研究与计划上下文、当前采用的计划、用户配置的资金边界、Halpha 发起的动作和未决责任，以及核对后的必要事实。跨工具交接必须说明对象、时间、适用计划和返回方式；外部工具中的“完成”不能替代 Halpha 对账户事实的核对。

# 4. 用户交互与控制【FLOW-UX-001】

## 4.1 六类用户任务域【FLOW-UX-001-DEF】

长期用户任务包括市场观察、策略研究、交易计划、交易执行与持仓管理、账户与交易记录、复盘学习。它们是任务范围，不是必须建立的页面、模块或服务。

## 4.2 用户任务域使用要求【FLOW-UX-001-REQ】

交互应保持任务上下文连续，让用户区分事实、推断、AI 建议、用户决定和外部结果。常用任务少步骤完成，来源、历史和技术细节按需展开。

计划内容固定、提高资金上限或扩大范围、激活计划、停止、退出和接管是不同用户决定。计划激活按第 0.2 节包含该次激活的有界机器执行授权；一个决定不得暗中改变另一个决定。

## 4.3 业务流程与交互形态无关【FLOW-UX-002】

### 平台无关业务流程要求【FLOW-UX-002-REQ】

不同交互形态必须读取同一权威状态、调用同一业务命令并返回同一结果语义。切换入口不得复制、猜测或分叉计划、权限、动作和结果。

# 5. 研究与计划【FLOW-TRADEPLAN-001】

## 5.1 AI 主导研究与最终人工选择【FLOW-AIR-001-REQ】

策略研究可以独立进行；只要不接入交易运行状态、账户写入或真实凭据，就不依赖交易执行链是否可用。用户只需先说明要研究的问题和当前可用的数据边界，AI 即可自主探索、失败、重跑和筛选，不要求逐项审批。准备形成可用于产品或资金决定的结论时，再补足与该结论强度相称的评价口径、成本、反证、资源和停止条件。没有合格结果是正常结束。

只有用户决定是否采用研究结果。采用研究结果不自动固定或激活交易计划，不改变资金边界，也不产生真实动作。

## 5.2 研究进入计划

研究结果用于计划时，应说明适用范围、主要依据、反证、成本、限制和失效条件。用户也可以用自己已经充分形成的判断直接开始计划；正式研究不是每个计划的必经步骤。

完整计划至少回答：为何行动、作用于什么、何时进入、何时失效、承担多少、持续多久、触发后怎样处理。内容不完整时只保留草案；实质变化形成新版本，不能在动作发生后补写原决定。

---

# 6. 运行、动作与核对【FLOW-RUN-001】

计划激活后，Halpha 可以观察条件并形成计划允许的动作意图。条件成立不等于动作获准；每次动作仍须读取当前事实，检查资金边界、激活范围和停止状态，再经唯一外部写入路径执行和核对。

计划激活是 Halpha 机器产生新增风险动作的唯一用户授权入口。触发后若出现计划外判断、范围不足或阻断性未知，Halpha 不临时请求逐动作确认来绕过原计划，而是等待、过期、不交易，或由用户修改并重新激活计划。针对已有风险的明确保护、撤销、减仓和退出指令仍进入同一检查、执行与核对链，不能转换为新增风险能力。

用户通过交易场所官方入口完成的操作属于外部活动。Halpha 返回后读取并核对事实，不把外部操作改写成自身动作。

用户可以停止 Halpha 新增风险、退出一个计划或转入人工接管。停止不会自动撤单或平仓；已有订单、持仓和保护责任必须继续核对或明确移交。

---

# 7. 结果与学习【FLOW-LRN-001】

计划结束或重要动作责任闭合后，复盘比较原判断、计划、实际结果和成本，并只形成能够改变后续决定的改进。复盘不得直接改写策略、计划、权限或资金上限，也不得因盈利自动扩大范围。

---

# 8. 接入、故障与退出【FLOW-REC-001】

接入先建立只读观察和交易场所官方入口的接管路径，再验证计划、动作、停止和核对。模拟环境用于验证同一业务链，但不能证明真实市场表现或真实资金可用性。

故障时采用短闭环：阻止受影响的新风险 → 核对订单、成交和持仓 → 修复或回退 → 决定继续、结束或接管。系统重启且用户没有停止该激活时，只有在事实与原授权一致且不会补发错过动作后，才可继续同一激活。

用户已经停止新增风险、退出或接管时，该激活不得恢复新增风险；已有风险继续按原计划保护或退出。旧责任闭合后如需再次交易，用户必须创建新激活。

Halpha 不可访问时，用户通过交易场所官方入口查看账户、撤销写入能力并处置风险。退出时停用新动作、看清遗留责任、撤销凭据并导出必要计划和交易记录。

---

# 9. 核心业务责任交接【FLOW-HOF-001】

## 9.1 横向业务责任【FLOW-HOF-001-DEF】

| 领域 | 产品级责任 |
|---|---|
| ALP | 研究策略、评价经济证据并形成可采用结果 |
| TRADEPLAN | 把依据变成可运行、可停止和可结束的计划 |
| EXE | 对 Halpha 发起的外部动作保持唯一执行和核对责任 |
| OUT | 从计划与结果形成有用的复盘和改进 |

CAP、DAT、UX、SYS 和 ENG 分别提供资金控制、事实、交互、运行和工程约束。责任名称不要求建立同名模块、服务或同等建设投入。

## 9.2 下位要求【FLOW-HOF-001-REQ】

下位设计只细化真实消费者需要的对象和失败处理，不复制整条流程，也不为未来可能性建立平台。当前支持范围、建设顺序和证据只由 L4 记录。
