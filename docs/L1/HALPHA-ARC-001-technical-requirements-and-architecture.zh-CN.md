# Halpha 总体技术要求与架构

**文档编号：** HALPHA-ARC-001  
**层级：** L1-D  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001  
**本文档负责：** 长期总体技术方向、权威状态、依赖、交易所变更请求、环境隔离和演进条件  
**本文档不负责：** 领域对象、接口字段、数据库结构、具体供应商、部署和当前配置

## 1. 总体结论

Halpha 采用个人可维护的模块化单体，使用关系型持久存储承载产品权威状态，通过少量外部适配连接数据源和交易场所，并把真实账户交易动作限制在一个可隔离的执行边界。

逻辑责任不等于独立服务或进程。进程、实例、组件和部署数量由详细设计与当前需要决定，不作为成熟度指标。

## 2. 质量取舍

技术选择依次保护用户控制、动作与事实正确性、核心产品价值、个人可维护性，再考虑通用性和未来扩展。

稳定性优先来自成熟组件、简单拓扑、少依赖、明确所有权、直接测试和可恢复数据。没有实际故障或规模需求时，不建设微服务、消息总线、分布式事务、集群、综合监控平台或机构级高可用。

## 3. 权威状态与所有权

每项业务状态和决定只有一个拥有模块。其他模块通过公开调用或已提交记录使用结果，不能直接改写私有状态，也不能把页面缓存、日志、邮件或第三方组件投影提升为权威事实。

本地状态变化使用明确事务；不能与外部系统共同回滚的动作先留下可恢复的唯一责任，再在提交后处理。结果未知时查询原外部身份，不盲目重发。

## 4. 依赖与组件

用户入口调用拥有业务语义的模块；业务模块调用基础设施和外部适配，基础设施不反向决定业务。研究、AI、报告、通知和配置能力不得成为停止、核对和真实动作的必要同步依赖。

Halpha 以集成成熟第三方组件为主要技术方式。高稳定性、高性能或实现复杂的通用基础能力优先由成熟组件承担；自研集中于第三方无法表达的产品语义、必要定制边界，以及需要项目自行研究和保有、外部没有可直接复用公开实现的 Alpha 逻辑。

成熟组件能够完整承担通用能力时直接采用。Halpha 只保留产品语义、必要转换和不可妥协的正确性边界，不复制组件内部状态机或为未来替换维护第二实现。

## 5. 环境与交易所动作

“本地持久化”是把产品状态提交到 Halpha 数据库；“交易所变更请求”是可能创建、修改或撤销订单，或者改变交易账户、持仓的交易所 API 请求；“真实账户交易动作”是在真实资金环境发起的交易所变更请求。设计和实现不得用笼统的“写入”同时指代这三件事。

研究、Demo、只读运行和真实资金环境必须具有可辨身份和能力。模拟与真实资金复用核心业务和执行语义，但账户、凭据、动作、事实和持久身份不能混用。

只有隔离的执行边界可以取得真实账户交易凭据并发起交易所变更请求。普通应用、研究、AI、页面和只读运行不得取得该凭据或能力。环境或执行身份无法确认时，不得发起真实账户交易动作。

具体存储隔离方式、运行角色和配置由 L3/L4 依据当前主机与组件决定，只要保持身份不混淆和真实账户交易能力隔离。

## 6. 数据、恢复与接管

产品保存不能低成本重建且承担交易责任的权威状态。可重建投影和缓存删除后应能恢复，陈旧时显示未知而不是冒充当前事实。

启动和恢复先读取持久状态与停止决定，再连接外部来源并核对未决责任。恢复不能扩大权限或补发错过动作。具体领域只处理自己拥有的状态，系统设计负责组合顺序。

用户始终保留交易场所官方入口、凭据撤销和数据导出路径。备份与回退恢复数据，不恢复已经失效的真实账户交易权限。

## 7. 安全与演进

安全投入按实际影响集中在真实账户交易凭据、资金边界、动作责任、账户事实和停止能力。单一所有者本地产品不默认建设企业身份平台、安全运营中心或可信计算登记。

只有实际并发、隔离、性能、故障或维护问题证明现有结构不足时，才增加进程、存储、适配层或运行平台。新增结构必须净降低全生命周期复杂度，并保持唯一所有权、事实权威、停止、恢复和退出路径。
