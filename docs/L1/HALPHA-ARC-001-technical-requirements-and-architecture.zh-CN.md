# 0. 架构结论【ARC-SUM-001】

1. Halpha 采用个人可维护的模块化单体，不预建微服务、消息总线、集群或机构级高可用。
2. 架构复杂度优先服务交易判断、计划、数据与执行正确性和 UX；稳定性主要通过成熟技术和简单结构取得。
3. 每个交易记录环境使用一个权威关系型数据库；模拟与真实资金环境使用彼此分离的权威存储或数据库。长期数据库产品及使用契约由 L3 选择，精确版本、实例和部署选择由 L4 记录。
4. Halpha 发起的交易所模拟盘与真实资金动作都由 EXE 拥有，使用同一环境限定 ExecutionAction、状态推进、CAP 检查、外部提交与核对链；真实资金动作仍额外受全部 Halpha 真实资金操作权限和真实写入门约束。
5. Halpha 不决定用户个人总投入，只执行用户已配置的资金使用上限与范围、适用决定依据、动作前检查、授权、停用和恢复；不为未来动作预先占用资金，不建设两阶段资金审批或独立生产准入产品。
6. 投影和缓存可以重建，不能反向成为权威事实；事实未知、冲突或结果未决时，受影响的新真实动作保持停用，只有用户明确指示且可证明不增加或转换风险的收缩动作可以继续。
7. 研究、AI、报告、管理和配置能力不得成为真实动作提交、停用、核对和恢复的必要同步依赖。
8. 六项横向业务责任与五项纵向约束责任组成十一项责任地图，但不要求十一模块、十一服务或同等建设深度。
9. 架构分别落实 CON-CMP-003 的三类控制要求，不用一个通用技术控制层复制或混淆其业务语义。
10. 通用技术能力先评价成熟第三方组件；仅影响可替换技术或策略选择的差异按组件实际语义调整并重新验证，Halpha 只补足不可妥协产品语义、功能正确性或安全边界的最小缺口；同一能力不保留平行实现。

# 1. 工程目标与取舍【ARC-QLT-001】

## 1.1 个人项目尺度的取舍依据【ARC-QLT-001-RAT】

目标环境是单一用户、少量账户、有限场所和有限策略的长期维护，允许计划内维护和短时中断，不以零停机或机构团队为前提。每项技术选择都必须同时承担开发、理解、测试、运行、故障、升级、迁移和退出成本。

## 1.2 质量优先级要求【ARC-QLT-001-REQ】

以下顺序只用于在全部满足 L0 不可突破边界、适用计划与授权以及基本功能正确性的候选方案之间分配复杂度；任何低位项目的必要边界都不能被高位目标豁免。

| 优先级 | 目标 | 典型失败 |
|---:|---|---|
| 1 | 交易判断、策略与计划质量 | 错误观点、条件或数量，无法形成盈利竞争力 |
| 2 | 数据与执行正确性 | 错误账户状态、重复动作、部分结果丢失或成本失真 |
| 3 | UX 便捷和用户控制 | 看不懂、操作慢、难停用、易误操作或无法接管 |
| 4 | 稳定性 | 组件脆弱、依赖过多或结构难理解，使常用能力不可靠 |
| 5 | 系统自身风险缓解 | 主机、依赖、网络、凭据或运行环境故障时无法隔离、停止、恢复或接管 |
| 6 | 金融风险控制实现 | 未执行适用决定依据、资金使用上限与范围、权限、金额或风险敞口检查 |
| 7 | 通用性、治理完整度和未来扩展 | 为假想规模增加当前维护负担 |

CON-CMP-003 已区分金融风险控制、功能正确性和系统自身风险缓解。基本功能正确性由拥有相应业务语义的模块实现，不能因资金或项目规模小而降级；系统故障影响优先由成熟选型、简单结构、隔离和外部控制限制；用户在 Halpha 外部作出的总资本决定不实现为系统流程。

未实现、未验证或不可信的自动控制不得表述为已具备。受影响的真实动作必须改用可行的人工或官方场所路径；人工也不可达时，保持停用、等待或不交易。辅助组件没有直接消费者或实际问题时，只保留边界或不建设。

## 1.3 成熟能力优先的总体技术取舍【ARC-QLT-002】

对跨领域或可能形成通用基础设施的能力，架构选择依次执行：明确当前消费者和不可妥协的产品语义、功能正确性与安全结果；评价适合用途的成熟第三方组件；比较实际能力、限制和全生命周期成本；最后才决定采用、按组件能力调整、保留最小补充或明确不支持。

组件与既有设计的差异如果只影响策略数学、内部结构、技术生命周期、命名或其他可替换选择，详细设计必须采用组件的实际语义并重新验证受影响结果，不得为维持旧草稿而复制实现。只有组件会破坏不可妥协产品语义、功能正确性或安全边界，或确实缺少所需能力时，Halpha 才保留最小边界翻译或产品语义补充。

同一能力只能有一个运行实现和一个事实权威。组件不可用或证据不足时，采用停止、等待、外部人工接管或明确不支持，不维护一套长期并行的 Halpha 自研替代。是否采用组件必须比较删除的自研复杂度与新增依赖、适配、配置、升级、恢复和退出成本；不能净降低复杂度且不负责补足不可妥协缺口的候选不进入总体方案。

# 2. Halpha 总体逻辑形态【ARC-TOP-001】

## 2.1 Halpha 总体逻辑形态定义【ARC-TOP-001-DEF】

~~~text
交互入口 / 通知
            │
            ▼
       应用命令与查询
            │
  ┌─────────┴─────────────────────────────┐
  │ 研究与判断 │ 计划 │ 事实 │ 执行动作与核对 │ 结果与交互 │
  └─────────┬─────────────────────────────┘
            │
   每个交易记录环境一个权威关系型数据库
            │
    模拟环境与真实资金环境的存储彼此分离
            │
   ┌────────┴────────┐
   │ 只读外部适配器 │ 单一且隔离的外部写入边界 │
   └────────┬────────┘
            │
      交易场所与数据源
~~~

图中方框表示逻辑责任，不表示独立部署，也不与责任地图一一对应。总体形态由模块化单体、按交易记录环境划分的权威关系型数据库、必要外部适配器，以及每个交易环境单一且隔离的外部写入边界组成；只有真实资金环境边界可以持有真实写入能力。

## 2.2 总体形态约束【ARC-TOP-001-REQ】

总体架构必须保持上述责任与隔离关系；长期数据库产品由适用 L3 选择，进程数、后台工作者、数据库实例、版本和实际部署隔离由 L4 决定。模拟与真实环境的外部写入边界使用同一受控实现路径但形成分离实例；真实写入权必须单一、最小，并与模拟凭据和一般应用能力隔离。

模块边界依据语义所有权、写入责任、故障影响和变化频率建立。只有测得的隔离、争用、部署或规模问题无法在单体内解决时，才评价拆分服务。

第三方组件完整承担某项模块内部能力时，语义所有者和公开边界仍由 Halpha 设计确定，但不要求存在对应的 Halpha 内部代码模块、组件状态机或通用包装层。长期采用的具体组件与使用契约进入相应 L3，精确版本、当前配置和资格证据进入 L4。

## 2.3 平台无关流程与交互入口适配【ARC-TOP-002】

### 交互入口适配的选择依据【ARC-TOP-002-RAT】

业务状态和流转若由交互形态分别持有，新增入口会变成状态迁移与业务重写，并引入分叉、冲突和误授权。Halpha 因此把平台无关的业务流程、对象身份和权威状态放在共同应用与领域边界中，把交互形态差异限制在入口适配层。

### 平台无关流程与入口适配要求【ARC-TOP-002-REQ】

每种受支持的交互形态只作为共同应用命令与查询的入口适配器，不得建立平行业务流程、业务写链、授权语义、权威状态或客户端专属迁移路径。新增交互形态必须复用同一对象身份、命令、处理回执、领域处理器和权威状态；入口本地状态只能是可删除、可重建且不影响业务流转的投影或草案。

上层架构只保留新增入口的兼容边界，不据此预建认证协议、网络暴露、通知、离线、同步、部署平台或客户端框架。只有出现已明确建设范围和真实消费者时，适用 L3 才定义该形态的安全与技术契约，L4 才记录当前启用范围和证据。

# 3. 业务模块与依赖【ARC-BND-001】

六项横向业务责任表达连续业务语义，五项纵向责任表达跨业务约束或支撑边界。它们指导模块划分，但不预设一项责任对应一个代码模块、进程或服务；运行保障由成熟技术选择、SYS、ENG 和相应业务所有者共同取得。

每项业务语义只有一个 L2 所有者。实现该语义的模块承担唯一业务写入责任；其他模块通过公开命令、查询或持久交接使用结果，不直接改写其状态。共享代码只容纳无业务语义所有权的稳定技术能力。

成熟第三方组件可以直接承担行情、指标、协议、持久化、调度、测试或其他通用技术能力；它不因承担实现而取得 Halpha 的产品决定、事实或持久业务状态所有权。Halpha 只在公开边界保留必要转换和不可妥协语义，不为未来更换供应商预建完整平行框架。

依赖方向保持为：

~~~text
交互入口 → 应用用例 → 业务逻辑 → 端口
                                 ↑
                         外部适配器实现端口

研究或用户判断 → 交易计划或允许的显式决定或指令 → 动作准备
动作准备 → 资金使用上限与范围及 Halpha 真实资金操作权限检查 → 执行 → 外部事实核对 → 结果与交互
~~~

研究、AI、报告、管理和配置功能不得成为真实动作提交、停用、核对和恢复的同步依赖。

# 4. 权威状态与数据【ARC-DAT-001】

## 4.1 来源、形成方式、证明力与所有权不得混淆

Halpha 必须区分带来源的外部观察、内部业务决定与责任、外部动作结果、经核对的可引用事实，以及可追溯派生结果。这些性质可以同时关联同一业务事项，但不能因保存或显示而互相转换。

派生结果和缓存不能成为账户事实。修正必须保留原观察、修正依据和关系，不得静默覆盖。事实缺失、过期、冲突或不可核对时，必须保留未知范围及其对动作的影响。

## 4.2 每个交易记录环境一个权威关系型数据库

每个交易记录环境使用一个关系型数据库承载该环境的权威业务状态、唯一约束、事务、未决处理责任、核对状态和必要历史。一个环境内的模块可以拥有不同写边界，但不为模块默认拆分数据库。

模拟与真实资金环境必须使用彼此分离的权威存储或数据库、凭据引用、账户身份和写入适配器实例；共享的是同一核心业务逻辑、ExecutionAction 契约、状态机、仓储接口和适配器实现路径，而不是账户、动作、授权或结果身份。长期数据库产品及使用契约进入 L3；精确版本、实例数量、宿主和部署位置属于 L4 当前选择。

任务投影服务读取和交互需要；缓存只改善性能。投影或缓存缺失时必须能重建，不得阻止停止真实动作、读取权威状态、外部核对和恢复。

## 4.3 时间与数值

会改变计划、资金使用上限与范围及 Halpha 真实资金操作权限检查、场所动作、核对或结果解释的数值，必须使用适合金额和数量精度的确定表示，并保持单位、币种和换算时点可追溯。需要区分时，事件时间、来源时间、接收时间、决定时间和提交时间不得混用。

数据源是否适合某用途由实际正确性、新鲜度，以及缺失、不完整和恢复时的行为决定；不建设覆盖所有来源的统一评级平台。

# 5. 计划、环境限定授权与执行动作【ARC-ACT-001】

## 5.1 可识别决定、明确启用与停用

启用的交易计划、允许的例外决定或指令、资金使用上限与范围、授权和停用决定都必须可识别，并保留历史决定与当前适用决定的边界。实质变化按相应语义所有者的规则形成新版本或新决定，不能静默改写已经支撑真实动作的内容。

人工授权不得被界面、程序或 AI 隐式转成机器授权；机器授权必须在触发前具有明确范围、期限和失败结果。停止和缩小范围立即生效；提高资金上限、扩大范围或扩大自动行动能力只能由用户显式操作。

## 5.2 统一执行动作主链

~~~text
已启用交易计划的当前分支，
或用户明确作出并已固定的允许保护/降低风险决定，
或用户明确提交且可证明不增加或转换风险的撤销、保护、划转/降低风险指令
→ 读取当前环境、关键事实、资金使用上限与范围、环境限定授权和停用决定
→ 确认当前环境的人工授权或机器授权路径有效；真实资金环境还必须具有适用的 Halpha 真实资金操作权限
→ Halpha 按 CAP 规则完成环境限定的上限、范围、授权效力和停用检查
→ EXE 在外部写入前持久形成环境限定 ExecutionAction，并建立唯一处理责任
→ 当前环境单一且隔离的外部写入边界刷新关键事实、重复 CAP 检查并处理该责任
→ 提交场所动作，查询外部结果并保持部分结果、超时和结果未决可见
→ 核对并更新可引用事实，把结果交回适用计划、保护责任或学习路径
~~~

交易所模拟盘与真实资金环境的所有允许来源必须进入同一主链。链内没有独立资金许可、未来资金预留或最终资金审批。人工授权与机器授权决定行动路径，CAP 检查决定当前动作是否越过当前环境的上限、范围和授权效力；二者不能互相替代。模拟验证授权对真实资本没有效力，不能满足真实资金检查。旧人工确认不能补足缺失的当前确认或机器授权，机器规则也不能替代需要用户承担的风险判断。

## 5.3 本地事务与交易场所或账户变化

关系型事务只原子提交内部状态，不能与外部场所或账户变化组成一个事务。Halpha 在任何交易环境外部写入前都必须留下可恢复的 ExecutionAction 和唯一处理责任；外部结果不明时优先按稳定外部身份查询，不盲目重发。

在任何可能相互影响外部动作身份或防重复能力的范围内，只能存在一个有效外部写入权。失去该权的运行实体不得继续提交。具体领取、幂等、并发和阻断机制由 EXE 与 SYS 定义。

## 5.4 交易所模拟盘、历史行情回放与真实资金环境隔离

历史研究和历史行情回放只复用纯条件求值、计划语义和结果分析。交易所模拟盘与真实资金路径则必须复用同一 TRADEPLAN→CAP→EXE→DAT→OUT 应用链、同一 ExecutionAction 契约与状态推进、同一仓储接口和同一场所执行客户端构造路径，同时保持环境、账户、动作、授权和结果身份不可混淆。

- 历史行情回放只用于研究和时效验证，不进入真实外部写入路径；
- 交易所模拟盘使用场所提供的模拟账户和结果，Halpha 不另建一套自撮合、自算盈亏的实时交易盘；其首要证据目标是系统流程和机制，策略行为证据次之；
- 模拟与真实资金环境使用分离的运行实例、权威存储或数据库、配置 profile、端点、凭据引用、账户身份和适配器实例；环境差异只能通过 L3 明确的配置允许清单表达，不散落在业务逻辑中；
- 两个环境的构建必须产生可核对的环境等价清单，证明源代码摘要、应用服务、ExecutionAction schema、仓储、状态机和场所执行客户端构造路径一致，并列出唯一允许变化的配置项；精确摘要和当前验证结果由 L4 记录；
- 模拟记录不能成为真实账户事实，模拟身份和凭据不能取得真实写入能力，模拟验证授权不能产生真实资本效力；
- 从模拟进入真实资金环境必须建立全新的真实环境、账户、动作和授权身份，并重新取得适用计划、资金使用上限与范围、Halpha 真实资金操作权限和所需授权；不得编辑、提升、复制或迁移模拟记录完成转换；
- 模拟结果不得被解释为已经证明真实流动性、排队、冲击、滑点、费用、资金费率、延迟、权限、可用性或真实 Alpha 表现。

无法证明模拟路径不能到达真实写入边界时，模拟能力不得与真实凭据同时启用。

# 6. 外部适配与工具边界【ARC-ADP-001】

外部适配器负责认证、协议、映射、限速、错误转换、外部身份和能力声明；核心业务不依赖供应商开发包对象或错误码。读取与写入能力尽量分离，普通研究、AI 和界面组件不持有真实交易写凭据。

场所或组件不支持的动作、事实或保护语义必须拒绝或显式降级，不能用统一接口伪装支持。只为实际使用的场所和数据源保留必要适配；成熟组件已提供的连接、协议、重试或状态能力不得在 Halpha 内重复实现。只有出现第二个真实消费者并证明共享抽象能净降低总复杂度后，才抽取 Halpha 通用框架。

---

# 7. UX 技术边界【ARC-UX-001】

UX 是核心产品能力。架构必须支持 FLOW 的用户任务、跨域当前状态、搜索和直接进入；模拟与真实环境持续可辨；当前决定、依据、未知、动作处理回执和外部结果可解释；停止新真实动作与外部接管保持可达；中断恢复突出期间变化；事实、估计、AI 建议、用户决定和场所结果在读取模型中可区分。

读取投影可以为 UX 优化，但所有写命令必须经过同一应用边界。界面或通知错误不能建立隐蔽写路径，也不能阻止后备停用和外部账户控制。

# 8. 稳定性技术选择与最小恢复【ARC-OPS-001】

## 8.1 稳定性技术选择依据【ARC-OPS-001-RAT】

稳定性以核心业务结果衡量：常用路径是否可用、事实是否新鲜、真实动作是否防重、核对是否推进、停止与重启是否安全。它首先来自成熟技术、简单结构、SYS 与 ENG 边界和业务领域自身正确性，而不是独立运行平台。

对于单一所有者维护的个人项目，关系型数据库事务、现成进程管理、普通日志以及数据库或托管环境自带的备份通常足以覆盖主要需要。重复故障出现前，不自建专门稳定性平台。

## 8.2 稳定性与恢复要求【ARC-OPS-001-REQ】

总体架构应优先使用成熟组件、模块化单体、按交易记录环境划分的权威关系型数据库、少量进程和少量外部依赖。现成能力足够时不重复建设。

组件故障、能力未知或资格证据失效时，受影响路径按其业务影响停止、降级为只读、交还用户或明确不支持；不得自动切换到一套未同时成为唯一设计和唯一验证对象的自研实现。恢复或替换组件前必须重新核对外部事实、兼容边界和受影响证据。

启动与恢复必须先读取权威状态和当前停用决定、资金使用上限与范围及 Halpha 真实资金操作权限，再连接只读外部来源、核对订单和持仓、处理未决真实动作责任。关键事实、外部结果或写入权不明时，受影响范围保持停用，只有用户明确指示且可证明不增加或转换风险的收缩动作可以继续；重启默认不重放真实动作。

恢复已经失效、撤销或收缩的 Halpha 真实资金操作权限只表示重新允许后续动作进入适用授权路径；人工路径的每个动作仍必须取得新的当前人工授权，旧确认不得复用。仍有效的机器授权与运行资格必须分开判断：只有当前阶段已接受设计明确允许，并且唯一写入、权威状态与数据库连续性、构建/配置/凭据/账户身份、外部订单/成交/仓位/保护/未知核对和不补发错过动作的证据门全部通过，系统才可在原授权范围继续；其他阶段或任一证据不明时，必须保持停止并等待用户显式继续、退出、接管或建立新授权。具体恢复条件和未决语义由 EXE、DAT 与 SYS 定义，当前 L4 必须选择其中一种阶段恢复方式并验证。

备份只保存不能低成本重建的权威状态和必要配置。默认不建设健康页、综合监控、告警中心、自愈、热备、多区域、自动故障转移、专门备份产品或运行事故平台；实际问题重复出现且现成方法不足时，才增加净降低长期成本的最小能力。

# 9. 身份与基本安全【ARC-SEC-001】

真实交易凭据采用实际可行的最小权限，默认不具有提现、转账和账户管理能力。真实凭据不得进入源代码、文档、AI 上下文、普通日志和开发环境。交互身份、运行身份、用户决定、机器授权和真实交易凭据必须分离；程序或 AI 不能凭自身输出形成授权，也不能把会话存在误判为用户已确认当前动作。

用户保留 Halpha 之外的官方账户访问、凭据撤销和人工处置路径。系统失效或怀疑凭据泄露时，优先停止写入、撤销或隔离凭据并核对账户。

安全投入按实际影响范围分级，重点保护能提交动作、读取真实凭据、改变资金使用上限与范围或 Halpha 真实资金操作权限、写入账户事实或阻止停用的能力。项目不建设企业 IAM、SOC、可信计算基线登记或机构级威胁治理平台。

---

# 10. AI 开发与技术策略【ARC-TEC-001】

AI 辅助开发遵循 ENG 的影响分级。交易判断、计划、数据、执行、常用 UX、停用和恢复变化需要更强验证；低影响辅助变更采用轻流程。不建立统一工程认可结论或生产准入委员会。

总体技术选择优先直接采用经用途资格验证的成熟语言、框架、量化或领域组件、关系型数据库、场所官方或成熟适配能力、标准日志、现成进程管理和备份，以及个人可以诊断和恢复的部署形态。现成组件能力与可妥协技术选择不一致时，按组件实际能力设计并重新验证，不以自研复制保持旧实现细节。

ENG 负责把能力检索、差异分类、许可证、目标平台、升级退出和复杂度净变化形成工程门槛；L3 按 DOC 记录长期采用的具体组件、Halpha 使用契约和最小补充边界，L4 记录精确版本、构建标识、当前配置、平台资格证据和启用状态。ARC 不选择具体供应商或版本。

没有实证触发前不引入微服务、分布式事务、事件溯源平台、通用消息总线、Kubernetes、多区域集群、通用插件系统、多租户权限、企业数据湖、综合可观测平台或机构级风险引擎。

---

# 11. 复杂度预算与演进【ARC-CMP-001】

复杂度优先投入交易判断与计划、事实与真实动作正确性，以及清楚可控的 UX；SYS、CAP、ENG、治理和辅助自动化保持满足当前真实消费者的最小深度。稳定性本身不构成建立独立运行平台或自动深化辅助领域的理由。

任何新平台、状态机、登记或流程都必须有真实消费者和已观察问题。拆分服务只在测得的扩缩、隔离、发布、争用或安全边界需要超过单体能力时进行；在此之前优先删除功能、减少同步依赖、优化查询和任务、隔离少量后台工作者或改善适配器。

引入第三方依赖时必须同时计算删除的自研代码、状态、测试、运行责任和维护面，以及新增的依赖、胶水、配置、许可证、升级、恢复和退出成本。净复杂度不能下降且没有不可妥协缺口需要补足时，不得引入新的包装层、平行实现或平台。

# 12. 十一项责任的长期架构关注点【ARC-L2D-001】

| 领域 | 长期架构关注点 |
|---|---|
| ALP | 核心判断、经济证据和策略质量；只对真实研究消费者深化 |
| TRADEPLAN | 完整、可执行、可结束的计划语义 |
| EXE | 环境限定 ExecutionAction 的唯一性、外部结果、保护、未知与核对；模拟与真实环境使用同一执行语义 |
| DAT | 带来源事实、时效、修正、未知和恢复 |
| UX | 高频任务的便捷、清楚、可控和可恢复 |
| SYS | 模块与运行边界、依赖方向、权威状态和简单恢复 |
| CTX | 候选低摩擦捕获和去向，不发展通用内容平台 |
| OUT | 从短复盘和可行动问题开始，按真实样本深化 |
| POR | 只有多个真实资金用途需要比较时产生实现需求 |
| CAP | 资金使用上限与范围、授权、动作检查、停用和恢复边界 |
| ENG | 影响分级验证、成熟能力检索与最小自研门槛、可重复构建、迁移、回退和依赖维护 |

责任地图是语义所有权地图，不是均衡投资计划。已启用核心链依赖的部分必须正确；没有消费者的部分不因逻辑闭环而建设。当前领域深度、支持范围、产品、实例和实现取舍由 L4 记录。

# 13. 下位设计清单【ARC-HOF-001】

L2、L3 和 L4 的通用职责、准入与现状记录边界以 HALPHA-DOC-001 为准。ARC 只把本文的总体逻辑形态、依赖方向、成熟能力优先门槛、每个交易记录环境一个权威关系型数据库、每个环境单一且隔离的外部写入边界、模拟与真实资金环境的统一执行实现和身份隔离、环境等价证明及恢复顺序交给适用下位设计；具体领域、模块、组件使用契约、接口、数据、状态、错误、幂等、并发、阻断与测试语义由相应 L2/L3 按真实消费者细化。

长期数据库产品和第三方组件使用契约按 DOC 进入相应 L3；数据库精确版本与实例、第三方组件精确版本与构建标识、当前配置与资格证据、进程与后台工作者、宿主与部署位置、场所和适配器支持范围等当前选择记录在 L4。候选设计、设计完成、代码存在或单次测试不能替代 L4 的当前能力与验证证据。
