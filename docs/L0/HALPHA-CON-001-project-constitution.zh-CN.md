# Halpha 项目宪法

**文档编号：** HALPHA-CON-001  
**层级：** L0  
**语言版本：** zh-CN  
**适用范围：** Halpha 的产品、设计、实现、运行与停用  
**权威范围：** 长期使命、不可突破边界、最高取舍和根本变更条件

## 1. 适用与权威

本宪法拥有 Halpha 的长期使命、不可突破边界和最高取舍。下层设计、实现和运行必须与之相容；代码、测试和运行结果可以证明符合、暴露偏差或触发修订，但不能自行改写这些根本规则。

## 2. 项目使命与角色

Halpha 是由一位所有者建设、维护和使用的个人交易决策、执行与学习系统。它帮助用户形成可检查的判断和计划，在用户设定的边界内可靠执行，并从真实结果中改进。

项目所有者决定产品方向、建设投入和维护取舍；用户决定是否使用产品、采用什么计划、允许系统使用多少资金以及何时停止或接管。两种责任可以由同一人承担，但产品不能利用建设者身份替代用户的资金决定。

AI 和其他工具可以帮助研究、设计、实现、测试和解释，但不能自行成为账户事实、用户决定、资金授权或真实动作来源。

## 3. 最高目标与取舍

Halpha 的首要目标是在真实成本和风险约束下为用户产生可归属的长期净价值。单次盈利、回测结果、运行时间、功能数量、文档完整度和工程投入都不能单独证明产品成功。

发生取舍时依次保护：

1. 用户对账户、资金、计划、停止和接管的控制；
2. 真实动作的功能正确性和事实可核对性；
3. 对用户有实际价值的交易判断、执行和体验；
4. 个人能够理解、运行、恢复和退出的总复杂度；
5. 通用性、形式完整和未来扩展。

## 4. 真实资金与动作边界

Halpha 不决定用户的个人总投入、可承受损失或系统外资金安排，也不承诺最大损失或未来收益。系统只能在用户明确设定的资金范围和当前决定内行动，不得因盈利、模型、策略、AI 输出或运行时间自行扩大范围。

任何可能改变真实交易场所或账户的动作都必须有可识别的用户决定依据、当前事实、适用边界和唯一处理责任。无法可靠确定是否允许或是否已经发生时，不新增风险；查询、已有风险保护、退出和外部人工接管仍应可用。

模拟、研究和只读结果不能被复制、改名或提升为真实资金决定、事实或动作。

## 5. 功能正确性与用户控制

真实动作必须防止重复、保留稳定身份、处理部分与未知结果，并以外部事实核对。停止、重启、重试、恢复和还原不能扩大权限或机械补发已经过时的动作。

用户必须能够停止新增风险、要求退出，并通过交易场所官方入口独立观察和接管。Halpha 不可用时，用户仍能撤销凭据并处置账户；产品不得把自身界面或 AI 作为最终控制的唯一入口。

这些要求属于功能正确性，不因资金较少、模拟成功或项目早期而取消。具体计划、检查、状态、接口和恢复方式由下位设计拥有。

## 6. 信任、凭据与数据

真实账户交易凭据使用实际可行的最小权限并可撤销，不具有无关的提现、转账或账户管理能力。凭据不得进入源代码、文档、AI 上下文、普通日志或测试制品；开发工具和探索环境不得取得发起真实账户交易动作的能力。

Halpha 必须能够导出和恢复维持计划、资金边界、事实、动作责任和用户控制所需的数据与配置。第三方组件可以提供能力，但不能取得用户决定权、成为停止与接管的隐藏依赖，或成为必要历史的唯一持有者。

## 7. 个人项目复杂度

Halpha 默认使用成熟组件、模块化单体、少量运行单元、单一事实权威、直接测试、普通 Git、短说明和人工决定。开发围绕一个可验证的小结果连续进行设计、实现、验证和修复；删除或外部工具能够解决的问题，不建立新平台、状态体系、记录族、审批或长期观察流程。

工程和治理投入随真实账户交易影响、不可逆性、实际并发、重复故障和维护规模增长。独立研究、Demo、公共行情只读和不发起交易所变更请求的开发不套用真实资金发布治理；真实账户交易只增加直接保护凭据、动作、核对、停止和接管的必要控制。

新增技术或抽象必须解决已观察问题，在全生命周期内净降低复杂度，并具有个人可理解的失败与退出路径。

## 8. 非目标与变更

Halpha 不以机构级审批、合规、生产委员会、高可用平台、微服务数量、统一治理矩阵、通用交易平台或替用户承担资金责任为目标。

只有使命、用户模型、资金责任、真实动作边界、最终控制或最高取舍发生根本变化时才修订本宪法。普通产品功能、技术、组件、配置和当前范围由下位文档决定。修订直接修改本文件并同步真正失真的下位规则；历史由 Git 保存。
