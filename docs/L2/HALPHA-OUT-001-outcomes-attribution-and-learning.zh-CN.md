# Halpha 单次激活复盘原则

**文档编号：** HALPHA-OUT-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** 用一份简短复盘连接一次计划激活的计划、当时事实、资金限制、动作、外部结果、成本、未知与开放责任，并记录可选的后续问题  
**本文档不负责：** 修改策略、计划、资金、权限、事实或动作；关闭相邻责任；自动证明因果或 Alpha；建立任务、改进交接、评分、审批或分析平台  

---

# 0. 唯一职责与复杂度上限【OUT-SCP-001】

OUT 只拥有一个稳定对象：复盘记录（`Review`）。一份 `Review` 回答这次激活发生了什么、哪些结论可靠、哪些仍未知、有哪些开放责任，以及用户下次是否需要改变决定。

复盘中发现的问题只写成可选的后续问题和建议语义所有者。只有项目所有者决定实际处理时，才按普通设计、开发或研究工作另行开始；OUT 不建立第二交接记录、任务、审批或自动反馈流程。

# 1. 复盘记录【OUT-REV-001】

每个 `PlanActivation` 最多有一条 `Review` 版本链。同一计划的多次激活分别复盘；后来事实修正或用户评价变化时追加新版本，不覆盖旧结论。

`Review` 引用已有权威记录，不复制相邻领域的完整状态。它至少连接计划与激活、评价使用的事实截止点、资金限制与检查、实际动作与场所结果、保护和退出、成本、用户接管、未知及开放责任。

# 2. 哪些结果可以复盘【OUT-CAS-001】

成交、部分成交、未触发、不行动、等待后结束、过期、拒绝、取消、结果未知和用户接管都可以复盘。非成交不是缺少结果，盈利也不是成功的充分条件。

结果未知时如实记录未知范围，不把数量、费用、盈亏或机会成本填为零。用户接管时可以评价 Halpha 接管前的行为，但无法归属的外部结果保持未知。

Demo 与 Live 的复盘身份、事实、动作和结论不得混合。模拟成功或盈利不能成为真实资金结论或权限依据。

# 3. 来源、时间与修正【OUT-EVD-001】

复盘只使用带来源且适合本次评价的计划、激活、场所事实、动作、费用、交互和运行材料。每个结论必须受其事实截止点限制；后来信息不能伪装成当时已经可知。

来源缺失、冲突或修正时，受影响结论保持有限或未知。后来修正足以改变结论时追加新 `Review` 版本，并保留原版本和变更原因；复盘不能改写源记录。

# 4. 简短评价与归因【OUT-EVL-001】

复盘只评价会改变下次决定的内容：账户结果及成本；计划与条件是否按固定规则工作；激活资金限制与机器执行范围是否生效；执行、保护和核对是否正确；用户控制和系统运行是否造成重要偏差。

盈利不能掩盖重复提交、保护缺口、未知结果或失去用户控制；亏损也不能单独证明计划、执行或系统错误。归因只写有证据支持的主要贡献和不确定性，不估算伪精确比例。

反事实只有在激活前已经固定，且能按当时可得事实和真实成本重建时才可比较。单次结果、模拟盈利或复盘评分不能自动提高资金限制、扩大机器执行范围或证明 Alpha。

# 5. 后续问题【OUT-FBK-001】

确有可能改变后续决定的问题，可以在 `Review` 中用普通文本记录：可观察问题、依据、影响范围、建议由哪个语义所有者重新判断。没有消费者、不会改变决定或只是重复描述的问题不记录。

该文本不自动创建任务，不要求目标领域接受，也不授权 OUT 修改其他记录。需要重新研究时，项目所有者可以把问题直接作为独立研究输入；需要产品修复时，按普通开发任务处理。

# 6. 完成、失败与下位边界【OUT-END-001】

一份复盘在来源、事实截止点、已知结论、未知和开放责任均已明确时即可记录；开放责任不必先关闭。复盘成功或失败都不能恢复激活、关闭订单或持仓责任、改变权限或阻止保护和接管。

L3 只定义 `Review` 的最小身份、引用、版本、幂等写入、查询、修正和关键反例。L4 只记录当前是否启用结构化复盘、当前字段选择和直接验证结果。没有当前消费者时，不增加聚合、评分、任务、worker、数据库或长期分析能力。
