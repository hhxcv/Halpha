# 0. 唯一职责【OUT-SCP-001】

## 0.1 OUT 责任定义【OUT-SCP-001-DEF】

OUT 只负责把一条交易计划启用记录及其引用的交易计划版本、事前决定、当时事实、用户选择、资金与权限检查、实际动作、外部结果、成本、开放责任和后续修正连接为简短复盘，并把可行动问题交给正确语义所有者重新决定。

## 0.2 OUT 责任边界要求【OUT-SCP-001-REQ】

OUT 可以形成复盘结论和复盘改进事项，但没有相邻领域的写权。历史计划、资金决定、动作、事实、用户指令和外部结果只能按各自拥有领域的规则形成新版本、修正或新决定。

盈利不能掩盖计划、执行、事实、交互或系统问题；功能正确也不能证明投资 Alpha。复盘完成不表示订单、持仓、保护、核对、事实未知或用户接管责任已经闭合。

---

# 1. 稳定对象【OUT-OBJ-001】

## 1.1 复盘记录定义【OUT-REV-001-DEF】

复盘记录，是对一条交易计划启用记录及其引用的交易计划版本、成交、非成交、结果未决或用户接管结局所形成的评价边界、来源事实、结论和开放责任的版本化记录。

## 1.2 对象责任边界【OUT-OBJ-001-REQ】

OUT 只保留复盘记录和复盘改进事项两个稳定对象。归因与产品增量价值只是适用复盘中的结论，不建立平行评价对象；没有当前消费者的其他复盘集合不建立对象。

---

# 2. 复盘边界与非成交结果

## 2.1 复盘建立要求【OUT-CAS-001-REQ】

一条交易计划启用记录在完成、失效、过期、拒绝、取消、未触发、本次不行动、等待后结束或其计划版本被替代时，可以建立或更新一份简短复盘。真实动作进入结果未决或用户接管、且该次启用不再产生新真实动作时，也可以开始复盘，但必须保持开放责任可见。

复盘边界必须固定交易计划启用身份、所引用的计划身份与版本、交易记录环境、账户、场所、评价时间、事实截止点和适用比较基准。同一个交易计划版本可以发生多次启用，每次启用分别复盘；同一条交易计划启用记录不因重复结束或责任闭合信号建立多份并行复盘。详细幂等和版本规则由 L3 定义。

## 2.2 非成交结果【OUT-NTD-001】

拒绝、本次不行动、等待后结束、条件未满足、事实未知、资金或权限检查拒绝、过期、外部变化前取消和用户接管，与成交结果具有同等复盘资格。

非成交结果不能补造成交价、费用、盈亏或机会成本。只统计成交会遗漏选择质量、正确拒绝、资金边界阻断、事实问题和系统停止，因此复盘必须说明没有产生真实动作的直接原因。

## 2.3 环境与结果不得混合【OUT-ENV-001】

复盘必须声明交易记录环境；涉及真实资金时还必须引用适用的 Halpha 真实资金操作权限。模拟与真实资金的账户、成交、费用、持仓、执行质量和结果不得合并，较弱环境的成功不能替代较强环境证据或提高权限。

# 3. 结果完整性

## 3.1 最小来源引用【OUT-EVD-001】

单次交易计划启用的简短复盘按适用范围引用：

- TRADEPLAN 的交易计划启用身份、固定计划版本、用户决定、结束原因和计划仍未闭合的责任；
- CAP 的资金使用上限与范围、Halpha 真实资金操作权限、停用状态和资金与权限检查结果；
- EXE 的待执行动作、场所尝试、部分或未知结果、保护、核对和用户接管责任；
- DAT 的当时事实、结果事实、成本、事实截止点、事实未知和后续修正；
- UX 的用户操作指令、处理回执、中断、误解与交互证据；
- 只有实际故障或维护问题影响本次结果时，才引用 SYS、ENG 或相应业务所有者的材料。

不适用内容明确排除；缺失内容必须标明范围和影响。复盘不得为补齐引用而制造事实、把未知写成零或复制相邻领域完整状态。

## 3.2 事实截止点与修正【OUT-TIM-001】

评价必须区分事前可知、决定时可知、真实动作时可知、结果后才知道和后来修正。事后信息不能用于美化当时判断；事实修正改变结论时形成新复盘版本，原版本及其当时依据继续保留。

---

# 4. 最小评价与归因

## 4.1 分开评价要求【OUT-QLT-001-REQ】

简短复盘至少分开回答：

| 评价范围 | 必须回答 |
|---|---|
| 账户结果 | 按 DAT 可引用事实，实际发生了什么；无法可靠计算时保持未知 |
| 计划 | 计划决定、条件、窗口和分支是否足以指导本次处理 |
| 资金与权限 | 已配置边界、交易计划当前确认、停用和检查是否按预期生效 |
| 执行与事实 | 动作是否唯一，场所结果、保护、核对、事实与修正是否正确 |
| 交互 | 用户是否看清环境、事实、动作、后果、回执和开放责任 |
| 系统与维护 | 只有实际系统或维护问题影响结果时，说明其失败和成本 |

没有材料或不会改变后续决定时，使用不适用或无法判断，不为填满评价收集额外数据。

## 4.2 最小归因【OUT-ATT-001-REQ】

归因只区分能够由证据支持的市场或外部因素、计划或用户决定、执行/数据/交互/系统因素，以及混合贡献或不足以归因。无法可靠拆分时保留混合或未知，不估算伪精确贡献比例，也不把剩余结果自动归给策略、用户或 Halpha。

## 4.3 反事实边界【OUT-CNT-001】

只有计划事前写明现金、不行动、等待或其他可行替代，且能够按当时事实和真实成本重建时，才进行反事实比较。不得在结果后挑选最有利价格、基准或交易方式，也不得把理论最优价格写成实际损失。

---

# 5. 产品增量价值的角色边界【OUT-VAL-001】

## 5.1 评价责任要求【OUT-VAL-001-REQ】

产品增量价值由项目所有者评价，比较项目所有者事前选定且用户实际可用的替代方式，并计入全部可归属建设、数据、运行和维护成本。用户仍以账户净结果和投资 Alpha 为主要结果；同一真人兼任两种角色时，两个判断视角也不得合并。

单份交易计划复盘可以提供产品使用证据，但不能独立证明稳定产品增量价值或决定项目是否继续。只有 L4 记录明确消费者后，才深化产品评价；本文件不预建评价周期、固定集合、指标体系或自动分析流程。

---

# 6. 复盘改进事项【OUT-FBK-001】

## 6.1 复盘改进事项定义【OUT-FBK-001-DEF】

复盘改进事项，是从复盘记录中形成、带证据与适用范围并提交给一个明确语义所有者决定是否接受的可行动问题。

## 6.2 交接要求【OUT-FBK-001-REQ】

复盘改进事项必须引用来源复盘、可观察问题、影响范围、证据和预期改变，并且只交给一个拥有相应决定或失败行为的语义所有者。OUT 不直接修改目标领域；涉及多个领域时分别交付，但保持共同复盘来源。

需要重新研究或形成新计划的问题可以交给 CTX；既有计划、资金、执行、事实、交互、系统或工程问题直接交给相应拥有领域。没有消费者、不会改变决定或失败行为且不能降低已观察成本的事项，应拒绝或删除，不建立长期待办平台。

---

# 7. 版本与评价完成【OUT-VER-001】

复盘的证据、事实判断、评价方法或适用范围发生实质变化时形成新版本，旧版本继续说明当时依据和结论。

只有适用来源已经取得或明确缺失、评价项已完成或声明不适用/未知、开放责任明确、实际问题已经交接或说明无需行动时，评价才可完成。评价完成不关闭任何相邻领域责任；后续新事实可以形成新评价版本，但不能重开已经结束的外部变化。

---

# 8. 明确输入与输出【OUT-HOF-001】

| 方向 | 内容 |
|---|---|
| 接收 | TRADEPLAN、CAP、EXE、DAT、UX 的单次交易计划启用复盘所需稳定引用；实际影响结果时再接收 SYS、ENG 材料 |
| 输出 | 单次交易计划启用的简短复盘记录，以及目标语义所有者明确的复盘改进事项 |

详细字段、查询、并发、测试和 UI 交接由 L3 定义；当前建设选择的复盘范围、指标/基准/成本口径和验收证据由 L4 记录，具体复盘事实与运行状态仍由 OUT 及来源领域保存。没有当前消费者的其他复盘范围和产品评价自动化不在本文件展开。
