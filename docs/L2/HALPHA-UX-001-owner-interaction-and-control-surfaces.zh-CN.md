# Halpha 用户交互与控制原则

**文档编号：** HALPHA-UX-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-VIS-001、HALPHA-FLOW-001  
**本文档负责：** 用户任务呈现、命令提交、结果反馈、中断恢复和控制可达性  
**本文档不负责：** 研究、计划、资金、事实、执行的业务语义，以及当前页面和技术组件

## 1. 唯一职责

UX 让用户正确理解当前任务、提交明确命令并看到真实处理结果。业务状态由拥有领域提供；页面、通知、图表和浏览器缓存只是视图或入口。

交互围绕用户任务，不围绕内部模块、证据或治理对象。默认用少量入口和直接路径完成高价值任务，只有真实使用暴露重复问题时才增加页面或定制。

## 2. 命令与结果

用户提交会改变状态的命令前，界面重新读取对象、事实、允许范围和时限，显示本次后果，再提交稳定请求身份。重复或并发请求得到原结果或明确冲突。

HTTP 成功、按钮禁用、通知送达和页面提示不能冒充场所动作或责任已经完成。结果无法确认时按原请求查询，不能盲目重发。

计划固定、计划启用、停止新增风险、退出和用户接管是不同决定。UX 只呈现和提交对应领域命令，不重新定义它们或把普通“确认”扩展为其他权限。

## 3. 当前任务与信息层次

默认先回答：当前怎样、是否需要行动、为什么、主要未知和下一步。来源、历史、模型、执行尝试和技术诊断按需展开；资金影响、事实未知、动作未知和保护缺口不能隐藏。

主视口空间按决策价值分配。当前敞口、保护、未知、时限和可执行动作优先；不改变当前判断的背景说明、重复身份和技术元数据不得占据主要任务区域，也不得因视觉容器或留白获得超过其价值的面积。

信息载体按用户任务连续性选择，而不按数据对象数量机械拆分。需要同时比较或连续操作的内容保持同页；不同时间范围或互斥视角不混排；补充说明和证据按需展开；只有独立任务、深链恢复或大量内容才使用独立页面。

可增长的历史或明细列表首屏必须有界，并通过显示更多或分页继续读取；不能把全部历史默认堆入主要任务页面。

界面区分来源材料、事实、推断、AI 建议、用户决定、Halpha 处理结果和场所结果，同时区分当前、过期、冲突、未知和已闭合。颜色不是唯一表达方式。

中断后恢复先显示对象版本、期间变化、过期和冲突。旧预览、旧通知和过期确认不能复活。

## 4. 环境与控制

Demo 和 Live 可以共享界面结构，但环境、账户、资金、动作和结果持续可辨。提高风险的决定不默认选中，不用收益、动画或紧迫感诱导行动。

停止新增风险、退出和交易场所官方入口接管始终直接可达。界面说明停止不等于撤单或平仓，接管不等于责任已经闭合。主要界面不可用时，用户仍能取得外部接管说明。

## 5. 视觉与验证

Halpha 使用明亮、克制、专业、数据优先的工作台视觉语言：中性表面承担大部分内容，单一结构强调色只标识当前选择、主要行动和焦点，语义色只表达成功、警告、危险和信息。数字、单位、正负号和时间稳定；面向用户的时间统一按项目所有者配置的时区转换并明确标注，来源时间和事实截止语义仍由 DAT 保存。危险动作、未知和保护缺口同时用文字与视觉区分；键盘、焦点、对比度和状态播报满足实际使用。

实际交互验证用户能否区分环境和结果、避免重复命令、看见未知、停止和接管，并在陈旧或失败页面中保持正确限制。当前页面、设备和直接结果由 L4 记录。
