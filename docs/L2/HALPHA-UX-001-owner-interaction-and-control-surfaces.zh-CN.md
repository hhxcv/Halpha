# 0. 非规范性摘要

本文固定十一项结论：

1. 用户围绕任务和决定工作，不围绕软件模块巡检。
2. 所有需要用户处理的事项进入一个列表；真实动作异常和有时限决定优先，正常状态让用户快速进入高价值交易任务。
3. 事实、观察、研究或交易计划候选、候选优势、决定、命令和外部结果必须在视觉和交互上可区分。
4. 页面、用户任务通知、图表和本地缓存都是从权威状态生成的只读视图或命令入口，不保存平行业务状态。
5. 计划版本固化、交易计划启用、交易计划当前确认、Halpha 真实资金操作权限和提高资金使用上限或扩大范围是不同决定。
6. 用户任务通知只把用户带回当前用户交互任务；旧通知、旧预览和重复操作不能增加真实资金风险。
7. 专业性通过准确语义、比较和可追溯上下文体现，不通过高密度、术语堆叠或操作数量体现。
8. 主要工作入口支持深度判断，限时操作入口支持短时决定；两者调用同一业务命令。
9. 高级定制可以改变呈现、筛选和草案效率，不能改变事实、资金使用上限与范围、交易计划启用、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态和外部动作结果。
10. 交互质量按任务正确完成率、步骤、耗时、理解负担、误操作、停用可达性和恢复连续性评价，不按页面数、停留时长或交易频率评价。
11. FLOW 的六类用户任务域、跨域当前状态入口、DAT 交易记录环境身份和 CAP 的 Halpha 真实资金操作权限是相互独立的分类维度；UX 只负责保持这种分类独立，不负责把它们固定为相同数量的页面或一级导航。

# 1. 职责边界与交互对象

## 1.1 UX 的唯一职责【UX-SCP-001】

### UX 责任定义【UX-SCP-001-DEF】

UX 只拥有以下交互语义的定义责任：用户现在看见什么、以什么顺序理解、能够提交哪些相应业务领域已经定义的命令、怎样获知处理结果、怎样在中断和多个交互入口之间保持同一用户交互任务。

### UX 责任边界要求【UX-SCP-001-REQ】

Halpha 按各拥有领域的规则向交互层提供对象、事实、决定、允许命令和失败结果。UX 不从显示状态推断或改变这些含义，也不因“更方便”增加业务能力。

## 1.2 四类交互对象【UX-OBJ-001】

### 四类交互对象定义【UX-OBJ-001-DEF】

| 对象 | 含义 |
|---|---|
| 用户交互任务 | 一个需要用户理解、决定或知悉的稳定任务身份 |
| 用户操作指令 | 用户对明确任务和版本提交的意图，不等于处理完成 |
| 用户操作处理回执 | Halpha 对接收、拒绝、处理中、过期、转交或处理状态无法确认的可追溯说明，不等于外部结果 |
| 用户任务通知 | 指向用户交互任务的最小送达载荷，不是任务状态、用户决定或真实动作 |

### 交互对象使用要求【UX-OBJ-001-REQ】

任务所需信息和当前决定所需信息是从拥有领域生成的只读视图，不建立 UX 业务对象。视图和图表同样只读取上述对象及业务领域状态，不建立平行状态。

### 四类交互对象的选择依据【UX-OBJ-001-RAT】

用户交互任务保持跨中断身份，用户操作指令表达用户请求，用户操作处理回执表达 Halpha 处理状态，用户任务通知只负责送达；四者的消费者和失效规则不同。任务所需信息和决定预览都可从拥有领域的权威记录重建，建立独立对象会造成陈旧副本和状态冲突，因此只作为只读视图。

## 1.3 用户操作命令闭环【UX-CMD-001】

### 用户操作命令闭环定义【UX-CMD-001-DEF】

```text
打开用户交互任务身份
→ 重新取得当前对象、事实、权限和时限
→ 形成当前决定预览
→ 用户提交明确的用户操作指令
→ 用户或 Halpha 按拥有领域规则校验、接受或拒绝
→ 返回用户操作处理回执
→ 需要交易场所或账户变化时继续进入 CAP / EXE
→ Halpha 以 DAT 事实和拥有领域终态更新任务
```

### 命令闭环使用要求【UX-CMD-001-REQ】

所有 Halpha 交互入口必须采用上述同一闭环。

按钮点击、用户任务通知快捷操作、页面成功提示、客户端请求完成和内部队列接收都不能表示成交、保护、核对或业务责任已经完成。

未取得可确认的用户操作处理回执时，不得推断指令未接收、成功或失败，也不得盲目重新提交；必须按原用户操作指令身份查询处理状态及关联业务责任。处理状态无法确认只说明 Halpha 不能确认该指令的处理状态；若外部动作可能发生，交互层另行显示 EXE 拥有的执行结果未决，不把二者合并。

## 1.4 用户任务域与交互入口相互独立【UX-WRK-001】

### 用户任务域与交互入口关系

FLOW 的市场观察与实时情报、策略研究与验证、交易计划与演练、交易执行与持仓管理、账户与交易记录、回顾分析与学习，是六类业务任务域。主要工作入口、限时操作入口、用户任务通知入口、技术后备入口和交易所官方入口，是交互形态。两者不是一一对应关系，也不预先决定菜单数量。

当前状态与用户待处理事项入口是跨任务域只读视图：它汇总有时限决定、活跃计划、订单或持仓异常、重要变化和用户订阅事项，并把用户送回相应用户交互任务。它可以成为最终产品首页，也可以由用户任务通知、搜索或保存视图承担。

### 跨任务域导航与环境区分要求【UX-WRK-001-REQ】

用户应能从搜索、用户任务通知、保存视图和对象关系直接进入计划、订单、持仓、交易记录、策略或复盘，并保留来源、返回点和未完成决定。正常工作不得要求先访问某个领域首页，也不得在从研究进入计划、从计划进入交易所模拟盘/真实动作、从持仓进入记录或从记录进入复盘时丢失上下文。

不得因 L2 使用某个入口名称而冻结最终信息架构。

交易所模拟盘与真实资金交易可以共享计划编辑、订单理解、记录查询和复盘交互，但交易记录环境身份必须持续可见。任何页面、用户任务通知、快捷键、默认值或颜色都不能让用户把模拟动作误认为真实动作，或在不知情时从模拟切换为真实资金。

# 2. 注意力与使用场景

## 2.1 一个用户待处理事项列表【UX-ATT-001】

### 用户待处理事项列表定义【UX-ATT-001-DEF】

用户待处理事项列表是 UX 对跨领域待处理事项形成的用户交互任务只读视图；它不改变来源领域的状态和优先级决定。

### 待处理事项内容与排序要求【UX-ATT-001-REQ】

每个任务必须让用户看清涉及什么业务和账户或计划、何时失效、当前处于什么状态、需要作出什么决定以及不响应的结果；具体字段和显示形式由 L3 按实际交互入口定义。

排序按实际后果和时限依次优先处理：可能重复的真实动作、执行结果未决、关键账户事实未知和用户接管；即将到期的计划或订单决定；活跃计划、持仓、交易计划启用、Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态变化；高价值研究或交易计划候选、研究、计划准备和复盘；最后才是摘要和低影响维护信息。同一来源的重复用户任务通知和待处理事项聚合为一个用户交互任务。无异常时，当前状态入口不得阻止用户直接进入实时情报、研究回测、计划演练、交易记录查询或复盘。

没有需要处理的事项时，界面应明确给出“当前无待办”、事实截止点和必要的运行状态及故障影响范围，并允许立即退出；不得用空内容流制造使用需求。

## 2.2 使用场景【UX-SES-001】

### 五种使用场景及交互要求【UX-SES-001-REQ】

| 使用场景 | 交互目标 |
|---|---|
| 无需交互时 | 无事不打扰 |
| 快速查看 | 判断当前状态和是否必须处理一项决定 |
| 完成普通任务 | 完成研究或交易计划候选筛选、标准计划、交易所模拟盘动作、交易记录查询或短复盘 |
| 集中处理复杂任务 | 实时情报综合、策略研究、回测或历史行情回放、策略比较和复杂计划 |
| 处理故障 | 先停止影响，再核对和恢复 |

| 使用场景 | 主要要求 |
|---|---|
| 无需交互时 | 只有真实需要用户处理的事项才进入用户待处理事项列表 |
| 快速查看 | 突出变化、账户或计划影响、时限、选项和无响应结果 |
| 完成普通任务 | 上下文连续，可保存任务检查点 |
| 集中处理复杂任务 | 支持比较、证据展开和跨工具工作，不成为日常使用前提 |
| 处理故障 | 未决 ExecutionAction、所属环境与账户事实、已停止范围、必须处理事项和用户接管最先出现 |

具体时间目标由 L3 按场景定义，实际使用时间进入 L4。任何正常流程不得假设用户长时间连续在线。

## 2.3 中断与恢复【UX-INT-001】

可暂停任务恢复后必须能够还原原对象版本、已完成决定、未决问题和当时来源截止点，且不得静默忽略中断期间的变化、过期或冲突；具体检查点字段由 L3 定义。界面先显示这些变化和当前限制，再允许继续、重做或丢弃。

时间受限的交易计划当前确认、旧用户任务通知和旧预览不能在恢复后继续使用。交易场所或账户变化可能发生时，恢复入口必须先进入查询、保护和核对，而不是恢复原操作步骤。

并发编辑、多个设备和重复提交必须产生可理解冲突或不会产生重复真实动作的结果；不得以最后写入覆盖已经生效的决定。

# 3. 信息层次与呈现

## 3.1 任务信息四层结构【UX-INF-001】

### 任务信息四层结构定义【UX-INF-001-DEF】

| 层次 | 默认回答 |
|---|---|
| 结论层 | 当前状态怎样、有什么值得做、必须做什么、时限是什么 |
| 决定层 | 发生了什么变化、资金影响、主要支持与反证、哪些未知会改变决定、有哪些选项 |
| 依据层 | 相关事实、计划、证据、风险、动作和历史版本怎样支持当前决定 |
| 诊断层 | 原始来源、完整时间线、技术状态、日志和重演材料是什么 |

### 使用场景与四层信息的选择依据【UX-SEM-001-RAT】

五种使用场景用于决定注意力、中断恢复和响应强度，四层信息用于决定同一任务默认展示多少依据与诊断细节；二者分别被任务入口和信息呈现消费，不是会话实体、页面类型或需要保存的业务状态。相邻场景或层次长期采用相同交互行为时，独立分类的价值依据消失；维持分类数量本身不会产生入口、页面或记录价值。

### 信息层次使用要求【UX-INF-001-REQ】

快速查看时默认停在前两层；复杂研究和故障可以深入。层次表示渐进取得，不允许把关键反证、资金使用上限与范围、事实未知状态、执行结果未决和外部责任永久藏到深层。

## 3.2 信息性质、状态与所有权必须可辨【UX-SEM-001】

界面必须使用一致的标签、位置、形态和时间说明，区分：

- 来源材料、带来源的外部观察、可引用事实判断、可追溯派生结果和其他业务记录；
- 研究或交易计划候选、候选优势、交易计划依据、交易计划和策略版本；
- 建议、用户决定、资金与权限检查、环境限定 ExecutionAction 和外部结果；
- 当前、过期、冲突、事实未知、执行结果未决、受限和已闭合；
- Halpha 动作、交易所人工操作和用户接管。
- DAT 定义的交易记录环境身份与 CAP 定义的 Halpha 真实资金操作权限，以及各自的资金、订单、持仓与结果；交互只读视图不复制二者定义。

颜色不得成为唯一区分手段。任何摘要必须能够回到来源、版本和事实截止点。

## 3.3 决策呈现【UX-COG-001】

涉及资金的当前决定预览默认显示：对象和当前版本、相对原决定的关键变化、当前资金影响、最强支持与反证、会阻断决定的未知、可选动作、各选项后果、剩余时间和不响应结果。

Halpha 生成的默认值和推荐必须标明来源、适用范围和是否可编辑。推荐不得默认选中增加风险的真实动作，不得以收益、紧迫动画、社交比较或交易次数诱导行动。

用户需要临场重新研究才能理解的时间受限决定，应拒绝、暂停或过期，而不是在快速查看场景中压缩成无上下文按钮。

# 4. 视觉与交互品味【UX-VIS-001】

Halpha 采用克制、专业、数据优先的视觉语言：

- 结构和留白承担主要层次，不依赖装饰、卡片数量和大面积强调色；
- 默认信息密度适合用户快速比较，但一屏只突出当前用户交互任务的少数决定性信息；
- 数字、单位、正负号、小数位和时间对齐稳定，表格和时间线优先保证可扫描与可核对；
- 颜色只表达少量稳定语义，并同时使用文字、图标或形态；盈亏颜色不等于风险、安全或建议；
- 动画只用于解释状态变化、位置关系或等待，不用于制造兴奋、紧迫和参与度；
- 主要动作、危险动作、停止控制和用户接管在层次与措辞上明确不同；
- 正常状态安静，异常强调与真实资金影响相称；长期高亮视为设计失败；
- 深色和浅色主题、设备尺寸和输入方式可以不同，但业务语义、顺序和停用可达性一致。

主要工作入口优先支持实时情报、研究回测、计划演练、交易管理、记录查询和复盘中的深度任务；限时操作入口优先支持当前状态扫视、时间受限的交易计划当前确认、持仓保护、停止和异常处理。限时操作入口不是主要工作入口的缩小复制，主要工作入口也不能成为时间受限响应的唯一入口。

键盘、指针和触控按实际交互入口提供高效路径；关键决定不能只依赖悬停、手势、精细拖拽或颜色。可访问性、缩放和可读性至少覆盖用户实际使用条件，并在 L3 验证。

# 5. 用户任务通知、确认与控制

## 5.1 用户任务通知【UX-NTF-001】

用户任务通知的处理方式由任务优先级、完整人工路径可行性和用户可达性决定。用户任务通知必须把用户带回同一用户交互任务，使其在不过度暴露敏感内容的前提下理解资金影响、时限和失效结果；具体载荷字段由 L3 定义。

用户任务通知失败不得扩大权限，也不得阻止事前确定的自动收缩。重复用户任务通知必须聚合；已关闭、过期或被替代任务的通知不能形成新决定。

用户任务通知快捷操作只有在重新认证、重新取得当前用户交互任务和当前版本、调用共同业务命令并接受拒绝或过期结果时才可用。本地载荷、缓存或送达回执不能直接形成真实动作。

## 5.2 不同决定保持不同命令【UX-CTL-001】

| 决定或控制 | 必须独立表达 |
|---|---|
| 停用 Halpha 新增真实动作 | 适用范围和可确认的用户操作处理回执；处理状态无法确认时不得显示停用已生效或未生效，必须按原指令身份查询并提示保持受影响范围受限；不暗示场所订单或仓位已改变 |
| 计划版本固化 | 形成不可变交易计划版本；不产生交易计划启用或 Halpha 真实资金操作权限变化 |
| 交易计划启用 | 使既有计划版本在明确适用范围和期限内承担观察、事件和交接责任 |
| 进入交易所模拟盘 | 选择交易所提供的模拟环境与账户；不得取得真实写入路径 |
| 由交易所模拟盘进入真实资金 | 明确真实账户、交易记录环境身份、Halpha 真实资金操作权限、资金使用上限与范围并重新启用交易计划；不得继承模拟环境中的交易计划当前确认或动作状态 |
| 交易计划当前确认 | 仅确认当前有效预览和一次拟执行真实动作 |
| 提高 Halpha 真实资金操作权限 | 明确账户、计划、期限和允许动作范围；不得与单笔确认合并 |
| 提高资金使用上限或扩大范围 | 新金额或比例、适用范围和立即撤回方式；不得与单笔确认合并 |
| 场所执行动作 | 适用计划、环境与资金权限检查、ExecutionAction 和 EXE 结果 |
| 用户接管 | 用户使用 Halpha 之外的交易场所或账户官方路径进行观察、撤权和处置 |

一次连续交互可以产生多个决定，但必须形成独立命令、结果和关联关系。不得用一个“确认”控件混合计划版本、交易计划启用、Halpha 真实资金操作权限和真实动作。

CON-CMP-003 已定义三类控制要求。Halpha 必须按 UX 规则把相应状态、命令和结果正确呈现并送入共同业务命令；同一用户交互任务可以组合三类信息，但不得改变其标签、结果、归因或语义所有者。

# 6. 异常、停用与用户接管【UX-INC-001】

异常入口按以下顺序显示：当前订单与仓位；执行结果未决和外部写入控制权；Halpha 新增真实动作停用状态及适用范围；仍处于事实未知状态的内容；用户可以采取的行动；用户接管方式；恢复 Halpha 真实资金操作权限前仍需核对的内容。

停用或降级状态必须说明适用范围、原因、禁止能力和仍在运行的查询/核对任务，不用一个红黄绿或复杂全局模式代替。

用户接管材料必须在主要界面、用户任务通知服务或 Halpha 主机不可用时仍可取得。用户接管始终指用户使用 Halpha 之外的交易场所或账户官方路径，UX 只定义找到和理解该路径的交互要求，不代表 Halpha 能够决定或观察接管结果；恢复后由执行器或核对后台任务按 DAT/EXE 规则核对外部事实。

# 7. 计划、交易所模拟盘与价格—时间视图【UX-CHT-001】

计划工作入口是用户发起半自动交易的主要入口。它必须允许从用户观点、研究或交易计划候选、研究结果、既有计划或持仓处置需求开始，在同一任务内完成草拟、比较、情景演练、环境选择、版本固定和启用准备；正式研究不是使用计划工作入口的必经步骤。

计划视图必须把同一交易计划版本的当前价格、进入、目标、保护、失效、期限、触发、订单、成交和实际结果放在一致时间语义中，并显示数据截止点、已知数据缺失或不完整范围、交易计划条件无法判定和事实未知状态。

计划视图必须持续显示当前交易记录环境身份和 Halpha 真实资金操作权限。历史行情回放、交易所模拟盘和真实资金使用不同且不可混淆的账户/资金标识、真实或模拟动作标识、订单/持仓状态和结果标签；从模拟进入真实环境时，七类计划决定及其决定性引用均未变化时可复用同一不可变交易计划版本，但必须建立新的真实环境启用记录；任一决定或引用变化时必须先形成并固定新版本。两种路径都不得复制确认、成交、持仓或已完成状态。

该视图服务计划理解和运营，不复制通用专业图表。它应能进入外部专业图表的同一市场和周期，也能从外部工具返回当前 CTX/TRADEPLAN 任务。

图形编辑只形成草案命令；不能直接改变已启用计划、条件或事实。可视化层必须可从 DAT 事实与相应业务记录重建，外部布局和浏览器状态不得成为唯一语义来源。

# 8. 跨工具与多个交互入口

## 8.1 跨工具交接体验【UX-HND-001】

UX 负责定义跨工具交接的连续性要求。Halpha 必须保持同一用户交互任务身份、返回位置、作出当前决定所需的信息和可理解失败；业务交接内容由 CTX、ALP、TRADEPLAN、EXE 或 OUT 定义，来源身份由 DAT 定义，传输与适配由 SYS 定义。

外发内容按任务最小化，不包含真实交易凭据。外部链接、令牌、页面和 AI 会话不能构成经济证据判断、交易计划启用或 Halpha 真实资金操作权限决定。返回内容首先进入相应材料、带来源的外部观察或外部活动路径，不因交互顺利而自动被接受。

## 8.2 多入口分工【UX-SRF-001】

### 五类交互入口及职责【UX-SRF-001-REQ】

| 交互入口 | 主要职责 |
|---|---|
| 主要工作入口 | 六类用户任务域中的深度工作；具体组合和导航由真实任务测试决定 |
| 限时操作入口 | 跨域状态扫视、有时限决定、当前交易、停止、持仓和保护 |
| 用户任务通知入口 | 把用户带回当前用户交互任务 |
| 技术后备入口 | 当前构建、部署说明、普通日志入口、现成备份方式和用户接管材料 |
| 交易所官方入口 | 场所观察、人工计划操作和独立最终控制 |

### 多入口一致性要求

主要工作入口不得成为时间受限响应、停用或接管的唯一前提；限时操作入口不承载完整研究、回测或复杂计划；用户任务通知入口不保存状态，也不得仅凭载荷确认或直接动作；技术后备入口不得形成专门稳定性产品，也不得替代业务核对与用户启用决定；交易所官方入口不得被 Halpha 视图取代。

所有 Halpha 交互入口共享相同任务、对象、命令和从权威状态生成的可重建只读视图。离线缓存只支持声明的读取或草案能力，不能恢复过期决定或绕过重新校验。

交互入口形态不得拥有独立业务状态、授权结论或业务流转。用户从一种受支持形态切换到另一种时，必须按同一对象身份读取当前权威状态并继续同一任务和命令闭环；不得复制、迁移或猜测业务状态，也不得因入口形态改变命令后果、失败、未知、停止、恢复或结束语义。新增形态的具体安全与技术契约只在真实建设范围出现后由适用 L3 定义。

## 8.3 AI 辅助交互【UX-AIX-001】

对话、自然语言输入和生成式解释可以作为研究或交易计划候选整理、研究、比较、计划草拟和结果理解的交互方式，但不建立一套以聊天记录为中心的平行业务流程。

- 每次 AI 协助必须关联用户交互任务、适用对象版本和事实截止点；跨会话继续读取任务所需信息，不依赖模型记忆。
- AI 输出必须区分事实引用、外部主张、推断、建议和明确未知，并能进入 CTX、ALP、TRADEPLAN、OUT 或其他拥有领域的正式接受边界。
- 用户的自然语言表达先形成可检查的草案或待检查的用户操作指令；任何涉及资金的变化都必须由 Halpha 按相应业务领域规则生成当前决定预览，并通过第 1.3 节的共同命令闭环明确提交。
- 模糊、冲突或无法绑定当前对象的表达不能执行；AI 自信度、会话连续性和用户在对话中的普通肯定均不构成事实、经济证据判断、交易计划启用、Halpha 真实资金操作权限或真实动作完成。
- 会改变决定的材料必须形成可追溯记录并保留来源；完整聊天、模型私有状态和外部会话标识不得成为计划、证据或结果的唯一载体。
- AI 不取得生产凭据；业务状态与对话摘要冲突时，以权威状态和拥有领域结论为准，并向用户显示差异。

# 9. 有界定制【UX-CUS-001】

允许定制：信息密度、排序和保存视图；研究或交易计划候选及研究偏好；用户任务通知渠道和静默窗口；图表显示；模板和草案默认；常用外部工具交接。

不得定制：事实和事实未知状态的含义；经济证据；用户设定的资金使用上限与范围、交易计划启用、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态和防重复要求；外部动作结果；必须可见的当前订单、仓位、执行结果未决和未解除责任。

模板和默认值只影响未来草案，必须有来源、版本、重置和退出方式。自定义脚本、表达式、模型和外部告警不能通过高级模式直接取得真实资金写入能力。

# 10. 交互质量【UX-QLT-001】

每个 L3 交互切片必须按适用的 FLOW 场景、交互入口、信息状态、责任阶段和命令后果验证：

- 用户能否快速识别当前状态、价值最高的任务、必须处理事项、时限、选项和无响应结果；
- 任务能否跨中断、设备和外部工具保持同一身份；
- 事实、研究或交易计划候选、候选优势、决定、命令、回执和外部结果是否没有混淆；
- 回测、交易所模拟盘和真实交易的环境、资金、订单、持仓和结果是否不会混淆或静默迁移；
- 过期、冲突、重复、缓存和恢复是否不会复活旧决定或产生重复交易场所或账户变化；
- 停止、异常、降级和用户接管能否直接到达；
- 真实渲染与完整操作链是否符合相应使用场景的目标。

以下属于硬失败：把事实未知状态、研究或交易计划候选、候选优势显示为当前事实；把命令接收显示为外部完成；旧用户任务通知或预览能够新增风险；隐藏当前风险敞口或保护缺口；让页面成为保护、核对或恢复的唯一边界；涉及资金的命令绕过共同业务命令。

# 11. 明确的输入输出【UX-HOF-001】

| 方向 | 内容 |
|---|---|
| 接收 | 各业务领域的稳定对象、当前决定所需信息、允许命令、业务结果和失败行为；DAT 的事实、时间和事实未知状态；当前可达性、凭据边界与外部账户入口等简单运行事实 |
| 输出 | 用户操作指令、用户操作处理回执的交互证据、可达性与用户任务通知偏好、任务中断和恢复信息、实际理解和交互质量证据 |

交互过程产生的信息只能作为拥有领域的命令输入或工程评价材料，不能自行形成业务决定。

# 12. L3 必须精确定义【UX-L3-001】

每个实际交互切片必须定义：

- 用户交互任务、用户操作指令、用户操作处理回执和用户任务通知的身份与契约，以及任务和决定只读视图的形成规则；
- FLOW 优先级到队列、聚合、用户任务通知和关闭条件的映射；
- 六类用户任务域、跨域当前状态入口、搜索/直达和回流路径到实际页面与导航的映射；
- 各使用场景的进入、退出、中断、冲突、过期和恢复；
- 信息性质与所有权、事实截止点、事实未知状态、执行结果未决、决定与外部结果的显示规则；
- 页面和导航结构、限时操作入口、视觉样式变量、排版、数值、颜色、动效和可访问性；
- 命令预览、确认、认证、防重复提交、拒绝和结果回流；
- 计划价格—时间视图、回测/交易所模拟盘/真实环境辨识、模拟转真实时重新启用交易计划的边界与外部图表交接；
- 多入口同步、缓存、离线、故障和用户接管；
- AI 辅助交互的任务绑定、信息性质与所有权、草案接受、命令确认、中断恢复和越界测试；
- 允许定制项、默认、版本、重置和越界检查；
- 真实渲染、关键任务和交互硬失败测试。

# 13. L4 当前事实范围【UX-L4-001】

L4 只在相应能力实际存在时记录当前页面、导航、组件、设备、交互入口、用户任务通知渠道、外部工具、AI 辅助交互入口、实际视觉样式变量、任务映射、时限、静默与可达性配置、模板和定制项、缓存与离线范围、交互测试、任务时长、中断、误解、重复、无效用户任务通知、已知限制和改进状态。L4 不建立新的业务或交互原则。
