# Halpha 用户交互与操作控制原则

**文档编号：** HALPHA-UX-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** 用户看见什么、按什么顺序理解、怎样提交领域已经定义的命令、怎样得知结果，以及怎样在中断和不同入口之间继续同一任务  
**本文档不负责：** 研究、计划、资金、授权、事实、执行和结果的业务语义；页面清单、内部记录、接口、技术平台、渠道和当前实现

---

# 0. 职责与复杂度上限【UX-SCP-001】

## 0.1 UX 责任定义【UX-SCP-001-DEF】

UX 只负责让用户正确理解当前任务、提交明确命令并看到真实处理结果。业务状态始终由拥有领域提供；页面、通知、图表和本地缓存只是可重建视图或入口。

## 0.2 UX 责任边界要求【UX-SCP-001-REQ】

交互围绕用户任务，不围绕内部模块和治理对象。页面与路径数量必须和实际使用规模及后果相称，默认用少量页面和直接路径完成高价值任务；只有真实使用暴露重复问题时，才增加入口、分类或定制能力。

---

# 1. 共同命令闭环【UX-CMD-001】

## 1.1 命令闭环定义【UX-CMD-001-DEF】

```text
打开当前任务
→ 重新读取对象、事实、权限和时限
→ 显示本次决定及后果
→ 用户提交明确命令
→ Halpha 接受或拒绝并返回当前处理结果
→ 涉及外部动作时继续执行与事实核对
→ 用业务终态更新任务
```

## 1.2 命令闭环要求【UX-CMD-001-REQ】

按钮点击、请求返回、通知送达和页面成功提示都不能冒充订单、保护或核对已经完成。处理结果无法确认时，界面必须保留原命令身份并查询，不能盲目重发或猜测成功/失败。

重复提交、多个入口和并发操作必须得到同一结果或明确冲突，不得产生重复真实动作。任何入口都调用同一业务命令，不保存平行业务状态。

---

# 2. 当前任务与上下文【UX-WRK-001】

用户进入 Halpha 时，先看见当前订单、持仓、活跃计划、有时限决定、重要变化、事实未知和 Halpha 是否可用。无事时明确显示“当前无待办”，并允许退出或直接进入市场观察、研究、计划、交易记录或复盘。

## 2.1 导航与环境区分要求【UX-WRK-001-REQ】

用户应能从搜索、通知、保存视图或对象关系直接进入相关任务，并保留来源、返回点和未完成决定。任务域不等于页面数量，也不要求先访问领域首页。

交易所模拟盘与真实资金可以共享计划和记录界面，但环境、账户、资金、授权、动作和结果必须持续可辨。不得用颜色、默认值、通知或入口切换让用户误把模拟动作当成真实动作。

## 2.2 待处理事项【UX-ATT-001】

待处理事项只收录真正需要用户知悉或决定的内容。每项显示对象、影响、时限、当前状态、用户可做什么和不响应的结果【UX-ATT-001-REQ】。

排序优先考虑可能重复的真实动作、结果未知、账户事实未知、保护缺口和用户接管，其次是即将到期的决定，再次是正常研究、计划和复盘。重复提醒合并，低影响维护信息不得挤占交易任务。

## 2.3 中断与恢复【UX-INT-001】

恢复任务时显示对象版本、期间变化、过期和冲突，再允许继续、重做或放弃。旧通知、旧预览和已经过期的确认不能复活；外部动作可能发生时，先查询和核对事实，不恢复原点击步骤。

# 3. 信息呈现【UX-INF-001】

## 3.1 信息层次使用要求【UX-INF-001-REQ】

默认先回答：现在怎样、是否需要行动、为什么、主要未知和下一步是什么。来源、历史、模型、执行尝试和技术诊断按需展开。关键反证、资金影响、事实未知、动作结果未知和开放责任不能被藏在深层。

## 3.2 信息性质与所有权必须可辨【UX-SEM-001】

界面必须区分来源材料、事实、推断、AI 建议、用户决定、Halpha 命令、Halpha 处理结果和交易场所结果；同时区分当前、过期、冲突、未知、受限和已闭合。颜色不是唯一手段，摘要必须能回到来源、版本和事实截止点。

## 3.3 涉及资金的决定【UX-COG-001】

当前决定预览至少显示对象和版本、环境与账户、资金影响、相对原决定的变化、主要依据与反证、阻断性未知、选项、时限和不响应结果。提高风险的动作不得默认选中，也不得用收益、紧迫动画或交易次数诱导行动。

如果用户必须临场重新研究才能理解一个时间受限决定，交互应让其等待、拒绝或过期，而不是压缩成无上下文按钮。

---

# 4. 视觉与操作效率【UX-VIS-001】

Halpha 使用克制、专业、数据优先的视觉语言：

- 结构和留白承担层次，默认只突出少数决定性信息；
- 数字、单位、正负号、小数位和时间稳定且便于核对；
- 颜色同时配合文字或形态，盈亏颜色不表示安全或建议；
- 动画只解释状态变化和等待，不制造兴奋或紧迫；
- 危险动作、停止和用户接管与普通动作明显不同；
- 键盘、指针和触控可以优化，但关键决定不能只依赖悬停、手势或颜色。

常用任务少步骤完成，诊断细节按需展开。用户完成任务的正确率、时间、理解负担、误操作、停止可达性和恢复连续性比页面数量和功能密度更重要。

---

# 5. 通知与不同命令【UX-NTF-001】

通知只把用户带回当前任务，不保存状态，也不直接产生真实动作。旧、重复或已关闭任务的通知不能形成新决定；快捷操作必须重新读取当前事实、权限和时限。

## 5.1 不同决定保持不同命令【UX-CTL-001】

以下决定必须让用户清楚区分：

- 固定计划内容；
- 提高资金上限或扩大范围；
- 激活计划；
- 停止新增风险；
- 按计划退出；
- 进入交易场所官方入口接管；

一次计划激活可以明确地同时授予只属于该激活的有界机器执行范围。预览必须说明固定计划、环境、账户、工具、本次资金限制、期限、允许动作和失败结果。它不改变计划、不适用于未来激活，也不调用交易场所执行动作。

除此之外，不得用一个“确认”暗中合并多个决定。降低范围、停止和撤销应快速完成。

---

# 6. 异常、停止与接管【UX-INC-001】

异常首先显示当前订单和持仓、可能重复或结果未知的动作、事实未知、保护缺口、已经停止的范围，以及用户可采取的行动。技术细节随后展开。

停止状态说明影响范围、禁止能力和仍在运行的只读/核对任务。停止不暗示场所订单或持仓已经改变；结果无法确认时显示仍受限并继续查询。

交易场所官方入口的接管说明必须在主要界面不可用时仍可取得。用户停止新增风险、退出或接管后，该激活不得再次新增风险；界面继续支持已有风险的保护和退出。旧责任闭合后如需再次交易，用户创建新激活。

如果只有系统重启且用户没有停止，界面必须先显示外部事实、原授权和连续性核对结果；全部一致且不会补发错过动作时，才显示同一激活继续运行。旧确认和过期授权不得复用。

---

# 7. 计划视图与跨工具交接【UX-CHT-001】

计划视图把计划依据、进入、目标、保护、失效、期限、当前价格、订单、成交和实际结果放在一致的时间语义中，并显示环境、账户、数据截止点、事实未知和开放责任。它帮助理解计划，不复制通用专业图表。

从模拟进入真实资金时可以复用内容未变的计划版本，但必须由用户重新选择真实环境、账户、资金边界和激活范围；确认、授权、动作、持仓和完成状态不得继承。

跨工具交接保持同一任务、对象和返回位置，并只发送所需内容，不包含真实交易凭据【UX-HND-001】。外部链接、图表、AI 会话和“完成”按钮不能成为事实、计划激活或真实动作结果。

---

# 8. AI 辅助与有界定制【UX-AIX-001】

AI 可以帮助整理材料、研究、比较、草拟计划和解释结果，但输出必须绑定当前任务、对象版本和事实截止点，并区分引用、推断、建议和未知。自然语言先形成可检查草案或明确命令预览；普通肯定、模型自信和会话连续性都不能替代涉及资金的用户决定。

允许定制信息密度、排序、保存视图、通知偏好、图表和草案模板；不得定制事实与未知的含义、资金边界、授权、停止、动作唯一和外部结果【UX-CUS-001】。没有真实需求时不增加脚本平台、入口类型或通用配置体系。

---

# 9. 验证与下位边界【UX-QLT-001】

实际交互必须验证：用户能否看清当前任务和环境；模拟与真实是否不会混淆；命令接收是否不会冒充外部完成；重复、过期、缓存和恢复是否不会增加风险；停止、退出和接管是否直接可达；事实未知和开放责任是否可见。

UX 接收业务领域提供的当前对象、事实、允许命令和结果，只输出用户命令及理解、可达性和交互质量证据【UX-HOF-001】。交互信息本身不能取得业务决定权。

L3 只需定义实际入口中的页面、命令预览、防重复、结果回流、环境辨识、中断恢复、停止和接管，以及真实渲染和关键任务测试【UX-L3-001】。L4 只记录当前页面、设备、通知、外部工具、实际配置、测试结果和已知限制【UX-L4-001】。
