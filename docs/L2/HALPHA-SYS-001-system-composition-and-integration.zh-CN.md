# Halpha 系统组成与集成原则

**文档编号：** HALPHA-SYS-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** 最小系统组成、模块所有权与依赖、应用边界、本地事务和外部动作分界、运行实体、配置、启动停止与重启约束  
**本文档不负责：** 重定义业务对象、决定、动作、事实或用户交互；规定当前模块、进程、组件版本、接口字段、数据库结构、主机配置、建设顺序、测试结果或发布流程

# 0. 系统组成结论【SYS-SUM-001】

Halpha 默认是个人可理解和维护的模块化单体。系统边界只为四件事服务：一个业务含义只有一个写入所有者；本地状态改变具有清楚的事务结果；外部动作先有持久责任并由唯一写者执行；停止、重启和还原不会扩大权限或重复外部影响。

模块、进程、数据库、后台工作和适配器的数量不是成熟度指标。一个现有边界能够正确承担责任时，不再建立平台、消息总线、通用任务、第二数据库、第二写路径或重复状态。只有实际消费者、并发冲突、隔离需要或重复故障证明现有边界不足时才增加系统结构。

# 1. 最小职责与概念【SYS-BND-001】

## 1.1 SYS 责任【SYS-BND-001-REQ】

SYS 只定义业务模块怎样组合、谁可以写什么、事务和外部动作怎样分界、运行单元怎样取得与失去能力，以及系统怎样启动、停止和在故障后恢复责任。业务含义仍由相应横向 L2 拥有；ENG 拥有开发、验证、构建、部署和回退方法。

## 1.2 四个必要概念【SYS-BND-001-DEF】

| 概念 | 含义 |
|---|---|
| 业务模块 | 承担一组内聚业务语义和可变状态的逻辑边界 |
| 应用边界 | 用户、后台工作或其他模块请求拥有模块执行命令或查询的公开入口 |
| Halpha 运行实体 | 可独立启动、停止或失去执行能力，并具有明确资源、秘密、依赖和故障后果的 Halpha 运行单元 |
| 隔离外部写入边界 | 持有某一环境最小写入能力并处理已持久化外部动作责任的唯一运行边界 |

这些是职责概念，不要求分别建立进程、表、基类、框架或治理记录。同一进程可以承载多个业务模块；没有当前消费者的分类、交接方式或状态不进入稳定设计。

## 1.3 建立新边界的条件【SYS-BND-001-REQ-002】

只有现有边界无法清楚表达唯一所有权、事务、外部故障隔离、独立启停或恢复责任，且这个不足已经影响当前消费者时，才新增模块、进程、存储或持久工作。新边界必须比保留单体更容易判断、测试、停止和恢复；否则保持或合并到现有边界。

# 2. 模块所有权与依赖【SYS-DEP-001】

## 2.1 单一所有权【SYS-DEP-001-REQ】

每项可变业务状态、决定和不变量只有一个拥有模块。拥有模块独占写入并通过应用边界返回确定结果、拒绝、冲突或未知；其他模块只能读取已提交状态或请求拥有模块改变它，不能直接更新私有表、复制状态机或从页面、缓存和日志反推权威结果。

模块所有权是实现责任，不是真人身份，也不替代 L2 语义所有权。共享代码只能提供无业务决定的通用能力；一旦代码判断业务状态或失败结果，它属于相应拥有模块。

## 2.2 依赖方向

依赖默认从入口进入拥有模块，再由拥有模块调用基础设施或外部适配。业务模块不依赖 UI、可重建视图或供应商对象；基础设施不反向决定业务。只保留当前调用需要的直接依赖，不为未来可能性建立全连接接口或中间协调层。

# 3. 命令、事务与外部动作【SYS-TXN-001】

## 3.1 直接调用优先

同一进程内需要立即结果的工作默认使用直接应用调用。读取不改变状态；改变状态的请求由一个拥有模块或一个明确的本地事务协调者提交。只有工作必须跨请求存活、能够独立失败并需要从持久状态恢复时，才使用持久异步交接；瞬时通知只能降低延迟，丢失后必须能够从权威状态重新发现。

不为这些差异建立通用命令、事件、任务或回执平台。L3 只在当前消费者确有需要时定义具体接口和持久表达。

## 3.2 本地事务【SYS-TXN-001-REQ】

一次业务请求的本地状态改变应在一个明确事务中全部成立或全部不成立。并发修改使用稳定身份、版本或唯一约束发现冲突；冲突返回可解释结果，不静默覆盖，也不靠全系统锁掩盖所有权不清。

一个请求若包含可以独立成功、查询或恢复的外部责任，应把该责任先持久化并在提交后处理，不把数据库和外部系统伪装成一个事务。

## 3.3 外部动作

任何可能改变交易场所或账户的 Halpha 动作都先由 EXE 建立唯一持久责任，再由相应环境唯一写者执行。超时或断线时查询原外部身份并核对，不盲目重发。Demo 与 Live 复用同一执行实现，但环境、账户、凭据、存储、激活和动作身份隔离；模拟路径不能取得真实写能力。

# 4. 后台工作、只读视图与外部适配【SYS-JOB-001】

后台工作只在责任需要离开当前请求继续时存在。它必须能从权威记录找到工作、使用稳定身份避免重复外部影响、在失去执行权后停止写入，并在重启后恢复责任而不是机械重放过期动作。短暂定时、进程内回调或可重算工作不因此变成持久业务对象。

可重建只读视图和缓存只为查询效率服务。它们必须说明权威来源，删除后可重建，陈旧或失败时显示未知或回退到权威读取，不能取得写入权。

外部适配只隔离协议、身份、单位、精度、时间、限频、能力和错误。供应商对象和缓存不成为 Halpha 业务模型；观察能力与写入能力分离，秘密和写接口只交给确需写入的运行实体。

# 5. 配置与运行【SYS-CFG-001】

业务限额、计划和动作规则由对应业务领域拥有；SYS 只拥有模块接线、运行实体、资源、运行环境和秘密注入边界。精确版本、当前参数、主机、进程命令和非秘密配置值由 L4 或普通配置记录。

配置不能创造业务权限、恢复被停止的激活、跳过动作检查或在身份缺失时提供危险默认值。秘密只通过目标运行实体所需的最小权限注入；日志、页面、文档和测试制品不得保存秘密值。

## 5.1 启动、停止与重启【SYS-RUN-001】

每个运行实体必须有直接可执行的启动、停止和识别故障方式。停止先阻断受影响范围的新外部写入并保留未闭合责任；备份与还原只恢复不能低成本重建的权威状态，不恢复旧执行权或扩大业务范围。

纯系统重启且用户未停止、退出或接管时，只有激活内资金限制与动作范围仍有效、当前外部事实已核对、唯一写者和持久连续性可证明、构建/配置/凭据/账户身份未变、未决责任已处理且不会补发停机期间错过的动作，才可在原激活范围继续。任一条件不足时，不可逆停止该激活新增风险；查询、既有保护、退出和用户接管继续。责任闭合后如需再次交易，建立新激活。

# 6. 演进与验证【SYS-CHK-001】

模块化单体是默认。只有实际规模、故障隔离、并发或独立启停需要证明收益大于部署、兼容、恢复和个人维护成本时，才增加进程、服务、存储或消息基础设施。拆分不能复制业务所有权，也不能依赖跨边界数据库事务维持正确性。

相关 L3 只回答会改变当前消费者的问题：所有者和消费者；必要输入、输出与失败；外部动作的持久、查询与核对；实际并发或后台恢复；只读来源或适配隔离；当前持久数据所需的最小兼容和测试。不适用内容直接略去，不为满足模板虚构接口、状态、worker 或证据。

验证直接围绕受影响不变量：禁止私有状态跨模块写入；本地提交不留下部分成功；外部动作先持久且只有一个写者；投影和缓存不成为权威；Demo 不能取得 Live 写入；停止和重启不扩大权限。SYS 不建立发布资格或治理结论。

# 7. 相邻领域边界【SYS-HND-001】

| 相邻领域 | 相邻领域拥有 | SYS 拥有 |
|---|---|---|
| 横向业务 L2 | 对象、决定、状态和业务终点 | 这些责任在模块、事务和运行实体中的组合方式 |
| DAT | 事实、来源、时间和未知 | 事实生产者、消费者与适配的系统边界 |
| UX | 用户任务、操作含义和呈现原则 | 交互进入共同应用边界的方式 |
| CAP、TRADEPLAN 与 EXE | 激活资金限制、动作、保护、核对和接管语义 | 本地事务、唯一写入和运行隔离结构 |
| ARC | 总体技术方向与增加基础设施的触发条件 | 当前系统组成对这些方向的落实 |
| ENG | 开发、验证、构建、部署和回退 | 被验证的所有权、事务、运行和恢复不变量 |

# 8. SYS 不变量【SYS-INV-001】

1. 一个稳定语义和可变状态只有一个拥有模块，其他模块不直接写入。
2. 本地事务有明确责任方；查询、视图、缓存、日志和通知不取得写入权。
3. 外部动作先有持久责任、唯一写者以及查询和核对路径；未知不盲重试。
4. 模拟与真实环境隔离，身份缺失或冲突时拒绝真实写入。
5. 停止、重启和还原不授予或扩大权限，不机械补发错过动作。
6. 新系统概念、进程、存储或平台必须有当前消费者和已观察的必要性；否则删除或不建设。

# 9. L4 当前事实【SYS-L4-001】

L4 只记录当前模块和进程、数据库、依赖、配置、主机、秘密注入方式、启动停止、备份还原、已启用边界和直接验证结果。它不新增业务所有权、事务语义、恢复权限或通用平台。
