# 0. 系统组成结论【SYS-SUM-001】

1. 运行时以模块化单体为主体；模块是业务责任、依赖和数据所有权边界，不是目录标签。
2. 每项可变业务状态和决定只有一个负责实现写入的模块；L2 语义所有权仍属于责任地图登记的领域，其他模块只能通过公开应用边界请求、查询或消费已提交事件。
3. 交互入口、通知、后台任务和外部适配器调用同一应用命令，不建立快捷写入路径。
4. 同步调用、提交后事件、持久化交接和瞬时通知各有固定用途；瞬时机制不能承载唯一请求或权威状态。
5. 一个改变状态的应用命令具有一个明确提交责任方；模块只写自己拥有的状态，跨模块原子提交仍由同一应用命令协调。
6. 数据库提交与不能随本地事务回滚的外部操作分离。可能改变外部系统的请求先持久化，再由唯一责任方处理和核对。
7. 可重建只读视图和缓存都可重建，不授予权限、不替代权威事实；写命令在必要时重新读取权威状态。
8. 外部系统经场所或工具特定适配器接入；供应商对象、错误和能力不得渗入核心业务语义。
9. 不因未来可能需要而拆服务、建消息总线或设计多供应商框架；只有明确触发条件成立后才增加边界。
10. SYS 定义运行时架构不变量，以及运行实体、宿主与进程、配置与秘密注入、启动停止、备份范围和恢复前置边界；ENG 按影响级别验证构建、发布、回退、迁移和部署说明；当前实例由 L4 记录，不建立独立运行可用性或恢复产品。
11. 历史研究、交易所模拟盘和真实资金可以复用纯业务语义与同一代码库；交易所模拟盘与真实资金分别绑定独立数据存储和场所接口/凭据适配器，交易记录环境身份必须先于适配器绑定确定；模拟路径缺失或失败时不得回退到真实写入。

# 1. 职责模型【SYS-BND-001】

## 1.1 SYS 纵向责任要求【SYS-BND-001-REQ】

SYS 贯穿 CTX、ALP、POR、TRADEPLAN、EXE 和 OUT 等横向业务组成。它规定这些业务在系统内怎样分界和交接，并拥有运行实体、宿主与进程、资源与运行时配置、秘密注入、启动停止、权威状态备份范围和恢复前置条件的稳定系统边界；它不重新定义研究或交易计划候选、候选优势、经济证据判断、固定策略依据、业务配置、计划、交易计划启用、Halpha 真实资金操作权限、真实动作、事实核对或结果的业务含义。

领域 L3 若主要解决某段业务，其主要语义所有者是对应横向业务文档，SYS 作为适用纵向约束；若领域 L3 主要解决公共应用交接、模块重构、后台调度、可重建只读视图、适配或兼容机制，SYS 可以成为主要语义所有者。涉及多个 L2 不自动变成编排产品。拟执行动作资金与权限检查结果与待执行动作记录在同一应用命令中提交，属于 EXE 业务命令使用 SYS 本地事务约束，不另建独立跨领域交接生命周期。

## 1.2 六类运行时构件定义【SYS-BND-001-DEF】

| 构件 | 定义 | 边界 |
|---|---|---|
| 业务模块 | 承载一组已有明确 L2 语义所有者的内聚业务语义、状态变化和公开能力 | 不改变 L2 语义所有权，也不等同于进程、页面、数据库数据结构或仓库 |
| 应用入口 | 把一个用户或 Halpha 意图编排为命令或查询 | 不持有独立业务规则，不绕过拥有模块 |
| 隔离外部写入边界 | 持有最小生产写入能力并处理持久化待执行动作记录 | 不生成策略、计划或权限，不依赖交互和 Alpha 能力 |

后台任务、可重建只读视图和外部适配器分别在第 6.1、7.1 和 8.1 节单独定义，本节不重复定义。这六项共同构成需要分清所有权和故障影响的运行时责任，不是互斥的部署类型。一个简单实现可以在同一进程或构件中承担多项责任，写入权、权威状态和失败结果仍按各项责任区分。

进程、主机、数据库连接和部署单元是运行实现，不自动形成业务模块。一个进程可以包含多个模块；一个模块只有在明确拆分条件成立时才跨进程。

## 1.3 六类运行时构件的选择依据【SYS-BND-001-RAT】

业务模块、应用入口、可重建只读视图、后台任务、外部适配器和隔离外部写入边界分别承担唯一实现写入责任、请求进入、只读呈现、无会话工作、外部协议隔离和最小写权，消费者与故障边界不同。L2 语义所有权仍属于责任地图登记的领域；进程、主机和部署单元没有业务语义所有权，实现形态本身不产生新构件概念。新增边界的价值依据是已经出现实际隔离或规模问题。

## 1.4 其余系统分类的保留依据【SYS-SEM-001-RAT】

应用交互类型区分请求、读取、已发生变化和可丢失信号；模块间交接方式决定原子性、持久性与恢复责任；后台任务类别决定任务丢失后的业务结果；配置分工决定谁可以解释和修改含义。它们分别被模块边界、失败处理、恢复和兼容检查消费，不是需要长期保存的业务对象。分类的保留依据是它仍会改变调用、持久化、恢复或所有权行为；框架或中间件变化本身不产生新的分类价值。

## 1.5 模块边界建立要求【SYS-BND-001-REQ-002】

一个模块必须同时具备：

- 单一、可命名的业务或公共应用责任；
- 明确拥有的状态与写入规则；
- 明确公开的命令、查询或已提交事件；
- 可声明的允许模块依赖方向；
- 可在不读取其他模块私有状态的条件下验证；
- 故障时可说明影响范围和安全结果。

仅因代码较多、使用同一技术、需要一个页面或预计未来扩展，不构成新模块依据。

# 2. 模块所有权与模块依赖方向【SYS-DEP-001】

## 2.1 模块所有权与模块依赖方向定义【SYS-DEP-001-DEF】

模块所有权是可变业务状态、决定和不变量到唯一负责实现模块的映射；这里的“所有权”是模块的语义与写入责任，不是真人身份、账号或运行授权；它落实但不替代 L2 语义所有权。模块依赖方向是入口经应用边界进入负责模块、再由基础设施实现模块端口的单向关系。无业务所有权且接口小而稳定的技术能力可以进入共享基础。

## 2.2 单一拥有模块要求【SYS-DEP-001-REQ】

每项可变业务状态、决定和不变量只有一个拥有模块。拥有模块负责解释、创建新版本、接受或拒绝变化以及发布已提交事件。其他模块不得：

- 直接写入拥有者的表、文件或缓存；
- 复制拥有者的状态机并独立推进；
- 依据本地推断把派生值提升为拥有者的决定；
- 通过共享工具函数隐藏跨模块业务规则。

跨领域查询可以组合多个只读来源，但组合视图不取得这些来源的写入权。

## 2.3 允许的模块依赖方向

稳定方向为：

~~~text
交互入口 / 外部入口 / 调度入口
              ↓
          应用命令或查询
              ↓
       业务模块公开应用边界
              ↓
       模块私有模型与持久化端口
              ↑
      基础设施和外部适配器实现
~~~

具体约束：

1. 入口适配器依赖应用边界，不能依赖模块私有模型或数据库实现。
2. 业务模块只依赖自己定义或由共享契约明确的最小端口；基础设施实现这些端口。
3. 模块之间通过公开应用边界、只读契约或已记录事实交接，不导入彼此私有类型。
4. 可重建只读视图依赖公开只读来源和已记录事实；业务写路径不依赖可重建只读视图实现。
5. 隔离执行器只依赖执行所需最小契约、持久化待执行动作记录和经过实际验证的场所适配，不反向依赖管理和配置功能、研究、AI 或一般工具适配。
6. 依赖环禁止以初始化顺序、服务定位器、回调注册或共享数据库查询伪装。

## 2.4 共享代码

只有同时满足以下条件的内容才进入共享基础：语义不属于某个业务领域；变化不要求多个模块同步修改业务规则；接口小且稳定；不会赋予跨模块私有状态访问权。

数值、时间、身份等共同表示仍由其拥有领域定义。把业务对象放入 `common`、`utils` 或共享 ORM 模型不改变其所有权。

# 3. 应用交互类型【SYS-INT-001】

## 3.1 固定分类

### 四类应用交互定义【SYS-INT-001-DEF】

| 类型 | 用途 | 结果含义 |
|---|---|---|
| 命令 | 请求拥有模块尝试改变状态 | 接受、拒绝、冲突、过期或状态未知；成功只表示声明变化已提交 |
| 查询 | 读取权威状态或可重建视图 | 返回来源、新鲜度、完整性和适用的事实未知状态 |
| 已提交事件 | 声明某项状态变化已经提交 | 供其他模块形成自己的后续决定 |
| 瞬时信号 | 加速唤醒、刷新或通知 | 丢失后仍可从权威来源恢复 |

### 应用交互使用要求【SYS-INT-001-REQ】

| 类型 | 禁止事项 |
|---|---|
| 命令 | 不以 HTTP 成功、按钮点击或消息送达代替业务接受 |
| 查询 | 不修改外部系统，不以查询触发隐式修复或启用 |
| 已提交事件 | 不表达“请执行”，不由消费者改写生产者历史；不等同 DAT 的可引用事实判断 |
| 瞬时信号 | 不承载唯一请求、顺序保证、权限或事实 |

外部网页事件、AI 输出、通知点击和定时器到期不是天然的业务命令。入口先保留来源和时效，再交给拥有模块分类与验证。

## 3.2 一条命令路径

同一业务请求无论来自主要工作入口、限时操作入口、运营工具、通知快捷操作、后台任务还是内部自动化，都调用同一应用命令和业务约束。适配器只负责认证后的调用转换、输入解码和结果呈现，不复制资金与权限检查、版本或状态迁移。

命令必须具有稳定业务身份、调用来源、目标适用范围和预期版本语义；是否需要防重复提交、有效期、交易计划当前确认或机器授权引用、当前事实复核，由适用业务领域规则决定，具体字段进入 L3。

## 3.3 已提交事件的使用

已提交事件使用过去时语义，只在生产者状态提交后可见。消费者可以据此：

- 更新自己的派生状态；
- 创建新的研究或交易计划候选、候选优势或交易计划草案；
- 使可重建只读视图失效或重建；
- 安排不改变生产者历史的后续工作。

消费者不得把事件到达本身当作新的资金操作权限、当前事实或外部动作结果。需要当前判断时，重新查询权威来源并执行适用业务检查。

# 4. 应用命令提交与业务状态冲突处理【SYS-TXN-001】

## 4.1 应用命令提交责任与业务状态冲突处理责任定义【SYS-TXN-001-DEF】

- **应用命令提交责任：** 一个改变状态的应用命令所对应的唯一逻辑提交责任。
- **业务状态冲突处理责任：** 对同一业务身份、版本或写入范围的冲突检测和唯一判定责任。

## 4.2 应用命令提交责任要求【SYS-TXN-001-REQ】

每个改变状态的应用命令必须声明一个提交责任方。责任方负责在一个逻辑提交边界内完成：当前版本检查、适用不变量、拥有模块写入、必要唯一约束、交接记录和已提交事件记录。

模块化单体可以在同一交易记录环境内使用同一关系型数据库完成跨模块原子提交，但必须满足以下条件；交易所模拟盘与真实资金环境不是同一交易记录环境，必须分别绑定独立数据存储。L3 约束事务、所有权与隔离，实际数据库产品、实例和连接由 L4 记录。

- 应用层明确协调该命令；
- 每个模块只通过自己的写入边界修改自己拥有的状态；
- 跨模块不变量有唯一判定责任，不由多个模块分别猜测；
- 提交失败不留下部分业务成功；
- 不以跨表直接更新绕过模块公开边界。

一个用户操作意图若包含多个可以独立成功、补偿或重试的责任，应拆为持久化交接，不扩大为长期大事务。

真实动作应用命令通过一次本地原子提交保存当前拟执行动作资金与权限检查结果和唯一待执行动作记录；具体存储和事务机制由 L3 规定。CAP 定义资金与权限检查规则，EXE 拥有命令与待执行动作记录生命周期，SYS 只保证提交原子性、唯一约束和失败恢复。采用模块化单体时不为此建立独立跨域对象、消息协调或分布式事务。

## 4.3 并发原则

- 可变业务对象使用明确版本或等价并发证据，禁止静默后写覆盖。
- 冲突结果是拒绝并重新取得当前状态、按业务规则重算，或由用户或 Halpha 按相应业务领域规则形成明确的无法判定或未决结果；不得自动合并会改变真实账户、持仓或资金使用的决定。
- 同时锁定多个对象时，L3 必须规定稳定顺序和最大适用范围；不使用全系统粗粒度锁掩盖边界不清。
- 后台工作使用限时执行权记录或旧实例写入阻断机制时，失去所有权的实例不得继续写入；进程存活不等于仍有执行权。
- 唯一约束承担身份和去重不变量时，应用层仍需把冲突转换为可解释业务结果。

## 4.4 不能随本地事务回滚的外部操作

数据库提交与外部场所或服务操作不视为一个原子事务。需要可靠执行、去重、查询恢复或责任闭合的外部动作，先建立待执行动作记录，再由唯一责任方执行。

真实动作使用 EXE 拥有的待执行动作记录、执行器、交易场所对象对应关系和核对路径。SYS 只规定交接必须持久、可领取、可隔离和可恢复，不重新定义动作状态或重试语义。

# 5. 模块间交接方式选择【SYS-HOF-001】

## 5.1 选择表

### 五种模块间交接方式定义【SYS-HOF-001-DEF】

| 模块间交接方式 | 稳定含义 |
|---|---|
| 同步公开接口 | 同一进程内需要立即结果、且结果属于当前事务判断的调用 |
| 同一事务内持久化交接记录 | 业务状态与后续责任在一个本地事务中共同建立 |
| 持久化异步交接 | 可延后、可领取、可恢复并可能跨进程或改变外部系统的工作 |
| 提交后内部事件 | 生产者提交变化后供多个消费者形成后续决定的通知 |
| 瞬时信号 | 只用于降低唤醒或刷新延迟、丢失后可恢复的信号 |

### 模块间交接方式选择要求【SYS-HOF-001-REQ】

| 场景 | 交接方式 | 约束 |
|---|---|---|
| 同一进程内需要立即结果，且结果属于当前事务判断 | 同步公开接口 | 调用链短、方向单一、不修改外部系统 |
| 一个提交需要原子记录后续责任 | 同一事务内持久化交接记录 | 状态与交接同时成功或同时失败 |
| 工作可延后、需要重试、可能跨进程或修改外部系统 | 持久化异步交接 | 有稳定身份、领取权、重试边界、结果和恢复路径 |
| 多个消费者只需获知已经发生的变化 | 提交后内部事件 | 生产者不等待消费者完成；消费者自行防止重复处理 |
| 只为降低延迟而唤醒工作者或刷新界面 | 瞬时信号 | 丢失无损，权威扫描或查询可以恢复 |

默认不引入独立消息总线。模块化单体和隔离执行器优先使用权威关系型数据库保存交接状态；瞬时进程信号只作加速。只有 ARC 的拆分条件成立且现有持久化方式已不能满足明确要求时，才评估新的消息基础设施。

## 5.2 交付和顺序

- 持久化异步交接默认按可能重复设计，消费者必须按业务身份防止重复处理。
- 不承诺全局严格顺序；只有业务确需的同一对象或适用范围声明顺序键和冲突处理。
- “恰好一次”不是跨存储和外部系统的设计前提；使用稳定身份、唯一约束、结果查询和核对取得等价业务效果。
- 交接失败必须留在可观察、可重试、可取消或可人工处置的状态，不允许无限静默重试。
- 消费者升级后不能解释旧交接时，先停止领取并完成兼容或迁移，不丢弃未完成责任。

# 6. 后台任务【SYS-JOB-001】

## 6.1 后台任务及其业务后果分类定义【SYS-JOB-001-DEF】

后台任务是在没有持续用户交互会话时，按稳定身份和明确责任执行的工作。后台任务分为真实动作保护与核对任务、业务连续任务和可重建任务；类别由任务失败或丢失的实际业务后果决定。

| 类别 | 稳定含义 |
|---|---|
| 真实动作保护与核对任务 | 保护、核对、未决待执行动作记录查询和责任恢复等不可静默丢失的责任 |
| 业务连续任务 | 条件观察、计划过期、策略复核和结果归集等需要按版本重评的工作 |
| 可重建任务 | 可重建只读视图、索引、摘要、非关键研究和缓存预热等可由权威来源重建的工作 |

## 6.2 后台任务契约要求【SYS-JOB-001-REQ】

每类后台任务的 L3 契约必须定义：

- 任务语义责任方和业务目的；
- 输入身份、版本和适用时效；
- 是否可以重算、重试、取消、合并或丢弃；
- 同时运行限制、领取与失权规则；
- 检查点、完成结果和失败终点；
- 产生的状态变化、交接和外部系统变化类别；
- 重启后从何处恢复，以及错过计划时间时怎样处理。

调度器只决定何时尝试运行，不决定任务是否仍可执行。任务开始和修改外部系统前，Halpha 按相应领域规则重新检查适用版本、明确的启用或停用决定、时效和阻断。

## 6.3 三类任务的处理要求

| 类别 | 例子性质 | 系统处理 |
|---|---|---|
| 真实动作保护与核对任务 | 保护、核对、未决待执行动作记录查询和责任恢复 | 持久化、优先、不可静默丢弃；失败进入明确受限或接管路径 |
| 业务连续任务 | 条件观察、计划过期、策略复核和结果归集 | 使用版本和检查点；错过后按业务规则重评，不补发失去时效的真实动作 |
| 可重建任务 | 可重建只读视图、索引、摘要、非关键研究和缓存预热 | 可暂停、合并、重跑或清理，不与真实动作保护与核对任务争夺最后资源 |

类别由实际后果决定，不能因同一调度框架而获得相同优先级。资源、进程和配额优先使用宿主环境的简单配置与现成诊断；不建设独立运行告警产品。

## 6.4 重启与追赶

重启后只恢复仍存在的责任，不按停机期间的时间序列机械重放业务动作。历史触发、过期通知和已失效计划不能因任务追赶而重新取得效力。是否补算研究、统计、可重建只读视图或不涉及真实资金的结果，由 Halpha 按相应业务领域规则判断。

# 7. 可重建只读视图【SYS-PRJ-001】

## 7.1 可重建只读视图定义【SYS-PRJ-001-DEF】

可重建只读视图是从声明权威来源生成、服务跨模块阅读和用户交互任务连续性、删除后可以重新形成的只读视图。

## 7.2 可重建只读视图与缓存使用要求【SYS-PRJ-001-REQ】

可重建只读视图必须携带覆盖范围、新鲜度、完整性和事实未知状态；可以组合多个领域内容，但不得直接修改来源，也不得成为命令、权限或事实的唯一载体。

缓存只改善性能。缓存命中不能提高来源可信度、新鲜度、权限或确定性；删除缓存后不能恢复的内容不属于缓存，而应转为有明确语义责任方的持久状态。具体缓存策略由 L3 在存在实际性能消费者时定义。

# 8. 外部适配【SYS-ADP-001】

## 8.1 适配器责任

### 外部适配器定义【SYS-ADP-001-DEF】

外部适配器是把 Halpha 公开端口连接到具体数据源、交易场所或外部工具，并隔离其协议、身份、能力、格式、错误和限频语义的运行时构件。

### 适配器责任要求【SYS-ADP-001-REQ】

每个外部系统或场所的适配边界负责：

- 认证方式与最小权限接口；
- 外部身份、时间、精度、分页、限频和错误语义；
- 能力探测及已验证适用范围；
- 原始请求、响应和观察的必要证据身份；
- 外部格式与 Halpha 公开契约之间的显式转换；
- 超时、重复、断线、部分结果和未知的保守映射；
- 外部版本变化的发现和兼容边界。

DAT、CAP、EXE 和 SYS 分别拥有事实可用性，资金使用限制、Halpha 真实资金操作权限与 Halpha 新增真实动作停用状态，场所动作、核对与真实写凭据消费，以及运行时配置与秘密注入边界。Halpha 或用户按相应规则作出具体判断；适配器不得因成功解析而代替他们作出结论。

## 8.2 观察与写入分离

观察能力和场所写入能力使用不同公开端口和权限边界。一般应用、可重建只读视图和研究可以取得经许可观察，只有隔离执行器可以取得生产写入能力。共用供应商开发包不等于共用凭据、进程或依赖图。

## 8.3 交易记录环境身份先于适配器绑定

涉及动作、订单、成交、仓位或结果的命令和持久化交接必须携带不可含混的交易记录环境身份。应用边界应先验证该身份与对象身份，再选择历史研究、交易所模拟盘或真实场所适配器；身份缺失、冲突或适配器不可用时拒绝，不允许使用真实场所适配器作为默认或回退。

交易所模拟盘适配器不能实现或取得真实写入端口，真实执行器也不能领取模拟动作。纯条件求值、计划语义和订单表示可以复用；账户、动作、订单、仓位、余额和结果状态必须隔离。模块化单体使用同一代码库，但模拟与真实分别绑定独立数据存储、接口和凭据，不要求复制业务逻辑或新增平台。

## 8.4 专用优先

首次接入一个外部系统时建立工具或场所特定适配，不预建通用多供应商框架。只有至少两个真实实现呈现稳定共同语义，且抽取后不会掩盖关键差异时，才提取共同端口；限频、订单、保护、时间和错误等场所差异仍显式保留。

外部服务不可用时，按依赖它的能力适用范围降级。不得为保持统一接口而伪造空成功、默认事实或看似通用但无法核对的状态。

# 9. 系统配置、版本与兼容【SYS-CFG-001】

## 9.1 配置分工

### 系统配置责任分工定义【SYS-CFG-001-DEF】

| 配置含义 | 语义所有者 |
|---|---|
| 业务规则、计划参数、交易计划启用、Halpha 真实资金操作权限和资金使用上限与范围 | 对应业务领域 |
| 模块启用、适配绑定和交接选择 | SYS |
| 运行实体、主机、进程、资源、运行时配置和秘密注入边界 | SYS |

当前工具、版本、部署和参数实例只是上述配置在某一时点的实际值，由 L4 记录；它不是第四种配置含义，也不改变原有语义所有者。

### 配置使用与兼容要求【SYS-CFG-001-REQ】

| 配置含义 | SYS 约束 |
|---|---|
| 业务规则、计划参数、交易计划启用、Halpha 真实资金操作权限和资金使用上限与范围 | 使用可识别版本和共同命令变化，不以环境变量或本地文件静默覆盖 |
| 模块启用、适配绑定和交接选择 | 显式声明依赖与兼容范围；变化不得制造第二套业务语义 |
| 运行实体、主机、进程、资源、运行时配置和秘密注入边界 | 通过受控运行配置提供，业务模块只取得最小端口，不读取无关秘密；实际秘密值只保存在授权的外部秘密工具中，L4 只记录注入方式和边界 |

真实动作配置缺失、来源不明或不兼容时，不使用方便默认值继续写入。默认只能停止相关自动动作，不能补造资金使用上限与范围、交易计划启用、Halpha 真实资金操作权限或账户事实。

## 9.2 兼容原则

- 持久化对象、未完成交接和已经生效的责任必须能够按其原版本解释，或先完成明确迁移。
- 新生产者不得写入当前消费者无法识别的语义版本。
- 添加可选内容只有在旧消费者忽略它不会改变业务含义时才算兼容；否则属于实质变化。
- 业务含义、默认、精度、时间、枚举、错误或缺失处理发生变化时，形成新契约版本并完成影响复核。
- 迁移不成功时保持旧版本可读、停止受影响写入或先解除责任，不进行静默就地重新解释。

单站点部署不要求为形式完整建设复杂滚动升级协议；但隔离进程、后台任务和未完成持久化交接之间仍必须声明实际兼容窗口。

# 10. 架构不变量与验证交接【SYS-CHK-001】

## 10.1 需要验证的架构不变量

SYS 定义以下不变量，开发者和项目所有者按 ENG 规则及变更影响选择自动检查、集成验证或实际环境验证的工具和深度：

- 模块依赖图无禁止方向和依赖环；
- 入口、可重建只读视图和适配器不能导入模块私有实现；
- 跨模块不能直接写入私有持久化状态；
- 每个命令、查询、事实和后台任务具有唯一明确的语义责任方，不得仅以“系统”或“后台”代称；
- 业务写入不从缓存、界面状态或通知载荷取得权威性；
- 修改外部系统前存在先持久化或由拥有领域规则明确允许的等价责任边界；
- 可重建只读视图可从声明来源重建，重建失败不会反向污染权威状态；
- 隔离执行器的依赖图不包含管理和配置功能、Alpha、AI 和一般外部工具适配；
- 模拟命令和对象不能绑定真实写入端口，环境缺失或冲突时不能通过默认值继续；
- 持久化交接和兼容迁移具有重启、重复、冲突和失败验证。

SYS 只负责相关设计必须满足的架构不变量，不拥有发布结论或例外审批。ENG 负责定义按影响级别组合验证结果的要求；项目所有者或开发流程据此构建、发布、停用或回退。

## 10.2 每篇相关 L3 必须回答

1. 文档是 `DOMAIN` 还是 `ORCHESTRATION`；主要语义所有者或已在 L2 拥有跨领域业务对象或交接生命周期的唯一协调所有者是谁，参与语义所有者及适用纵向约束是什么；
2. 哪个模块拥有本设计改变的状态和决定；
3. 允许模块依赖方向、公开边界和禁止访问是什么；
4. 使用命令、查询、已提交事件、持久化交接还是瞬时信号，为什么；
5. 应用命令提交责任、业务状态冲突处理、唯一性和重试边界是什么；
6. 是否会修改不能随本地事务回滚的外部系统，怎样持久化、查询、核对和闭合责任；
7. 后台任务怎样领取、失权、恢复、取消和结束；
8. 可重建只读视图或缓存的来源、新鲜度、事实未知状态和重建方式是什么；
9. 外部适配怎样隔离能力、错误、时间、精度、权限、交易记录环境身份和版本；
10. 持久化内容、未完成责任和相邻进程怎样保持兼容；
11. 哪些架构检查和故障场景构成完成证据。

具体接口字段、数据结构、锁、索引、超时、批量大小、错误码和实现库进入 L3 或 L4，不进入 SYS。

# 11. 运行实体、启动停止与恢复前置边界【SYS-RUN-001】

Halpha 运行实体，是可独立启动、停止或失去执行权，且具有明确能力、秘密、资源、依赖和故障后果的 Halpha 运行单元。用户浏览器、数据库和外部系统是运行依赖，不计作 Halpha 运行实体；同进程业务模块也不自动成为运行实体。

实际部署必须具有可重复的启动和停止方式、可识别的构建与配置、能够定位直接错误的普通日志、不可低成本重建权威状态的现成备份方式、不会泄露秘密的注入边界，以及 Halpha 之外的账户控制路径。SYS 拥有运行实体及其宿主、进程、资源、依赖顺序、启动、停止和重启约束，明确哪些权威状态进入备份及还原后哪些系统前置条件必须重新成立；ENG 拥有构建、迁移、发布、回退和部署说明的可重复验证。

停止或重启必须先阻断受影响范围的新外部写入，并保留未闭合责任。还原只恢复不能低成本重建的权威状态和必要配置，不自行授予或扩大交易计划当前确认、机器授权、权限、领取权、外部写入控制权或提交许可。已经持久化且仍在原作用域和期限内有效的机器授权，只有在 CAP 重新检查当前授权与资金边界、DAT 提供当前事实、EXE 重新证明写入唯一性后才可继续；过期、撤销或无法复核的授权不得复活。还原后由 DAT 提供当前外部事实，EXE 处理未决、核对、业务恢复和用户接管，CAP 处理资金权限，UX 呈现停止与恢复结果；缺少必要事实、唯一写入边界或用户接管路径时保持停止。

当前主机、服务管理器、监听地址、数据库连接、秘密注入方式、日志位置、备份与还原命令、实际发布和回退步骤只由 L4、普通部署说明或现有工具配置记录。SYS 不据此建立专门健康页、综合监控、告警中心、自研备份恢复、自愈、容量、工单或事故平台。

# 12. 架构演进后的系统边界【SYS-EVO-001】

ARC 拥有保持模块化单体或增加进程、服务和存储边界的触发条件；SYS 不建立第二套拆分标准。当 ARC 条件要求评价拆分时，开发者和项目所有者按 SYS 规则验证新的系统边界是否仍具有单一语义所有者、版本化持久交接、明确重复与顺序行为、独立停止和恢复方式、迁移与退出路径，并且不依赖跨边界数据库事务。

拆分后仍不能复制业务所有权。若跨边界需要共同原子性，应先重划业务责任或改为持久化交接，不把分布式事务作为默认补救。边界长期没有独立责任、只转发调用或制造重复模型时，项目所有者和开发者按 ARC 规则评估合并或删除；当前演进决定和迁移状态由 L4 记录。

# 13. 与相邻领域的固定边界【SYS-HND-001】

| 相邻领域 | 相邻领域拥有 | SYS 拥有 |
|---|---|---|
| 横向业务 L2 | 对象、决定、状态含义和业务终点 | 模块落实、模块依赖方向和应用模块间交接方式 |
| DAT | 观察、事实、来源、质量、时间、事实未知状态和数据生命周期 | 数据生产者与消费者之间的运行时端口、可重建只读视图及适配结构 |
| UX | 任务、信息层次、交互命令含义和多入口原则 | 各交互适配器进入共同应用命令和可重建只读视图的方式 |
| CAP 与 EXE | 资金使用上限与范围、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态、拟执行动作、保护、核对和执行责任 | 统一调用边界、本地事务、持久化交接与隔离依赖结构 |
| ARC | 稳定性技术选择、简单结构、少依赖、现成能力优先和专门能力进入门槛 | 对这些选择的系统组成、运行实体和恢复前置边界 |
| ENG | 开发流程、验证方法、可重复构建、发布、回退、迁移和部署说明 | 被验证的依赖、事务、交接、运行实体、可重建只读视图、适配和兼容不变量 |

发生边界问题时修改拥有语义的文档：业务含义改横向 L2；事实与时间改 DAT；交互决定改 UX；运行时组成、宿主、进程、配置、启动停止、备份范围和恢复前置规则改 SYS；构建、发布、回退、迁移、部署说明和验证规则改 ENG；真实写凭据消费、核对、业务恢复与接管改 EXE。不得在某个模块 L3 中建立一条只有本模块知道的全系统规则。

# 14. SYS 不变量【SYS-INV-001】

1. 一个稳定语义、状态和决定只有一个拥有模块。
2. 模块只能通过公开应用边界影响其他模块，不直接写私有状态。
3. 所有入口使用共同命令；查询、可重建只读视图、缓存和通知不获得写入权。
