# 0. 非规范性摘要

本文固定十一项结论：

1. Halpha 按 TRADEPLAN 规则从 CTX、ALP 或用户已经明确选择并交付的交易计划依据开始；TRADEPLAN 不拥有更早的研究或交易计划候选池。
2. 交易计划必须完整回答七类决定；缺失内容只能保留为草案，不能由默认值或 AI 猜测补齐后启用。
3. 交易计划版本一经固定不可改写；交易计划启用只建立观察与交接责任，不提高 Halpha 真实资金操作权限或资金使用上限，也不扩大适用范围。
4. 条件评价、交易计划事件和真实动作是三个边界；条件成立不等于已经获准形成真实动作。
5. 人工路径只有在完整响应链可行时才能进入关键路径；否则必须使用已取得适用机器授权的系统路径或不行动。
6. 交易计划当前确认只针对一次当前预览；需要无人在线处理的计划分支必须另行取得 CAP 定义的有效机器授权，计划固定、启用或条件成立均不产生该授权。
7. 失效、过期、暂停、关键事实未知和交易计划条件无法判定默认停止形成新风险；已存在或可能存在的交易场所或账户变化继续由 EXE 闭合。
8. 一个计划状态不得同时替代版本、值守、条件、交互、交易场所或账户变化和结束责任。
9. 计划结束与真实动作责任闭合分开判断；不能因为计划到期而遗忘订单、仓位、保护或未决结果。
10. Halpha 只按 TRADEPLAN 规则保存供复盘使用的交易计划事实；跨计划、策略和产品的综合归因由 OUT 负责定义。
11. 交易计划可以被不同交易记录环境重新引用，但每次启用必须固定交易记录环境身份；交易所模拟盘环境的交易计划启用不得自然延续为真实资金环境的交易计划启用。

# 1. 职责边界与稳定计划语义

## 1.1 TRADEPLAN 的起点与终点【TRADEPLAN-SCP-001】

### 交易计划依据与责任终点定义【TRADEPLAN-SCP-001-DEF】

交易计划依据是由上游明确选择、带来源和用途、用于进入计划设计的输入；它不是交易计划，也不产生交易计划启用、监控、提醒、Halpha 真实资金操作权限或资金使用上限与范围变化。

TRADEPLAN 定义的责任终点是计划不再形成新事件、计划责任已经闭合或明确移交，并且 Halpha 已按 OUT 输入要求保存可评价的计划结果材料。

### TRADEPLAN 起点与终点要求【TRADEPLAN-SCP-001-REQ】

Halpha 形成交易计划时必须使用有效交易计划依据。该依据可以来自：

- 用户已经明确选择并交付的交易计划依据，包括用户原创判断或明确的真实验证目标与测试问题；
- CTX 的研究或交易计划候选去向决定所选择的交易计划起始材料；
- ALP 的有边界经济证据判断，或固定策略提供的交易计划依据；
- 多项真实资金用途需要比较时，用户按 POR 边界完成比较后选择的计划、资金范围和主要限制；
- 已有交易计划、外部账户与资金事实或结果反馈所触发的新计划需求，但必须先由用户或 Halpha 按相应业务领域规则形成新的明确输入。

TRADEPLAN 不定义普通新闻、图表观察、AI 建议和未归类想法的筛选规则；这些材料只有在用户已经形成并明确选择交易判断，或 Halpha 按 CTX、ALP 规则形成有效交易计划依据后，才能进入计划形成。

DAT 拥有外部动作事实语义，EXE 拥有执行责任闭合语义；Halpha 按两者规则取得更新事实和结论，再按 CAP 规则重新完成资金与权限检查。

## 1.2 稳定计划语义【TRADEPLAN-OBJ-001】

### 稳定计划语义定义【TRADEPLAN-OBJ-001-DEF】

| 稳定计划语义 | 唯一含义 |
|---|---|
| 交易计划草案 | 正在形成七类决定、允许修改且无运行效力的工作对象 |
| 交易计划资金需求 | 用户确需比较资金用途时，从交易计划依据或草案按需投影出的资金规模和用途需求；不独立维护，本身不占用资金、不提高资金使用上限与范围，也不授予形成真实动作的权限 |
| 交易计划版本 | 固定一次完整计划决定及全部决定性引用的不可变版本 |
| 交易计划启用 | 使一个计划版本在明确适用范围和期限内承担观察、事件和交接责任的决定 |
| 交易计划条件判断 | 对计划中一个类型化条件在给定事实截止点的判定 |

交易计划事件在第 4.3 节单独定义，与上述五项共同构成需要分离的六项稳定计划语义，本节不重复定义。

### 稳定计划语义边界要求【TRADEPLAN-OBJ-001-REQ】

交易计划草案不得替代研究或交易计划候选、固定交易计划版本或用户交互任务；交易计划资金需求不得替代交易计划、用户资金用途选择或资金使用上限与范围，也不允许 Halpha 形成、提交真实动作，并且只在用户确需比较资金用途时从交易计划依据或草案按需生成，不建立独立维护的生命周期；交易计划版本不得由可编辑表单、策略版本或用户资金用途选择替代；交易计划启用不得替代 Halpha 真实资金操作权限、资金使用上限与范围或真实动作；交易计划条件判断不得替代市场事实或交易计划事件，也不允许 Halpha 形成、提交真实动作；交易计划事件不得由订单、成交、通知或事实判断修正替代。

交易计划启用后，Halpha 必须持续承担相应的观察、事件和交接责任，直到计划不再形成新事件且外部责任已经闭合或明确移交。该责任由上述稳定计划语义共同产生，不建立独立“计划责任”对象。计划结果材料只是相关计划记录和相邻领域事实的只读汇集，不建立平行业务状态或独立评价对象。

### 六项稳定计划语义的选择依据【TRADEPLAN-OBJ-001-RAT】

草案与固定版本需要不同可变性；资金需求只在用户比较资金用途时按需生成和消费；启用决定承担运行范围；交易计划条件判断和交易计划事件分别表达当前判断与不可变历史。六者都会改变下游行为，同一状态字段会丢失这些差异。持续责任由相关记录和决定产生，Halpha 按 OUT 规则根据计划事实评价质量，因此独立责任或评价对象没有新增消费者。

六项稳定计划语义要求身份、历史与不变量能够独立判断，不要求实现为六个物理记录族。L3 可以在同一交易计划启用记录中表达启用决定和当前条件判断投影，也可以由交易计划事件携带形成该事件的条件判断输入摘要与历史；但不得因此覆盖不可变事件、丢失事实截止点、混合当前判断与历史判断，或使任一语义的拒绝、重放、恢复和结束结果无法单独验证。

## 1.3 状态必须分轴【TRADEPLAN-AXS-001】

### 交易计划六条状态轴定义【TRADEPLAN-AXS-001-DEF】

| 状态轴 | 独立表达的范围 |
|---|---|
| 内容 | 是草案还是不可变版本，哪个版本替代哪个版本 |
| 运行 | 是否未启用、值守中、暂停形成新事件或已结束 |
| 条件 | 条件为成立、不成立、无法判定、不适用、已错过还是已失效 |
| 交互 | 交易计划只通知路径、等待交易计划当前确认，还是由 CAP 定义的有效机器授权支持无须当前人工响应的处理路径 |
| 外部责任 | 尚无交易场所或账户变化、可能有交易场所或账户变化、已有交易场所或账户变化、结果未决、已核对或已移交 |
| 结果 | 为何结束、局部材料是否完整、是否已交付 OUT |

### 状态分轴要求【TRADEPLAN-AXS-001-REQ】

L3 不得用一个单一综合状态同时回答上述问题。

停止新事件不等于已有外部责任消失；服务恢复不等于计划恢复；计划版本有效不等于当前条件、资金使用上限与范围、Halpha 真实资金操作权限或 Halpha 新增真实动作停用状态允许继续运行。

### 六条状态轴的选择依据【TRADEPLAN-AXS-001-RAT】

内容、运行、条件、交互、外部责任和结果会独立变化，并分别影响编辑与版本管理、Halpha 的计划值守和条件评价、用户交互、执行处理和复盘评价。用一个状态合并会产生“计划暂停即外部责任结束”或“版本有效即条件有效”等错误。某条轴如果不再改变任何拥有领域的行为，应从计划状态中删除或改为普通派生信息，不新增综合总状态。

## 1.4 与 POR 的有界往返【TRADEPLAN-POR-001】

当存在多个现实资金用途、共同风险敞口或既有规则要求比较时，Halpha 可以按 TRADEPLAN 规则从交易计划依据或无运行效力的草案形成版本化交易计划资金需求，表达拟使用资金范围、时间、流动性、退出、用户注意力需求和会改变选择的未知。它不是交易计划、资金用途选择或资金使用上限与范围，也不允许 Halpha 形成、提交真实动作。

用户在多项资金用途比较的进入条件成立时，按 POR 定义的人工交接边界比较交易计划资金需求与其他真实资金用途，并向 Halpha 返回所选择的计划、资金范围和主要限制，或选择现金、不配置和修改草案。Halpha 只在返回内容与当前需求仍匹配时按 TRADEPLAN 规则把选择固化进交易计划；计划需求发生实质变化时重新比较，不能自动反复调整直到获得非现金结果。

单一计划在当次决定中不需要比较多个资金用途，且相应进入条件未成立时，可以直接形成交易计划；是否需要比较由 POR 规则和计划影响决定，不把该往返强加给全部计划。

## 1.5 计划合并、拆分与拟执行动作映射【TRADEPLAN-GRN-001】

一个交易计划必须具有统一的七类决定和连续责任生命周期，不能把原始信号、一次策略评价、一个订单或一个真实动作直接当作交易计划。一次策略评价可以产生零个、一个或多个交易计划依据；一个调仓建议可以按显式映射形成一个协调计划或多个独立计划。

### 拟执行动作定义【TRADEPLAN-HOF-002-DEF】

拟执行动作是 Halpha 根据适用来源形成的、可能造成交易场所或账户变化的不可变拟执行内容。适用来源包括：固定且已启用的交易计划版本、当前交易计划条件判断和计划内允许分支；已有风险敞口的保护或风险减少决定；以及用户明确提交的撤销、保护、划转或风险减少指令。来源、对象、范围、期限和不得扩大风险敞口的边界必须固定；Halpha 按 CAP 规则运行资金与权限检查时使用该内容。检查和本地提交成功前，尚未形成由 EXE 拥有的执行责任，成功后其固定内容形成待执行动作记录。拟执行动作不是第三种长期对象，提交失败时不留下执行责任。

### 计划合并、拆分与拟执行动作映射要求【TRADEPLAN-GRN-001-REQ】

多个输入只有在事前接受的确定性规则下，且计划目的、形成依据、适用的经济证据判断或固定策略依据、事实截止点、有效窗口、处理路径、七类决定和责任终点一致时，才可合并。合并必须保留全部来源，不得掩盖净额或冲突。交易计划固定后不得追加或合并新输入；新信息只能按既定条件评价，或形成新输入、新草案或新版本。

一个交易计划可以包含有序或条件性的多个动作，但每个真实动作仍分别经过资金与权限检查、唯一待执行动作记录、提交前事实刷新和执行器处理。L3 必须为每种已支持计划类型定义输入到计划的零/一/多映射、批量、拆分、合并、去重、并发、乱序、重放、形成计划和启用的最大延迟，以及多动作的顺序和部分结果；过期输入不得补执行。

# 2. 交易计划七类决定【TRADEPLAN-DEC-001】

## 2.1 交易计划七类决定定义【TRADEPLAN-DEC-001-DEF】

TRADEPLAN 负责把完整交易计划定义为以下七类决定的组合：

| 决定 | 决定内容 | 权威边界 |
|---|---|---|
| 1. 为什么承担风险 | 计划目的、所依赖的论点或配置目的、主要支持、最强反证、会改变决定的未知和失效依据 | 有边界经济证据判断或固定策略提供的计划依据引用 ALP，不由 TRADEPLAN 重评 |
| 2. 作用于什么 | 账户、场所、交易工具、方向、计划归属及与现有持仓和其他计划的关系 | 身份与当前事实引用 DAT；多项资金用途比较引用用户按 POR 边界作出的选择 |
| 3. 何时进入 | 必须成立的市场、账户、事件、时间或组合条件，以及每项条件的用途和事实来源 | TRADEPLAN 定义条件语义；Halpha 按 DAT 规则判断事实是否可用 |
| 4. 何时失效 | 论点失效、路径不可行、依赖失效、时间届满、禁止再进入及已有风险敞口需重评的条件 | Halpha 按 CAP 规则读取当前停用状态；其他原因引用其拥有领域定义的结论 |
| 5. 承担多少 | 拟使用金额或数量、成本与可接受损失假设、动作范围、分支上界和部分结果后的剩余责任 | 需要比较时引用用户选择的资金范围；Halpha 按 CAP 规则检查用户设定的资金使用上限与范围 |
| 6. 持续多久 | 生效、观察、触发、行动、复核和结束窗口，以及是否允许再次触发 | 具体期限下放 L3/L4 |
| 7. 触发后怎样处理 | 交易计划只通知路径、交易计划当前确认、有效机器授权支持的处理路径或其他明确的输入输出；未送达、超时、拒绝、部分结果、执行结果未决和交易场所或账户变化后的处理 | UX 定义呈现要求；Halpha 按 CAP 规则完成资金与权限检查，执行器按 EXE 规则处理外部动作 |

### 七类决定的选择依据【TRADEPLAN-DEC-001-RAT】

七类决定分别回答理由、对象、进入、失效、规模、期限和触发后处理，缺少任一项都会使资金使用后果、有效窗口或责任终点无法判断。它们是完整计划的检查维度，不是七个独立对象；某类对特定计划确实不适用时可以说明理由，不为形式完整保存空字段。

## 2.2 完整计划要求【TRADEPLAN-DEC-001-REQ】

除已有风险敞口的保护或风险减少、以及用户明确提交的撤销、保护、划转或风险减少指令外，FLOW 要求真实动作具有完整计划；每个交易计划版本都必须固定上述七类决定。

每个可能产生不同资金使用后果的分支都必须能够回溯到七类决定。只写进入条件，不写失效、期限、部分结果、保护要求和触发后路径，不构成完整交易计划。

决定可以声明“不适用”，但必须给出适用于本计划类型的理由；不得以空值、模板默认值、策略名称、AI 置信度或页面上下文制造完整性。

真实验证链路仍使用同一七类决定，但第一类决定表达要观察的策略、执行或稳定性问题和停止条件，不得伪装为已经成立的投资论点。它引用验证目标所对应的真实环境和用户配置的资金使用上限与范围，并把经济结果与工程行为分栏评价；一次结果不形成投资 Alpha。

# 3. 草案、固定版本与交易计划启用

## 3.1 草案允许不完整【TRADEPLAN-DRF-001】

草案必须显式保留已决定内容、未决内容、冲突、来源变化和下一项需要人决定的问题。草案可以暂停、恢复、复制或放弃，但不得：

- 进入持续监控、触发或真实动作链；
- 使用通知紧迫感暗示已存在行动责任；
- 把模板、历史计划或 AI 建议自动提升为用户决定；
- 在恢复时忽略期间发生的事实、经济证据判断、固定策略依据、资金使用上限与范围或 Halpha 真实资金操作权限变化。

## 3.2 固定版本不可改写【TRADEPLAN-VER-001】

七类决定、决定性来源、适用的经济证据判断或固定策略依据、事实截止点、条件解释规则、交互路径和适用依赖固定后，形成不可变交易计划版本。

每个固定版本必须记录用户对当前版本的明确决定。来自用户、CTX、ALP 或用户按 POR 边界作出的资金用途选择的交易计划依据默认只形成草案，必须由用户决定后才能固定；模板、AI 建议、历史计划和固定策略依据不能取得计划形成权。

任何改变对象、方向、计划使用资金的方式、进入或失效条件、期限、动作范围、保护要求、人工参与方式或事实解释的修改，都必须形成新版本。拼写、显示和不改变计划含义的修正可以通过独立说明处理，但不得覆盖历史内容。

新版本不继承旧版本的交易计划启用决定、交易计划当前确认、机器授权适用结论或事件去重状态。旧版本必须保留其事件、外部责任和结果链。

## 3.3 交易计划启用建立运行责任【TRADEPLAN-ACT-001】

交易计划启用前必须确认：

1. 七类决定完整且相互一致；
2. 交易计划依据、事实以及适用的经济证据判断或固定策略依据、用户资金用途选择和其他决定性引用可以识别且在计划用途下有效；
3. 承担计划观察与条件运行的构建、配置和适用范围已经完成与主要影响相称的工程验证，运行所需的直接依赖在启用时可用；不要求独立运行领域或运行可用性结论；
4. 条件能够按声明的时间、新鲜度和完整性要求评价；
5. 声明的交易计划当前确认（人工授权）或机器授权路径在保守时延下可行；
6. 值守缺口、用户不可达、交易计划条件无法判定、拒绝和超时均有明确终态；
7. 交易计划事件可以去重、追溯并交付后续领域；
8. 计划结束后可以形成局部结果材料并保留未闭合责任。

交易计划启用必须记录用户对当前计划版本的明确决定。普通交易计划依据、模板、AI 建议、用户资金用途选择、固定策略依据和计划已固定均不能自行启用交易计划。

交易计划启用只表示 Halpha 开始承担声明的观察、事件、交互和交接责任。它不证明策略具有 Alpha，不占用或提高资金使用上限或扩大范围，不改变 Halpha 真实资金操作权限，也不保证计划最终产生真实动作。

## 3.4 交易计划启用必须绑定交易记录环境身份【TRADEPLAN-ENV-001】

同一个不可变交易计划版本在内容完全相同时可以被不同交易记录环境重新引用，但每次交易计划启用都必须明确交易记录环境身份、适用账户和计划范围、有效窗口及复盘所需结果范围。Halpha 真实资金操作权限在形成和提交拟执行动作时按 CAP 规则重新检查，不由交易计划启用固定或授予。

交易所模拟盘环境的交易计划启用可以产生模拟动作、订单、成交、仓位和结果，但不形成真实待执行动作记录，不触发真实场所责任，也不通过真实资金与权限检查。用户决定从交易所模拟盘转入真实资金时，必须形成新的真实资金环境交易计划启用记录，并重新作出适用的用户决定；不得把模拟环境的交易计划启用或其状态直接改名为真实资金环境的交易计划启用。

若真实资金环境与交易所模拟盘环境的费用、成交、保护、数量或其他七类计划决定不同，必须先形成新的交易计划版本。两个环境中的条件状态、交易计划当前确认、拟执行动作与结果分别记录和结束，不相互继承。

# 4. 条件、评价与事件

## 4.1 条件必须类型化【TRADEPLAN-CND-001】

### 交易计划六类条件用途定义【TRADEPLAN-CND-001-DEF】

计划条件按用途分为：

- 进入：允许形成下一步拟执行动作；
- 失效：禁止计划继续形成相应新的真实动作，并决定已有风险敞口是否需要重评；
- 到期：时间窗口结束，不再接受过时触发；
- 复核：要求重新取得事实、证据、配置或人工决定；
- 退出或保护：形成既定的待检查风险处置动作，不等同于真实动作已获准；
- 通知：只形成用户交互任务，不表示用户已经作出资金使用决定或交易计划当前确认。

### 六类条件用途的选择依据【TRADEPLAN-CND-001-RAT】

同一观察可能用于进入、停止、到期、复核、退出保护或通知，而这些用途对资金使用、真实动作和失败处理的影响不同，混合后会产生错误处理。用途只决定条件成立后可以进入哪类计划处理，六套条件对象没有独立生命周期消费者。没有不同后果时，独立用途没有保留价值；新增用途的价值依据是存在无法由现有六类表达的责任终点。

### 条件声明与引用要求【TRADEPLAN-CND-001-REQ】

每个条件必须同时声明“观察什么”和“用于什么”。

同一观察用于多个目的时必须分别声明用途。没有不同资金使用、真实动作或失败处理后果的用途必须合并；新增用途必须说明现有六类为何无法表达其责任终点。

条件可以依赖市场、账户、组合、计划关系、外部事件、时间、能力或人工决定，但必须引用相应拥有领域的结论。TRADEPLAN 不复制市场真值，不把页面显示、推送到达或 AI 解释当作条件事实。

## 4.2 交易计划条件无法判定【TRADEPLAN-EVL-001】

### 交易计划条件无法判定定义【TRADEPLAN-EVL-001-DEF】

交易计划条件无法判定，是所需事实、时间或依赖当前不足以得出成立或不成立结论的一种交易计划条件判断结果。

### 条件评价与冲突处理要求【TRADEPLAN-EVL-001-REQ】

条件评价至少能够区分成立、不成立、无法判定、不适用、错过有效窗口和条件本身失效。所需数据缺失、不完整、无法确认连续性、冲突、过期、时间不确定或依赖失效不得静默变成“不成立”，更不得变成“继续沿用上次成立”。

同一时点多个条件冲突时，失效、停用 Halpha 新增真实动作和处理已有责任优先于进入。DAT 的关键事实处于事实未知状态时，相关条件评价为无法判定，默认不得形成新的真实动作事件；是否允许只读观察、通知或待检查的处置动作，由计划和当前 Halpha 真实资金操作权限共同限定。

## 4.3 交易计划事件形成不可变历史记录【TRADEPLAN-EVT-001】

### 交易计划事件定义【TRADEPLAN-EVT-001-DEF】

交易计划事件是由条件、人工决定、外部结果或时间变化形成，并固定计划版本和事实截止点的不可变计划历史记录。

### 交易计划事件保存与使用要求【TRADEPLAN-EVT-001-REQ】

交易计划事件必须让计划值守、执行交接和复盘能够确定：它来自哪个计划版本与条件用途、依据哪个事实截止点、为何及何时形成、是否仍在有效窗口内、怎样识别重复以及下一步怎样处理。只保存会改变上述行为的信息，具体字段进入 L3。重新计算可以形成修正或新事件，不能覆盖原事件。

条件成立只允许形成一个交易计划事件。交易计划事件进入资金与权限检查或人工决定路径后，仍可能被拒绝、过期或因最新事实失效。通知送达、用户打开页面和订单结果分别属于 UX、交互决定和 EXE/DAT 事实，不得合并为“触发成功”。

# 5. 时间可行性与连续值守

## 5.1 可行性按完整链路判断【TRADEPLAN-TIM-001】

计划必须对每个时间敏感分支评估以下完整链路：

> 取得适用事实 → 条件评价 → 形成去重事件 → 必要通知与理解 → 交易计划当前确认（人工授权）或机器授权路径 → CAP 资金与权限检查与待执行动作记录本地提交 → 执行器刷新关键事实并运行同一资金与权限复核 → 场所提交 → 必要保护与核对

局部数据延迟、模型计算时间或接口平均响应不能替代完整链路证据。路径只在保守尾部、失败重试边界和用户实际可达性下仍能在有效窗口内完成时才可采用。

L1 不限定交易时间尺度，TRADEPLAN 也不按策略名称排除时间受限机会。若完整链路不能可靠完成，则该计划必须选择更早的信号、更宽的窗口、已经取得有效机器授权的路径，或不行动。

## 5.2 连续观察与用户参与【TRADEPLAN-WAT-001】

已启用的交易计划必须声明其值守范围：哪些条件持续观察、哪些只在会话中计算、允许的数据新鲜度、观察缺口怎样处理、计划何时自动暂停或到期，以及用户不可达时采取何种终态。

只要计划仍声称会自动触发，就必须保持相应观察能力。真实验证或其他计划可以明确要求用户在线或及时响应；是否减少参与由已验证的流程和授权路径决定。观察中断时，停止受影响的新事件，保留已发生事件和外部责任，并按适用范围进入重评、停用或人工交接。

# 6. 交易计划当前确认与机器授权边界

## 6.1 交易计划当前确认【TRADEPLAN-HUM-001-REQ】

交易计划只通知路径只传递注意力，不表达真实动作授权。交易计划当前确认绑定当前有效的不可变计划版本、一次交易计划事件、当前事实和一次动作预览，只对该次拟执行内容成立；动作预览、事实或有效窗口发生实质变化时失效，拒绝、撤回、超时和未送达均不得形成新的待执行动作记录。撤回与 EXE 建立待执行动作记录的本地提交必须有可验证先后：撤回先成立时不得建立记录；记录先成立或无法排除已经建立时，不得改写交易计划当前确认或删除执行责任，必须禁止盲目重发并按 EXE 规则查询、核对或由用户接管。

交易计划固定、启用、条件成立和固定策略提供计划依据均不产生机器授权。需要用户不在当前链路中响应的计划分支，必须在触发前另行取得 CAP 定义的有效机器授权，并在形成和提交每个拟执行动作时重新检查；TRADEPLAN 只提供不可变计划来源、分支、时效和失败结果，不建立其他授权语义。

UX 负责定义用户理解、交易计划当前确认、拒绝、撤回和查看后果的交互要求；TRADEPLAN 负责验证该确认所绑定的计划对象、失效条件和未响应分支。任何界面默认值、快捷键或 AI 回复都不能扩大交易计划当前确认或机器授权范围。

具体 L3/L4 支持切片可以只实现事前取得机器授权的路径，无须同时实现交易计划当前确认路径；未实现的人工路径必须明确不支持，不能在触发后临时等待用户响应并将其视为机器授权。

## 6.2 计划决定与资金使用决定分离【TRADEPLAN-AUT-001】

用户可以确认计划内容，但 CAP 仍可因当前资金使用上限与范围、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态或关键事实拒绝动作。第一次检查不保证最终提交；事实变化后必须重新运行同一轻量规则。

Halpha 不得因计划规模较小、已有持仓或历史成功而绕过资金与权限检查。检查拒绝只形成计划内拒绝或重评事件，Halpha 不得修改拟执行动作后沿用旧交易计划当前确认继续尝试。

# 7. 部分结果、暂停、失效与结束

## 7.1 外部结果只能驱动预定义分支【TRADEPLAN-RES-001】

Halpha 读取按 EXE 规则形成的待执行动作记录接受、拒绝、部分结果、执行结果未决、保护和核对结论，以及按 DAT 规则形成的相应事实。Halpha 只能按原计划已经定义的分支继续观察、形成后续拟执行动作、请求新决定、暂停或结束。

部分成交、滑点、费用、场所拒绝、保护缺口、外部人工活动、执行结果未决或关键事实未知改变七类决定的含义时，必须停止旧版本形成新的真实动作，并形成新交易计划依据或新交易计划版本。不得在运行记录中临时改写计划以适配结果。

## 7.2 暂停、失效与过期【TRADEPLAN-STP-001】

| 结果 | 对新事件的效果 | 对已有外部责任的效果 |
|---|---|---|
| 暂停 | 停止声明范围内的新事件，等待恢复条件满足后重新决定 | 不撤单、不平仓、不解除保护与核对责任 |
| 失效 | 禁止依赖该前提形成新的真实动作；按计划判断已有风险敞口是否需处理或新计划 | 执行器和核对后台任务继续按 EXE/DAT 规则闭合责任，Halpha 按 CAP 规则重算已使用资金 |
| 过期 | 时间窗口结束，不补发错过的真实动作 | 已可能发生的交易场所或账户变化继续核对 |
| 被新版本替代 | 旧版本不再接受新的交易计划启用、交易计划当前确认或机器授权引用 | 历史事件和未决责任仍归旧版本 |

只有能够证明尚未越过交易场所或账户变化边界的交易计划事件，才能直接拒绝或过期。是否已经越过该边界由 Halpha 按 EXE 规则判断，不得按 TRADEPLAN 本地超时推断。

## 7.3 计划结束与责任闭合【TRADEPLAN-END-001】

计划可以因完成、失效、过期、拒绝、取消、计划不再形成新的真实动作或被新版本替代而结束，并停止形成新事件。结束必须记录唯一主要原因及适用的次要原因。

计划结束后仍可能存在订单、仓位、保护、已使用资金估算、执行结果未决或用户接管责任。计划责任只有在以下一种结果成立时才闭合：

- Halpha 按相邻领域规则证明从未产生交易场所或账户变化；
- 全部外部事实、保护和核对责任已经闭合；
- 未闭合责任已经以明确对象、适用范围和接收者明确移交。

不得为了关闭计划而把开放仓位、未决待执行动作记录或风险敞口保护任务改称“新计划负责”。新计划可以接续管理，但必须保留责任转移链。

# 8. 供复盘使用的交易计划事实【TRADEPLAN-RVW-001】

Halpha 按 TRADEPLAN 规则在计划结束或关键分支完成时保存以下事实：

- 原交易计划版本、交易计划依据和七类决定；
- 实际条件评价、事件、人工决定和时间路径；
- 计划预期分支与实际分支的差异；
- 未触发、拒绝、过期、计划不再形成新的真实动作、部分结果、执行结果未决和事实未知；
- 用户可补充但 Halpha 无法观察的当时理由；
- 仍未闭合或已经移交的责任；
- 值得提交给拥有领域的问题，不直接修改其对象。

Halpha 将这些事实与计划版本和事实截止点一并用于复盘，并按 OUT 规则评价计划是否完整、是否按计划运行、是否存在偏离，以及这些问题是否影响综合结果；OUT 拥有综合评价语义，TRADEPLAN 不定义本领域评价对象，也不负责策略 Alpha、组合最优性、资金风险管理质量或 Halpha 增量产品价值的评价。事实需要修正时，Halpha 按拥有该事实的领域规则形成新版本，并据新事实更新复盘记录。

# 9. 失败闭合【TRADEPLAN-FAL-001】

| 失败位置 | TRADEPLAN 的确定结果 |
|---|---|
| 草案或固定版本形成失败 | 保持无效草案或拒绝固定，不产生值守责任 |
| 交易计划启用前依赖缺失或冲突 | 拒绝启用；保留缺口和拥有领域 |
| 值守、数据或时间能力不足 | 暂停受影响事件；不得沿用旧评价 |
| 通知未送达、交易计划当前确认超时或用户拒绝 | 不形成相应拟执行动作；记录明确事件 |
| 资金与权限检查或提交前资金与权限复核拒绝 | 不提交；计划按预定分支结束、等待或重评 |
| Halpha 按 EXE 规则能够证明未提交 | 拒绝、过期或重评，不补执行错过事件 |
| 无法按 EXE 规则排除交易场所或账户变化 | 保持计划责任开放，停止重复真实动作，交给执行器或核对后台任务核对，并按可能已发生计入风险敞口 |
| 已有交易场所或账户变化 | 计划停止不删除责任；继续保护、核对、结束或明确移交 |

TRADEPLAN 的失败终态必须是“无真实动作且责任闭合”或“可能/已有交易场所或账户变化且责任主体明确”。不得留下只有页面报错、任务消失或计划状态未知的中间结果。

# 10. 明确的输入输出【TRADEPLAN-HOF-001】

| 方向 | 内容 |
|---|---|
| 接收 | 用户已经明确选择并交付的交易计划依据；CTX 的研究或交易计划候选去向决定所选择的交易计划起始材料；ALP 的有边界经济证据判断或固定策略提供的计划依据；适用时用户按 POR 边界作出的资金用途选择；DAT 的事实、时间、交易记录环境身份和事实未知状态；会改变计划可行性的已知系统能力限制；CAP 的资金使用上限与范围、Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态；EXE 的交易场所结果、执行结果未决、保护与核对结论 |
| 输出 | 供 Halpha 按 UX 规则呈现的用户交互任务、有效预览、时限和允许决定；供用户按 POR 边界比较且不占用资金、不授予权限的交易计划资金需求；供 Halpha 按 CAP 规则检查的不可变计划来源与拟执行动作语义；供执行器按 EXE 规则处理的拟执行动作来源（计划路径引用计划来源，例外路径引用对应决定或指令）、时效、分支和保护要求；供 Halpha 按 OUT 规则形成复盘记录的计划版本、条件、事件、决定、偏离、结束和责任闭合事实 |

每项领域 L3 必须声明一个主要语义 L2 所有者并列出直接依赖。这里的“所有者”是语义责任归属，不是真人身份。只有设计本身拥有跨领域业务对象或交接生命周期时才使用编排 L3，并按 DOC 指定唯一协调所有者和参与语义所有者；普通多领域接口不因此变成编排 L3。任何 L3 都不得重新定义相邻领域的稳定对象和决定。

# 11. L3 必须精确定义【TRADEPLAN-L3-001】

L3 按实际支持的计划类型分别定义：

- 交易计划草案、版本、启用、交易计划条件判断和交易计划事件的数据表达、身份、状态转换与不变量，以及这些语义如何表达计划责任；允许由启用记录或事件内嵌结构承载相关表达，但必须保留各语义的独立身份、不可变历史、事实截止点和可验证结果；
- 交易计划启用记录的交易记录环境身份、账户和计划范围、有效窗口及复盘所需结果范围，以及交易所模拟盘转真实时的重新启用与禁止状态继承规则；Halpha 真实资金操作权限必须在形成和提交拟执行动作时重新检查，不由启用记录绑定；
- 交易计划资金需求如何按需生成、何时重算或不再适用，以及为何不建立独立生命周期、不绑定用户的资金用途选择；
- 七类决定的完整性规则、不可变引用、版本替代和形成边界；
- 交易计划依据与交易计划的零/一/多映射、批量、拆分、合并、来源保留、去重、并发、乱序、重放、形成计划和启用的最大延迟；
- 计划形成依据、用户固定版本与启用的明确决定，以及相应绑定、拒绝和失效契约；
- 各类条件的输入契约、事实用途、新鲜度、无法判定、错过、失效、冲突和去重规则；
- 交易计划启用、暂停、恢复、过期、结束和责任移交的状态转换；
- 交易计划当前确认的预览、绑定、时效、撤回与重放防护，以及机器授权只由 CAP 提供的边界；
- 完整时间链路的预算、测量、保守判定和不满足时的降级；
- 多动作计划的动作顺序、逐动作资金与权限检查、部分结果、停止与剩余责任；
- TRADEPLAN 与当前直接生产者和消费者之间的机器可读交接；每项交接只包含对方当前需要的语义，并声明缺失、过期和拒绝结果，不建立面向全部领域的统一交接对象；
- 供 Halpha 按 OUT 规则形成复盘记录的计划事实范围、事实截止点、补充、修正和完整性规则；
- 上述每个契约的正常、拒绝、过期、无法判定、并发、重启和历史版本测试。

字段名、事件格式、数据库事务、队列、缓存、接口和页面属于 L3 技术选择，不在 L2 固定。

# 12. L4 选择边界【TRADEPLAN-L4-001】

L4 只记录当前建设选择的交易计划依据来源、计划/条件/事件类型、交互与时效边界、明确不支持范围和验收证据。具体计划、版本、启用、事件、交易计划当前确认、结束和开放责任属于 TRADEPLAN、EXE 等拥有领域的运行记录，不写入 L4；L4 也不得从当前成功记录反向扩大 L2 支持范围。
