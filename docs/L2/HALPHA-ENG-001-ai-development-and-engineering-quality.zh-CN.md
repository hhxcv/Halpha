# Halpha AI 开发与工程质量原则

**文档编号：** HALPHA-ENG-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** AI 辅助开发的人机边界；个人项目的相称验证；最小开发闭环；成熟组件优先；可重复构建、部署、回退和依赖维护的稳定要求  
**本文档不负责：** 产品和交易语义、当前建设范围、精确组件版本、当前主机与配置、运行可用性、资格状态，或机构级工程治理

# 0. 定位与复杂度预算【ENG-SCP-001】

ENG 是服务产品交付的辅助能力，不能比核心交易闭环更难理解、运行或修改。对单一所有者项目，默认使用普通 Git、直接测试、短说明和人工决定；流程只有在真实外部写入、不可逆影响、多人或多实例并发、重复故障已经出现时才增强。

复杂度按两个维度分别处理：

- 动作唯一、防重复、事实核对、保护、停止和人工接管属于功能正确性，不因模拟、资金少或项目早期而降低；
- 评审层级、状态、记录、持续观察和治理自动化属于工程投入，必须由当前风险和真实消费者证明必要。

删除、普通语言、已有工具、直接测试和人工步骤能解决的问题，不建立新概念、状态、记录、流程或平台。文档和检查的数量不是质量指标。

# 1. AI、开发者与项目所有者【ENG-AID-001】

## 1.1 可信但可能犯错【ENG-AID-001-REQ】

在用户授权的任务、路径、工具、数据和凭据边界内，AI 按开发者工具使用。AI 可以分析、实现、测试、评审和整理材料，但其输出不天然正确，也不产生设计接受、产品启用或资本决定。

AI 不得：

- 扩大任务、真实资金范围或外部写入权限；
- 持有真实交易凭据或自行触发真实交易；
- 声称运行了未运行的检查；
- 删除不利测试、失败和已知限制；
- 用篇幅、置信度、代理数量或同源赞同代替实际结果。

项目所有者决定工程和产品建设；用户决定交易计划启用、资金限制和真实动作。AI 或开发者不能代替两类决定。

## 1.2 防错原则【ENG-AID-001-RAT】

当前主要风险是善意开发中的误解、遗漏和实现错误，因此优先使用范围核对、静态检查、自动测试、实际运行、差异复核和可回退修改。只有出现不同信任主体、恶意输入或简单控制无法限制的高成本失败时，才增加隔离、专用协议或额外运行环境。

# 2. 按影响选择验证【ENG-IMP-001】

## 2.1 影响分类【ENG-IMP-001-DEF】

| 影响 | 典型内容 | 默认验证 |
|---|---|---|
| 核心 | 交易判断、计划、订单、保护、账户事实、凭据、停止、恢复和常用控制界面 | 相关自动测试、关键反例、近真实链路检查、回退或停用 |
| 一般 | 普通业务功能、内部接口、数据投影和非关键交互 | 相关测试、实际运行或渲染检查 |
| 轻量 | 文案、辅助脚本、研究探索和格式调整 | 作者检查与最小相关验证 |

分类只用于选择投入，不形成资格或准入状态。边界不清时先检查可能的高影响结果；证明影响已隔离后即可采用较轻验证。

## 2.2 验证底线【ENG-IMP-001-REQ】

核心影响变更至少覆盖正常路径、一个关键反例、重复或重试、停止或人工接管，以及失败后的可识别结果。模拟和小额真实运行可以补充联合行为信息，但不能替代动作唯一、保护、核对和用户控制。

# 3. 最小开发闭环【ENG-LCY-001】

一个变更只需完成以下闭环，简单变更可以合并步骤：

1. 说明实际问题、消费者、完成结果和不做范围；
2. 读取直接适用设计、实现和测试；
3. 优先删除或复用成熟能力，只为不可妥协缺口保留最小代码；
4. 以可检查的小改动实现；
5. 运行与影响相称的验证并检查实际差异；
6. 说明失败、限制和回退或停用方式；
7. 集成后检查直接相关结果。

不为这些步骤分别创建状态、表单、审批或证据对象。

# 4. 小切片与并行【ENG-WRK-001】

任务应小到所有者可以理解差异、验证行为并在失败时回退。已授权的一个清楚结果可以包含多个可独立测试的小切片，不要求每个切片重新取得授权或建立新任务。

并行只在语义、路径、数据库、凭据和外部影响互不重叠时使用；共享文件和产品集成保持串行。研究只写独立工作区且不影响产品运行、数据库、凭据或场所写时，可以与产品建设并行。

第二视角只在核心影响、跨领域歧义、重复逃逸缺陷或高成本失败时使用。复核从预期与反例重新推导，不以代理数量制造独立性。

# 5. 验证与实际证据【ENG-TST-001】

验证按真实失败方式选择：

- 规则和边界例用单元测试；
- 事务、状态和模块交接用集成测试；
- 外部数据与场所语义用沙盒、录制样本或受控观察；
- 常用交互用实际渲染和连续操作；
- 启动、重启、追赶、备份和恢复用实际进程或还原检查。

证据只有在一个当前决定明确消费它时才成为前置。最小记录说明所支持的决定、参与结果的输入或运行身份、结果、时间和限制。只有这些输入、测量方法或结论发生相关变化时才重验；普通文档、独立研究或其他未进入产品构建的变化不会自动使产品结果失效。

自然时间只证明实际观察到的行为，不替代直接测试。没有当前发布、部署或真实资金决定需要时，不设置固定时长观察或额外资格流程。

# 6. 评审与接受【ENG-REV-001】

作者先检查实际差异和测试输出。必要复核的结论只有：可集成、限定范围可集成、需修复。未知项直接写清，不建立多层工程认可状态。

工程检查通过不授权真实资金。开始或扩大真实资金使用仍由用户依据产品内计划、限制和控制作出决定。

# 7. 构建、部署与回退【ENG-REL-001】

可部署产品必须能够识别其源码、锁定依赖、数据库迁移、前端制品和非秘密运行配置。用一个产品构建标识追溯这些输入即可，不为同一构建再建立平行身份或状态。

部署说明至少覆盖构建、迁移、启动、停止、重启、日志、凭据注入、备份、还原、回退和用户接管。当前主机、精确命令、组件版本、配置值和实际检查结果属于 L4。

核心影响发布必须有可执行的回退或停用方式；不可逆迁移必须先有可用备份与还原检查。长期观察只有在当前消费决定需要且直接测试无法回答时才增加。

# 8. 第三方能力与最小自研【ENG-DEP-001】

对准备长期维护、重复使用或进入核心路径的通用能力，先快速检查成熟框架、库、官方适配器或操作系统能力。重要组件核对公开契约、支持边界、许可证、维护状态和退出方式；一次性低影响胶水不要求形成正式选型材料。

处理顺序是：

1. 组件满足产品结果时直接采用；
2. 只影响未固定技术选择时，按组件行为调整；
3. 组件缺少不可妥协的产品语义时，只补最小边界；
4. 补充成本高于当前价值时，明确不支持或人工处理。

同一能力只保留一个运行实现和一个事实权威。不得为了未来可能性长期维护双实现、影子栈或通用适配平台。

# 9. L3 与 L4 边界【ENG-LVL-001】

L3 只在当前实现确实需要时固定组件使用契约、模块边界、产品构建标识的最小输入、部署接口、秘密加载顺序和失败结果。L4 记录当前精确版本、配置、主机、构建标识、验证结果和已知限制。

任何新增工程概念都必须回答“当前哪个消费者会因此改变行为”。没有消费者，或删除后用户行为、安全结果和实现判断不变，就不进入设计。
