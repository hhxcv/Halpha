# 0. 定位与复杂度预算【ENG-SCP-001】

ENG 是辅助交付能力，其流程复杂度不得超过核心交易业务，也不得为了形式完整消耗本应投入 ALP、TRADEPLAN、DAT、EXE、UX 和系统稳定性的维护注意力。

Halpha 采用“影响越大，验证越强；影响越小，流程越轻”的个人项目工程方式。默认不建立独立工程认可结论、生产准入委员会、多层证据包、固定数量 AI 助手评审或覆盖所有变更的统一质量矩阵。

ENG 的成功标准是：核心缺陷更少、修改反馈更快、发布可恢复、系统更稳定、项目所有者能够理解和维护，且开发成本与产品价值相称。文档、检查、制品或评审数量本身不是质量。

通用能力的默认工程顺序是删除、直接复用成熟第三方组件、按组件实际能力收缩可妥协技术选择、保留不可妥协缺口的最小胶水，最后才是明确依据的自研。现有设计已经写出自研对象不构成继续自研的证据。

# 1. AI、自动化、开发者与项目所有者【ENG-AID-001】

开发者身份、可兼任关系及资本与账户权限边界直接继承 CON-USR-002；ENG 只规定开发、验证、发布和反馈等工程行为。

项目所有者授权并在适用仓库规范、任务或工作包范围、路径、工具权限、数据许可与凭据边界内工作的 AI，按可信开发者工具处理。“可信”表示开发者所使用的 AI 不以作恶为目的，并会尽最大努力保障工作正确；它不保证 AI 不会理解错误、遗漏、误用工具、错误判断或产生实现缺陷，也不使其输出天然正确、已验证或具有接受效力。

### 开发者与项目所有者边界要求【ENG-AID-001-REQ】

开发者可以按 ENG 规则提出、修改、验证和发布工程内容；项目所有者决定工程变更是否符合项目设计和当前建设计划，用户决定产品使用和资本控制事项。涉及业务语义或真实资金路径的变更，必须分别取得项目所有者的建设决定和用户的交易决定；两者不能由开发者或 AI 代替。项目不为开发者另建第二种真人身份或生产准入流程；工程验证结果只能作为相应角色决定的输入。

AI 可以搜索、分析、实现、测试、评审、整理文档和提出方案，但输出需要核对。AI 不得：

- 持有真实交易凭据或直接产生真实资金交易场所或账户变化；
- 自行扩大任务范围、资金使用上限与范围，改变经济证据判断、固定策略提供计划依据的边界或交易计划启用，或提高 Halpha 真实资金操作权限；
- 声称运行了未运行的检查；
- 用置信度、篇幅、AI 助手数量或同源赞同代替实际结果；
- 删除或覆盖不利测试、真实故障和已知限制。

AI 防错控制必须按场景重要程度、潜在错误成本和可恢复性选择。优先使用范围审查、输入与输出 schema、静态检查、自动测试、实际运行或渲染、独立重演、差异与证据复核、回退，以及进入真实资金使用前适用的策略审查、研究验证和产品内重新资格化。不得仅因代码由 AI 编写，就把它额外视为敌对代码并建立研究局部安全边界；只有实际错误场景与风险估算能够证明新增隔离降低的预期损失高于其实现、运行、升级、维护和退出成本时，才采用该控制。成本投入明显高于风险价值的隔离、重复运行时和专用协议不得进入当前设计。

只有用户可以提高 Halpha 真实资金操作权限、提高系统内资金使用上限或扩大交易适用范围。开发者和 AI 可以提出风险，但不得触发这些变更；项目所有者可以决定是否继续建设或修改项目设计，但不能以项目所有者身份代替用户的资本决定，也不建立额外审批层。

## 1.1 可信 AI 与相称防错的选择依据【ENG-AID-001-RAT】

Halpha 的产品、研究和工程文档本来就由开发者使用 AI 共同建设；只在 `research` 对同一 AI 建立敌对代码假设，不能解决其在其他项目路径中也可能犯错的事实，反而增加 sandbox、身份、运行时、协议、资格化和长期维护责任。当前主要风险是善意开发过程中的误解、遗漏和实现缺陷，因此审查、schema、测试、重演、证据复核、回退和真实资金前重新资格化更直接地针对错误影响。若未来出现可观察的恶意输入、不同信任主体或简单控制无法限制的高成本失败，再按实际净风险重新设计，而不是预先支付明显高于当前风险价值的隔离成本。

# 2. 三档影响级别【ENG-IMP-001】

## 2.1 工程影响级别定义【ENG-IMP-001-DEF】

| 级别 | 稳定适用范围 |
|---|---|
| 核心影响 | 交易判断、经济证据判断与固定策略形成计划依据的行为、交易计划、账户或行情事实、订单、保护、核对、凭据、停用、常用 UX 和启动恢复 |
| 一般影响 | 普通业务功能、内部接口、数据投影、后台任务和非关键交互 |
| 轻量影响 | 文案、低影响报告、辅助脚本、非生产研究工具和格式整理 |

## 2.2 分级与验证要求【ENG-IMP-001-REQ】

每项变更只选择当前最高适用级别：

| 级别 | 默认验证要求 |
|---|---|
| 核心影响 | 明确预期；相关自动测试；独立于实现步骤的复核；在接近真实环境中验证关键链路；可回退 |
| 一般影响 | 相关测试；实际运行或渲染检查；变更说明；必要回退 |
| 轻量影响 | 作者检查和最小相关验证 |

资金规模小不等于执行代码影响小，但真实验证可以作为核心影响变更联合链路验证的一部分。验证应围绕最可能造成真实错误的地方，而不是机械覆盖每个质量维度。

若分类不明，先按较高一级处理；一旦证明影响被隔离，可以降级。分类只用于选择工程投入，不授予 Halpha 真实资金操作权限。

## 2.3 三档影响级别的选择依据【ENG-IMP-001-RAT】

核心影响、一般影响和轻量影响分别对应可能改变真实动作或核心可用性、普通业务行为和低后果辅助内容三种验证强度，直接被开发者和项目所有者用于选择检查与回退投入。三级已经足以避免所有修改都走同一重流程；分类不形成工程认可或准入对象。相邻级别长期采用相同验证时，独立分级的价值依据消失；只有重复缺陷证明现有分级无法选择相称验证时，增加级别或规则才有额外价值。

# 3. 最小开发闭环【ENG-LCY-001】

每项进入正式代码的变更采用以下闭环，步骤可在小变更中合并：

1. 说明要解决的实际问题、用户、项目所有者或其他 Halpha 消费者、完成条件和不做范围；
2. 读取直接适用的规范、现有实现和测试，确认不会复制相邻领域语义；
3. 对拟自研能力及其中可抽离的通用部分检索成熟组件，比较实际能力、限制、许可证、目标平台、退出和总复杂度；
4. 按第 8 节决定直接采用、按组件能力调整、最小补充或明确不支持，再以可独立检查的小切片实现；
5. 运行与影响级别相称的自动和人工验证；
6. 检查实际差异、失败输出、数据迁移和回退方式；
7. 集成到唯一变更版本；
8. 对核心影响变更在模拟、沙盒或用户配置的较小真实资金使用范围内观察联合行为；
9. 发布后观察关键结果，发现问题立即回退、停用或修复。

流程允许重复和回到前一步，不要求为每个步骤建立独立状态、角色、表单或制品。

---

# 4. 小切片与并行【ENG-WRK-001】

任务应尽量小到项目所有者能够理解差异、验证行为并在失败时回退；用户的产品使用和资本决定不由任务拆分替代。并行只用于边界清楚、能够独立产出且不会争夺同一语义或文件的工作。

多个 AI 助手的输出必须回到一个集成变更版本中检查。AI 助手数量不产生独立性；真正有价值的独立复核是从需求或失败场景重新推导预期，而不是重复阅读同一实现后表示同意。

对于个人项目，默认优先顺序是：

1. 修复核心业务错误和重复运行故障；
2. 改善交易决策、执行、数据、UX 和稳定性；
3. 删除重复自研并复用经验证的成熟组件；
4. 补足直接阻塞上述工作的最小工程工具；
5. 最后才建设通用平台、模板或治理自动化。

# 5. 验证策略【ENG-TST-001】

验证按变更风险选择最小充分组合：

- 纯函数和规则使用单元测试与边界例；
- 模块交接、事务和状态推进使用集成测试；
- 场所、数据源和真实外部语义使用沙盒、录制样本或受控真实观察；
- 常用交互使用实际渲染和连续操作检查；
- 启动、重启、后台追赶和故障恢复使用实际进程验证；
- 时效和吞吐只在存在真实瓶颈或明确交易需求时测量完整链路；
- 第三方组件使用公开契约、固定候选版本和代表性输入验证关键语义、精度、身份、错误、重启、升级与退出；可妥协差异改用组件语义后重新生成预期，不沿用自研实现的成功结果。

核心影响变更至少验证正常路径、一个关键反例、重复或重试行为、停止或回退路径。测试不得持有生产提现或转账能力，不得产生超出用户为真实验证配置的资金使用上限与范围的交易场所或账户变化。

回测、模拟、自动测试、AI 评审、真实验证和用户观察各有盲区。它们可以共同支持对功能与实现是否经过充分验证的判断，但不需要合并成统一工程认可结论。单次成功和单次盈利不得触发系统自动提高资金使用上限或扩大范围。

# 6. 评审与接受【ENG-REV-001】

作者或实现 AI 助手必须先检查实际差异和测试结果。只有以下情况默认需要第二视角：

- 核心影响变更；
- 需求或规范含义不清；
- 修改跨越多个领域边界；
- 失败可能重复下单、误报账户状态、阻止停用、泄露凭据或破坏恢复；
- 已经出现过同类逃逸缺陷。

第二视角应重点验证业务预期和反例，不要求固定角色数量或完整评审矩阵。低影响变更可以由作者完成后直接集成。

接受结论只有：可集成、限适用范围可集成、需修复。未知项写清即可，不建立多层工程认可状态机。用户按 CAP 规则决定 Halpha 真实资金操作权限；核心影响验证结果、真实验证观察和当前系统表现只能作为输入，不能替代适用的经济证据、交易计划、资金使用限制或用户命令，ENG 不批准该权限。

# 7. 构建、发布与回退【ENG-REL-001】

正式发布必须能够识别源版本、主要依赖的精确版本与构建来源、配置版本和数据库迁移。构建应可重复到足以定位差异；许可证及必要归属信息必须随锁定依赖可追溯，但不要求为个人项目建设企业级供应链证明平台。

发布采用小批次。核心影响变更发布前必须有可执行回退或停用路径；数据库迁移若不可逆，必须有备份和恢复验证。发布后观察与该变更直接相关的业务和稳定性信号。

每个可部署构建必须附带可重复执行的部署说明，至少说明构建与配置身份、迁移、启动、停止、重启、普通日志、现成备份与还原、凭据注入边界、发布、回退和用户接管入口。说明只规定可重复步骤和验证要求；当前主机、服务管理器、监听地址、连接、日志位置和实际命令由 L4 或外部工具记录，秘密值只保存在授权的外部秘密工具中。

Halpha 不设统一生产准入流程。ENG 只提供当前构建、相称验证、回退与已知限制等工程输入；开始或扩大真实资金使用仍由用户按 CON、ALP、TRADEPLAN 和 CAP 的适用规则决定，不得由“可集成”结论、发布完成或工程检查通过触发。

# 8. 第三方能力、依赖与最小自研【ENG-DEP-001】

## 8.1 能力检索门槛【ENG-DEP-001-REQ】

任何拟自研能力及其中可抽离的通用部分，在进入详细实现前必须检索适合当前用途的成熟框架、库、官方适配器或标准基础设施。检索优先使用官方文档、固定版本发布说明、公开 API 或源码、许可证和维护状态等一手证据；不能凭记忆、既有草稿、自研代码较短或未来可能更换组件认定“没有可复用能力”。

最小能力比较必须写清：真实消费者与所需结果、组件可直接承担的能力、缺失或语义差异、失败影响、集成与退出成本，以及采用、妥协、最小补充或不支持的结论。证据不足时保持未知或阻断，不把未知当作组件缺失，也不直接转为自研。

## 8.2 差异分类与妥协判定【ENG-DEP-001-CLS】

组件差异只分为两类：

- **不可妥协缺口：** 会改变当前适用规范或项目所有者明确固定的产品语义，破坏动作唯一、状态正确、事实核对、停止与恢复等功能正确性，或突破安全、权限、凭据和用户控制边界；
- **可妥协差异：** 只改变尚未被当前适用规范或项目所有者固定的策略数学、算法种子、内部结构、技术生命周期、命名、性能实现或其他技术选择。

可妥协差异必须按组件实际语义调整设计并重新验证受影响结果，不能为了维持旧文档、旧测试或个人偏好复制组件能力。无法判断产品语义是否可妥协时，开发者必须请求拥有该语义的主体澄清；澄清前保持阻断，不以自研回避未知。

## 8.3 采用、最小胶水与单一实现【ENG-DEP-001-SEL】

决策顺序固定为：

1. 组件满足所需产品结果和正确性时直接采用，删除对应自研设计、代码和只验证自研内部细节的测试，保留产品契约与组件资格测试；
2. 差异只属可妥协技术选择时按组件实际能力设计并重验；
3. 组件无法满足不可妥协缺口，且没有另一成熟组件能以更低总复杂度满足时，只保留补足该缺口的最小胶水；
4. 最小补充成本仍高于当前价值时，明确拒绝、停止、用户接管或不支持。

最小边界补充（以下简称“最小胶水”）只能承担必要边界翻译、不可妥协产品语义或功能正确性补充，不得复制组件的通用能力、内部缓存、总线、状态机、网络生命周期或完整适配器。组件完整替代模块内部能力时，Halpha 只保留调用、输入输出、事实权威、错误映射、资格化和退出边界，不建立对称内部实现。

同一能力只能有一个运行实现和一个事实权威。不得同时保留组件实现与自研实现作为静默、自动或长期平行 fallback，也不得以“未来可替换”为理由预建完整包装平台。组件不可用或资格失效时按业务设计停止、只读、用户接管或明确不支持；恢复或换用另一组件必须形成新的唯一选择并重验。

## 8.4 资格证据、退出与复杂度净变化【ENG-DEP-001-EVD】

重要组件进入当前建设选择前，L3 必须明确长期使用契约和最小自研边界，L4 必须固定并记录：

- 精确版本、构建或源码身份、锁定方式和升级/回退兼容边界；
- 许可证、必要归属、传递依赖及其与发布方式的兼容结论；
- 目标平台、安装、导入、关键 API、精度、身份、错误、重启和资源边界的实际资格结果；L4 选择 Windows 时，必须在目标 Windows 版本和实际安装形态上验证，不得用其他操作系统成功替代；
- 数据与必要配置的导出、迁移、恢复和退出条件，以及组件停止维护、许可证变化、平台不兼容或关键语义改变时的替换路径；
- 删除的自研代码、状态、测试、进程、数据库、记录、写入路径和维护责任，与新增依赖、胶水、配置、运行、验证、升级和退出成本的可核对比较。

只有新增总复杂度由删除的重复自研或明确新增价值抵消，且项目所有者能够理解、验证、停止、升级和退出时，组件或最小胶水才可进入建设。资格结果属于 L4 当前证据；未通过时不声称可用，也不启动平行自研路径。

依赖更新按影响级别验证：直接进入交易、数据持久化、认证和外部适配的更新按核心影响；普通工具按一般影响或轻量影响。不要求为全部传递依赖建立人工审批，但不得静默自动更新会改变真实交易语义的组件。

删除无人使用的工具、模板、流水线、适配层和抽象与新增能力同等重要。工程系统自身必须证明净节省维护时间，否则应简化或退出。

# 9. 反馈与经验【ENG-LRN-001】

重复出现的缺陷、回退、测试盲区和维护瓶颈应形成简短经验，并优先固化为更好的接口、测试、默认值或自动检查。一次性问题不必形成永久流程。

新增工程控制前必须回答：

1. 它针对已经观察到的什么失败；
2. 为什么现有简单检查不足；
3. 它每次会增加多少认知与运行成本；
4. 何时删除或降级。

若控制产生大量误报、拖慢核心迭代或需要持续人工维护而没有减少实际缺陷，应合并或退役。

# 10. 与相邻领域的边界【ENG-BND-001】

开发者和项目所有者按 ENG 规则验证实现是否符合各领域给出的行为契约，但不重新判定 ALP 的经济证据、TRADEPLAN 的计划质量、CAP 的资金使用上限与范围、DAT 的账户事实、EXE 的执行结果或当前部署是否可用。

SYS 定义模块、运行实体、宿主与进程、事务、运行时配置、秘密注入和恢复前置边界；EXE 定义真实写凭据的唯一消费、外部写入控制和业务恢复责任；UX 定义交互验收要求。ENG 负责第三方能力检索与差异证据、最小自研门槛、构建、迁移、发布、回退与部署说明的可重复验证，不取得组件所承载的业务语义或事实所有权；实际宿主、精确依赖版本、部署、凭据注入、日志、备份和外部入口由 L4 记录。项目不建立独立运行状态产品或跨领域准入机构。

# 11. L3 与 L4 交接【ENG-HOF-001】

只有实际开发流水线需要 L3，最少定义：

- 三档影响级别怎样标记；
- 当前仓库的必跑检查和核心影响变更额外检查；
- 长期采用的具体第三方组件、公开使用契约、Halpha 与组件的责任边界，以及不可妥协缺口的最小胶水；
- 构建版本、迁移、发布、回退、停用命令和可重复部署说明；
- 凭据如何与开发及测试隔离；
- 哪些核心链路用沙盒或真实验证观察。

L4 记录当前工具、第三方组件精确版本与构建来源、许可证结论、Windows 等目标平台资格结果、当前配置、复杂度净变化、退出条件、构建方式、构建与配置身份、实际主机与服务管理、发布位置、凭据注入、日志、备份/回退状态、用户接管入口、已知核心缺陷和最近真实验证。不记录假想工程认可与准入状态。
