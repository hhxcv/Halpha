# Halpha 外部动作执行原则

**文档编号：** HALPHA-EXE-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** 外部动作责任、唯一提交、防重复、未知结果、保护和核对  
**本文档不负责：** 策略、计划与激活生命周期、资金计算、事实定义、页面和当前场所配置

## 1. 唯一职责

EXE 拥有 Halpha 发起的每个外部动作从本地建立、场所提交、查询与事实核对到明确闭合或交给用户的唯一处理责任。计划、页面、任务、日志和供应商缓存不能替代或推进该责任。

## 2. 先持久后外写

每个拟执行动作最多建立一个稳定动作责任。任何场所写入都发生在该责任成功持久化之后；调用前重新读取当前计划、CAP 检查、事实和停止结果。

同一来源和内容重复时返回原责任；身份相同但内容冲突时拒绝。真实写入范围同时只允许一个 Halpha 写者，无法证明唯一性时停止新写入。

## 3. 结果与未知

外部调用只有三类业务结论：能够证明没有外部变化；能够由稳定身份和事实确认结果；或者结果未知。超时、断线、崩溃、部分响应和金额较小都不能被猜成成功或失败。

结果未知时按动作可能已经发生处理，只查询原外部身份，不创建第二动作或改换条款。受影响新增风险停止；可靠查询、已有保护、退出和用户接管继续。

部分成交、迟到成交、费用和场所修正都关联原动作。需要撤销、补保护、减仓或退出时，计划形成新的明确动作，而不是修改或重发原动作。

## 4. 保护与闭合

风险出现后，保护是持续责任。只有场所事实能够证明保护实际有效；“已经提交”或页面成功不能替代。保护缺失、冲突或未知时停止受影响新增风险，并优先处理保护、减仓、退出或用户接管。

动作只有在能够证明未产生变化、实际结果及影响已经核对，或剩余责任已明确交给用户时闭合。计划结束、订单局部终态和进程停止本身都不能关闭动作责任。

## 5. 系统交接与复杂度

SYS 拥有运行实体、单写者和重启组合顺序；EXE 在恢复时只负责读取未闭合动作、查询原身份并返回本领域结果，不复制系统恢复流程。

EXE 保持一个动作责任模型和一条场所写路径，不建立提交尝试、保护任务、核对事项、通用重试或恢复平台。详细状态、字段和场所映射进入 L3，当前范围进入 L4。
