# 0. 非规范性摘要

本文固定十三项结论：

1. Halpha 进入 EXE 执行处理的唯一正常入口，是通过当前资金使用上限与范围、Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态检查后持久化形成的待执行动作记录；执行器不能自行提高资金使用上限、扩大范围或补造计划。
2. `拟执行动作资金与权限检查结果` 与唯一待执行动作记录应在一次本地原子提交中共同形成，避免“检查通过但没有待执行动作记录”或同一拟执行动作产生重复记录；具体存储和事务机制由 L3 规定。
3. 任何可能产生交易场所或账户变化的命令，都必须先有不可变、持久化且可去重的待执行动作记录。
4. 每个写入适用范围同时只能有一个可证明有效的 Halpha 执行器；无法证明唯一性即停止新写入。
5. 超时、断线和未知响应不等于失败；不能排除交易场所或账户变化时进入执行结果未决，先查询核对，不盲目重发。
6. 场所确认、订单状态、成交、仓位、保护和费用分别形成证据，不能用任一项推定其余项。
7. 部分成交、取消与成交同时发生、迟到结果和重复推送是正常执行情况，必须归入同一待执行动作记录的完整处理过程。
8. 保护是持续责任，不是一次下单；替换保护时先确认新保护成立，再解除旧保护。
9. 用户可以随时通过官方场所独立操作或接管；Halpha 必须识别、隔离并核对这些外部活动。
10. DAT 拥有事实语义，EXE 拥有真实动作证据和差异闭合语义；执行器与核对后台任务提交真实动作证据、推进差异闭合并读取 Halpha 按 DAT 规则形成的事实，不建立平行账户真相。
11. 宿主环境可以使用现成能力恢复进程，执行器按 EXE 规则恢复未决待执行动作记录与交易场所或账户变化责任；进程恢复不得自动重放命令或恢复写入权。
12. Halpha 只有在交易场所或账户变化已排除或全部外部责任已核对闭合、必要保护已处理且剩余订单与持仓事实已更新时，才按 EXE 规则结束待执行动作记录责任。
13. 交易所模拟盘执行可以复用交易语义和部分实现，但必须与真实写入端口、凭据、对象身份和交易场所或账户变化处理过程隔离，且不得回退到真实执行。

# 1. 职责边界与稳定执行语义

## 1.1 EXE 的唯一入口【EXE-SCP-001】

Halpha 进入 EXE 执行处理的唯一正常入口，是第 2 节本地提交成功后形成的待执行动作记录。进入提交前，不可变拟执行动作必须已经具备：

- 有明确交易计划、已有风险敞口所需的保护或风险减少决定，或用户按已接受规则提交的撤销、保护、划转或风险减少指令；
- 拟执行动作对象、方向、数量边界、期限和可能产生的交易场所或账户变化范围已经固定；
- EXE 规则可以明确所需场所能力、唯一执行适用范围、保护和核对责任。

当前 `资金使用上限与范围`、`Halpha 真实资金操作权限`、`Halpha 新增真实动作停用状态` 和 `拟执行动作资金与权限检查结果` 是建立待执行动作记录的输入。提交未成功时，Halpha 不建立执行责任；成功后，待执行动作记录必须能够引用所用资金使用上限与范围版本和检查结果。

用户或 Halpha 可以通过 CAP 定义的 Halpha 新增真实动作停用状态立即停止未来由 Halpha 发起的真实动作；执行器必须在提交前读取当前 Halpha 新增真实动作停用状态。任何需要改变场所订单、仓位、保护或余额的处置，仍必须作为独立拟执行动作经过同一资金与权限检查后进入 EXE，或由用户在外部场所直接处理。

## 1.2 稳定执行语义【EXE-OBJ-001】

### 交易场所或账户变化与稳定执行语义定义【EXE-OBJ-001-DEF】

本文所说的“交易场所或账户变化”，是指对交易场所或账户已经或可能造成的实际变化，例如订单、成交、持仓、保护或余额变化；它不是泛指有害后果。

| 稳定执行语义 | 唯一含义 |
|---|---|
| 待执行动作记录 | 对一次可能产生交易场所或账户变化的不可变拟执行内容及其处理范围 |
| 外部写入控制权 | 某个写入适用范围内当前唯一 Halpha 执行器的可验证控制权 |
| 交易场所提交记录 | 一个待执行动作记录向场所发起的一次可追溯提交尝试 |
| 交易场所结果引用 | 场所订单、成交、划转、保护或其他交易场所或账户变化的稳定外部身份及关联 |
| 交易执行核对事项 | 把待执行动作内容、提交尝试、外部身份、场所观察、DAT 事实以及保护和外部变化责任闭合的工作对象 |

风险敞口保护任务在第 8.1 节单独定义，与上述五项共同构成需要分离的六项稳定执行语义，本节不重复定义。

### 稳定执行语义保存与责任要求【EXE-OBJ-001-REQ】

待执行动作记录不得由交易计划分支、接口请求或订单状态替代；外部写入控制权不得由进程存活、页面开关或凭据存在替代；交易场所提交记录不得由重试计数或日志文本替代；交易场所结果引用不得由本地临时编号或推送顺序替代；风险敞口保护任务不得由一张止损单或“已保护”布尔值替代；交易执行核对事项不得由定时任务运行结果替代。因执行前提不足形成的限制是对当前写入范围的结论，不建立独立业务对象，也不得由无适用范围的全局维护模式替代。

上述记录、控制权与任务可以有新版本、修正和终态，但历史待执行内容、提交尝试、外部身份和处理责任不得改写。

CON-CMP-003 已定义三类控制要求。EXE 拥有待执行动作记录唯一、防重复提交、部分结果处理、真实写凭据消费、会话隔离和核对推进等执行正确性；执行器按 CAP 的金融风险边界和 ARC、SYS 的系统故障影响限制工作，但 EXE 不重新定义这些边界。

### 稳定执行语义的选择依据【EXE-OBJ-001-RAT】

待执行动作记录、提交尝试和交易场所结果引用分别证明 Halpha 已持久记录拟执行内容并承担处理责任、已经发起场所调用和已经取得外部结果；外部写入控制权、风险敞口保护任务和交易执行核对事项分别被执行器、风险敞口处理和差异闭合消费。它们是重启后防止重复交易场所或账户变化并恢复责任所需的最小分离。当前限制只是一项适用范围结论，Halpha 按 OUT 规则根据事实评价执行质量，二者不建立独立对象。

六项稳定执行语义要求各自的身份、证据和不变量可独立验证，不要求实现为六个物理记录族。L3 可以让一条待执行动作记录内嵌提交尝试、场所结果、保护和核对证据，也可以让交易计划启用记录承载当前保护的只读投影；外部写入控制权也可由满足本节不变量的宿主能力表达。上述合并不得覆盖历史提交、混合外部身份、把只读投影变成保护事实，或使重复、未决、保护、核对与恢复结果无法单独判断。

## 1.3 模拟执行不进入真实交易场所或账户变化处理过程【EXE-SIM-001】

交易所模拟盘可以复用适用的订单、成交、保护和持仓语义，但模拟动作、订单、成交、仓位和保护只属于模拟环境，不形成真实待执行动作记录、外部写入控制权、交易场所提交记录、交易场所结果引用或交易执行核对事项。

交易所模拟盘适配器不得持有真实交易凭据或调用真实写入端口。交易记录环境身份必须在选择执行适配器前已经确定；身份缺失或冲突时拒绝执行，不允许把真实执行作为模拟盘不可用时的默认或回退路径。

从模拟转入真实资金必须建立新的真实资金交易记录环境身份，并收到用户显式授予相应 Halpha 真实资金操作权限的命令，再经过 CAP 资金与权限检查、真实待执行动作记录、唯一外部写入执行器和真实核对过程。模拟订单、成交、仓位、保护和执行结果未决不得被真实执行继承。

# 2. 拟执行动作资金与权限检查结果和待执行动作记录的本地提交【EXE-ING-001】

Halpha 必须在一次本地原子提交中保存当前 `拟执行动作资金与权限检查结果` 并建立唯一待执行动作记录；具体存储和事务机制由 L3 规定。CAP 拥有资金使用限制、Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态的含义；EXE 拥有待执行动作记录与执行生命周期；SYS 只负责定义提交原子性和并发约束。该本地提交只有两类可观察结果：

1. 待执行动作记录未进入执行：没有可提交的待执行动作记录；
2. 待执行动作记录已进入执行：检查结果、资金使用上限与范围版本和唯一待执行动作记录相互引用，责任明确。

不得出现“同一拟执行动作建立多个有效待执行动作记录”或“失败后无法判断记录是否已经建立”的中间状态。提交结果未决时读取同一权威状态恢复，不盲目重试。采用模块化单体时不为此建立跨域编排对象、消息协议或分布式事务。

待执行动作记录建立后仍不等于可以提交。执行器领取后、越过交易场所或账户变化边界前，必须刷新关键账户、订单和仓位事实，并使用同一资金与权限规则复核资金使用上限与范围、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态、待执行动作记录的时效和执行唯一性；后一次可以否决前一次。该复核是防止陈旧状态的执行正确性措施，不是独立的最终资金检查步骤或生产准入流程。领取身份、限时执行权、执行器版本序号、过期、接管和旧实例写入阻断由 EXE/SYS L3 按实际并发需要定义。

适用的 Halpha 新增真实动作停用状态、Halpha 真实资金操作权限降低或资金使用范围收缩决定，与执行器越过交易场所或账户变化分界之间必须具有可验证先后。前述收缩决定先成立时，执行器不得越过分界；待执行动作已经先越过分界时按可能已发生处理；无法证明先后时，保持受影响范围停用、禁止重发，并进入查询、核对或用户接管。

# 3. 待执行动作记录与提交边界

## 3.1 先持久化，后交易场所或账户变化【EXE-INT-001】

每个待执行动作记录必须让执行器和核对任务能够确定：不可变拟执行动作及其来源、按 CAP 规则形成的资金与权限检查结果、预期产生的交易场所或账户变化、允许的时间与写入范围、部分结果与保护边界、唯一身份及失败处理。只保存会改变提交、拒绝、防重复、保护或恢复行为的信息，具体字段进入 L3；执行前仍按第 2 节刷新事实和资金与权限状态。

修改、取消、替换保护、划转或其他可能改变场所状态的命令，各自需要独立待执行动作记录，并通过因果关系连接原动作。不得修改原记录后把新命令伪装为同一次提交。

在待执行动作记录持久化、唯一外部写入执行器成立、提交前资金与权限复核通过且场所命令已经确定之前，不得发起任何可能产生真实交易场所或账户变化的调用。日志、内存任务、队列消息或界面状态不能作为唯一记录凭据。

## 3.2 必须识别交易场所或账户变化分界【EXE-BND-001】

### 交易场所或账户变化分界定义【EXE-BND-001-DEF】

交易场所或账户变化分界，是 Halpha 已经保存该次待执行动作记录，但交易场所或账户的不可回滚实际变化尚未开始或可能已经开始的协议边界。越过该分界后，本地回滚不能证明外部没有变化。

### 交易场所或账户变化分界识别要求【EXE-BND-001-REQ】

每种场所命令必须明确：

- 在哪个边界前可以证明尚未产生交易场所或账户变化；
- 哪些响应只能证明请求被接收，不能证明订单、成交或保护结果；
- 超时、连接中断、客户端取消和进程崩溃各自可能落在边界哪一侧；
- 使用什么稳定外部身份查询，而不是重新提交；
- 何时只能维持执行结果未决并进入用户接管。

如果协议或适配器无法可靠区分交易场所或账户变化分界，该能力不得用于自动新增或转换风险；可以缩小到只读、交易计划当前确认、更低自主性或完全不支持。

# 4. 唯一外部写入执行器【EXE-OWN-001】

## 4.1 唯一外部写入执行器定义【EXE-OWN-001-DEF】

唯一外部写入执行器，是在一个可能相互影响真实订单或场所防重复提交能力的写入范围内，当前唯一获准持有外部写入控制权并发出场所命令的 Halpha 运行实体。

## 4.2 执行唯一性要求【EXE-OWN-001-REQ】

每个写入范围同时只能有一个有效 Halpha 执行器。该范围至少覆盖共享凭据、账户、子账户、场所会话和订单身份的实际传播边界，不能只按进程或服务实例划分。

单宿主实现可以使用操作系统互斥机制表达当前外部写入控制权，无须为此建立专属记录族；前提是其范围覆盖实际写入传播边界，陈旧实例无法绕过，重启和异常退出后的状态能够保守判定，并且无法确认唯一性时仍按本节停止写入和核对。仅有进程名、端口占用或“通常单实例”不足以证明该不变量。

唯一性必须能抵御并发启动、网络分区、陈旧实例、限时执行权更新延迟、版本切换和恢复接管。旧执行器在失去控制权后即使仍运行，也不能继续发出场所命令。

无法证明唯一性时：

- 停止受影响适用范围的新写入和风险转换；
- 保留可信只读查询、证据保存、保护状态检查和核对准备；
- 把所有可能仍有效的待执行动作记录视为未决，不由新实例重发；
- 扩大核对范围直到能够排除并行交易场所或账户变化；
- 必要时撤销或隔离 Halpha 写入凭据并转用户接管。

执行器唯一性不阻止用户在官方场所独立操作；后者作为外部活动进入第 9 节的识别和核对路径。

# 5. 场所适配与身份【EXE-VEN-001】

## 5.1 交易场所对象对应关系定义【EXE-VEN-001-DEF】

交易场所对象对应关系是 Halpha 内部待执行动作与场所客户端、订单、成交、保护及其他外部对象之间的稳定对应关系。

## 5.2 场所适配与身份要求【EXE-VEN-001-REQ】

每个受支持场所和动作类型都必须把 Halpha 的业务含义明确映射到场所的实际命令、精度、最小量、时效、防重复提交能力、响应、查询、推送、部分结果、取消与成交冲突、保护和费用含义。

执行能力必须通过真实场所行为验证，不能从开发包类型、示例成功或统一接口外观推定。场所不支持或语义不等价的能力必须显式拒绝、收缩或采用经过验证的替代路径，不得静默模拟为“已支持”。

内部待执行动作记录身份一经形成保持稳定；场所客户端身份、订单身份、成交身份和保护身份分别保存并建立关系。外部身份缺失、冲突或复用时进入执行结果未决，不用时间接近、数量相似或推送顺序强行匹配。

适配器只转换协议和场所含义，不拥有 Halpha 真实资金操作权限、计划分支、可引用事实判断或执行责任的最终状态。

---

# 6. 防重复提交、重试与执行结果未决

## 6.1 防重复提交绑定待执行动作记录【EXE-IDM-001】

同一待执行动作记录无论经历调度重复、进程重启、消息重投、网络重连或场所重复响应，都只能产生一次被允许的交易场所或账户变化。

优先使用场所原生防重复提交标识，并由 Halpha 的稳定待执行动作记录身份派生或绑定。场所不支持可靠防重复提交时，执行器必须使用“查询既有外部身份或场所事实后再决定”的路径；无法排除已经提交时不得重发。

重试策略按失败语义决定，不按异常类型统一重试。读取重试、查询重试和场所写入重试必须分开；任何通用网络重试、开发包自动重试或任务框架重放都不得包裹未证明防重复提交的写入。

## 6.2 执行结果未决【EXE-UNK-001】

### 执行结果未决定义【EXE-UNK-001-DEF】

执行结果未决是待执行动作已经或可能越过交易场所或账户变化分界，但当前无法确认其是否产生以及产生何种交易场所或账户变化的执行状态。

### 未决进入与处理要求【EXE-UNK-001-REQ】

以下情况必须进入或保持执行结果未决：

- 请求可能已经离开 Halpha，但没有可靠接受或拒绝结论；
- 场所响应、推送、查询和账户事实互相冲突；
- 外部身份缺失、重复或无法唯一匹配；
- 取消、修改与成交并发，最终外部状态尚未稳定；
- 数据、时间、凭据或唯一外部写入执行器可信性不足；
- 恢复后无法证明旧实例是否仍可能写入。

执行结果未决期间不得以“通常失败”“金额很小”“稍后会一致”或人工重复点击推断无交易场所或账户变化。未决待执行动作记录按可能已发生计入账户风险敞口，新自动写入停止或收缩，查询、保护、记录保存和用户接管保持可用。

# 7. 接受、部分结果、取消与迟到结果【EXE-PRT-001】

场所接受请求只表示场所承认某个外部对象，不表示成交、目标数量完成、保护已建立或责任已闭合。执行器和核对后台任务必须按 EXE 规则分别处理：明确拒绝、等待、部分成交、完全成交、部分取消、取消失败、过期、迟到成交、费用修正和最终仍未决。

部分结果发生后：

- 立即按已确认交易场所或账户变化更新尚未闭合的风险敞口、剩余数量和账户事实，不等待待执行动作记录处理完成；
- 只按交易计划已定义分支形成后续拟执行动作；没有预定义分支时停止新增动作并请求决定；
- 需要取消剩余量、补充保护或处置超额时，分别形成新的拟执行动作并经过 CAP；
- 不把“期望数量减已知成交”直接当作仍可安全提交数量，必须考虑在途、迟到和未知结果。

取消请求与成交可能同时成立。取消成功只能按场所确切含义解释，不能抹去取消前或迟到产生的成交。修改若在场所实际表现为取消加新建，必须保留两个外部责任以及两者同时发生时的处理结果。

# 8. 风险敞口保护任务【EXE-PRT-002】

## 8.1 风险敞口保护任务定义【EXE-PRT-002-DEF】

风险敞口保护任务是风险敞口开始后，用于持续记录退出、止损、取消和异常收缩所需保护及其建立、验证、维持、替换和移交状态的工作对象。执行器或核对后台任务按该对象记录的当前责任推进保护。

## 8.2 保护落实要求【EXE-PRT-002-REQ】

TRADEPLAN 定义计划中的保护要求，CAP 负责资金使用限制、Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态的检查规则；执行器必须把允许的保护动作转换为真实场所命令，Halpha 持续核对其外部状态。

新增或扩大风险敞口前，必须证明适用保护能够在要求的时间和失败场景下建立。优先使用场所原生、在 Halpha 离线后仍有效且已验证语义的保护；Halpha 仍需独立核对，不能因“场所托管”而停止责任。

风险敞口保护任务至少覆盖：

- 首次风险敞口、部分成交和增量风险敞口后的保护数量与适用范围；
- 保护被拒绝、过期、触发、取消、修改、部分执行或场所规则变化；
- 保护与仓位方向、数量、账户、工具和其他保护之间的冲突；
- Halpha 断线、重启、失去写权或用户外部操作后的持续有效性；
- 保护替换期间的空窗和双重保护风险。

替换保护时，旧保护在新保护已经由场所确认并经事实核对前保持有效；若两者并存可能造成更大风险，则必须采用经 L3 验证的有界替换协议或不支持该能力。不得先取消旧保护，再依赖后续调用一定成功。

保护缺失、冲突或状态不明时，执行器停止受影响新增或转换风险，保留可靠既有保护，并在相关风险敞口保护任务或交易执行核对事项中记录禁止或收缩的写入范围及解除条件；待检查的风险减少拟执行动作仍须由 Halpha 按 CAP 规则检查。执行器不得因“保护更安全”自行获得动作权限。

# 9. 外部人工活动与用户接管【EXE-EXT-001】

用户始终可以通过交易所、托管方或其他官方路径独立观察账户并处置资金、订单和持仓。Halpha 不得成为撤权、撤单、保护或退出的唯一入口。

观察到非 Halpha 执行器发起的订单、成交、仓位、保护、划转或权限变化时，必须保留原始来源并区分：

- 与已知计划和人工交接明确匹配的计划内活动；
- 用户已知但不属于当前计划的外部活动；
- 来源或意图无法确认的未知活动。

相似的价格、数量和时间不足以自动归入计划。未知或冲突活动不得由 Halpha 自动反向交易、删除或吸收到现有计划或待执行动作记录；先停止可能冲突的新写入，再由执行器或核对后台任务提交观察，Halpha 按 DAT 规则形成事实、按 CAP 规则重算已使用资金，最后由用户或 Halpha 按 TRADEPLAN/POR 规则决定是否需要新计划或配置。

进入用户接管时，Halpha 停止受影响写入、让出或隔离外部写入控制权、提供未决待执行动作记录和已知外部责任的只读材料，并继续保存可获得的证据。用户无需等待 Halpha 交接成功即可通过官方路径处置紧急风险。

---

# 10. 执行核对

## 10.1 交易执行核对推进【EXE-REC-001】

交易执行核对的目标是把待执行动作记录、提交尝试、外部身份、场所观察和 DAT 事实推进为可确认关系，并明确闭合差异或移交未决责任。

### 核对推进要求【EXE-REC-001-REQ】

Halpha 必须按 EXE 规则推进以下关系闭合，并按 DAT 规则形成其中的可引用事实判断：

1. 待执行动作记录与交易场所提交记录是否一一对应且没有重复交易场所或账户变化；
2. 交易场所提交记录与场所订单、成交、取消、保护和费用是否建立稳定身份关系；
3. 场所动作与账户、仓位、余额、保护和资金占用是否一致；
4. 实际结果与交易计划分支、资金使用上限与范围、拟执行动作数量和预期外部效果是否一致；
5. 外部人工活动、迟到结果、修正和未知对象是否已归属或隔离。

执行器或核对后台任务向 Halpha 的事实记录入口提交带来源、原始身份和时间的真实动作证据与场所观察，并读取按 DAT 规则形成的事实、修正和事实未知状态结论。EXE 负责定义核对差异和处理进度的语义，但 Halpha 不得维护一套与 DAT 并行的订单、仓位或余额真相。

## 10.2 必须触发核对的时点【EXE-TRG-001】

核对至少在以下情况启动或扩大：

- 每次可能产生交易场所或账户变化的提交、响应、推送、查询和超时之后；
- 部分成交、取消与成交冲突、保护变化、费用或场所修正之后；
- 用户外部活动、入出金、划转或未知对象出现时；
- 执行器切换、服务重启、断线恢复、凭据变化、版本发布和备份恢复后；
- 当前事实或质量结论表明所需数据缺失、不完整、冲突、过期、映射变化或账户与资金事实存在差异时；
- 计划结束、已使用资金重算、用户接管或控制返回前。

定期核对用于发现遗漏，不能替代关键边界上的即时核对。高频轮询也不能替代稳定身份、来源追溯关系和执行结果未决处理。

## 10.3 差异只能显式闭合【EXE-DIF-001】

每个差异必须以以下一种方式闭合：

- 经权威来源确认 Halpha 记录正确；
- Halpha 按 DAT 规则形成事实判断修正，核对任务按 EXE 规则推进责任；
- 识别为外部人工活动并完成归属；
- 识别为场所修正、迟到结果或映射变化并保留来源追溯关系；
- 无法安全解释，保持执行结果未决，在相关交易执行核对事项中记录当前禁止或收缩的写入范围和明确责任；
- 明确移交用户接管。

不得使用自动余额调整、删除未知订单、覆盖历史事件、强行匹配或“以最新快照为准”静默消除差异。

# 11. 失败时停止或收缩写入【EXE-FAL-001】

| 失败或不确定性 | 新写入 | 必须继续的责任 |
|---|---|---|
| 提交前资金与权限复核拒绝，且证明未越过交易场所或账户变化边界 | 不提交并关闭待执行动作记录 | 保存拒绝原因，并由 Halpha 按 DAT 规则更新账户与资金事实 |
| 提交前本地失败，且能证明无交易场所或账户变化 | 不重放过时动作 | 关闭待执行动作记录或将其标记过期，记录原因 |
| 可能越过交易场所或账户变化边界 | 禁止重发及冲突真实动作 | 查询、保护、按可能已发生计入暴露、核对和接管准备 |
| 无法证明唯一外部写入执行器成立 | 停止受影响场所写入 | 扩大执行结果未决的适用范围、隔离旧写权、只读核对 |
| DAT 事实缺失、过期或冲突 | 不新增或转换风险 | 保留既有可靠保护，获取事实并核对 |
| 场所写入或查询不可用 | 不盲目切换写入提供方 | 保留场所原生保护，按适用范围阻断并允许用户接管 |
| 保护缺失或状态不明 | 不扩大受影响的风险敞口 | 核对现有保护，提交经 Halpha 按 CAP 规则检查通过的风险减少拟执行动作 |
| 差异长期无法闭合 | 保持当前禁止或收缩的写入范围，不以超时自动清除 | 保留记录、尚未闭合的风险敞口和明确责任，转人工决定 |

禁止或收缩写入的结论只能减少执行器的写入范围，不能授予动作。其原因、证据、影响范围、禁止内容、仍允许的只读和责任处理、解除条件必须保存在相关外部写入控制权、风险敞口保护任务或交易执行核对事项中，不建立独立对象。

多个禁止或收缩写入的结论同时适用时采用更严格结果。无法证明影响边界时，扩大到覆盖全部合理传播路径；不得用一个无适用范围的全局开关替代事实记录和恢复条件。

# 12. 动作责任恢复与控制返回

## 12.1 现成设施恢复进程，执行器恢复执行责任【EXE-RCV-001】

SYS 规定运行实体、宿主与进程、运行时配置、秘密注入和恢复前置边界，ENG 规定可重复构建、发布、回退与部署说明；Halpha 按 EXE、SYS 与 DAT 规则处理重启、防重复、未决责任和核对恢复，并直接复用现成宿主与数据库能力。

Halpha 在设施可用后按 EXE 规则依次恢复以下责任：

1. 确认旧实例、旧会话和凭据写权已隔离，作废旧外部写入控制权并保持业务写入阻断；该事实可直接来自当前部署、SYS 边界或场所，不要求独立运行对象；
2. 重新建立可验证的唯一外部写入控制权；
3. 载入全部待执行动作记录、交易场所提交记录、风险敞口保护任务和交易执行核对事项，包括其中当前禁止或收缩的写入范围；
4. 从场所取得只读观察并交由 Halpha 按 DAT 规则形成当前事实与事实未知状态；
5. 对每个未决待执行动作记录区分“确定未提交”“可能有交易场所或账户变化”和“已确认有交易场所或账户变化”；
6. 验证既有保护，识别保护缺口和外部人工活动；
7. 闭合或明确移交全部旧责任；
8. 只有相关领域重新给出当前结论，并且新的拟执行动作重新经过 CAP 时，才接受新的风险增加或转换动作。

恢复对象是责任和事实，不是旧命令队列。失去时效的拟执行动作不补执行；可能已提交的拟执行动作先查后判；备份中的旧执行权、旧交易计划当前确认、旧会话以及已经过期、撤销或无法复核的机器授权和权限不得恢复。重启前已经持久化且仍在原计划、账户、动作、期限和额度作用域内的机器授权，可以在 CAP 重新检查当前授权、资金与权限边界，DAT 提供当前事实，并且 EXE 重新证明写入唯一性后继续生效；重启本身不得创建、扩大或延长授权，也不得使失效授权复活。

## 12.2 控制返回【EXE-TKO-001】

用户接管后，Halpha 重新取得外部写入控制权前必须确认：

- Halpha 与外部控制不会并行争夺执行权；
- 接管期间的订单、成交、仓位、保护、余额和划转已经由 Halpha 按 DAT 规则形成事实；
- 旧待执行动作记录、外部身份、执行结果未决和尚未闭合的风险敞口已经闭合或继续由明确责任方承担；
- 当前保护和退出能力明确；
- 所有新增或转换风险的动作重新形成拟执行动作并经过当前资金与权限检查。

控制返回建立新的执行责任边界，不继承接管前未使用的提交许可，也不自动恢复交易计划启用或 Halpha 真实资金操作权限。

# 13. 动作责任闭合与结果材料【EXE-END-001】

待执行动作记录只有在以下一种情形成立时才能结束责任：

- 可证明从未越过交易场所或账户变化边界，并已明确拒绝、过期或取消；
- 外部订单、成交、费用、仓位、保护和相关人工活动已经取得并核对，全部差异闭合；
- 未闭合外部责任已经连同记录、适用范围、尚未闭合的风险敞口和控制权明确移交用户。

责任闭合前，未决待执行动作记录必须继续按可能已发生计入账户风险敞口，不得因本地任务完成、计划结束、订单显示终态或时间经过而忽略。执行器或核对后台任务向 Halpha 的事实记录入口提交闭合结论；Halpha 按 DAT 规则更新账户与资金事实，后续拟执行动作再按 CAP 规则完成资金与权限检查。

Halpha 按 EXE 规则输出两类事实材料：

- 执行器或核对后台任务向 Halpha 的事实记录入口提交待执行动作记录、场所尝试、响应、查询、推送、外部身份、对应关系、差异和修正输入，用于按 DAT 规则形成可引用动作事实；
- Halpha 保存预期与实际路径、事实截止点、时延、拒绝、滑点、部分结果、费用、保护、人工活动、执行结果未决、偏离、核对和恢复材料，并按 OUT 规则在复盘记录中评价执行质量。

EXE 只负责执行事实的范围，不定义本领域评价对象；Halpha 不根据单次成交或执行表现提高策略证据，不在执行过程中评价计划、组合或产品价值，也不改写事实和资金使用决定。

# 14. 明确的输入输出【EXE-HOF-001】

| 方向 | 内容 |
|---|---|
| 接收 | CAP 的资金使用上限与范围、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态和拟执行动作资金与权限检查结果；TRADEPLAN 的不可变计划来源，以及例外路径的固定决定或指令来源、时效、分支和保护要求；DAT 的当前事实、交易记录环境身份、修正、外部活动与事实未知状态；需要时读取当前部署和凭据可用事实；ENG 的已测试构建与发布记录；UX 的明确用户操作指令与用户交互证据，但不把它们视为场所动作或外部结果 |
| 输出 | 执行器或核对后台任务提交给 Halpha 事实记录入口的真实动作证据、场所观察和差异输入；供 Halpha 按 CAP 规则检查的执行前提、未决交易场所或账户变化、保护、限制和责任闭合；供 Halpha 按 TRADEPLAN 规则更新计划的接受、拒绝、部分结果、执行结果未决和结束结论；供 Halpha 按 UX 规则呈现的未决责任、保护、限制和接管材料；供 Halpha 按 OUT 规则形成复盘记录的、带事实截止点的执行事实材料 |

每项领域 L3 必须声明一个主要语义 L2 所有者。场所适配、保护、核对和恢复领域 L3 以 EXE 为主要语义所有者时，只引用 DAT、CAP、ENG、TRADEPLAN 及必要部署边界，不重定义其规则。本地事务交接属于相关 EXE L3 与 SYS 事务约束，不另建跨领域编排产品。

# 15. L3 必须精确定义【EXE-L3-001】

L3 按实际支持的场所和动作类型定义：

- 待执行动作记录、外部写入控制权、交易场所提交、交易场所结果、风险敞口保护和交易执行核对的数据表达、身份、状态转换与不变量，以及现有对象如何表达当前禁止或收缩的写入范围；允许由待执行动作记录、交易计划启用记录内嵌结构或满足要求的宿主互斥能力承载，但必须保留各语义的独立证据与可验证结果；
- 模拟与真实执行的交易记录环境身份、适配器和凭据隔离，禁止回退到真实写入以及模拟状态转入真实责任链的约束；
- EXE 侧待执行动作记录的接收、唯一性和生命周期契约，以及拟执行动作资金与权限检查结果与待执行动作记录建立的本地事务；提交前资金与权限复核与失败闭合；
- 执行器领取、排他、失效、隔离、版本切换和旧实例阻断契约；
- 每个场所的命令转换、精度、时效、外部身份、防重复提交、响应、查询、推送、限频、错误和交易场所或账户变化分界；
- 写入重试、查询重试、重复、乱序、迟到、部分结果、取消与成交冲突、修改和执行结果未决处理；
- 保护建立、验证、替换、触发、缺口、冲突和场所离线语义；
- 外部人工活动识别、归属、隔离、核对、用户接管和控制返回；
- 与 DAT 的事实输入、修正、差异和来源追溯关系契约；
- 供 Halpha 按 OUT 规则形成复盘记录的执行事实范围、事实截止点、补充和修正交接；
- 重启、备份恢复、网络分区、执行器竞争和未决责任重建；
- 正常、拒绝、重复、乱序、超时、崩溃、部分、未知、恢复和用户接管场景的验证证据。

L3 可以选择技术实现，但不得删除待执行动作记录、提交前资金与权限复核、唯一外部写入执行器、执行结果未决、保护、核对和用户接管中的任何适用责任；这些责任不要求分别建立专属记录族。

# 16. L4 选择边界【EXE-L4-001】

L4 只记录当前建设选择的场所/账户/动作/订单/保护范围、适配与外部变化分界、执行器与凭据隔离边界、参数和验收证据。具体待执行动作、未决结果、保护缺口、核对差异、控制范围与接管责任属于 EXE、DAT 等拥有领域的运行记录，不写入 L4；L4 也不得从历史成功或小额交易反向推定能力已经合格。
