# Halpha 执行、保护与核对原则

**文档编号：** HALPHA-EXE-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** Halpha 如何把一个已检查的拟执行动作变成唯一可追踪的场所动作责任，以及如何防重复、处理未知、维持保护、核对结果和交还用户控制  
**本文档不负责：** 策略、计划条件、激活资金限制、账户或场所事实的定义、页面、具体字段与状态枚举、场所命令映射、当前组件、进程、凭据或配置  

---

# 0. 设计结论与复杂度上限【EXE-SCP-001】

EXE 只用一个稳定业务对象：`ExecutionAction`。它表示 Halpha 对一次可能改变交易场所或账户的动作所承担的唯一处理责任，从本地建立、场所提交、查询与事实核对一直持续到明确闭合或交给用户。

提交尝试、外部身份、部分成交、保护和核对证据都是该动作及其引用事实的内容，不另建通用提交记录、保护任务、核对事项、命令回执或恢复对象。只有实际场所能力无法用这一结构安全表达时，才在 L3 增加最小实现细节；不得先把技术步骤提升为稳定产品概念。

# 1. `ExecutionAction`【EXE-ACT-001】

一个 `ProposedAction` 最多建立一个有效 `ExecutionAction`。EXE 是它的唯一写入所有者；计划、页面、任务框架、日志、供应商订单状态和进程存活都不能替代或推进它。

`ExecutionAction` 固定所属环境、账户、激活、拟执行内容、允许时限、稳定动作身份和必要保护责任。原始动作意图与外部身份不得被后续结果覆盖；修改、撤单、补保护、减仓和平仓等新的场所动作各自建立新的 `ExecutionAction`，并引用原因动作。

# 2. 建立与提交【EXE-INT-001】

Halpha 在一个本地事务中保存当前 CAP 检查结果并建立唯一 `ExecutionAction`。事务只有全部成立或全部不成立两种结果；失败或调用方不知道结果时，先按同一稳定身份查询本地权威状态，不重复创建。

任何场所写入都必须发生在 `ExecutionAction` 已持久化之后。执行器真正调用场所前，重新读取当前激活、资金限制、停止结果、账户事实和开放责任，并运行同一 CAP 检查。检查不满足或为未知时，原动作记录拒绝或未知结果，不改换动作、不新建替代动作，也不把旧检查当成当前许可。

# 3. 单一写者与场所身份【EXE-WRT-001】

在共享账户、凭据、场所会话和客户端身份的实际传播范围内，同一时刻只允许一个 Halpha 运行实体发出场所写入。无法证明单写者时停止该范围的新写入；只读查询、事实保存、既有可靠保护和用户接管仍可继续。

只有这个写者可以取得真实写凭据。交互进程、普通后台工作、只读路径、研究和 Demo 运行不得取得真实写能力。具体互斥、领取、失效和旧实例阻断方式由 L3/SYS 定义，不要求独立业务记录。

每个场所动作使用能稳定映射回 `ExecutionAction` 的客户端身份。场所订单、成交、费用和保护各自保留真实外部身份；价格、数量或时间接近不能替代身份核对。

# 4. 结果、重复与未知【EXE-OUT-001】

场所调用只能得到三类业务结论：能够证明未产生外部变化；能够由外部身份和事实确认实际结果；或者结果未知。网络异常、进程崩溃、超时、取消与成交竞争、部分响应或身份冲突都不能被猜成失败或成功。

对可能已经提交的动作，先用原客户端身份查询场所并按 DAT 事实核对。只有场所契约和当前事实能够证明原调用没有产生变化，且动作仍有效并再次通过 CAP 检查时，才允许按同一动作身份继续；通用网络重试和任务重放不得包裹场所写入。

结果未知时，该动作按可能已发生计入风险，停止可能冲突的新风险，不创建第二动作猜测原结果。查询、核对、既有保护、退出和用户接管保持可用。时间经过或金额较小不会自动关闭未知。

部分成交、迟到成交、费用修正、取消失败和场所修正都更新同一动作的事实关系。需要额外撤单、保护或退出时，形成新的明确动作并再次经过 CAP，而不是修改原动作或把剩余量静默重发。

# 5. 保护、核对与用户接管【EXE-PRT-001】

风险出现后，保护是持续责任。Halpha 必须从场所事实确认保护的工具、方向、数量和有效状态，不能用“已提交”或一个布尔值推定保护已经成立。替换保护时，除非经验证的场所操作能够原子替换，否则先确认新保护有效再解除旧保护；无法安全替换时不支持该路径。

保护缺失、冲突或未知时，停止受影响新增风险，保留可信既有保护，并优先执行经 CAP 证明不会增加风险的撤单、减仓或退出。Halpha 不能因为动作意图更安全而自行获得权限。

EXE 把动作意图、场所调用、外部身份和 DAT 事实核对到一致结果。以下情况至少需要立即核对：场所响应不完整或未知；部分成交、取消、保护或费用变化；断线、进程重启、写者变化；用户在官方场所操作；激活准备结束。差异只有被权威事实解释、明确保留为未知或交给用户时才算处理，不能靠本地状态覆盖。

用户可随时通过场所官方入口观察或处置。用户接管后，Halpha 停止该激活全部新场所写并继续提供只读事实和未闭合责任；用户无需等待 Halpha 完成交接。接管结束不恢复旧激活的新增风险能力，责任闭合后如需交易由用户创建新激活。

# 6. 重启与责任闭合【EXE-RCV-001】

重启恢复的是已持久动作责任和当前事实，不是旧命令队列。已经过时的动作不补发，可能已提交的动作先查询，备份中的旧写入权、旧会话和过期权限不恢复。

纯系统重启只有在用户未停止、退出或接管，原激活仍有效，唯一写者、持久状态连续、产品构建/配置/凭据/账户身份未改变，所有未决动作、订单、成交、仓位和保护已经核对，且不会补发停机期间错过动作时，才可在原激活范围继续。任一条件无法证明时，该激活不可逆停止新增风险；查询、既有保护、退出和用户接管继续。

一个 `ExecutionAction` 只有在以下任一结果明确时闭合：能够证明没有发生场所变化；预期及实际外部结果、费用和保护均已核对；或剩余订单、持仓、保护、未知和处置责任已经清楚交给用户。计划结束、本地任务结束、订单显示终态或复盘建立本身都不能替代责任闭合。

# 7. 环境隔离与下位边界【EXE-ENV-001】

Demo 与 Live 使用同一 `ExecutionAction`、提交、防重复、保护和核对实现，但环境、账户、凭据、存储、激活、动作和场所端点完全隔离。Demo 不得持有或回退到 Live 凭据和端点；模拟动作、结果和未知不能迁移或提升为真实资金记录。

L3 只定义当前实现确实需要的内容【EXE-L3-001】：

- `ExecutionAction` 的最小字段、状态转换、唯一约束和本地事务；
- 当前场所动作、客户端身份、查询、部分结果、保护、限频和错误映射；
- 单写者、重启、事实核对、责任闭合与关键反例；
- Demo/Live 的代码复用和身份隔离。

L4 只记录当前场所、账户、动作范围、组件版本、参数、进程、凭据注入和直接验证结果【EXE-L4-001】。没有当前消费者时，不增加动作类型、场所、执行进程、记录族或恢复流程。
