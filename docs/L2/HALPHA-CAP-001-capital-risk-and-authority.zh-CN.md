# 0. 定位与复杂度上限【CAP-SCP-001】

## 0.1 CAP 定位定义【CAP-SCP-001-DEF】

CAP 是拥有 L0 金融风险控制在 Halpha 内的落实规则，以及 Halpha 真实资金操作权限语义的纵向领域。它负责资金使用上限与范围的记录与执行规则、策略激活额度配额、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态和拟执行动作资金与权限检查结果；资金使用上限与范围本身的概念含义仍由 CON 拥有。

CAP 定义五个问题的判断语义：Halpha 当前最多可以动用多少资金并用于哪些范围；一次策略激活独占哪些额度；Halpha 是否可以向真实账户写入以及是否需要用户逐次确认；Halpha 新增真实动作是否已停用；拟执行动作是否越过这些边界。

## 0.2 CAP 责任与复杂度要求【CAP-SCP-001-REQ】

用户在 Halpha 外部决定向项目提供哪些账户和多少资金；CAP 不拥有该个人资金安排的决定、验证或保证责任，只定义 Halpha 如何记录并执行用户输入的系统内边界。

CAP 不得成为比 TRADEPLAN、EXE、DAT、UX 或稳定性能力更复杂的子系统。

CAP 不建设风险总分、完整情景库、机构级审批或形式化证明，也不把系统外人工资金安排包装为产品能力。

# 1. 负责的稳定语义【CAP-OBJ-001】

## 1.1 CAP 负责的稳定语义定义【CAP-OBJ-001-DEF】

CAP 负责以下五项语义在 Halpha 内的记录、检查和停用规则；其中资金使用上限与范围的概念含义仍由 CON 定义：

`Halpha 真实资金操作权限` 在第 3.1 节单独定义，本节只定义其余四项并说明五项之间的关系。

| 稳定语义 | 含义 |
|---|---|
| `资金使用上限与范围的记录与执行规则` | Halpha 如何记录 CON 定义的资金使用上限与范围、识别其适用版本，并在拟执行动作检查和提交前复核中执行 |
| `策略激活额度配额` | 用户在一次策略激活前分配的最大保证金、最大名义仓位和最大允许损失三轴互斥额度，以及该激活结束时的释放边界 |
| `Halpha 新增真实动作停用状态` | 当前适用范围按新增资金动作、保护动作或全部 Halpha 动作是否禁止发起新的真实动作，以及由谁、因何停用 |
| `拟执行动作资金与权限检查结果` | 某个拟执行动作在当前事实下是否超出资金使用上限与范围、策略激活额度配额、Halpha 真实资金操作权限或 Halpha 新增真实动作停用状态 |

## 1.2 CAP 稳定语义责任边界【CAP-OBJ-001-REQ】

CON-CMP-003 已定义三类控制要求。CAP 只拥有金融风险控制中资金使用上限与范围的系统内落实、策略激活额度配额和动作前检查，以及 Halpha 真实资金操作权限和 Halpha 新增真实动作停用状态；状态呈现由 UX 负责，停用与真实动作正确性由 EXE、DAT 负责，系统故障影响限制由 ARC 与 SYS 负责，工程验证由 ENG 负责。

Halpha 真实资金操作权限不描述交易记录所属环境、资金规模或用户在官方场所的独立操作。历史研究、历史行情回放、交易所模拟盘和真实资金记录由 DAT 的交易记录环境身份区分；真实验证的资金范围由 CAP 的资金使用上限与范围表达；用户外部操作由 EXE 作为外部活动接入。

除用户为已经启用的策略分配互斥策略激活额度配额外，CAP 不为未来真实动作预先占用资金，不建立独立资金许可、通用风险分数、完整授权图、资金风险事故对象或生产准入对象。并发占用、订单冻结资金和部分成交后的可用余额属于 EXE、DAT 与场所适配需要共同正确处理的交易事实，不上升为独立资金风险管理平台。

## 1.3 五项稳定语义的选择依据【CAP-OBJ-001-RAT】

资金使用上限与范围的记录与执行规则决定 Halpha 怎样落实 CON 定义的账户、规模和简单限额；策略激活额度配额隔离多个同时启用策略对用户已分配额度的使用；Halpha 真实资金操作权限决定 Halpha 是否可以写入以及是否需要用户逐次确认；Halpha 新增真实动作停用状态允许在不改写长期配置时按动作类别立即阻断适用范围；拟执行动作资金与权限检查结果把一次拟执行动作与前述边界当时有效的版本和事实联系起来，并由执行器按 EXE 规则消费。五者的变化频率、消费者和责任终点不同；未经动作类别标注的页面设置或总开关会混合这些含义。每项语义的保留依据是它仍会改变额度隔离、资金检查、写入权限、停用或执行责任，前提消失时分离价值也随之消失。配额之外的资金预留、风险总分和独立许可没有独立高价值消费者且会复制 EXE/DAT 事实，故不创建。

# 2. 用户设定的资金使用上限与范围【CAP-ENV-001】

## 2.1 资金使用上限与范围继承【CAP-ENV-001-REQ】

`资金使用上限与范围` 继承 `CON-CAP-001-DEF` 的含义。CAP 只定义 Halpha 记录和执行该系统内边界的要求，不重新定义用户个人总投入或最大损失保证。

## 2.2 配置与使用要求

它必须使 Halpha 能够判断当前允许使用的账户或资金池、最高资金规模、适用策略或计划范围、用户实际配置的简单限额及其是否生效。只有会改变动作检查、停止呈现或用户复核的信息才保存；具体字段由下位设计根据这些消费者确定。

资金使用上限与范围不需要覆盖全部市场、流动性、相关性、尾部、对手方、托管、网络和操作风险类别。只有已观察到、与当前交易方式直接相关且能形成真实动作限制的项目才应加入。

资金使用上限与范围只限制 Halpha 的主动使用规模，不保证最大损失。账户与资金隔离、场所权限、资金留存和实际投入规模由用户在 Halpha 外部处理；CAP 不建设或验证这些机制。

## 2.3 用户在 Halpha 外投入的交易资本、Halpha 全局上限与单笔上限【CAP-ENV-002】

三种额度定义如下：

1. **用户在 Halpha 外投入的资金控制**是用户在 Halpha 外部决定的实际入金、出金、账户余额、资金留存和账户隔离。它不是 Halpha 的上限配置，也不是 Halpha 可以保证的损失范围；CAP 只读取 DAT 提供的当前可用余额、保证金和费用事实。
2. **Halpha 全局资金上限**是一个资金使用上限与范围版本内的系统总边界，约束 Halpha 已确认敞口、开放订单、未决动作保守占用和拟执行动作后的保守敞口之和。
3. **单笔资金上限**是单个不可变拟执行动作的金额上限。它在场所量化价格、数量和费用后仍须满足，不能通过拆分、并发或部分成交绕过。

### 三种额度使用要求【CAP-ENV-002-REQ】

Halpha 必须把上述三种额度分开记录和判断。全局上限和单笔上限都只在用户明确配置的环境、场所、账户、币种、工具、动作和计划范围内生效。提高任一上限或扩大适用范围都必须形成新的用户版本；外部余额变化不会自动提高 Halpha 上限。具体字段、累计口径和生效表达由 L3 定义；当前值属于 CAP 与 DAT 的权威运行记录，不写入 L4。

### 三种额度分离依据【CAP-ENV-002-RAT】

外部实际余额由用户和 DAT 负责，全局上限由 CAP 约束一个资金使用上限与范围版本内的累计风险，单笔上限由 CAP 约束一次拟执行动作；三者的来源、变化时机和失败结果不同，混成一个“余额上限”会导致外部入金被误当成授权、并发动作漏算或单笔动作绕过。风险减少或保护动作不占用新增资金额度，但仍经过场所和事实检查，是在不建设资金预留或复杂风险模型的前提下避免额度挡住退出路径的最小规则。

## 2.4 策略激活额度配额【CAP-ENV-003】

用户启用一个可以无人在线持续运行的策略前，必须为该次激活同时分配最大保证金、最大名义仓位和最大允许损失。分配时三轴都不得超过当前资金使用上限与范围中尚未分配的相应容量；每项额度只能归属于一个仍有效的策略激活，其他策略不得使用这部分额度。资金不转入、不冻结到 Halpha，仍留在用户指定的交易场所账户；互斥只表示 Halpha 不把同一用户已分配额度重复承诺给多个策略。

最大保证金和最大名义仓位只阻止 Halpha 主动增加风险的动作。行情和费用变化导致已经存在的仓位自然越过任一额度时，CAP 不因此强制减仓或平仓；后续保护、风险减少和退出仍按该策略计划、EXE 与交易场所事实处理。任何继续主动增加风险的动作仍须在当前事实下重新满足这两项额度。

最大允许损失只计算该次策略激活自身形成的动作和持仓，不受其他策略损益影响。该激活净结果按 `activation_net_result = realized_pnl + unrealized_pnl + funding - commission` 计算，累计损失按 `loss = max(0, -activation_net_result)` 计算；其中资金费保留实际收付正负号，手续费为非负成本。达到最大允许损失时，Halpha 必须停止该激活继续增加风险，并按其既定退出路径退出该激活的全部持仓；退出动作和结果仍经过 EXE/DAT 的事实、唯一动作、保护与核对规则。激活完全平仓并结束后释放其额度；结果未决或仍可能存在仓位时不得提前释放。

# 3. Halpha 真实资金操作权限【CAP-MOD-001】

## 3.1 Halpha 真实资金操作权限定义【CAP-MOD-001-DEF】

Halpha 真实资金操作权限，是用户授予 Halpha 在明确范围内向真实交易场所或账户发起写入的最高权限。它只约束 Halpha，不限制用户通过官方入口独立操作。

本 L2 把 L0 的两种授权路径落实为三档可操作权限：`逐次确认后写入` 是人工授权，`计划内自动写入` 是机器授权，`禁止真实写入` 表示当前没有可用的 Halpha 真实写入授权。保护、工程验证、风险增加、风险减少和订单管理是动作目的或效果，不是与人工授权、机器授权并列的第三类授权。

| 权限级别 | 稳定含义 |
|---|---|
| `禁止真实写入` | Halpha 不向真实交易场所或账户发起写入；仍可只读观察、形成计划并核对用户的外部活动 |
| `逐次确认后写入` | Halpha 只能在用户针对当前计划、事实和动作预览作出有效交易计划当前确认后发起该次真实写入 |
| `计划内自动写入` | Halpha 可以在用户已启用的固定计划分支、账户、期限和动作范围内自动发起真实写入 |

## 3.2 Halpha 真实资金操作权限进入与转换要求【CAP-MOD-001-REQ】

| 权限级别 | 进入条件 |
|---|---|
| `禁止真实写入` | 默认级别；Halpha 没有有效真实写入权限或权限已被停止、撤销或无法确认 |
| `逐次确认后写入` | 用户已明确账户和资金使用上限与范围；每次写入绑定当前计划、事实、动作预览和有效交易计划当前确认 |
| `计划内自动写入` | 用户已明确启用固定计划分支、账户、期限和动作范围，并且停止、核对和用户接管路径可用 |

逐次确认路径只有在用户能够在线理解当前事实、作出交易计划当前确认，并且从通知到提交的完整人工路径在适用时限内可行时才可进入关键路径；用户不在线、身份无法确认、响应时间不足或人工路径不可达时，不得把该路径静默转换为机器授权。计划内自动写入必须在计划触发前已经取得有界的机器授权，触发后不得再依赖人工响应；若运行中出现需要主观判断、关键事实未知、授权边界不清或路径不可行，只能停止、等待用户接管或以不交易结束。

交易所模拟盘属于 DAT 定义的交易记录环境，不是上述权限级别；模拟盘适合验证工作流、计划条件、交互和部分策略行为，但模拟资金、成交、费用、流动性和心理影响均不等于真实资金。真实验证是受控真实环境中的验证活动，不是权限级别；它可以采用逐次确认后写入或计划内自动写入，但不能被包装为已经证明 Alpha 或系统成熟，其实际资金仍由资金使用上限与范围表达。

Halpha 只有在收到用户显式选择的真实账户、资金使用上限与范围和相应权限命令后，才能从 `禁止真实写入` 进入更高权限。模拟订单、持仓、交易计划当前确认、交易计划启用状态和检查结果不得迁移成真实资金使用决定或 Halpha 真实资金操作权限。

机器授权所需控制未实现、未验证或不可信时，不得进入计划内自动写入。逐次确认路径只有在该路径所需的资金、权限、事实、唯一提交、停止和核对控制均已验证时才能使用；交易计划当前确认不能替代缺失控制。受影响的 Halpha 真实动作保持禁止真实写入，用户需要处置时使用交易场所官方入口。

降低或撤销权限应立即生效。进入或扩大 `计划内自动写入` 必须由用户显式完成，不得由盈利、模型、策略或后台任务自动发生。

具体 L3/L4 支持切片可以只实现 `计划内自动写入`，无须同时实现 `逐次确认后写入`；未实现的路径必须明确不支持，不能用界面确认或临时人工步骤模拟为可用授权。

# 4. 简单限额检查【CAP-CHK-001】

Halpha 准备拟执行动作时，按 CAP 规则只检查：

1. 当前 Halpha 真实资金操作权限是否允许该来源写入；若为人工路径，用户是否在线且存在仍有效的交易计划当前确认；若为机器路径，触发前是否存在仍在作用域和期限内的机器授权；
2. 账户、策略、工具和拟执行动作是否在资金使用范围内；
3. 拟执行动作归属的策略激活是否存在仍有效的三轴互斥额度配额；
4. 按当前可得账户与订单事实估算后，拟执行动作是否明显超过最大投入、策略激活额度配额和适用简单限额；
5. 相关适用范围是否处于 Halpha 新增真实动作停用状态。

额度检查先按动作对保守敞口的预期影响区分新增资金、保护、风险减少和订单管理。只读查询、模拟和用户通过官方入口的独立操作不是 Halpha 真实动作；具体动作代码和当前支持范围分别由 L3、L4 记录。

- **新增资金动作**（开仓、加仓或其他会增加保守敞口的动作）同时受 Halpha 全局资金上限、单笔资金上限和 DAT 当前可用余额/保证金/费用事实约束；余额加上场所要求的费用或保证金不足时拒绝，余额事实未知或过期时为未知。
- **风险减少或保护动作**（减仓、平仓、止盈、止损或保护替换）只有在 EXE/DAT 能证明动作后的保守敞口不增加时，才不占用上述“新增资金”额度；它仍必须满足场所数量、费用、保证金和工具规则。若保护动作会新增保证金、扩大敞口或无法证明风险减少，则按新增资金动作检查，不能绕过额度。
- 保护动作因余额或场所条件无法执行时，不得静默丢弃保护责任：已有场所保护订单不得因 Halpha 停用而自动撤销；没有有效保护或保护存在缺口时，必须保持 EXE 的开放责任、阻止受影响范围继续新增风险，并提供用户通过官方场所入口处理的路径。

这些检查属于金融风险和资金操作权限边界：它们判断一个按交易计划、保护/风险减少决定或用户明确指令形成的拟执行动作是否允许承担相应资金与风险敞口，不证明订单实现正确。待执行动作记录身份、去重、部分成交、保护落实和核对仍是 EXE/DAT/SYS 的基本功能正确性。

任何一项明确不满足时，Halpha 拒绝该拟执行动作；无法可靠计算时结果为无法判定并拒绝提交，交易计划当前确认不得覆盖未知。需要处置时，Halpha 只能在取得新事实后重新检查，或按 FLOW 另行形成并证明不增加或转换风险的收缩动作；否则由用户使用交易场所官方入口。CAP 不负责 TRADEPLAN 的计划决定、POR 的组合比较、ALP 的收益证据、EXE 的订单可执行性或 DAT 的事实质量判断。

为了避免陈旧状态和重复真实动作，Halpha 可在拟执行动作准备时检查一次，并在实际提交前刷新关键账户事实后再次运行同一轻量规则。这是同一规则在两个时点的技术复核，不形成“首次检查—预先占用资金—最终审批”的两阶段治理流程，也不需要跨域审批对象。

# 5. 停用真实动作、缩小范围与恢复权限【CAP-CTL-001】

Halpha 必须向用户提供少量步骤完成以下操作：

- 全局或按账户、策略停止 Halpha 新增资金动作；
- 按账户、策略或指定保护计划停止 Halpha 新的保护、止盈、止损或保护替换动作；
- 按账户、策略停止 Halpha 新的风险减少或订单管理动作；
- 全局停止 Halpha 通过当前写入链路发起的全部真实动作；
- 缩小最大投入或适用范围；
- 降低或撤销某项 Halpha 真实资金操作权限；
- 查看停止命令是否已经阻止适用范围内的拟执行动作领取和提交；无法确认时继续保持受限并交给用户；
- 转向 Halpha 之外的账户或场所路径人工处理。

单独停止新增资金动作时，不自动停止现有持仓的保护、风险减少或订单管理动作。发现保护计划本身有问题时，用户可以指定保护计划；全部 Halpha 真实动作停用后，Halpha 可以继续启动、读取事实、展示、模拟和核对，但不得通过当前写入链路发起新增、撤销、平仓、保护或修改。该停用不撤销或改变已经提交的场所订单、保护或持仓，也不改变用户通过官方入口的独立操作能力；场所仍可能自行执行已有订单，外部处置转由用户或官方场所路径处理，并保留未闭合责任。

如果用户的意图是处置已经存在的止盈止损订单，命令必须额外明确选择保留既有保护、撤销指定保护或撤销全部保护，默认保留。撤销选择只形成独立撤销意图；全部 Halpha 真实动作停用时 Halpha 不得提交该撤销动作，只能由用户通过官方场所路径处理，撤销失败、未知或部分完成时保护责任继续开放。

全部 Halpha 真实动作停用不是某一动作类别的额度限制。它优先于逐次确认、计划启用和其他动作类别停用；解除必须由用户显式执行，不能由系统、计划、盈利或后台任务自动解除。

停止和收缩立即生效，不需要审批、事故结案或证据包。CAP 只拥有按动作类别记录、检查和传播停用状态，不拥有保护订单的提交、撤销或执行结果。

Halpha 只接受用户显式恢复权限的命令，并应在提交前显示直接问题和关键账户核对状态；CAP 不定义个人投入决定、恢复会议或工程验收结论，系统也不得自动恢复权限。

# 6. 与核心业务的边界【CAP-BND-001】

| 相邻领域 | CAP 定义的交付语义 | CAP 不拥有的语义 |
|---|---|---|
| ALP | 研究确需资金规模约束时提供当前资金使用上限与范围 | Alpha、经济主张的资金规模适用范围和策略有效性 |
| POR | 用户允许纳入比较的最高资金使用上限与范围 | 计划、现金和不配置选项的比较 |
| TRADEPLAN | 策略激活额度配额、Halpha 真实资金操作权限、资金使用限制和 Halpha 新增真实动作停用状态 | 七类计划决定和条件生命周期 |
| DAT | 要求 Halpha 读取必要账户事实 | 余额、订单、仓位、行情和时间真实性 |
| EXE | 轻量 `拟执行动作资金与权限检查结果` | 待执行动作记录、订单提交、部分成交、保护和核对 |
| UX | 当前 Halpha 真实资金操作权限、Halpha 新增真实动作停用状态、资金使用限制及相关用户命令的效果 | 页面结构、交互步骤、反馈体验和交易计划启用语义 |
| SYS | 运行实体、宿主与进程、运行时配置、秘密注入和恢复前置边界 | 交易计划启用、Halpha 真实资金操作权限、Halpha 新增真实动作停用状态和外部动作结果 |
| ENG | 需要验证的简单行为 | 发布流程、质量体系和生产准入 |

若某项 CAP 设计需要复制上述领域的业务对象，或其维护成本开始接近核心交易链，应优先删除、合并或退回配置，而不是继续扩展 CAP。

# 7. L3 必须精确定义【CAP-L3-001】

只有真实资金切片需要 L3。相关 L3 最少定义：

- `资金使用上限与范围` 的字段、金额口径和更新命令；
- 用户在 Halpha 外投入的资金、Halpha 全局资金上限、单笔资金上限和保护动作额度处理的区别及适用范围；
- 策略激活额度配额的身份、三轴口径、互斥分配、检查、损失计算、结果未决时保留和激活结束后释放规则；
- 三档 `Halpha 真实资金操作权限`、适用范围、交易计划当前确认与权限降低；
- `拟执行动作资金与权限检查结果` 的输入、无法计算、拒绝原因和当前事实变化后的失效规则；
- 检查与待执行动作记录持久化在本地关系型事务中的顺序；
- 停止信号怎样按新增资金动作、保护动作或全部 Halpha 动作到达执行链，以及怎样证明已经生效；
- 全部 Halpha 真实动作停用怎样阻止全部 Halpha 真实写入、允许只读、模拟和核对继续运行，以及怎样证明解除后才可恢复；
- 恢复 Halpha 真实资金操作权限的用户命令和最小核对清单。

除非已发生重复实际问题，不得新增资金风险事故对象、审批角色、通用风险规则语言、独立准入包或多阶段授权状态机。

---

# 8. L4 选择边界【CAP-L4-001】

L4 只记录当前建设选择的环境/账户/动作支持范围、采用的限额与计算规则、授权路径、明确不支持范围和验收证据。具体余额、资金边界版本、权限、停用、检查与用户命令属于 CAP、DAT 等拥有领域的运行记录，不写入 L4；只有新增真实动作类型无法由现有边界表达时，才复核 CAP 深度。
