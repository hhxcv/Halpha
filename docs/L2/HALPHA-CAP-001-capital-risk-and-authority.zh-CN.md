# Halpha 用户资本边界与使用控制原则

**文档编号：** HALPHA-CAP-001  
**层级：** L2  
**语言版本：** zh-CN  
**上位文档：** HALPHA-CON-001、HALPHA-DOC-001、HALPHA-VIS-001、HALPHA-FLOW-001、HALPHA-ARC-001  
**本文档负责：** Halpha 怎样把用户为一次计划激活设置的资金边界与有界机器执行范围落实到动作检查，以及怎样响应停止、退出和接管  
**本文档不负责：** 决定用户个人总投入或可承受损失；研究收益、交易计划内容、订单执行、事实生产、页面设计、工程发布和当前配置

---

# 0. 定位与复杂度上限【CAP-SCP-001】

## 0.1 CAP 定位定义【CAP-SCP-001-DEF】

CAP 只回答三个长期问题：本次计划激活允许 Halpha 在哪个账户和范围内最多使用多少资金；当前动作是否在该边界和有效授权内；用户停止、退出或接管后哪些新动作必须被阻止。

用户在 Halpha 外决定向哪个账户投入多少资金。Halpha 只执行用户在系统内配置的更小边界，不把外部余额当成授权，也不承诺最大损失。

## 0.2 CAP 责任与复杂度要求【CAP-SCP-001-REQ】

CAP 使用简单的激活资金限制、动作前检查和少量用户控制，不建设风险总分、完整情景库、多级审批、资金预留平台或独立准入流程。只有实际交易方式出现新的、无法由现有边界表达的问题时才增加规则。

---

# 1. 激活资金边界【CAP-ENV-001】

用户激活计划时，在同一个决定中为明确的环境、账户、工具和计划设置最大保证金、最大名义仓位、最大允许损失、单动作上限、动作范围和期限。外部余额是账户事实，不表示 Halpha 可以使用全部余额；外部入金、盈利、模型结论或运行时间都不能自动扩大本次激活边界。

动作后的保守敞口必须同时计入本激活已有持仓、开放订单、尚未核对的动作和当前拟执行动作。拆分、并发、重试和部分成交不能绕过任一限制。具体金额口径由下位设计定义，并保持同一口径用于动作形成和提交前复核。

为避免在个人项目当前需求中引入账户级分配与竞争，本契约一次只允许同一真实账户存在一个未闭合激活。需要多个激活并发共享同一账户时，必须先由真实消费者证明必要，再另行设计共享上限；不得提前建立账户额度版本、分配或预留对象。

---

# 2. 有界执行授权【CAP-MOD-001】

## 2.1 Halpha 真实资金操作权限定义【CAP-MOD-001-DEF】

Halpha 对真实资金新增风险只有一种用户授权入口：一次明确的计划激活。没有有效激活时禁止新增风险；激活后，机器只能在该激活已经固定的范围内执行。

一次明确的计划激活可以同时授予该激活的有界机器执行范围。用户必须看到并确认环境、账户、固定计划、工具、本次资金限制、期限、允许动作和失败结果。授权只属于该激活，不是账户总开关，不适用于后续激活，也不允许 Halpha 自行扩大范围。

## 2.2 权限进入与转换要求【CAP-MOD-001-REQ】

- 默认禁止新增真实风险；只有用户明确激活完整计划后，机器才取得该激活内的有界执行范围。
- 界面、程序、AI 输出、会话存在或临时确认都不能绕过计划激活或扩大其范围。
- 降低范围、停止或撤销立即作用于后续动作；扩大范围或重新取得新增风险能力必须修改并重新激活计划。
- 关键事实、授权边界或处理结果无法确认时，保持禁止新增风险。

交易所模拟盘与真实资金环境使用同一检查和执行语义，但身份和效力完全分离。模拟环境中的计划、确认、授权、资金限制、动作和结果不得复制、改名或提升为真实资金记录；进入真实资金环境必须由用户重新选择真实账户、资金限制和计划激活范围。

# 3. 动作前检查【CAP-CHK-001】

Halpha 对每个可能改变交易场所或账户的动作检查：

1. 环境、账户、工具、计划和授权范围是否一致；
2. 当前动作是否属于仍有效的计划激活范围，或属于用户针对已有责任作出的明确且可证明不增险的收缩指令；
3. 计入本激活已有持仓、开放订单、未核对动作和当前动作的保守结果后，是否仍满足激活内的保证金、名义仓位、最大损失和单动作限制；
4. 当前余额、保证金、费用、价格、数量和订单事实是否足够新且可核对；
5. 用户是否已经停止相应新增风险、退出该计划或进入接管。

任一明确不满足时拒绝动作。无法可靠计算或核对时，结果是未知并拒绝新增风险；用户确认不能覆盖未知。

能够证明不会增加或转换风险的撤销、保护、减仓和平仓，可以不增加本激活的已计风险，但仍须满足当前事实、授权范围、动作唯一和场所规则。不能证明时按新增风险处理。保护无法执行时保持责任可见，阻止继续增加受影响风险，并向用户提供交易场所官方入口的接管路径。

Halpha 在形成动作时检查，并在真正提交前刷新关键事实后复核同一规则。这是防止事实漂移的同一检查，不形成两级审批或资金预留。

---

# 4. 停止、退出、接管与重启连续性【CAP-CTL-001】

用户必须能够用少量步骤：

- 停止 Halpha 为指定账户或计划新增风险；
- 要求计划按已固定的退出路径退出；
- 转入交易场所官方入口人工接管；
- 缩小资金上限或撤销机器执行范围；
- 看清系统重启后的事实核对结果和该激活是否仍可继续。

停止新增风险不自动撤单、平仓或改变交易场所已有保护；这些处置必须作为明确动作执行，或由用户在官方入口完成。停止命令的处理结果无法确认时，Halpha 按仍处于停止状态处理，并查询原命令，不盲目重发。

用户停止新增风险后，该激活永久不得再次新增风险；已有风险仍按原计划保护或退出。退出同样保持新增风险关闭，并由 Halpha 按计划完成退出与核对；接管则停止该激活全部新的 Halpha 场所写，由用户通过场所官方入口处置。旧订单、持仓、保护和未决责任闭合后，如需再次交易，用户必须创建新激活。

系统仅发生重启且用户没有停止该激活时，可以在订单、成交、持仓、保护、未决结果和原授权均核对一致，并确认不会补发错过动作后继续同一激活。事实或连续性未知时保持不新增风险。

# 5. 相邻领域边界【CAP-BND-001】

| 相邻领域 | CAP 提供 | CAP 不拥有 |
|---|---|---|
| TRADEPLAN | 激活资金边界、授权和动作检查结果 | 计划内容、条件和结束决定 |
| DAT | 对当前账户与市场事实的需求 | 事实的生产与质量结论 |
| EXE | 动作是否通过资金和授权检查 | 动作唯一提交、场所结果和核对推进 |
| UX | 应呈现的边界、授权、停止和拒绝理由 | 页面、交互步骤和反馈形式 |
| SYS、ENG | 需要运行和验证的简单行为 | 运行实现、发布和当前验证结果 |

CAP 不复制相邻领域的对象和状态。维护成本接近核心交易链时，优先删除或退回简单配置。

---

# 6. L3 与 L4 边界【CAP-L3-001】

L3 只需精确定义激活内资金限制的金额口径、计划激活怎样绑定有界机器执行范围、动作形成与提交前怎样复用同一检查，以及停止、退出、接管和重启连续性怎样到达执行链。正常、重复、事实未知、拒绝、重启和外部接管必须可测试；同一真实账户的第二个未闭合激活必须拒绝。

L4 只记录当前支持的环境、账户、动作、精确限额与验证结果【CAP-L4-001】。没有当前消费者时，不为未来场景增加资金限制维度、授权状态或控制流程。
