# 0. 提案结论【ALIGN-P0-SUM-001】

本提案把长期建设方向与 P0 实现目标分开，但保持一套不矛盾的权限语义：

1. **长期方向**：普通故障重启本身不撤销、扩展或续期仍有效的机器授权。未来阶段只有在单写者、构建与配置身份、数据库连续性、场所核对、授权范围、停用状态和外部责任全部得到确定证明后，才可以在原授权范围内自动继续；用户停止、撤销、接管、授权到期、身份变化或连续性未知永不自动解除。
2. **P0 目标**：不实现自动继续计划。唯一 Executor 或主机重启后，全部未完成 `PlanActivation` 在策略回调和新场所写入前进入 `run_state=PAUSED`；只读恢复与核对自动进行。恢复同一计划的判断和后续动作必须等待 `RESUME_ACTIVATION`，直接退出由 `EXIT_STRATEGY` 触发，用户接管由 `USER_TAKEOVER` 触发；三者都是用户显式命令，不要求先恢复计划再退出。
3. **P0 复杂度约束**：继续复用原 `MachineAuthorizationVersion`、`PlanActivation`、`PlanEvent`、`PlanAllocation`、`PendingAction` 和 `Command/Receipt`；不建立重新授权版本链、恢复批准记录、数据库 epoch、恢复平台、额外进程、额外 worker 或第二写路径。
4. 同一协调集还关闭 Binance Demo 使用真实 `PendingAction` 身份、CAP L3 缺少独立单次动作上限与完整停用类别、以及完整 React 工作台成为唯一限时入口三项冲突。

这里的“等待用户命令”只指 Halpha 在重启后再次发起场所变化。场所中已经存在的订单和原生保护不会因本地暂停而撤销；已跨写边界的增险订单仍可能在暂停期间成交。启动 reconciliation、按原 UUID 查询、持仓/成交/费用事实更新和 UNKNOWN 闭合仍自动运行，但不能借核对结果自动产生新的写入。若重启时存在无保护敞口、保护缺口或需撤单/减仓的责任，P0 必须显著告警并等待用户从 Halpha 限时入口显式继续、退出或接管；在能够无人值守自动收口之前，B05 不得宣称具备无人在线故障恢复能力。

在全部目标版本正式接受前：

- D00 保持 `BLOCKED`；
- B00 只可作为无产品语义、无真实写能力的隔离资格化；
- B01–B05 不得开始；
- `real_write_status` 保持 `DISABLED`；
- 本提案中的对象名、字段、测试和阶段安排都只是候选。

# 1. 决定边界【ALIGN-P0-DEC-001】

## 1.1 采用方向

采用以下解释顺序：

```text
L0 区分“授权仍有效”和“当前运行是否具备执行资格”
→ L1 固定长期恢复证据门和绝不自动解除的边界
→ L2/L3 固定动作身份、核对、防重和停止语义
→ L4 可以选择比长期上限更窄的阶段能力
→ P0 选择 MANUAL_PLAN_RESUME，未来经新阶段评审后才可选择 AUTO_CONTINUE
```

普通重启后的自动继续不是创建、提高或恢复一项已经失效的授权；它只可能是在原授权仍有效、范围不变且恢复证据门通过时恢复运行资格。反之，用户显式停止/撤销/接管、授权到期、构建或身份变化，以及任何无法证明连续性的情况，都需要新的用户决定。P0 即使满足长期自动继续的必要条件，也固定停在 `run_state=PAUSED` 等待人工命令；这是长期边界允许的更保守阶段选择，不是 L4 对上位规则的例外。

## 1.2 复杂度边界

本候选不增加进程、权威数据库产品、真实写路径、持久 worker 或恢复记录族。候选实现应优先复用：

- 原 `MachineAuthorizationVersion` 及其原 deadline/范围；重启不创建新授权版本，也不延长、扩大或替换原授权；
- `PlanActivation.run_state=PAUSED`、一个稳定暂停原因和现有 `PlanEvent`，不改写生命周期或建立独立恢复状态机；
- 现有 `Command/Receipt` 增加幂等的 `RESUME_ACTIVATION`，成功时继续同一 `PlanActivation`，不创建第二激活；
- 用一个中性的动作物理记录族替代当前仅以 `PendingAction` 命名的物理记录族，通过不可变环境与逻辑类型区分模拟和真实动作，不增加记录族数量或第二执行引擎；
- 现有 `AccountCapitalLimitVersion` 和 `StopStateVersion`，补齐字段和检查，不增加 CAP worker；
- 现有 `halpha-app`、认证、`Command/Receipt` 和命令处理器，增加最小页面而不增加进程或第二业务 API。

P0 不预建长期自动恢复资格引擎、数据库世代/租约、部署编排或无人值守恢复控制面。未来实施 `AUTO_CONTINUE` 时，应依据实际组件和运行证据另行细化 L3/L4，而不是把未被 P0 消费的状态和平台提前写入产品。

除现有“记录族、worker、进程、数据库产品、真实写路径”数量外，L4 的实现复点还应报告：独立不变量、状态轴、跨域原子事务、恢复路径、UNKNOWN 解除路径、每个记录族的写入者数量，以及协调器承担的责任种类。数量上升本身不自动拒绝，但必须说明消费者和删除条件。

# 3. 长期恢复方向与 P0 人工继续候选【ALIGN-P0-RCV-001】

## 3.1 长期建设方向

长期语义区分两件事：

- `MachineAuthorizationVersion` 是否仍在原 deadline、原账户/工具/动作类别和原资本范围内有效；
- 当前 Executor 是否经过恢复证据门，具备在该有效授权内继续运行的资格。

普通同构建故障重启不会单独使前者失效。未来阶段可以在以下条件全部得到权威证明后自动恢复后者：唯一写者已取得、数据库与计划版本连续、构建/配置/凭据/账户身份未变、原授权仍有效且未被撤销、当前停用和接管状态允许、订单/成交/持仓/保护/UNKNOWN 已核对并分类、不会补发错过信号，也不会重复任何可能已提交动作。自动继续只能恢复原授权范围，不能延长 deadline、扩大额度、改变动作类别、创建第二激活或绕过当前 CAP/DAT/EXE 检查。

长期恢复资格按动作后果分级，不以“核对已完成”一次性恢复全部能力：只读核对最先恢复；存在 `PROTECTION_GAP`、未闭合 UNKNOWN、仍可能增险的开放订单或未终结外部责任时，只能恢复经原授权允许的防重复查询与安全收口，不得恢复计划判断或新增风险。只有保护已确认、UNKNOWN 已闭合且残余增险责任已终结后，才允许在其余证据门也通过的前提下自动恢复计划判断和新增风险。

以下情况无论未来阶段都不得自动继续：用户显式停止、撤销或接管；授权到期；构建、配置、凭据、运行身份、账户或工具范围变化；数据库恢复点或写连续性不能证明；存在无法安全分类的外部活动；阶段计划明确选择人工继续。此时系统保持更窄状态，等待用户决定。

## 3.2 P0 触发和能力边界

P0 固定 `recovery_mode=MANUAL_PLAN_RESUME`。以下任一事件使所有未完成且非 `USER_TAKEOVER` 的 `PlanActivation` 在策略回调、READY 动作领取和任何新场所写入前进入 `run_state=PAUSED`：

- 唯一 Executor 运行连续性发生任何中断后重新启动或被替换，包括计划内停止和非计划退出；
- 主机重启；
- PostgreSQL 写连续性中断并重新恢复。

单独重启 App、浏览器、完整 SPA 或 NotificationDispatcher 不暂停仍连续运行的 Executor。版本部署、构建/配置/凭据/身份变化不走普通人工继续；它们保持阻断，并按变更评审或新激活处理。

P0 明确不具备无人在线时自动继续计划或自动新增场所写的能力。首次 LIVE 必须按第 3.4 节主动在线监控；只有在运行连续、入场终态与场所原生保护均已确认且未发生恢复触发后，用户才可按原计划离开。一旦发生恢复触发，继续速度取决于用户重新在线并提交明确命令。该限制必须在 B05 准入、运行状态和通知中显著展示，不得把 Task Scheduler 已自动拉起进程描述为交易已自动恢复。

## 3.3 P0 恢复顺序

恢复顺序候选固定为：

```text
取得单实例门
→ 读取权威本地状态
→ 以 CAS 把未完成激活的 run_state 持久转为 PAUSED，保留原 lifecycle/外部责任/UNKNOWN 轴
→ 只读启动数据与场所适配器并完成启动 reconciliation
→ 按原 UUID 查询订单、成交、仓位、保护和未决动作，保存权威事实
→ 保持策略判断、READY 领取和全部新的 Halpha 场所写暂停
→ 用户提交 RESUME_ACTIVATION、EXIT_STRATEGY 或 USER_TAKEOVER
→ 对原授权、事实、停用、接管、计划 deadline 与外部责任重新执行当前检查
→ 仅 RESUME_ACTIVATION 成功后才继续同一激活；其他命令进入各自既有路径
```

重启暂停不创建新的 `MachineAuthorizationVersion`、`PlanAllocation` 或 `PlanActivation`，也不改变原授权 deadline、范围或额度。进程健康、核对完成、用户登录、通知送达和旧授权尚未到期都不能代替用户命令。恢复事务结果未知时按原 `command_id` 和现有记录查询，不创建第二激活或第二授权。

`RESUME_ACTIVATION` 只解除该激活的重启暂停并恢复同一计划的允许路径，不得顺带解除任何 `StopStateVersion`、用户停止、授权撤销、`ALL_WRITES` 或接管状态。`EXIT_STRATEGY` 不先解除暂停或恢复增险判断，而是直接进入既有退出路径；`USER_TAKEOVER` 直接冻结 Halpha 写并移交责任。原授权过期/撤销、范围变化、核对未完成、外部活动冲突或计划已不可继续时，继续命令必须被拒绝并返回可理解回执；用户仍可选择退出或官方接管。

## 3.4 暂停期间的动作与责任

- 已跨越场所写边界且结果未知的动作只能按原持久身份查询和核对，不能因新旧授权状态重发。
- 尚未跨越写边界的 `READY` 动作不得在恢复后自动领取或提交；人工继续后仍须使用当前事实和原授权执行提交前检查。入场窗口、deadline 或来源条件已经错过时闭合为未提交/错过，不得补发。
- 已在场所 WORKING 的普通订单和原生保护保持原状；P0 自动查询和展示，不因本地暂停自动撤销、替换或重复提交。已跨写边界的普通增险订单仍可能在暂停期间成交；`run_state=PAUSED` 只证明 Halpha 没有发起新的增险写，不能证明场所敞口不再增加。
- 暂停期间出现的 ACK、晚到成交、费用、仓位和保护变化必须自动写入权威事实，并允许原 UNKNOWN 推进到确定结果，但这些本地状态推进不能形成第二场所调用。晚到成交形成的新敞口立即进入可见 `PROTECTION_GAP`。
- 新的保护、风险减少、撤单和订单管理仍是 Halpha 场所写；在 `run_state=PAUSED` 时不得自动形成或提交。已确认敞口但保护尚未 WORKING 时必须发送最高优先级通知，并在 `/operations` 提供继续、退出和用户接管。用户未处理前，Halpha 不发起新的增险写，也不得把只读核对冒充风险已闭合。
- 只读核对、展示和保存核对结果不构成恢复真实写权限。
- 用户可随时转向交易所官方入口；Halpha 不因外部人工处置自动恢复权限。

由于 P0 把所有新的 Halpha 写入置于人工命令之后，B05 首次真实闭环必须把“用户从激活开始主动在线监控，已验证 `/operations` 与官方入口可用，并持续到入场终态与场所原生保护均已确认”为准入前提；这是一项受控 LIVE 资格验证，不是无人值守运行。若项目需要用户离线时自动补保护、撤单、减仓或继续计划，必须先实施并资格化长期自动恢复能力；不能把该能力暗含在 P0。

## 3.5 最小状态和幂等

P0 只增加以下最小语义：

- `PlanActivation.run_state` 取 `ACTIVE | PAUSED`，与现有 lifecycle、外部责任和 UNKNOWN 轴正交；暂停只保存稳定 `pause_reason=WRITER_CONTINUITY_LOST`、发生时间和 `PlanActivation` 预期行版本，不新增恢复 epoch。因此 `PAUSED + EXITING`、`PAUSED + UNKNOWN` 等组合保留原责任，不被暂停覆盖。
- 现有 `PlanEvent` 保存幂等暂停和继续事实；同一预期行版本重复启动不追加第二事件。若用户已继续后再次崩溃，行版本已经推进，下一次 CAS 再把 `run_state` 置为 `PAUSED`。
- 现有 `Command/Receipt` 增加幂等 `RESUME_ACTIVATION`；同一命令重复返回原结果。
- 原 `MachineAuthorizationVersion`、`PlanAllocation`、`PendingAction`、UUID、条款摘要和外部责任身份保持不变。
- `run_state=PAUSED` 不复用或覆盖 `UNKNOWN`、`EXITING`、`USER_TAKEOVER` 或 `StopStateVersion`：它是确定的运行轴状态，不是事实未知、生命周期、永久接管或资本控制权威。

暂停状态必须在重建可产生策略 proposal 的适配器和领取 READY 动作之前提交；若该事务结果未知，启动过程只能查询同一来源身份并保持写门关闭。P0 不新增恢复专用表、worker、队列、租约、fencing token 或授权投影。

## 3.6 最小验收

- `RCV-T-P0-PAUSE-001`：每个未完成激活在策略回调、READY 领取和外部写入前幂等进入 `run_state=PAUSED`，且保留原 lifecycle/外部责任/UNKNOWN 状态。
- `RCV-T-P0-NO-WRITE-002`：未收到相应用户命令时，暂停期间只读核对和本地事实更新继续，但保护、撤单、减仓、退出和新增风险均不得产生新场所调用；收到 `EXIT_STRATEGY` 后只允许既有退出/减险路径写入，不恢复计划判断或新增风险。
- `RCV-T-P0-RESUME-003`：只有用户命令能继续同一激活；不创建新授权、新额度或新激活。
- `RCV-T-P0-CHECK-004`：授权过期/撤销、范围或身份变化、显式停止、接管、核对未完成或计划不可继续时拒绝继续。
- `RCV-T-P0-RESTART-005`：继续前后再次崩溃都重新暂停，同一生命周期预期版本不重复 PlanEvent、PendingAction 或场所调用。
- `RCV-T-P0-UNKNOWN-006`：重启前未知提交只查询原 UUID，绝不重发；权威结果可在暂停期间自动保存并把原动作/外部责任推进到确定状态，但不形成新场所写。
- `RCV-T-P0-MISSED-007`：停机期间错过的入场窗口闭合为错过，继续后不补开仓、不延长 deadline、不重置已消费来源。
- `RCV-T-P0-VENUE-008`：暂停不撤销或修改已有场所原生订单和保护。
- `RCV-T-P0-GAP-009`：成交后、保护 WORKING 前崩溃时保持 `run_state=PAUSED + PROTECTION_GAP`，告警并只允许用户显式继续、退出或接管。
- `RCV-T-P0-STOP-010`：继续命令不能解除用户停止、授权撤销、`ALL_WRITES` 或用户接管。
- `RCV-T-LONG-BOUNDARY-011`：长期自动继续的候选实现只能恢复原有效范围；任何身份/范围/连续性失败都必须转人工且无场所写。
- `RCV-T-P0-LATE-FILL-012`：重启前已提交/未知的增险订单在暂停后才成交时，系统保存新敞口与 `PROTECTION_GAP`、不重发原动作、不自动补保护或退出，并等待用户继续、退出或接管。

# 4. Demo 与真实动作身份候选【ALIGN-P0-DEMO-001】

## 4.1 逻辑身份

候选允许 Demo 与 LIVE 复用名为 `ExecutionAction` 的中性 schema/ORM 持久化载体，但该名称不是新的业务对象、领域语义或共同所有者。业务上仍只有两个不可互换的逻辑动作身份：

- `SimulatedAction`：属于 TRADEPLAN 现有拟执行动作语义，只属于历史回放或交易所 Demo 环境，可以驱动模拟订单、成交、仓位、保护结果和复盘；不构成真实 `PendingAction`、真实写权限、真实提交责任或真实核对责任。
- `PendingAction`：仍由 EXE 唯一拥有，只属于 LIVE 环境，必须引用当前真实计划、CAP 检查、机器授权和权威事实，才可进入唯一真实写边界。

二者可以复用 `ExecutionAction` schema/ORM、公共值对象和无副作用状态推进函数，但不得出现可写的通用 `ExecutionAction` 领域仓库或跨域共同写对象。TRADEPLAN 只能通过自己的公开边界写 `SimulatedAction` 行，EXE 只能通过自己的公开边界写 `PendingAction` 行；任何模块都不能写另一类型。DEMO 与 LIVE 使用独立数据库，任一运行数据库中只允许出现与其环境匹配的动作类型。该替换不增加物理记录族数量；不得为了形式分离复制执行适配器、协调器或状态机。

## 4.2 不可绕过约束

- `environment=DEMO` 只能形成 `SimulatedAction`，不得引用 LIVE 凭据、真实机器授权或真实额度分配。Demo 与 LIVE 可以复用同一适配器实现和工厂代码，但必须按环境实例化独立 adapter、port 和 credential binding；Demo 实例的依赖图中不得存在 LIVE 写端口或 LIVE 凭据。
- `environment=LIVE` 的场所写只接受 `PendingAction`；缺少真实授权或 CAP 引用即失败停止。
- 环境和动作逻辑类型在创建后不可修改；Demo 转真实必须新建真实 `PlanActivation`、授权、额度和 `PendingAction`。
- Demo 的订单、成交、仓位、保护、费用和结果保留 Demo `environment_id`，不得提升、复制或聚合成真实账户事实。
- 共享 schema/ORM 时必须使用数据库约束、环境专属数据库和代码封闭构造器共同证明，不能只依赖 UI 标签或配置约定。

## 4.3 B03 含义

B03 可继续使用 Binance USDⓈ-M Demo 验证产品代码路径，但只能形成 `SimulatedAction`。它证明公共纯逻辑、状态推进、场所适配能力和失败处理可复用，不证明真实 `PendingAction` 已发生，不授予真实写资格。B05 必须新建 LIVE 身份，不能原地升级 B03 数据。

## 4.4 最小验收

- `DEMO-T-TYPE-001`：Demo 输入不能构造或持久化 LIVE `PendingAction`。
- `DEMO-T-WRITE-002`：Demo 动作不能取得 LIVE 凭据或进入真实写门。
- `DEMO-T-MIGRATE-003`：修改环境或动作类型被数据库与领域构造器同时拒绝。
- `DEMO-T-NEW-LIVE-004`：从已完成 Demo 发起 LIVE 时生成全新激活、授权、额度和动作身份。
- `DEMO-T-FACT-005`：同名 Demo 与 LIVE 订单、成交和仓位默认查询、导出与投影相互隔离。

# 5. CAP 候选【ALIGN-P0-CAP-001】

## 5.0 L2 内部对齐

`HALPHA-CAP-001` 候选必须把概念表、控制正文和 L3 下放统一为相同四类：新增资金或风险、保护、风险减少或订单管理、全部 Halpha 真实写入。任何摘要、交接清单或责任登记都不能再省略第三类。该 L2 修订只关闭文内遗漏，不改变现行 `CAP-CTL-001` 已经明确的四类行为、`ALL_WRITES` 优先级和用户显式解除要求。

## 5.1 独立单次动作上限

`AccountCapitalLimitVersion` 候选新增 P0 可执行的独立单次动作金额上限。L3 必须定义其 Decimal 口径、币种、适用环境/账户/工具/动作范围和量化时点；它不能复用每激活 `max_notional` 或外部账户余额。

对主动增加风险的不可变拟执行动作，CAP 在场所价格、数量和费用量化后计算同一动作金额，并同时检查：

- 单次动作金额不超过当前单次动作上限；
- 动作后的全局保守占用不超过 Halpha 全局上限；
- 动作后的激活占用不超过 `max_margin` 和 `max_notional`；
- 当前机器授权、事实截止点和动作类别仍允许提交。

P0 不支持把一次拟执行动作自动拆分成多笔以规避单次上限；相同计划事件的并发或重试必须共享不可变动作身份和累计检查。保护或风险减少动作不消耗新增资金额度，但仍须通过其动作类别停用、当前持仓、场所和事实检查。

## 5.2 完整停用类别

`StopStateVersion` 候选用一个不可变的适用范围集合表达至少四类控制：

1. `NEW_FUNDING`：停止新增资金或风险动作；
2. `PROTECTION`：停止新的保护、止盈、止损或保护替换动作；
3. `RISK_REDUCTION_OR_ORDER_MANAGEMENT`：停止新的风险减少、撤单或订单管理动作；
4. `ALL_WRITES`：停止 Halpha 通过当前写链发起的全部真实动作。

同一动作命中多个状态时采用限制并集，`ALL_WRITES` 优先于其他类别。`ALL_WRITES` 生效后仍可启动、读取、展示、模拟和核对，但不得新增、撤销、平仓、保护或修改；已有场所订单和持仓不被自动改变，用户仍可使用官方入口。只有用户显式命令可以解除 `ALL_WRITES`，系统、计划、盈利、重启、核对完成或后台任务都不能自动解除。

## 5.3 最小验收

- `CAP-T-SINGLE-ACTION-001`：量化后的单次动作金额越界时拒绝提交。
- `CAP-T-NO-SPLIT-BYPASS-002`：拆单、并发和部分成交不能规避同一不可变动作上限。
- `CAP-T-STOP-CATEGORY-003`：四类动作逐一被相应停用范围阻止，未命中类别不被错误扩大。
- `CAP-T-ALL-WRITES-004`：全部写停用后任何 Halpha 场所写调用失败停止，但只读核对继续。
- `CAP-T-EXPLICIT-RESUME-005`：全部写停用只能由用户显式解除，重启和后台流程不能解除。

# 6. 限时操作入口候选【ALIGN-P0-OPS-001】

任何步骤发现第三方组件无法表达候选约束时，结果是保持阻断并回到设计修订，不得通过私有 patch、第二写适配器、第二状态机或 L4 例外绕过。
